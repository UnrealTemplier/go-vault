
Данные в бэкенде обрабатываются двумя фундаментально разными способами: **пакетно (batch)** и **потоково (stream)**. Выбор между ними — это не просто техническое решение, а архитектурный компромисс, влияющий на задержку, пропускную способность, стоимость инфраструктуры и даже на способ мышления команды. В этой статье мы сравним оба подхода, разберём их реализацию в Go и покажем, как они вписываются в экосистему Data Pipeline, рассмотренную в предыдущей статье ([[41. Data Pipeline и потоковая обработка]]).

### Что такое Batch Processing

**Batch Processing** — это обработка данных, собранных за определённый период, единым блоком (пакетом, батчем). Пайплайн запускается по расписанию (каждый час, раз в сутки), читает накопленные данные, обрабатывает их и записывает результат.

В Go batch-обработка часто реализуется как фоновая задача (cron, job), которая читает данные из БД или файлового хранилища, выполняет вычисления и сохраняет агрегаты.

```go
// Простейший batch-процессор
func runDailyAggregation(ctx context.Context, db *sql.DB) error {
    start := time.Now().Add(-24 * time.Hour)
    rows, err := db.QueryContext(ctx, 
        "SELECT user_id, COUNT(*) FROM orders WHERE created_at >= $1 GROUP BY user_id", start)
    if err != nil {
        return err
    }
    defer rows.Close()
    for rows.Next() {
        var userID string
        var count int
        rows.Scan(&userID, &count)
        // сохраняем агрегат в БД аналитики
    }
    return nil
}
```

### Что такое Stream Processing

**Stream Processing** — это непрерывная обработка данных по мере их поступления. Каждое событие (или небольшой микро-батч) обрабатывается немедленно, без ожидания накопления. Система всегда находится в активном состоянии, слушая источник.

В Go это реализуется через consumer'ы Kafka, каналы, горутины, обрабатывающие события в реальном времени.

```go
func runStreamProcessor(ctx context.Context, consumer *kafka.Reader, handler func(msg kafka.Message) error) {
    for {
        msg, err := consumer.ReadMessage(ctx)
        if err != nil {
            break
        }
        if err := handler(msg); err != nil {
            log.Error("processing failed", "error", err)
        }
        consumer.CommitMessages(ctx, msg)
    }
}
```

### Ключевые отличия

| Критерий | Batch | Stream |
|----------|-------|--------|
| **Задержка** | Высокая (часы, дни) | Низкая (мсек, сек) |
| **Окно данных** | Фиксированное (весь объём сразу) | Скользящее (последние N минут) |
| **Пропускная способность** | Максимальная за счёт оптимизации под большие объёмы | Ограничена необходимостью обрабатывать on-the-fly |
| **Состояние** | Обычно безсостоятельное | Часто требует хранения промежуточного состояния |
| **Сложность обработки** | Проще отладка, перезапуск с чекпоинта | Сложнее, требует обработки дубликатов и гарантий порядка |
| **Примеры в Go** | CRON jobs, `dataflow` ETL | Kafka consumer, каналы, горутины |

### Batch в Go: паттерны и инструменты

**Worker pool с диспетчеризацией задач.** Для batch-обработки большого массива записей используется пул горутин, который параллельно обрабатывает элементы батча.

```go
func processBatch(ctx context.Context, items []Item, workers int, handler func(Item) error) error {
    var wg sync.WaitGroup
    itemsCh := make(chan Item, len(items))
    for _, item := range items {
        itemsCh <- item
    }
    close(itemsCh)

    for i := 0; i < workers; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for item := range itemsCh {
                if err := handler(item); err != nil {
                    log.Error("batch item failed", "error", err)
                }
            }
        }()
    }
    wg.Wait()
    return nil
}
```

**Использование внешних систем.** Для крупных batch-задач часто применяются Spark, Flink или облачные сервисы (AWS Batch, GCP Dataflow). Go может быть использован для написания легковесных jobs, читающих из S3/GCS и пишущих в ClickHouse, но для петабайтовых объёмов обычно подключают JVM-инструменты.

### Stream в Go: горутины, каналы и библиотеки

Потоковая обработка в Go строится вокруг трёх ключевых примитивов:

1. **Горутины** — параллельные исполнители.
2. **Каналы** — коммуникация между этапами пайплайна.
3. **Контексты** — управление временем жизни и отменой.

```go
func streamPipeline(ctx context.Context, input <-chan Order, output chan<- EnrichedOrder) {
    for {
        select {
        case <-ctx.Done():
            return
        case order, ok := <-input:
            if !ok {
                return
            }
            enriched, err := enrich(order)
            if err != nil {
                continue
            }
            output <- enriched
        }
    }
}
```

Для production-ready потоковой обработки в Go используют интеграции с Kafka (`segmentio/kafka-go`, `confluent-kafka-go`) и библиотеки вроде `goka` (аналог Kafka Streams), которая управляет состоянием в RocksDB и гарантирует восстановление после сбоев.

### Mechanical Sympathy: влияние на Go-рантайм

**Batch** — это периодическая высокая нагрузка. Во время работы batch-процессор интенсивно утилизирует CPU, создаёт много аллокаций и нагружает GC. Но между запусками нагрузка нулевая. Это позволяет агрессивно выделять ресурсы (много горутин, большие буферы) и быстро высвобождать их.

**Stream** — это постоянная, предсказуемая нагрузка. Горутины-обработчики живут долго, и любые аллокации в горячем пути накапливаются, вызывая регулярные циклы GC. Нужно минимизировать аллокации через `sync.Pool`, выбирать эффективные форматы сериализации (Protobuf, Avro), контролировать число горутин.

> [!info] Под капотом
> В stream-обработке важно не создавать горутину на каждое сообщение. Вместо этого используют фиксированный пул, читающий из канала. Это снижает нагрузку на планировщик и даёт предсказуемое потребление памяти. Каждая долгоживущая горутина потребляет минимум 2 КБ стека, но при агрессивной оптимизации GC это приемлемо.

### Лямбда- и Kappa-архитектуры

- **Лямбда-архитектура** объединяет batch и stream: холодный путь (batch) пересчитывает исторические данные для точности, горячий путь (stream) даёт приближённый результат в реальном времени. Сложность — поддержка двух кодовых баз и логики слияния.
- **Kappa-архитектура** предлагает отказаться от batch, обрабатывая всё как stream. Исторические данные перечитываются из того же брокера (Kafka с долгим retention). В Go такая архитектура проще, если брокер позволяет хранить весь объём данных.

### Когда что выбирать

**Batch подходит для:**
- Периодической аналитики (дневные отчёты, ежемесячные биллинги).
- Трансформации больших архивных данных.
- Задач, где задержка не критична, а объём огромен.

**Stream подходит для:**
- Обнаружения мошенничества в реальном времени.
- Дашбордов с метриками «прямо сейчас».
- Систем алертинга и мониторинга.
- Непрерывного обогащения событий (Event-Driven архитектура).

**Часто их комбинируют:** стрим для оперативных метрик, batch — для пересчёта и коррекции исторических данных.

### Антипаттерны и ошибки

- **Использовать batch для низколатентных задач.** Если результат нужен за секунды, ночная job не поможет.
- **Использовать stream для задач, где порядок и полнота данных важнее задержки.** Пропуск события или дубликат могут исказить аналитику, тогда как batch с repeatable read и транзакциями гарантирует точность.
- **Игнорирование идемпотентности в stream.** Из-за at-least-once доставки каждое событие может прийти несколько раз ([[27. Idempotency и exactly once семантика]]).
- **Отсутствие мониторинга лага consumer'а** в stream — можно не заметить, что обработка отстаёт от поступления.

> [!tip] Собеседование
> **Вопрос:** Вам нужно построить систему аналитики для интернет-магазина. Заказы должны отображаться в дашборде с задержкой не более 5 секунд, а также нужна точная выручка за прошлый месяц. Как вы спроектируете пайплайн?
> **Ответ:** Для реального времени я построю stream-пайплайн на Kafka и Go-сервисе, который агрегирует заказы в Redis (скользящее окно). Для точной месячной отчётности — batch-процессор, который раз в сутки читает все заказы за день из БД, корректирует возможные расхождения из-за дубликатов и пишет в аналитический ClickHouse. Таким образом, stream даёт скорость, batch — точность.

### Итог

Batch и Stream — не конкуренты, а взаимодополняющие подходы. Batch обеспечивает максимальную точность и эффективность на больших объёмах, stream — низкую задержку и реактивность. Go позволяет реализовать оба: cron-задачи с worker pool'ами для batch и долгоживущие consumer'ы с каналами для stream. Выбор зависит от бизнес-требований к задержке, допустимой погрешности и архитектурной сложности.

Теперь, когда данные непрерывно поступают и обрабатываются, критически важно не допустить перегрузки обработчиков. В следующей статье мы разберём механизм, который защищает систему от захлёбывания данными: [[43. Backpressure и контроль нагрузки]].
