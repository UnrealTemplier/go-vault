
Timeout, Retry, Circuit Breaker и Bulkhead защищают сервис от медленных и падающих зависимостей, а также изолируют ресурсы. Но есть класс угроз, который не связан с отказами компонентов, — **слишком большой поток легитимных или злонамеренных запросов**, который превышает пропускную способность системы. Никакой Bulkhead не спасёт, если разрешённое количество горутин уже занято, а новые запросы продолжают поступать с той же скоростью. Необходим механизм, ограничивающий сам **темп входящего трафика**.

**Rate Limiting** — это техника контроля частоты запросов к сервису или его отдельным операциям. Она работает на уровне запросов в секунду (RPS), запросов в минуту или других временных интервалах и отсекает избыточный трафик, прежде чем он достигнет уязвимых компонентов.

### Зачем нужен Rate Limiting

1. **Защита от DDoS и флуда.** Ограничение числа запросов с одного IP или от одного клиента предотвращает исчерпание ресурсов злоумышленником.
2. **Честное распределение ресурсов (Fair Use).** В многотенантных системах одни клиенты не должны потреблять всю пропускную способность, оставляя других без обслуживания.
3. **Соблюдение SLA и SLO.** Если downstream-сервис имеет ограниченную пропускную способность, вызывающий сервис должен сам ограничивать себя, чтобы не нарушать чужие лимиты ([[4. SLA, SLO, SLI и как они влияют на дизайн]]).
4. **Предотвращение каскадных перегрузок.** Когда один сервис начинает слать запросы быстрее, чем downstream успевает обрабатывать, без rate limiting'а запросы будут накапливаться в очередях, вызывая лавину таймаутов и повторных попыток.

### Основные алгоритмы

#### Token Bucket

Самый гибкий и распространённый алгоритм. В «ведре» хранятся токены, которые пополняются с постоянной скоростью и отбираются при каждом запросе. Если токенов нет — запрос отбрасывается. Размер ведра позволяет накапливать токены до определённого предела, что даёт возможность кратковременных всплесков (bursts) сверх установленной средней скорости.

```mermaid
flowchart LR
    Incoming["Входящие запросы"]
    Bucket["Ведро токенов<br>вместимость N<br>пополнение: r токенов/сек"]
    Downstream["Downstream-сервис"]

    Incoming -->|"Забирает 1 токен<br>на запрос"| Bucket
    Bucket -->|"Пропускает, если есть токен"| Downstream
    Bucket -->|"Отклоняет 429, если пусто"| Incoming
```

#### Leaky Bucket

Запросы попадают в очередь фиксированного размера и обрабатываются с постоянной скоростью (протекают через «дырку»). Если очередь переполнена, запросы отбрасываются. В отличие от Token Bucket, не допускает bursts — темп обработки жёстко задан.

#### Fixed Window

Счётчик запросов сбрасывается через фиксированные интервалы (например, каждую минуту). Очень прост в реализации, но страдает от «эффекта границы»: два последовательных запроса могут попасть в соседние окна, удвоив нагрузку.

#### Sliding Window

Улучшенный вариант Fixed Window: учитывается не только текущее окно, но и взвешенный вклад предыдущего. Обеспечивает более плавное ограничение без резких скачков.

### Реализация в Go

Go-экосистема предоставляет как стандартный пакет, так и готовые библиотеки для rate limiting.

#### `golang.org/x/time/rate` — Token Bucket

Пакет реализует Token Bucket с возможностью ожидания токена (блокирующий вариант) или немедленного резервирования.

```go
limiter := rate.NewLimiter(rate.Limit(100), 200) // 100 RPS, burst 200

func middleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if !limiter.Allow() {
            w.Header().Set("Retry-After", "1")
            http.Error(w, "rate limit exceeded", http.StatusTooManyRequests)
            return
        }
        next.ServeHTTP(w, r)
    })
}
```

Метод `Allow()` не блокирует горутину — он мгновенно проверяет наличие токена и возвращает `false`, если его нет. Это минимальные накладные расходы.

Для более сложных сценариев (лимиты на клиента) используют `Reserve()` с ожиданием:

```go
r := limiter.Reserve()
if !r.OK() {
    http.Error(w, "rate limit exceeded", http.StatusTooManyRequests)
    return
}
time.Sleep(r.Delay()) // ждём до появления токена
```

#### Распределённый Rate Limiting через Redis

В кластере из нескольких экземпляров сервиса локальный Token Bucket не работает — каждый инстанс будет пропускать свою квоту, и суммарный лимит превысится. Нужно общее хранилище счётчиков. Redis идеально подходит благодаря атомарным операциям `INCR` и TTL.

```go
func isAllowed(ctx context.Context, rdb *redis.Client, key string, limit int, window time.Duration) (bool, error) {
    pipe := rdb.TxPipeline()
    incr := pipe.Incr(ctx, key)
    pipe.Expire(ctx, key, window)
    _, err := pipe.Exec(ctx)
    if err != nil {
        return false, err
    }
    return incr.Val() <= int64(limit), nil
}
```

Это реализует Fixed Window. Для скользящего окна используют сортированные множества (`ZSET`) или готовые библиотеки вроде `go-redis/redis_rate`, реализующие алгоритм Generic Cell Rate.

#### Middleware для популярных роутеров

Для `chi`, `gin`, `echo` существуют готовые middleware, но написать свой — дело нескольких строк:

```go
func RateLimitMiddleware(limiter *rate.Limiter) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            if !limiter.Allow() {
                w.Header().Set("Retry-After", "1")
                w.Header().Set("X-RateLimit-Limit", "100")
                http.Error(w, "too many requests", http.StatusTooManyRequests)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}
```

> [!warning] Ловушка / Gotcha
> При использовании локального лимитера в многоинстансном развёртывании каждый под будет иметь свой бакет. Общий лимит в N раз выше запланированного (N — число подов). Для внешних клиентов обязательно использовать распределённый трекинг счётчиков через Redis или централизованный API Gateway ([[35. API Gateway и BFF]]).

### Mechanical Sympathy: влияние Rate Limiting на рантайм Go

**Локальный Token Bucket (пакет `rate`):**
- Проверка `Allow()` — это атомарное вычитание токенов с минимальным количеством тактов.
- Не создаёт аллокаций в горячем пути.
- Не блокирует горутину, быстро возвращает `false`.
- При использовании `Reserve()` и `time.Sleep` горутина засыпает, но это сознательное решение для сглаживания.

**Распределённый Rate Limiting с Redis:**
- Каждый запрос требует сетевого вызова к Redis, что добавляет задержку.
- Можно батчировать инкременты или использовать локальный кэш для снижения обращений.
- Соединения к Redis управляются пулом (`go-redis`), netpoller обеспечивает эффективный I/O.

### Уровни Rate Limiting в архитектуре

1. **На уровне API Gateway:** единая точка входа ограничивает трафик до попадания в микросервисы. Подходит для внешних клиентов.
2. **На уровне отдельного сервиса:** защита от перегрузки конкретного downstream. Может использовать локальный лимитер, если сервис один.
3. **На уровне клиента:** вызывающий сервис сам ограничивает исходящие запросы, уважая лимиты downstream. Реализуется через `rate.Limiter` в коде клиента.

Комбинирование этих уровней даёт эшелонированную защиту: Gateway держит фронтальный лимит, сервис — локальный, клиент — исходящий.

### Связь с другими паттернами

- **Circuit Breaker** ([[36. Circuit Breaker, Retry, Timeout и Backoff]]) реагирует на ошибки, Rate Limiting предотвращает перегрузку, которая могла бы эти ошибки вызвать.
- **Bulkhead** ([[37. Bulkhead и изоляция отказов]]) изолирует ресурсы, Rate Limiting ограничивает темп, с которым эти ресурсы потребляются.
- **Backpressure** ([[43. Backpressure и контроль нагрузки]]): отбрасывая запросы, Rate Limiter создаёт обратное давление, заставляя клиентов замедляться.

### Антипаттерны и типичные ошибки

- **Жёсткий лимит без возможности burst'а.** Небольшие всплески трафика естественны; Token Bucket позволяет их обработать, не отбрасывая запросы.
- **Отсутствие информирования клиента.** Сервер обязан возвращать `429 Too Many Requests` с заголовком `Retry-After`, чтобы клиент мог корректно повторить попытку.
- **Rate Limiting без мониторинга.** Не видно, срабатывает ли лимит, кто упирается в него, насколько лимиты адекватны нагрузке.
- **Использование глобального мьютекса для счётчиков в распределённой системе.** Это становится узким местом; используйте атомарные операции Redis.

> [!tip] Собеседование
> **Вопрос:** Как реализовать Rate Limiting для внешнего API, к которому обращаются разные клиенты с разными квотами?
> **Ответ:** Я бы использовал распределённый лимитер на основе Redis — скользящее окно на каждую пару `client_id:endpoint`. Клиенты и их квоты хранятся в конфигурации. Middleware извлекает `client_id` из API-ключа или заголовка, делает `INCR` в Redis с TTL окна и проверяет, не превышен ли лимит. В ответ добавляются заголовки `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `Retry-After`. Для burst-защиты алгоритм Token Bucket, реализованный через Lua-скрипт в Redis или библиотеку `redis_rate`.

### Итог

Rate Limiting — это первая линия обороны против перегрузок, DDoS-атак и нечестного потребления ресурсов. В Go он реализуется легко и элегантно: от простого локального Token Bucket через `golang.org/x/time/rate` до распределённых лимитеров с Redis. В сочетании с Bulkhead и Circuit Breaker он формирует завершённую стратегию устойчивости, позволяя системе держать нагрузку в рамках SLO и деградировать предсказуемо, а не обрушиваться лавиной.
