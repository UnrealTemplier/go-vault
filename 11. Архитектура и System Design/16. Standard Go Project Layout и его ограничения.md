## Эволюция структуры Go-проекта

Когда разработчик впервые клонирует популярный open-source проект на Go или изучает примеры "best practices", он неизбежно сталкивается с определённой схемой расположения директорий: `cmd/`, `internal/`, `pkg/`, `api/` и так далее. Эта схема известна как **Standard Go Project Layout** и фактически стала отраслевым стандартом, хотя официально не закреплена командой Go. В этой статье мы разберём, что это за структура, какие проблемы она решает, в чём её ограничения и как адаптировать её к современным архитектурным подходам вроде Clean Architecture ([[14. Clean Architecture и Dependency Rule]]) и модульного монолита ([[10. Модульный монолит как компромисс]]).

### Что такое Standard Go Project Layout

Standard Go Project Layout — это набор соглашений о структуре директорий, популяризированный репозиторием [golang-standards/project-layout](https://github.com/golang-standards/project-layout). Важно понимать: это **не официальный стандарт**, а обобщение успешных практик из крупных проектов (Kubernetes, Prometheus, Docker и др.). Тем не менее, он настолько распространён, что знание этой структуры обязательно для любого Go-разработчика, претендующего на позицию Middle+.

Основные цели такой организации:
- **Разделение ответственности**: отделение исполняемых файлов от библиотечного кода, внутренних пакетов от публичных API.
- **Упрощение сборки и деплоя**: стандартные пути для Dockerfile, конфигураций, скриптов CI/CD.
- **Удобство навигации**: новый член команды быстрее понимает, где что лежит.

### Обзор ключевых директорий

Рассмотрим типичное дерево проекта, придерживающегося этого соглашения:

```
myapp/
├── cmd/                    # Точки входа (main)
├── internal/               # Приватный код приложения
├── pkg/                    # Публичные библиотеки
├── api/                    # Контракты API (OpenAPI, Protobuf)
├── configs/                # Файлы конфигурации и шаблоны
├── scripts/                # Вспомогательные скрипты
├── build/                  # Упаковка и CI (Dockerfile, packer)
├── deployments/            # Манифесты развёртывания (k8s, docker-compose)
├── test/                   # Внешние тесты и тестовые данные
├── docs/                   # Документация
├── go.mod
└── Makefile
```

#### `/cmd`

Содержит поддиректории для каждого исполняемого файла, который собирается из проекта. Каждая поддиректория содержит `main.go` и, возможно, вспомогательные файлы, специфичные для этого бинарника. Имя директории соответствует имени бинарника.

```
cmd/
├── server/      # Основной HTTP/gRPC сервер
│   └── main.go
├── worker/      # Фоновый воркер
│   └── main.go
└── cli/         # Утилита командной строки
    └── main.go
```

**Важно:** код в `cmd/` должен быть минимальным. Обычно он только парсит конфигурацию, инициализирует зависимости (Dependency Injection) и запускает приложение. Вся бизнес-логика и даже настройка роутинга должны быть вынесены в `internal/`.

```go
// cmd/server/main.go
func main() {
    cfg := config.MustLoad()
    logger := log.New(cfg.LogLevel)
    db := postgres.MustConnect(cfg.Database)
    
    orderRepo := postgres.NewOrderRepository(db)
    orderService := order.NewService(orderRepo)
    orderHandler := http.NewOrderHandler(orderService)
    
    srv := server.New(cfg.HTTP, logger, orderHandler)
    if err := srv.Run(); err != nil {
        logger.Fatal("server failed", "error", err)
    }
}
```

#### `/internal`

Сердце философии Standard Go Project Layout. Компилятор Go **запрещает** импортировать пакеты, лежащие внутри `internal/`, извне модуля. Это принудительная инкапсуляция: вы гарантируете, что код в `internal/` не станет публичным API и не будет использован другими проектами без вашего ведома.

Внутри `internal/` обычно организуют структуру по функциональным модулям (доменам) или слоям, в зависимости от выбранной архитектуры.

**Вариант с доменной структурой (рекомендуется для DDD и модульного монолита):**
```
internal/
├── order/          # Модуль заказов
│   ├── handler/
│   ├── service/
│   ├── repository/
│   └── model/
├── user/           # Модуль пользователей
├── payment/        # Модуль платежей
└── shared/         # Общий код внутри приложения
    ├── db/
    └── logger/
```

**Вариант со слоистой структурой (используется реже, но встречается):**
```
internal/
├── delivery/       # = handlers
├── usecase/        # = business logic
├── repository/     # = data access
└── entity/         # = domain models
```

> [!warning] Ловушка / Gotcha
> **Не кладите слишком много в `pkg/`.** Часто начинающие разработчики выносят почти весь код в `pkg/`, оставляя `internal/` пустым. Это лишает вас защиты компилятора и открывает внутренние детали всему миру. Используйте `pkg/` только для кода, который **действительно предназначен для повторного использования другими проектами** (например, SDK к вашему API, общие утилиты типа `retry` или `paginator`). По умолчанию весь код приложения должен быть в `internal/`.

#### `/pkg`

Предназначен для библиотечного кода, который **безопасно использовать извне**. Идея в том, что другие проекты могут импортировать `github.com/yourname/myapp/pkg/something`, и вы гарантируете обратную совместимость этого API.

**Примеры содержимого `pkg/`:**
- Клиент к вашему сервису: `pkg/client/`
- Общие структуры ответов/ошибок: `pkg/apierrors/`
- Утилиты, стабильные и документированные: `pkg/retry/`, `pkg/pagination/`

> [!info] Под капотом
> Исторически Go позволял импортировать любой пакет в репозитории. Это приводило к тому, что случайные внутренние пакеты становились публичным API, и разработчики боялись их менять. Появление `internal/` в Go 1.4 решило эту проблему на уровне компилятора. `pkg/` остался как «безопасная зона» для осознанно публичного кода, но его использование не обязательно и часто оспаривается.

#### `/api`

Содержит контракты API: OpenAPI (Swagger) спецификации, Protobuf файлы (`.proto`), GraphQL схемы. Вынос контрактов в корень проекта упрощает кодогенерацию и позволяет хранить их отдельно от реализации.

```
api/
├── proto/
│   └── order/
│       └── v1/
│           └── order.proto
├── openapi/
│   └── order.yaml
└── graphql/
    └── schema.graphql
```

#### `/configs`

Шаблоны конфигурационных файлов или файлы конфигурации по умолчанию. Например, `config.yaml.example`. Реальные конфиги (с секретами) не должны коммититься, но шаблоны — да.

#### `/scripts`

Вспомогательные скрипты для сборки, кодогенерации, запуска линтеров, миграций. Часто это bash-скрипты или Go-утилиты в `tools.go`.

#### `/build`

Файлы для упаковки приложения: `Dockerfile`, `packer.json`, файлы для CI (GitLab CI, GitHub Actions можно и здесь, но чаще в корне `.github/`).

#### `/deployments`

Манифесты для развёртывания в инфраструктуре: Kubernetes YAML, docker-compose.yaml, Terraform, Helm charts.

#### `/test`

Данные для интеграционных тестов, внешние тесты (не модульные тесты с `_test.go`, которые лежат рядом с кодом). Сюда можно положить `testdata/` или скрипты для поднятия окружения.

### Ограничения и критика

Несмотря на широкое распространение, Standard Go Project Layout имеет ряд недостатков и не рекомендуется командой Go как обязательный.

1. **Избыточность для малых проектов.** Для микросервиса из 2-3 эндпоинтов такая структура — огромный оверхед. Проще всё положить в пакет `main` или в пару файлов в корне.

2. **Неявное поощрение "слоистой" архитектуры.** Начинающие разработчики часто воспринимают `internal/handler`, `internal/service`, `internal/repository` как догму, даже когда это не нужно. В итоге получается много пакетов с одним файлом и высокая связанность через общие модели. Вместо этого рекомендуется группировать по модулям/доменам (как в первом варианте).

3. **Путаница с `pkg` и `internal`.** Новички часто не понимают разницы и кладут всё в `pkg`. Это ломает инкапсуляцию, ради которой всё затевалось.

4. **Не является официальным.** Команда Go неоднократно заявляла, что не поддерживает этот layout. Они рекомендуют начинать с простой структуры и усложнять её по мере необходимости. Расс Плато (Russ Cox) писал: *"A little copying is better than a little dependency."*

5. **Сложность с версионированием.** В монорепозиториях с несколькими сервисами в `cmd/` версионирование всего модуля становится проблемой: изменение в одном сервисе тянет за собой смену версии всего модуля. В таких случаях лучше держать каждый сервис в отдельном модуле (отдельный репозиторий или подмодуль).

### Адаптация к современным архитектурным подходам

Как совместить Standard Go Project Layout с Clean Architecture или Hexagonal Architecture?

**Для модульного монолита с DDD:**
- Весь код модулей лежит в `internal/{module}/`
- Внутри модуля можно выделить подпакеты: `domain/` (сущности), `application/` (use cases), `infrastructure/` (репозитории, клиенты), `interface/` (handlers).
- Пакет `pkg/` содержит только то, что действительно экспортируется (общие события, API клиент).

**Для микросервисного проекта:**
- Если каждый сервис — отдельный модуль Go (отдельный `go.mod`), вся эта структура воспроизводится в корне каждого сервиса.
- Если это монорепозиторий с общей кодовой базой для нескольких сервисов (монорепо с несколькими Go-модулями), структура усложняется, и часто `internal/` становится общим для всех, а `cmd/` содержит точки входа.

> [!tip] Собеседование
> **Вопрос:** Что вы думаете о Standard Go Project Layout? Стали бы вы использовать его в новом проекте?
> **Ответ:** Я знаком с этим соглашением и считаю его полезным ориентиром для средних и крупных проектов, особенно с несколькими командами. Однако я не применяю его слепо. Для стартапа или микросервиса из одного `main.go` я бы начал с плоской структуры и рефакторил по мере роста. Я также предпочитаю группировать код по доменам внутри `internal/`, а не по техническим слоям, так как это лучше соответствует DDD и модульному монолиту. И я всегда кладу основной код в `internal/`, оставляя `pkg/` только для действительно переиспользуемых компонентов.

### Mechanical Sympathy: влияние на компиляцию и сборку

С точки зрения компилятора Go, структура директорий не влияет на производительность скомпилированного кода. Однако она влияет на **скорость сборки** и **удобство кэширования**. Разделение на небольшие пакеты (без фанатизма) ускоряет инкрементальную компиляцию — меняется один пакет, перекомпилируется только он и те, что от него зависят. Размещение всего кода в одном гигантском пакете `main` лишает этого преимущества.

Также размещение `go.mod` в корне и использование `internal/` позволяет инструментам вроде `go mod vendor` и `go mod tidy` работать предсказуемо.

### Альтернативные подходы

- **Плоская структура**: `main.go`, `handlers.go`, `services.go`, `repos.go` в корне. Идеально для прототипов и маленьких утилит.
- **Domain-driven package structure**: всё, что связано с заказами, лежит в `order/`, с пользователями — в `user/`, и всё это в корневом модуле или в `internal/`. Этот подход набирает популярность и более соответствует принципам модульности.
- **Структура на основе фреймворка**: если используете Buffalo, Goa или другие кодогенераторы, они навязывают свою структуру.

### Итог

Standard Go Project Layout — полезный набор конвенций, упрощающий жизнь в средних и крупных проектах. Ключевые принципы: `cmd/` для бинарников, `internal/` для приватного кода, `pkg/` для осознанно публичного, контракты и деплой-манифесты в корне. Однако важно не превращать это в догму. Меньшие проекты выигрывают от простоты. Лучшая структура — та, которая соответствует размеру и архитектуре вашего конкретного приложения (модульный монолит, Clean Architecture, трёхслойка). Главное — осознанно выбирать, а не копировать слепо.

Теперь, когда у нас есть представление о физической раскладке файлов, пора углубиться в то, как связывать компоненты внутри этой структуры, не создавая жёстких зависимостей. Об этом — в следующей статье: [[17. Dependency Injection в Go на уровне архитектуры]].
