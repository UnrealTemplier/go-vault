## Внедрение зависимостей как архитектурная дисциплина

В предыдущих статьях мы разобрали несколько архитектурных стилей — от трёхслойки до Clean Architecture — и увидели, что ключом к гибкости и тестируемости является **инверсия контроля** и её частный случай — **внедрение зависимостей (Dependency Injection, DI)**. В Go, где нет встроенного фреймворка для DI, эта задача решается идиоматичными, явными средствами языка, что принципиально отличает подход от Java/C# с их контейнерами. В этой статье мы детально рассмотрим DI на уровне архитектуры Go-приложения: от простых конструкторов до кодогенерации, обсуждая компромиссы и влияние на сопровождаемость.

### Что такое Dependency Injection и зачем оно нужно

**Dependency Injection** — это техника, при которой компонент (клиент) получает свои зависимости извне, а не создаёт их самостоятельно. Это позволяет:

- **Ослабить связанность**: клиент зависит от абстракции (интерфейса), а не от конкретной реализации.
- **Упростить тестирование**: легко подставить моки или фейковые реализации.
- **Гибко заменять компоненты**: сменить БД, логирование, метрики без правок бизнес-логики.

В архитектурном разрезе DI является практической реализацией **правила зависимостей** из Clean Architecture ([[14. Clean Architecture и Dependency Rule]]): внутренние слои определяют интерфейсы, внешние поставляют конкретные реализации, а точка сборки (`main`) связывает всё воедино.

```mermaid
flowchart LR
    Main["main<br>Композиционный корень"]
    Core["Ядро<br>интерфейсы"]
    Adapter["Адаптер<br>реализация"]

    Main -->|"внедряет"| Core
    Main -->|"создаёт"| Adapter
    Core -.->|"зависит от"| Adapter
    Adapter -->|"реализует"| Core
```

### Идиоматичный DI в Go

В Go принято использовать **явное внедрение зависимостей** без магии рефлексии и тяжёлых контейнеров. Вот основные паттерны в порядке возрастания сложности.

#### 1. Конструктор (Constructor Injection)

Самый простой и предпочтительный способ. Структура получает зависимости через параметры конструктора и сохраняет их в полях. Это делает зависимости явными, а структуру — готовой к использованию сразу после создания.

```go
// internal/order/service.go
package order

type Repository interface {
    Save(ctx context.Context, order *Order) error
}

type Service struct {
    repo   Repository
    logger Logger
}

func NewService(repo Repository, logger Logger) *Service {
    return &Service{repo: repo, logger: logger}
}

func (s *Service) Create(ctx context.Context, req CreateRequest) (*Order, error) {
    // использование s.repo, s.logger
}
```

**Преимущества:**
- Максимальная прозрачность.
- Проверка зависимостей во время компиляции.
- Легко тестировать.

**Недостатки:**
- При большом количестве зависимостей (более 5-7) конструктор становится громоздким. Это сигнал, что структура делает слишком много и её стоит разделить.

#### 2. Функциональные опции (Functional Options)

Паттерн для опциональных зависимостей или конфигурации. Позволяет создавать объекты с удобным API, не перегружая конструктор.

```go
type ServiceOption func(*Service)

func WithMetrics(m Metrics) ServiceOption {
    return func(s *Service) {
        s.metrics = m
    }
}

func NewService(repo Repository, opts ...ServiceOption) *Service {
    s := &Service{repo: repo, logger: defaultLogger}
    for _, opt := range opts {
        opt(s)
    }
    return s
}

// Использование
svc := NewService(repo, WithMetrics(prom), WithTracer(jaeger))
```

#### 3. Интерфейсы для инверсии зависимостей

Ключевой архитектурный момент: **интерфейс определяется клиентом**. Не репозиторий объявляет интерфейс, который реализует, а сервис объявляет интерфейс, который ему нужен. Благодаря неявному удовлетворению интерфейсов в Go, любой объект с нужными методами автоматически подходит.

```go
// core/domain/order/service.go
type OrderSaver interface {
    Save(ctx context.Context, order *Order) error
}

type OrderService struct {
    saver OrderSaver
}

// adapter/postgres/repo.go
type Repo struct { db *sql.DB }
func (r *Repo) Save(ctx context.Context, order *Order) error { ... }

// Repo автоматически удовлетворяет OrderSaver, 
// хотя явно не объявляет implements.
```

Это фундаментальное свойство Go позволяет избежать централизованных интерфейсов и держать зависимости направленными внутрь.

### Композиционный корень: main.go

В Clean/Hexagonal архитектуре точка сборки всех зависимостей выносится в начало исполнения — обычно в `main.go` или отдельный пакет `di`. Это единственное место, где код знает обо всех конкретных реализациях. Вся остальная система работает с интерфейсами.

```go
// cmd/server/main.go
func main() {
    cfg := config.MustLoad()
    logger := log.New(cfg.LogLevel)
    db := postgres.MustConnect(cfg.Database)
    
    // Внедрение зависимостей явно, но упорядоченно
    orderRepo := postgres.NewOrderRepository(db)
    paymentClient := stripe.NewClient(cfg.StripeKey)
    
    orderSvc := order.NewService(orderRepo, paymentClient, logger)
    orderHandler := http.NewOrderHandler(orderSvc, logger)
    
    srv := http.NewServer(cfg.HTTPPort, orderHandler)
    srv.Start()
}
```

> [!warning] Ловушка / Gotcha
> Не пытайтесь имитировать контейнер типа Spring через глобальные переменные и `init()`. Это ведёт к скрытым зависимостям, усложнению тестирования и неконтролируемому порядку инициализации. В Go глобальные переменные приемлемы только для констант и крайне редко для singleton-зависимостей, которые не влияют на бизнес-логику (например, метрики или трассировка).

### Масштабирование DI: от явного к кодогенерации

Когда проект разрастается, ручное связывание десятков зависимостей в `main.go` становится утомительным и подверженным ошибкам. Возникает желание автоматизировать DI, но сохранив явность и производительность Go.

#### Wire (от Google)

`wire` — инструмент кодогенерации, который анализирует аннотации в коде и генерирует функцию `wire_gen.go` с полным графом зависимостей. Это **не рантайм-контейнер**: весь код разрешения зависимостей компилируется статически, без рефлексии.

```go
// +build wireinject

package main

import (
    "github.com/google/wire"
)

func InitializeApp() (*App, error) {
    wire.Build(
        postgres.NewOrderRepository,
        stripe.NewPaymentClient,
        order.NewService,
        http.NewOrderHandler,
        http.NewServer,
        // wire свяжет всё автоматически
    )
    return nil, nil
}
```

После запуска `wire` генерируется файл:

```go
func InitializeApp() (*App, error) {
    db := postgres.MustConnect()
    orderRepo := postgres.NewOrderRepository(db)
    paymentClient := stripe.NewPaymentClient()
    orderSvc := order.NewService(orderRepo, paymentClient)
    orderHandler := http.NewOrderHandler(orderSvc)
    server := http.NewServer(orderHandler)
    return &App{Server: server}, nil
}
```

**Преимущества Wire:**
- Полностью статический, ошибки на этапе генерации.
- Легко интегрируется с `go generate`.
- Поддерживает интерфейсы и может автоматически выбирать реализацию.
- Сохраняет читаемость сгенерированного кода.

#### Fx (от Uber)

`fx` — полноценный DI-фреймворк, работающий в рантайме с использованием рефлексии. Он больше похож на классические контейнеры из Java. Предоставляет управление жизненным циклом (start/stop), группировку зависимостей, аннотации.

```go
func main() {
    fx.New(
        fx.Provide(
            postgres.NewOrderRepository,
            stripe.NewPaymentClient,
            order.NewService,
            http.NewOrderHandler,
            http.NewServer,
        ),
        fx.Invoke(func(*http.Server) {}),
    ).Run()
}
```

**Когда использовать Fx вместо Wire:**
- Нужно сложное управление жизненным циклом: запуск/остановка фоновых процессов, graceful shutdown.
- Большое количество опциональных зависимостей, которые удобно группировать.
- Проект уже использует Uber-стек (например, Zap, fx). 

> [!info] Под капотом
> С точки зрения производительности, явный конструктор и Wire практически идентичны — всё разрешается на этапе компиляции/линковки. Fx использует рефлексию для вызовов конструкторов и внедрения, что добавляет небольшие накладные расходы при старте (порядка десятков-сотен миллисекунд), но в рантайме после построения графа разницы нет. Для высоконагруженных сервисов критичного удара нет, но стартапы ценят fast boot, поэтому Wire предпочтительнее.

### DI и архитектурные границы

DI тесно связано с выбранной архитектурой. Рассмотрим, как DI проявляется в разных стилях.

| Архитектура | Способ DI | Расположение интерфейсов |
|-------------|-----------|--------------------------|
| **Трехслойка** | Часто прямой вызов, DI слабо выражено | Интерфейсы редко или определены в repository |
| **Чистая архитектура** | Явный DI через конструкторы, интерфейсы в usecase слое | Интерфейсы в ядре, реализации в адаптерах |
| **Hexagonal** | То же, что и Clean; порты = интерфейсы в ядре | Адаптеры реализуют порты |
| **Модульный монолит** | Ручной DI в main, возможно Wire | Интерфейсы внутри модулей |

В модульном монолите с DDD ([[10. Модульный монолит как компромисс]], [[12. Domain Driven Design. Bounded Context и Aggregate]]) каждый модуль может экспортировать конструктор, а главный `main` связывает модули через их публичные API (интерфейсы). Внутри модуля можно использовать простые конструкторы или даже глобальные переменные для приватных зависимостей, если это упрощает код и не вредит тестированию.

### Тестирование с DI

Главный бенефит DI — модульное тестирование. Подмена реального репозитория на in-memory заглушку становится тривиальной.

```go
type mockRepo struct {
    orders map[string]*Order
}

func (m *mockRepo) Save(ctx context.Context, order *Order) error {
    m.orders[order.ID] = order
    return nil
}

func TestCreateOrder(t *testing.T) {
    repo := &mockRepo{orders: make(map[string]*Order)}
    svc := order.NewService(repo, logger) // внедряем мок
    _, err := svc.Create(ctx, validRequest)
    assert.NoError(t, err)
}
```

Без DI пришлось бы либо поднимать тестовую БД, либо патчить глобальные переменные, что ненадёжно и усложняет параллельный запуск тестов.

### Mechanical Sympathy: аллокации и DI

Использование интерфейсов и внедрение зависимостей через конструктор не создаёт значительных накладных расходов в Go. Вызов метода через интерфейс — это разыменование указателя на itab (interface table), что хорошо предсказуемо для CPU. Единственное, о чём стоит помнить: если конструктор возвращает интерфейс, а не конкретный тип, это может привести к дополнительной аллокации на куче из-за упаковки значения в интерфейс. Поэтому **конструкторы должны возвращать конкретный тип, а интерфейс принимать**. Это идиома Go, называемая "accept interfaces, return structs".

```go
// Хорошо: принимает интерфейс, возвращает структуру
func NewService(repo Repository) *Service { ... }

// Плохо: возвращает интерфейс без необходимости
func NewService(repo Repository) OrderService { ... }
```

Исключение — когда нужно вернуть несколько реализаций в зависимости от конфигурации, но и тогда лучше возвращать конкретный тип, который уже реализует нужный интерфейс.

### Антипаттерны DI в Go

1. **Service Locator (глобальный контейнер)** — создание глобальной переменной `App.Services` и вызов `Get("order")` в коде. Скрывает зависимости, ломает тестируемость, мешает статическому анализу.

2. **Интерфейсы ради интерфейсов** — определение интерфейса на каждую структуру, даже когда он не нужен. В Go интерфейсы нужны только там, где есть полиморфизм или тестирование. Не создавайте `type UserServiceInterface interface`, если есть одна реализация.

3. **Внедрение через поля (Field Injection)** без конструктора:
```go
svc := &Service{}
svc.Repo = repo // неочевидно, что Repo обязателен
```
Используйте конструктор.

4. **Переусложнение DI для маленьких сервисов** — для микросервиса с тремя обработчиками ручное связывание проще и понятнее, чем генерация через Wire. Начинайте с простого, усложняйте по мере необходимости.

> [!tip] Собеседование
> **Вопрос:** Как бы вы организовали внедрение зависимостей в Go-проекте с 20+ сервисами? Использовали бы вы фреймворк вроде `wire` или обошлись бы конструкторами?
> **Ответ:** Я бы начал с явных конструкторов, группируя зависимости в `main.go`. Когда их количество превысит 10-15 или появятся вложенные графы, я интегрирую `wire`. Этот инструмент сохраняет статическую проверку, генерирует читаемый код и не добавляет рантайм-оверхеда. `fx` я бы рассматривал только если нужны сложные хуки жизненного цикла для множества компонентов. Кроме того, я бы рассмотрел модульную структуру: для каждого модуля — свой подграф зависимостей, а в `main` — связывание модулей.

### Итог

Dependency Injection в Go — не магия, а дисциплина явного связывания компонентов. Идиоматичный подход с конструкторами, интерфейсами и точкой сборки в `main` обеспечивает хорошую тестируемость и гибкость без лишних фреймворков. С ростом проекта можно применять кодогенерацию (Wire), сохраняя преимущества статической типизации. Ключевой архитектурный принцип: **зависимости направлены внутрь, интерфейсы определяются потребителем, конструкторы возвращают конкретные типы**. Это создаёт прочный фундамент для любой архитектуры — от трёхслойки до чистой.
