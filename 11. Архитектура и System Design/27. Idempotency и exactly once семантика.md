## Определение и важность

В распределённых системах, особенно построенных на микросервисах и асинхронной коммуникации, сетевые сбои и таймауты неизбежны. Клиент может отправить запрос, не получить ответ из-за обрыва связи и повторить попытку. Сервер, получив повторный запрос, должен гарантировать, что бизнес-операция не выполнится дважды. Именно для этого существует **идемпотентность**.

**Идемпотентная операция** — это операция, которую можно выполнить несколько раз, но результат будет таким же, как после первого успешного выполнения.

В архитектуре Go-сервисов идемпотентность критична для:
- Обработчиков HTTP/gRPC (retry от клиентов).
- Консьюмеров очередей (at-least-once доставка).
- Оркестраторов Saga ([[26. Saga Pattern. Оркестрация и хореография]]), где компенсации могут повторяться.
- Записи в Event Store, чтобы избежать дублирования событий ([[24. Event Sourcing. Хранение событий вместо состояния]]).

### Exactly-Once семантика

Под **exactly-once** понимают гарантию, что сообщение будет обработано ровно один раз, не потеряется и не дублируется. В реальности достичь абсолютной exactly-once обработки в распределённой системе невозможно из-за законов физики (сети отказывают), поэтому чаще говорят об **эффективной exactly-once**: система гарантирует отсутствие дублирования эффектов даже при повторной доставке сообщений. Достигается это через **идемпотентность обработчика** и **идемпотентность инфраструктуры**.

Три уровня семантики доставки сообщений:

| Семантика | Описание | Пример в Go |
|-----------|----------|-------------|
| **At-most-once** | Сообщение доставляется ноль или один раз, без гарантий | `channel <- msg` (без буфера, non-blocking) |
| **At-least-once** | Сообщение будет доставлено, но может дублироваться | RabbitMQ, Kafka (автокоммит до обработки) |
| **Exactly-once** | Эффект обработки применится ровно один раз | Идемпотентный consumer + Kafka transactional producer |

### Реализация идемпотентности в Go

Идемпотентность — это всегда дополнительная логика и, как правило, дополнительное состояние. В Go есть несколько проверенных паттернов.

#### 1. Идемпотентный ключ (Idempotency Key)

Клиент генерирует уникальный ключ для каждой команды и передаёт его в заголовке (`Idempotency-Key`). Сервер сохраняет ключ вместе с результатом операции в атомарной транзакции. При повторе ключа возвращается ранее сохранённый результат.

```go
// Обработчик HTTP
func (h *PaymentHandler) Charge(w http.ResponseWriter, r *http.Request) {
    key := r.Header.Get("Idempotency-Key")
    if key == "" {
        http.Error(w, "missing idempotency key", http.StatusBadRequest)
        return
    }

    // Проверяем, не был ли ключ уже использован
    cached, err := h.idempotencyStore.Get(r.Context(), key)
    if err == nil {
        w.Header().Set("Idempotent-Replayed", "true")
        json.NewEncoder(w).Encode(cached)
        return
    }

    // Выполняем бизнес-операцию и сохраняем результат + ключ атомарно
    result, err := h.paymentService.Charge(r.Context(), chargeReq)
    if err != nil {
        http.Error(w, err.Error(), http.StatusInternalServerError)
        return
    }
    
    // Сохраняем в БД ключ + результат (одной транзакцией с зарядом)
    err = h.idempotencyStore.Set(r.Context(), key, result)
    if errors.Is(err, ErrKeyExists) {
        // редкий случай гонки — перечитываем
        cached, _ := h.idempotencyStore.Get(r.Context(), key)
        json.NewEncoder(w).Encode(cached)
        return
    }
    json.NewEncoder(w).Encode(result)
}
```

> [!info] Под капотом
> Идемпотентный ключ обычно хранится в той же БД, что и бизнес-данные, чтобы гарантировать атомарность записи ключа и изменений. В PostgreSQL это реализуется через `INSERT INTO idempotency (key, result) ... ON CONFLICT DO NOTHING` в одной транзакции с изменением данных.

#### 2. Уникальные ограничения базы данных

Если бизнес-операция естественно содержит уникальный идентификатор, можно положиться на ограничение `UNIQUE` в БД. Например, повторный запрос на создание заказа с тем же `OrderID` вызовет ошибку нарушения уникальности, которую можно обработать — вернуть уже существующий заказ.

```go
func (s *OrderService) Create(ctx context.Context, order *Order) error {
    err := s.repo.Insert(ctx, order)
    if errors.Is(err, ErrDuplicateKey) {
        return s.repo.GetByID(ctx, order.ID) // возвращаем существующий
    }
    return err
}
```

#### 3. Версионирование и оптимистическая блокировка

В паттернах Event Sourcing идемпотентность достигается через уникальность номера версии агрегата. Попытка добавить событие с уже существующим номером версии отбрасывается.

```go
// Event Store
func (es *EventStore) Append(ctx context.Context, aggID string, version int, events []Event) error {
    _, err := es.db.ExecContext(ctx, 
        `INSERT INTO events (aggregate_id, version, ...) VALUES ($1, $2, ...) ON CONFLICT (aggregate_id, version) DO NOTHING`,
        aggID, version)
    return err
}
```

#### 4. Дедупликация в брокерах сообщений

Некоторые брокеры предоставляют встроенную поддержку exactly-once. Пример с Kafka:
- **Idempotent Producer**: продюсер назначает `Producer ID` и нумерует сообщения. Брокер отбрасывает дубликаты по PID + sequence number.
- **Transactional API**: позволяет атомарно записать сообщения в несколько топиков и коммитнуть оффсет.

В Go идиоматично комбинировать дедупликацию на уровне брокера и идемпотентную обработку в коде, так как гарантии брокера не абсолютны.

### Mechanical Sympathy: идемпотентность и рантайм Go

**Хранение состояния идемпотентности.** Каждый идемпотентный ключ требует записи в БД или кэш. Это дополнительный сетевой вызов или запрос к диску. Для снижения задержки можно использовать быстрый кэш (Redis), но тогда теряется гарантия атомарности с основной бизнес-транзакцией — при сбое после сохранения ключа в Redis, но до коммита бизнес-операции, повторный запрос увидит ключ и решит, что операция выполнена, хотя это не так.

**Аллокации.** Идемпотентный обработчик часто сериализует ключ и результат для хранения. Если ключей много, сериализация создаёт нагрузку на GC. Используйте `sync.Pool` для буферов, если сервис обрабатывает десятки тысяч ключей в секунду.

**Конкурентность.** Один и тот же идемпотентный ключ может прийти одновременно в двух запросах. Атомарная вставка в БД (`ON CONFLICT DO NOTHING`) разрешает гонку корректно, но один из запросов получит ошибку конфликта, которую нужно обработать, перечитав результат. В Go это может выглядеть как несколько горутин, одновременно пытающихся записать один ключ.

```go
var mu sync.Mutex
mu.Lock()
defer mu.Unlock()
// Проверка и вставка под мьютексом — но плохо масштабируется
```
Лучше полагаться на БД, а не на in-memory блокировку, чтобы система оставалась распределённой.

### Exactly-Once в Kafka на Go

Для консьюмера Kafka exactly-once достигается, если обработка сообщения идемпотентна, а коммит оффсета происходит в той же транзакции, что и запись результатов обработки. В Go популярные клиенты (`segmentio/kafka-go`, `confluent-kafka-go`) поддерживают transactional API.

```go
// Пример с segmentio/kafka-go
producer := kafka.NewWriter(kafka.WriterConfig{
    Brokers:     []string{"localhost:9092"},
    Topic:       "orders",
    Idempotent:  true, // включает идемпотентного продюсера
})
producer.WriteMessages(ctx, msg...)
```

Консьюмер в транзакционной группе читает сообщения внутри транзакции и атомарно коммитит оффсеты после обработки:

```go
// Чтение в изоляции read_committed
reader := kafka.NewReader(kafka.ReaderConfig{
    Brokers:       []string{"localhost:9092"},
    Topic:         "orders",
    GroupID:       "order-processor",
    IsolationLevel: kafka.ReadCommitted,
})
for {
    msg, err := reader.ReadMessage(ctx)
    // идемпотентная обработка + коммит оффсета (авто)
}
```

> [!warning] Ловушка / Gotcha
> Включение транзакционного API в Kafka увеличивает задержку и снижает пропускную способность, потому что брокер должен координировать несколько топиков/партиций. Не применяйте exactly-once глобально, если это не критично.

### Связь с другими паттернами

- **Saga** требует, чтобы компенсации были идемпотентны. Если сервис склада отменил резерв, повторная отмена не должна создать отрицательное количество.
- **Outbox Pattern** гарантирует, что событие будет отправлено ровно один раз, если потребитель идемпотентен (удаление дубликатов по ID сообщения).
- **Circuit Breaker** ([[36. Circuit Breaker, Retry, Timeout и Backoff]]) при размыкании может вызывать повторные попытки — идемпотентность позволяет безопасно ретраить.
- **CQRS** ([[23. CQRS. Разделение чтения и записи]]): проекторы, строящие Read Model, должны быть идемпотентными, так как события могут доставляться повторно.

### Идемпотентность и UX

Иногда идемпотентность должна учитывать клиентский опыт. Если клиент нажал «Оплатить» дважды, хорошо бы ответить одинаковым статусом, не создавая два платежа. Идемпотентный ключ решает это, но важно помнить, что ключи могут быть разными для разных попыток, если клиент не сохраняет их. Лучший подход — генерировать ключ на фронтенде и передавать в запросе.

> [!tip] Собеседование
> **Вопрос:** Как реализовать идемпотентный REST API на Go?
> **Ответ:** Я бы потребовал от клиентов передавать заголовок `Idempotency-Key`. На сервере:
> 1. Проверяю наличие ключа.
> 2. Если ключ уже существует в хранилище идемпотентности — возвращаю сохранённый ответ.
> 3. Иначе выполняю бизнес-операцию и в той же транзакции вставляю ключ с результатом.
> Хранилище идемпотентности может быть отдельной таблицей в PostgreSQL с уникальным constraint по ключу. В случае конфликта при вставке (гонка) — перечитываю результат.

### Итог

Идемпотентность — это не просто «хорошая практика», а фундаментальное требование к надёжным распределённым системам. В Go она реализуется через атомарные вставки в БД, идемпотентные ключи и уникальные ограничения. Современные брокеры вроде Kafka предоставляют exactly-once семантику, но архитектор должен всегда строить защиту на уровне приложения — идемпотентные обработчики, компенсации и тщательное отношение к дубликатам. Без этого невозможны надёжные Saga, Event Sourcing и отказоустойчивые API.

Теперь, когда мы освоили гарантии атомарности и согласованности в распределённых операциях, перейдём к технике, которая ускоряет системы и снижает нагрузку на БД, но вносит свои сложности: [[28. Кэширование. Cache Aside, Write Through, Write Back]].
