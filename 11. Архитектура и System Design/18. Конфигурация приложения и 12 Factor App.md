## Конфигурация как краеугольный камень эксплуатации

В предыдущих статьях мы спроектировали слои приложения, определили границы модулей и научились связывать их с помощью Dependency Injection. Однако ни одно приложение не работает в вакууме: ему нужны строка подключения к базе данных, порт для HTTP-сервера, ключи доступа к внешним API, таймауты и множество других параметров, которые различаются на dev-, staging- и production-окружениях. Управление этими параметрами — отдельная архитектурная дисциплина, от которой напрямую зависят надёжность, безопасность и переносимость системы.

В этой статье мы разберём, как правильно организовать конфигурацию Go-приложения в соответствии с принципами **The Twelve-Factor App**, какие инструменты предоставляет экосистема Go, и как конфигурация интегрируется в Clean Architecture и DI.

### 12-Factor App: Конфигурация как переменные окружения

Манифест 12-Factor App, написанный инженерами Heroku, стал золотым стандартом для создания облачных приложений. Третий фактор гласит: **«Конфигурация должна храниться в переменных окружения»**. Это означает категорический отказ от хранения конфигурации в коде, в файлах, «привязанных» к репозиторию, или в проприетарных форматах.

Основные принципы:

- **Строгое разделение кода и конфигурации.** Кодовая база неизменна для всех окружений. Всё, что меняется — вынесено в окружение.
- **Переменные окружения (environment variables) — универсальный интерфейс.** Они поддерживаются всеми ОС и платформами (контейнеры, оркестраторы), легко переопределяются без пересборки приложения.
- **Гранулярность.** Каждый параметр — отдельная переменная, а не один файл с настройками. Это упрощает управление и снижает риск утечки секретов (нельзя случайно закоммитить весь `.env` с продакшен-ключами, если он не хранится в репозитории).

Для Go-разработчика это означает, что приложение должно получать настройки преимущественно из `os.Getenv()` с разумными значениями по умолчанию для локальной разработки.

### Реализация в Go: от флагов до сложных структур

Go из коробки предоставляет несколько механизмов конфигурирования, которые можно комбинировать.

#### Флаги командной строки (пакет `flag`)

Удобны для локального запуска и отладки, но не являются основным способом для продакшена (трудно передавать в контейнерах без обёрток). Тем не менее, флаги полезны для указания пути к файлу конфигурации или переопределения отдельных параметров.

```go
var configPath = flag.String("config", "config.yaml", "path to config file")
flag.Parse()
```

#### Переменные окружения

Стандарт де-факто для облачных приложений. Ручной парсинг `os.Getenv` быстро становится громоздким, поэтому используют библиотеки.

#### Библиотеки для работы с конфигурацией

Наиболее популярны:

- **`spf13/viper`** — де-факто стандарт в Go-мире. Поддерживает множество источников (файлы YAML/JSON/TOML, переменные окружения, флаги, удалённые хранилища типа etcd/Consul), live watching (hot reload), значения по умолчанию.
- **`caarlos0/env`** — минималистичная библиотека для маппинга переменных окружения на структуру через теги. Быстрая, без зависимостей.
- **`kelseyhightower/envconfig`** — простой маппинг env → struct с поддержкой префиксов и обязательных полей.

Пример с `caarlos0/env` (часто используемый в связке с 12-factor):

```go
type Config struct {
    HTTPPort    int           `env:"HTTP_PORT" envDefault:"8080"`
    DatabaseURL string        `env:"DATABASE_URL,required"`
    LogLevel    string        `env:"LOG_LEVEL" envDefault:"info"`
    Timeout     time.Duration `env:"TIMEOUT" envDefault:"30s"`
}

func Load() (*Config, error) {
    var cfg Config
    if err := env.Parse(&cfg); err != nil {
        return nil, err
    }
    return &cfg, nil
}
```

**Viper** уместен, когда нужны сложные иерархические конфигурации или горячая перезагрузка без рестарта приложения (например, изменение уровня логирования или порогов Circuit Breaker). Однако для микросервисов часто достаточно простого env-маппинга, что ближе к идеям 12-factor.

> [!info] Под капотом
> При использовании переменных окружения важно помнить, что они доступны всему процессу и могут быть прочитаны дочерними процессами. Это одновременно и удобство (передаются в вызываемые внешние команды), и риск утечки (секреты могут попасть в логи при дампе окружения). Никогда не логируйте `os.Environ()` в продакшене.

### Интеграция конфигурации в архитектуру

Конфигурация должна быть загружена **в композиционном корне** (`main.go`) и затем передана в те компоненты, которые в ней нуждаются. В Clean Architecture конфигурация относится к внешнему слою, но её значения могут использоваться в адаптерах (например, параметры подключения к БД) или даже в ядре (например, бизнес-правило «скидка не более 50%», задаваемое через конфиг).

**Антипаттерн:** глобальная переменная `var AppConfig *Config`, доступная из любого пакета. Это нарушает инверсию зависимостей и делает тестирование кошмаром.

**Правильный подход:** внедрять только необходимые параметры в конкретные компоненты через конструкторы.

```go
// cmd/main.go
func main() {
    cfg := config.MustLoad()
    
    db := postgres.MustConnect(cfg.DatabaseURL)
    orderRepo := postgres.NewOrderRepository(db)
    
    // Передаём только нужные параметры
    paymentClient := stripe.NewClient(cfg.StripeKey, cfg.PaymentTimeout)
    
    orderService := order.NewService(orderRepo, paymentClient)
    // ...
}
```

Иногда удобно создать отдельную структуру `Dependencies`, которая агрегирует все готовые к использованию компоненты, но конфигурация всё равно должна быть распарсена и передана явно.

### Секреты и безопасность

Конфигурация часто содержит секреты: токены доступа, пароли БД, ключи шифрования. 12-factor рекомендует переменные окружения, но это не решает проблему хранения секретов в безопасности при развёртывании.

В production-среде секреты должны управляться специализированными системами:

- **Менеджеры секретов оркестраторов** — Kubernetes Secrets (с шифрованием etcd).
- **Внешние хранилища** — HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager.
- **Зашифрованные переменные в CI/CD** (GitLab CI Variables, GitHub Secrets).

Приложение Go в идеале вообще не знает, откуда секреты — оно просто читает переменную окружения `DATABASE_PASSWORD`, которая была заполнена безопасным способом. Никогда не коммитьте файлы `.env` с реальными секретами в репозиторий.

> [!warning] Ловушка / Gotcha
> В Go иногда используют флаг `-ldflags` для внедрения конфигурации во время сборки (например, `-X main.version=...`). Это подходит для версии, но **категорически не подходит для секретов или URL окружений**. Внедрённые значения запекаются в бинарник и становятся доступны любому, кто его скачает. Сборка должна быть отделена от конфигурации.

### Конфигурация и тестирование

DI облегчает подмену конфигурации в тестах. Вместо загрузки из env, тест напрямую создаёт структуру конфигурации и передаёт её компоненту.

```go
func TestService(t *testing.T) {
    cfg := &order.Config{
        MaxItemsPerOrder: 10,
    }
    svc := order.NewService(mockRepo, cfg)
    // ...
}
```

Для интеграционных тестов можно использовать библиотеки типа `testcontainers-go`, которые автоматизируют запуск реальной БД в контейнере и подставляют динамические строки подключения через переменные окружения или прямое конфигурирование.

### Структурирование конфигурации в коде

В крупных проектах конфигурация может насчитывать десятки параметров. Хорошая практика — разбить её на логические блоки, соответствующие доменам или инфраструктурным компонентам.

```go
type Config struct {
    HTTP     HTTPConfig
    Database DatabaseConfig
    Payment  PaymentConfig
    Log      LogConfig
    Metrics  MetricsConfig
}

type HTTPConfig struct {
    Port         int           `env:"HTTP_PORT" envDefault:"8080"`
    ReadTimeout  time.Duration `env:"HTTP_READ_TIMEOUT" envDefault:"10s"`
    WriteTimeout time.Duration `env:"HTTP_WRITE_TIMEOUT" envDefault:"30s"`
}

type DatabaseConfig struct {
    URL             string        `env:"DATABASE_URL,required"`
    MaxOpenConns    int           `env:"DB_MAX_OPEN_CONNS" envDefault:"25"`
    MaxIdleConns    int           `env:"DB_MAX_IDLE_CONNS" envDefault:"10"`
    ConnMaxLifetime time.Duration `env:"DB_CONN_MAX_LIFETIME" envDefault:"5m"`
}
```

Каждый адаптер при инициализации получает свою часть конфигурации. Это сохраняет принцип единственной ответственности и облегчает тестирование.

### Горячая перезагрузка конфигурации

В некоторых сценариях требуется изменять конфигурацию без перезапуска приложения (например, динамически менять уровень логирования или параметры rate limiting). В Go это можно реализовать с помощью:

- **Viper + WatchConfig()** — подписка на изменения файла.
- **Подписка на изменения в Consul/etcd**.
- **Атомарное обновление глобальной конфигурации** через периодический polling источника.

Однако это усложняет код (нужно делать конфигурацию потокобезопасной — `atomic.Value` или `sync.RWMutex`) и нарушает принцип неизменности конфигурации в рамках одного процесса (immutable infrastructure). Большинство 12-factor приложений просто перезапускаются с новой конфигурацией (graceful restart), что проще и надёжнее. Перезапуск нескольких инстансов оркестратор делает незаметным.

> [!tip] Собеседование
> **Вопрос:** Как бы вы организовали конфигурацию в Go-микросервисе, который должен работать в разных окружениях (dev, staging, prod) с разными параметрами подключения к БД и сторонним сервисам? Какие инструменты и паттерны выберете?
> **Ожидаемый ответ:** Я бы следовал 12-factor, храня конфигурацию в переменных окружения. Для парсинга использовал бы `caarlos0/env` или `envconfig` — это минималистично и без внешних зависимостей. Структура `Config` была бы определена во внешнем слое, а через конструкторы в `main.go` я передавал бы нужные параметры компонентам. Секреты получал бы из Kubernetes Secrets или Vault. Для локальной разработки использовал бы `.env` файл и утилиту `godotenv` (или просто `export`). Флаги командной строки оставил бы только для указания пути к файлу конфигурации или переопределения порта для удобства отладки. Горячую перезагрузку рассматривал бы только при явной необходимости (например, огромное количество инстансов, которые дорого перезапускать).

### Mechanical Sympathy: влияет ли способ загрузки конфигурации на производительность?

Чтение переменных окружения — системный вызов, но выполняется один раз при старте приложения, поэтому его стоимость пренебрежимо мала. Использование библиотек с рефлексией (почти все парсеры env) добавляет время инициализации (порядка десятков-сотен микросекунд), что также не критично. Важно не парсить конфигурацию на каждый HTTP-запрос — результат парсинга должен быть сохранён в структуре и переиспользован.

### Файлы конфигурации: разрешены ли?

12-factor не запрещает файлы конфигурации категорически, но настаивает, что они не должны быть частью кодовой базы и не должны различаться для разных окружений. Файл может использоваться как **источник значений по умолчанию** или как **промежуточное звено**, например, в Kubernetes ConfigMap монтируется как файл, который затем читается приложением. Однако сам формат — YAML, JSON, TOML — вторичен по отношению к конечному интерфейсу (переменным окружения, которые могут переопределять значения из файла). Viper как раз реализует такую каскадную модель: defaults < file < env < flag.

### Итог

Конфигурация — не просто техническая деталь, а архитектурное решение, влияющее на переносимость, безопасность и эксплуатацию системы. Следование принципу «конфигурация из окружения» в духе 12-Factor App и идиоматичная реализация в Go с помощью структур, тегов и явного внедрения параметров обеспечивают чистый композиционный корень и лёгкое тестирование. Выбор конкретной библиотеки (`env`, `envconfig`, `viper`) зависит от сложности требований, но неизменным остаётся главное: код отделён от настроек, секреты не попадают в репозиторий, а окружение диктует поведение приложения.

Теперь, когда приложение умеет настраиваться под любое окружение, мы переходим к способам его взаимодействия с другими сервисами. В следующей статье мы сравним два фундаментальных подхода к коммуникации: [[19. Синхронное vs асинхронное взаимодействие сервисов]].
