
Timeout, Retry и Circuit Breaker защищают систему от зависших и постоянно отказывающих downstream-сервисов. Но есть ещё одна угроза: медленный или проблемный компонент может **исчерпать общие ресурсы** и парализовать весь сервис, даже если остальные части работают идеально. Представьте, что на одном HTTP-сервере обрабатываются и быстрые запросы к профилю, и тяжёлые запросы к отчётам. Если генерация отчётов потребляет все горутины и соединения с БД, запросы к профилям тоже встанут. **Bulkhead** — паттерн, который изолирует ресурсы, предотвращая распространение отказов.

### Что такое Bulkhead

Название происходит от корабельных переборок (bulkheads), которые делят судно на водонепроницаемые отсеки. Пробоина в одном отсеке не топит весь корабль. В архитектуре ПО **Bulkhead** — это техника резервирования или изоляции ресурсов (пулов соединений, горутин, памяти) для разных операций или компонентов, чтобы отказ или деградация одного не затронули остальные.

Существует два основных подхода к реализации:

- **Изоляция по ресурсам:** выделение отдельных пулов горутин, соединений к БД, брокерам для разных типов операций.
- **Изоляция по времени:** ограничение максимального времени ожидания ресурса; если ресурс не получен — отказ.

### Реализация Bulkhead в Go

Go с его горутинами и каналами предоставляет естественные и легковесные инструменты для Bulkhead.

#### Семафор как ограничитель конкурентности

Самый простой способ — ограничить количество одновременных горутин, обрабатывающих определённый тип запросов. В Go семафор реализуется через буферизированный канал.

```go
type Service struct {
    // семафор для тяжёлых запросов, максимум 10 одновременно
    heavySem chan struct{}
}

func NewService() *Service {
    return &Service{
        heavySem: make(chan struct{}, 10),
    }
}

func (s *Service) HandleHeavyRequest(ctx context.Context, w http.ResponseWriter, r *http.Request) {
    select {
    case s.heavySem <- struct{}{}:
        defer func() { <-s.heavySem }()
        // выполнение тяжёлого запроса
    case <-ctx.Done():
        http.Error(w, "too many heavy requests", http.StatusServiceUnavailable)
        return
    }
    // ...
}
```

Если все 10 слотов заняты, новый запрос немедленно получает `503 Service Unavailable`, не блокируя горутину. Быстрые запросы продолжают обрабатываться в других горутинах без задержек.

> [!info] Под капотом
> Буферизированный канал в Go — это легковесная структура в рантайме (`hchan`). Отправка в канал без блокировки (не заполнен) — это атомарная операция с небольшим числом тактов. Получение — аналогично. Семафор на канале не создаёт дополнительных аллокаций и практически не нагружает GC.

#### Изолированные пулы соединений

Другой аспект Bulkhead — разделение пулов соединений к внешним сервисам. Если сервис А ходит в БД для аналитики и для оперативных запросов, не стоит использовать один пул соединений. Тяжёлые аналитические запросы могут занять все соединения, и оперативные встанут.

```go
type Repositories struct {
    AnalyticsDB *sql.DB // отдельный пул
    MainDB      *sql.DB // отдельный пул
}

// Настройка отдельных пулов
analyticsPool, _ := pgxpool.Connect(ctx, "postgres://.../analytics?pool_max_conns=5")
mainPool, _ := pgxpool.Connect(ctx, "postgres://.../main?pool_max_conns=20")
```

Go-библиотеки вроде `pgxpool` и `go-redis` позволяют создавать несколько независимых пулов с разными лимитами подключений.

#### Worker Pool с разделением по типам задач

Более сложный вариант — выделенные пулы воркеров для разных операций. В Go это могут быть горутины, читающие из разных каналов.

```go
func startWorkerPool(ctx context.Context, tasks <-chan Task, concurrency int, handler func(Task) error) {
    for i := 0; i < concurrency; i++ {
        go func() {
            for {
                select {
                case <-ctx.Done():
                    return
                case task, ok := <-tasks:
                    if !ok {
                        return
                    }
                    handler(task)
                }
            }
        }()
    }
}
```

Для различных API-эндпоинтов или топиков Kafka можно создать отдельные пулы с разным числом воркеров. Если пул для топика `events` перегружен, сообщения накапливаются в канале, создавая backpressure, но не влияя на обработку `orders`.

### Mechanical Sympathy: влияние на рантайм Go

**Память и горутины.** Главная опасность, от которой защищает Bulkhead, — неконтролируемый рост числа горутин. Без семафора на пике трафика может создаться по горутине на каждый входящий запрос. При медленном downstream они все заблокируются на I/O, потребляя память (стек ~2 КБ) и структуры планировщика. Семафор ограничивает максимальное количество таких горутин, удерживая потребление памяти в предсказуемых рамках.

**Планировщик.** Огромное количество горутин увеличивает время сканирования планировщиком очередей. Уменьшение числа конкурентных операций через Bulkhead снижает эту нагрузку, делая latency более предсказуемым.

**Системные вызовы и дескрипторы.** Изоляция пулов соединений предотвращает исчерпание лимита файловых дескрипторов одним компонентом. Операционная система накладывает лимиты на количество открытых сокетов, и Bulkhead распределяет их между потребителями.

### Связь с другими паттернами устойчивости

- **Circuit Breaker** ([[36. Circuit Breaker, Retry, Timeout и Backoff]]) и Bulkhead дополняют друг друга. Circuit Breaker быстро обнаруживает отказ downstream и размыкает цепь, но если цепь замкнута и запросы идут, именно Bulkhead ограничивает количество одновременных вызовов, не давая им заполонить горутины.
- **Rate Limiting** ([[38. Rate Limiting и защита системы]]) ограничивает частоту запросов от внешних клиентов, Bulkhead — внутренние ресурсы.
- **Backpressure** ([[43. Backpressure и контроль нагрузки]]) — Bulkhead создаёт backpressure, отбрасывая запросы при заполнении семафора или очереди, заставляя вызывающую сторону замедлиться или повторить позже.

### Bulkhead на уровне архитектуры

Bulkhead применим не только на уровне кода, но и на уровне инфраструктуры:
- Разные поды/контейнеры для разных типов задач (аналитика, real-time API).
- Разные кластеры БД или реплики для чтения и записи.
- Отдельные балансировщики для разных групп сервисов.

В Go-микросервисах это соответствует выделению отдельных инстансов с разными конфигурациями, управляемых через Kubernetes и Service Discovery ([[34. Service Discovery. Client side и Server side]]).

### Антипаттерны

- **Слишком жёсткая изоляция:** создание отдельного пула на каждый эндпоинт ведёт к фрагментации ресурсов и простою.
- **Отсутствие лимитов:** один глобальный пул соединений и неограниченные горутины — классический способ дождаться каскадного отказа.
- **Игнорирование backpressure:** если Bulkhead отбрасывает запросы, вызывающая сторона должна корректно обрабатывать `503` или `429` (повтор с backoff, fallback).

> [!tip] Собеседование
> **Вопрос:** Как Bulkhead помогает предотвратить каскадные отказы? Приведите пример реализации в Go.
> **Ответ:** Bulkhead изолирует ресурсы для разных операций, чтобы перегрузка или отказ одного компонента не исчерпали общие ресурсы и не заблокировали остальные. Например, в Go я могу использовать буферизированный канал как семафор для ограничения числа одновременных вызовов к внешнему сервису отчётов. Если все слоты заняты, новый запрос немедленно получает `503`, а горутины для быстрых запросов профилей продолжают работать нормально. Также я разделю пулы соединений к БД для аналитики и для оперативных запросов.

### Итог

Bulkhead — это простая, но мощная концепция изоляции ресурсов, которая превращает потенциально катастрофический отказ в управляемую деградацию. В Go семафоры на каналах, раздельные пулы соединений и worker pool'ы реализуют Bulkhead без внешних библиотек, с минимальным оверхедом. Вместе с Timeout, Retry и Circuit Breaker он образует полный набор инструментов для построения по-настоящему устойчивых сервисов.
