## Почему модульных тестов недостаточно

Модульные тесты (unit tests) проверяют корректность отдельной функции или структуры в изоляции. Они быстры, дёшевы и составляют основу пирамиды тестирования. Но они никак не проверяют, что:
- Микросервисы соблюдают контракты API и не ломают клиентов при изменении схемы.
- Интеграция с реальной базой данных или брокером сообщений работает в условиях, приближенных к продакшену.
- Архитектурные границы (порты, адаптеры, слои) действительно соблюдаются в рантайме.

**Контрактное тестирование** и **интеграционное тестирование** закрывают эти пробелы. Для Senior/Lead Go-разработчика это неотъемлемая часть обеспечения надёжности распределённой архитектуры, рассмотренной в предыдущих статьях ([[9. Микросервисы. Преимущества, недостатки и мифы]], [[46. Версионирование сервисов и API]], [[47. Миграции архитектуры без downtime]]).

### Контрактное тестирование

**Контракт** — это явное соглашение между поставщиком (продюсером API) и потребителем (консьюмером) о формате и семантике взаимодействия. Контрактное тестирование проверяет, что обе стороны соблюдают это соглашение, независимо друг от друга.

В отличие от End-to-End (E2E) тестов, контрактные тесты не требуют поднятия всей системы. Поставщик проверяет, что его ответы соответствуют контракту. Потребитель проверяет, что его запросы и обработка ответов соответствуют тому же контракту. Контракт выступает как «истина посередине».

#### Pact: стандарт контрактного тестирования

**Pact** — самый распространённый фреймворк контрактного тестирования. Его модель основана на двух ролях:

- **Consumer** (клиент, например, фронтенд или другой микросервис) определяет ожидания («я ожидаю, что GET /users/42 вернёт JSON с полями id и name»). Эти ожидания записываются в **pact-файл**.
- **Provider** (сервер) загружает pact-файл и проверяет, что его реальные ответы соответствуют ожиданиям.

```mermaid
flowchart LR
    Consumer["Клиент<br>Определяет ожидания"]
    Pact["Pact-файл<br>Контракт"]
    Provider["Поставщик<br>Верифицирует ожидания"]

    Consumer -->|"создаёт"| Pact
    Pact -->|"верифицирует"| Provider
```

#### Контрактное тестирование в Go с Pact

Библиотека `pact-foundation/pact-go` реализует Consumer-Driven Contract Testing.

**Шаг 1: Тест потребителя (Consumer)**

```go
func TestConsumer_GetUser(t *testing.T) {
    // Создаём Pact мок-сервер на случайном порту
    pact := dsl.Pact{
        Consumer: "UserMobileApp",
        Provider: "UserService",
    }
    defer pact.Teardown()

    // Описываем ожидаемое взаимодействие
    pact.AddInteraction().
        Given("User 42 exists").
        UponReceiving("A request for user 42").
        WithRequest(dsl.Request{
            Method: "GET",
            Path:   dsl.String("/users/42"),
        }).
        WillRespondWith(dsl.Response{
            Status: 200,
            Body: map[string]interface{}{
                "id":   "42",
                "name": "Alice",
            },
            Headers: dsl.MapMatcher{
                "Content-Type": dsl.String("application/json; charset=utf-8"),
            },
        })

    // Запускаем тест клиента, который обращается к Pact-мок серверу
    err := pact.Verify(func() error {
        client := NewUserClient(fmt.Sprintf("http://localhost:%d", pact.Server.Port))
        user, err := client.GetUser(context.Background(), "42")
        if err != nil {
            return err
        }
        // Проверяем, что клиент корректно обработал ответ
        if user.ID != "42" || user.Name != "Alice" {
            return fmt.Errorf("unexpected user")
        }
        return nil
    })
    require.NoError(t, err)
}
```

Этот тест генерирует pact-файл, который фиксируется и передаётся провайдеру (например, через Pact Broker).

**Шаг 2: Тест поставщика (Provider)**

```go
func TestProvider_UserService(t *testing.T) {
    pact := &dsl.Pact{
        Provider: "UserService",
    }

    // Загружаем pact-файл от потребителя
    _, err := pact.VerifyProvider(t, dsl.VerifyRequest{
        ProviderBaseURL: "http://localhost:8080",
        PactURLs:        []string{"./pacts/UserMobileApp-UserService.json"},
        // State handlers: подготавливаем данные в БД
        StateHandlers: dsl.StateHandlers{
            "User 42 exists": func() error {
                return seedUser("42", "Alice")
            },
        },
    })
    require.NoError(t, err)
}
```

Пакет `VerifyProvider` поднимает реальный HTTP-сервер приложения (или его часть) и бьёт по нему запросами, описанными в pact-файле, сверяя ответы.

> [!info] Под капотом
> Pact-тест поставщика запускает настоящий Go-сервер (обычно в том же процессе, через `httptest.NewServer`). Это значит, что в тесте поднимаются все middleware, сериализация, доступ к БД (через заглушки или testcontainers). Такой тест проверяет полный путь запроса от адаптера до БД, но только для конкретного контракта, а не для всех возможных сценариев.

#### Контракты и версионирование

Контрактное тестирование идеально дополняет стратегию версионирования API ([[46. Версионирование сервисов и API]]). Если провайдер меняет MAJOR-версию, он может поддерживать два набора контрактов (для v1 и v2). Pact Broker хранит версионированные контракты и показывает, какие потребители используют какую версию API, что критично при объявлении deprecation.

### Интеграционное тестирование

Интеграционные тесты проверяют взаимодействие между несколькими реальными компонентами, включая базы данных, брокеры сообщений и внешние сервисы. В отличие от контрактных, они не требуют формального контракта, а проверяют реальное поведение системы в связке.

#### Testcontainers: реальные зависимости в Go

Библиотека `testcontainers-go` позволяет поднимать в Docker-контейнерах PostgreSQL, Redis, Kafka и другие сервисы прямо из Go-тестов. Это даёт изолированную, воспроизводимую среду для каждого теста или набора тестов.

```go
func TestOrderRepository_Integration(t *testing.T) {
    ctx := context.Background()

    // Запускаем PostgreSQL в контейнере
    pgContainer, err := testcontainers.PostgresContainer(ctx, testcontainers.PostgresContainerRequest{
        Database: "testdb",
        Username: "test",
        Password: "test",
    })
    require.NoError(t, err)
    defer pgContainer.Terminate(ctx)

    dsn, _ := pgContainer.ConnectionString(ctx)
    db, err := sql.Open("postgres", dsn)
    require.NoError(t, err)
    defer db.Close()

    // Накатываем миграции
    migrateDB(db)

    // Выполняем интеграционный тест
    repo := NewOrderRepository(db)
    order := &Order{ID: "1", Status: "new"}
    err = repo.Save(ctx, order)
    require.NoError(t, err)

    loaded, err := repo.GetByID(ctx, "1")
    require.NoError(t, err)
    require.Equal(t, "new", loaded.Status)
}
```

> [!warning] Ловушка / Gotcha
> Интеграционные тесты с testcontainers **медленные**. Каждый запуск контейнера занимает 1-5 секунд. Запускать сотни таких тестов на каждый коммит — значит ждать CI часами. Разделяйте тесты по уровням: модульные — быстро, интеграционные — выборочно на PR или в отдельной стадии пайплайна.

#### Интеграционные тесты для брокеров сообщений

Для проверки взаимодействия с Kafka можно использовать `testcontainers-go/kafka` или встроенные in-memory решения (вроде `sarama/mocks` или `franz-go` с мок-брокером). Контрактное тестирование для асинхронных событий сложнее, так как Pact-подход исторически ориентирован на HTTP. Здесь чаще используют схемы Avro/Protobuf и Schema Registry, а также кастомные тесты на сериализацию/десериализацию.

```go
func TestOrderCreatedEvent(t *testing.T) {
    // Проверяем, что структура сериализуется и десериализуется без ошибок
    event := &OrderCreated{OrderID: "123"}
    data, err := proto.Marshal(event)
    require.NoError(t, err)

    var decoded OrderCreated
    err = proto.Unmarshal(data, &decoded)
    require.NoError(t, err)
    require.Equal(t, event.OrderID, decoded.OrderID)
}
```

### Mechanical Sympathy: цена тестов в CI/CD

Архитектурные тесты не бесплатны, и Senior-инженер должен уметь балансировать покрытие и время пайплайна.

**Контрактные тесты:**
- Тест потребителя (Consumer) быстр: поднимается Pact-мок сервер, генерируется JSON. Зависимости не нужны.
- Тест поставщика (Provider) медленнее: нужно поднять сервер (возможно, с БД). Но он запускается только при изменении контракта или кода провайдера.

**Интеграционные тесты:**
- Каждый контейнер — это скачивание образа (если отсутствует), запуск, ожидание готовности, выполнение тестов, остановка. Суммарно 5-15 секунд на тест-файл.
- Для ускорения можно переиспользовать контейнеры между тестами через `testcontainers.Reuse()` или вынести общий пул контейнеров на уровень CI-джобы.

**Стратегия CI:**
1. Быстрый этап: линтинг, юнит-тесты (секунды).
2. Средний этап: контрактные тесты, интеграционные тесты без тяжёлых зависимостей (минуты).
3. Медленный этап: интеграционные тесты с реальными БД/очередями (до 15-20 минут). Запускается при мёрдже в основную ветку, а не на каждый пуш.

### Тестирование архитектурных границ

Контрактные и интеграционные тесты напрямую связаны с архитектурными паттернами:

- **Hexagonal Architecture** ([[13. Hexagonal Architecture. Ports and Adapters]]): тесты поставщика проверяют, что адаптер правильно реализует порт. Тесты потребителя — что вызов порта корректен.
- **Clean Architecture** ([[14. Clean Architecture и Dependency Rule]]): контрактное тестирование позволяет протестировать use case через реальные адаптеры, не нарушая границ.
- **Event-Driven Architecture** ([[21. Event Driven Architecture]]): схема событий (Protobuf, Avro) — это контракт между сервисами. Контрактный тест проверяет, что событие валидно и может быть обработано подписчиком.
- **CQRS** ([[23. CQRS. Разделение чтения и записи]]): команды и запросы — это тоже контракты, которые можно верифицировать раздельно.

> [!tip] Собеседование
> **Вопрос:** В чём разница между интеграционным и контрактным тестированием? Когда вы выберете одно, а когда другое?
> **Ответ:** Интеграционное тестирование проверяет, что реальные компоненты работают вместе (например, сервис + PostgreSQL). Контрактное — что API поставщика удовлетворяет конкретным ожиданиям потребителя, зафиксированным в контракте. Я выберу контрактное, когда у API есть несколько потребителей, и я хочу гарантировать, что изменение не сломает их, без развёртывания всей системы. Интеграционное — для проверки взаимодействия с инфраструктурой, которую сложно или нельзя замокать (например, транзакционное поведение БД, гарантии доставки очереди).

### Антипаттерны

- **E2E тесты как замена контрактным.** E2E тесты дороги, нестабильны и не указывают, кто именно нарушил контракт. Контрактные тесты изолируют ответственность.
- **Использование продакшен-данных в тестах.** Интеграционные тесты должны работать с синтетическими, изолированными данными.
- **Игнорирование контрактного тестирования для внутренних API.** «Внутри микро» — не значит «никогда не изменится». Внутренние API тоже должны быть версионированы и законтрактованы.

### Итог

Тестирование архитектуры выходит далеко за рамки юнит-тестов. Контрактное тестирование гарантирует совместимость сервисов и предотвращает поломки при эволюции API. Интеграционное тестирование проверяет работоспособность системы с реальными зависимостями. Вместе они создают защитную сеть, которая позволяет итеративно менять архитектуру без страха сломать продакшен — что является конечной целью всех практик, рассмотренных в этом разделе.
