
Невозможно построить распределённую систему, которая одновременно даёт низкую задержку, высокую доступность и строгую согласованность данных при наличии сетевых разделений. Это диктует CAP-теорема, которую мы разберём в следующей статье. Здесь мы сфокусируемся на самих моделях согласованности: **Strong**, **Eventual** и **Causal**. Понимание их механик и компромиссов напрямую влияет на выбор архитектурных паттернов — от кэширования до Saga и CQRS.

### Что такое консистентность в распределённых системах

**Консистентность** определяет правила, по которым данные, записанные в систему, становятся видимыми для последующих операций чтения. В локальной базе данных это обычно ACID: после коммита транзакции все последующие запросы видят изменения. В распределённой системе данные могут находиться на разных узлах, а сеть — задерживать или терять сообщения. Модель консистентности описывает, насколько «свежими» данными готов пожертвовать разработчик ради доступности и производительности.

Различают три основные модели: **сильная (Strong)**, **согласованность в конечном счёте (Eventual)** и **причинная (Causal)**.

### Strong Consistency (Сильная согласованность)

**Определение:** После того как запись подтверждена, все последующие операции чтения (с любого узла) возвращают записанное значение или более новое. Система ведёт себя так, будто существует только одна копия данных.

Самый строгий подвид — **линеаризуемость**. Для неё выполняется требование: каждая операция чтения и записи происходит мгновенно, в определённый момент времени между её началом и концом, и все операции упорядочены глобально.

**Как достигается:**
- **Синхронная репликация:** транзакция считается завершённой, только когда большинство реплик подтвердили её. В Go это может быть вызов `etcd` с ожиданием ответа от кворума.
- **Алгоритмы консенсуса (Raft, Paxos):** реплицированный лог гарантирует, что лидер обслуживает все записи, а читатели видят одинаковый порядок. Библиотека `hashicorp/raft` на Go позволяет встроить Raft-кластер в приложение.
- **Распределённые блокировки и лидерство:** перед записью захватывается блокировка, обеспечивающая глобальный порядок.

**Влияние на архитектуру:**
- **Latency:** каждая запись ждёт подтверждения от реплик или лидера. Добавляются сетевые задержки (RTT).
- **Availability:** при сетевом разделении система может отказать в записи (жертвует доступностью ради консистентности).
- **Производительность Go-сервиса:** горутины, выполняющие запись, дольше заблокированы на ожидании ответа, что увеличивает потребление памяти и снижает throughput при высоком конкурентном доступе.

**Пример на Go (etcd-подобный вызов):**
```go
// Получение строго консистентного значения из etcd с использованием sequential consistency
ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()
resp, err := client.Get(ctx, key, clientv3.WithSerializable())
if err != nil {
    return fmt.Errorf("failed to get: %w", err)
}
// Значение гарантированно актуально для кворума
```

### Eventual Consistency (Согласованность в конечном счёте)

**Определение:** Если в систему не поступают новые обновления, то в конечном счёте все реплики вернут последнее записанное значение. Между записью и моментом, когда все реплики её увидят, существует окно несогласованности (inconsistency window). Это модель **BASE** (Basically Available, Soft state, Eventually consistent) в противовес ACID.

**Примеры из реального мира:**
- **DNS:** изменение записи может распространяться часы.
- **Amazon DynamoDB, Cassandra:** по умолчанию обеспечивают eventual consistency с настраиваемыми уровнями консистентности (ONE, QUORUM, ALL).
- **Кэши** (Cache-Aside): данные в кэше обновляются по TTL или инвалидации.

**Влияние на архитектуру Go-сервиса:**
- **Допускается расхождение:** чтение может вернуть устаревшие данные. Бизнес-логика должна быть готова к этому.
- **Снижение задержки:** запись может подтверждаться после фиксации на одном узле, не дожидаясь репликации.
- **Конфликты:** одновременные записи на разных узлах могут создать рассинхронизацию. Требуется разрешение конфликтов (LWW, CRDT, ручное слияние).

```mermaid
flowchart LR
    Client["Клиент"]
    Node1["Реплика A<br>значение = X"]
    Node2["Реплика B<br>значение = Y"]

    Client -->|"Запись X"| Node1
    Node1 -.->|"асинхронная репликация"| Node2
    Note[Окно несогласованности:<br> читатель B может видеть Y] 
    Node2 --- Note
```

**Пример кода Go с рассинхронизацией:**
```go
// Пишем в основной узел, не ждём репликации
func (s *Service) Write(ctx context.Context, key string, val []byte) error {
    _, err := s.master.Set(ctx, key, val, 0).Result()
    return err
}

// Читаем с любого реплика
func (s *Service) Read(ctx context.Context, key string) ([]byte, error) {
    // reader может вернуть устаревшее значение, если репликация не завершена
    return s.replica.Get(ctx, key).Bytes()
}
```

### Causal Consistency (Причинная согласованность)

**Определение:** Операции, связанные причинно-следственной связью, видны всем узлам в одном и том же порядке. Операции, не связанные причинно (конкурентные), могут быть видны в разном порядке. Это модель сильнее eventual, но слабее strong.

**Причинно-следственная связь** возникает, когда:
- Процесс или пользователь видит результат предыдущей операции (например, прочитал комментарий и удалил его).
- Запись основана на прочитанных данных (read-modify-write).
- Операции следуют в рамках одного потока выполнения.

Пример: пользователь загружает фотографию (запись A), затем добавляет к ней описание (запись B). Для любого наблюдателя описание не должно появиться раньше фотографии. Но лайк от другого пользователя на другой фотографии (запись C) может прийти в любом порядке.

**Реализация:**
- **Векторные часы (Vector Clocks):** с каждым сообщением передаётся счётчик операций по каждому узлу. При получении можно определить причинный порядок и обнаружить конкурентные обновления.
- **Гибридные логические часы (HLC):** комбинация физических и логических часов.

**Влияние на Go-сервисы:**
- Требуется передача метаданных (векторных часов) в каждом запросе. Можно встроить их в `context.Context` или в заголовки gRPC.
- Разрешение конфликтов: если обновления конкурентны, система может автоматически слить их (CRDT) или передать на разрешение выше (приложению).
- Связь с Event Sourcing: события содержат причинные идентификаторы, позволяя восстановить точный порядок зависимых операций.

**Пример Go с передачей причинных метаданных:**
```go
type ContextWithClock struct {
    Clock VectorClock
}

// В middleware распаковываем векторные часы из заголовка
func CausalityMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        clock := ParseVectorClock(r.Header.Get("X-Causality-Clock"))
        ctx := context.WithValue(r.Context(), clockKey, clock)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

// В бизнес-логике инкрементируем счётчик и встраиваем в ответ
func (s *Service) Update(ctx context.Context, key string, value []byte) error {
    clock := FromContext(ctx)
    clock.Increment(s.nodeID)
    // сохраняем данные и метаданные
    return s.db.WriteWithClock(ctx, key, value, clock)
}
```

### Связь с другими архитектурными концепциями

- **CAP теорема** ([[30. CAP теорема и реальные компромиссы]]): strong consistency выбирает CP (консистентность + partition tolerance), eventual — AP, causal — промежуточный вариант.
- **PACELC теорема:** расширение CAP, утверждает, что даже при отсутствии разделения есть компромисс между latency и consistency. Strong даёт высокую задержку, eventual — низкую.
- **Кэширование** ([[28. Кэширование. Cache Aside, Write Through, Write Back]]): Cache-Aside — eventual по умолчанию; Write-Through — приближается к strong.
- **Saga** ([[26. Saga Pattern. Оркестрация и хореография]]): использует eventual consistency, компенсации вместо отката.
- **CQRS** ([[23. CQRS. Разделение чтения и записи]]): Read Model — eventual относительно Write Model.

### Mechanical Sympathy: влияние на рантайм Go

**Strong Consistency:**
- Каждая операция записи ждёт ответа от нескольких узлов; горутина заблокирована на I/O. Планировщик Go открепляет её от потока ОС, но память под стек и дескрипторы остаются заняты. При сотнях тысяч одновременных запросов потребление растёт.
- Сетевые вызовы (`net.Dial`, `syscall.Write`) требуют переключения в режим ядра.
- Полезно использовать пул горутин с ограниченным параллелизмом, чтобы не перегрузить сеть.

**Eventual Consistency:**
- Операции быстрые, горутины быстрее освобождаются.
- Возможно увеличение числа конфликтов, которые приходится разрешать дополнительным кодом (CPU overhead).
- Кэширование в памяти (локальное) снижает задержку, но требует инвалидации.

**Causal Consistency:**
- Дополнительные метаданные (векторные часы) увеличивают размер сообщений → больше аллокаций при сериализации.
- Проверка причинности может быть ресурсоёмкой при большом числе узлов (сравнение векторов длиной N).

> [!tip] Собеседование
> **Вопрос:** Вам нужно спроектировать систему для интернет-магазина. Каталог товаров должен быть строго консистентен? А статус заказа?
> **Ответ:** Каталог товаров может быть eventual consistent: задержка в несколько секунд при обновлении цены приемлема, зато мы получаем быструю загрузку страниц через кэш. Статус заказа должен быть причинно-согласованным: если склад подтвердил резервирование, то оплата должна увидеть это обновление до списания денег, но строгая глобальная линеаризуемость не нужна — достаточно причинной связи между шагами.

### Итог

Выбор модели согласованности — один из важнейших архитектурных компромиссов. **Strong** даёт простую ментальную модель, но ограничивает масштабируемость и доступность. **Eventual** максимально производительна, но перекладывает ответственность за конфликты на разработчика. **Causal** — золотая середина для многих бизнес-сценариев, где важен порядок причинно-связанных событий.

Go с его горутинами и явной работой с сетью позволяет точно реализовать любую из моделей, но заставляет платить ресурсами (память, CPU) за более сильные гарантии. В следующей статье мы формализуем этот компромисс: [[30. CAP теорема и реальные компромиссы]].
