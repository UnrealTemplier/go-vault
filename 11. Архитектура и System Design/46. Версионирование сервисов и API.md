## Почему версионирование — это архитектурная необходимость

Микросервисы дают независимость команд и скорость разработки ([[9. Микросервисы. Преимущества, недостатки и мифы]]), но за это приходится платить: каждый сервис становится публичным API для других сервисов. Изменение схемы данных, сигнатуры метода или семантики эндпоинта может мгновенно сломать десятки потребителей. Без дисциплинированного версионирования система превращается в «распределённый монолит», где любое изменение требует синхронного деплоя всех участников.

**Версионирование сервисов и API** — это архитектурная практика, позволяющая сервисам эволюционировать и ломать обратную совместимость контролируемо, не обрушивая работу зависимых систем.

### Что именно мы версионируем

В распределённой системе под «версией» понимают три разных сущности, которые часто путают:

1. **API-версия** — версия контракта взаимодействия (схема запросов/ответов, эндпоинты, протокол). Именно её видят клиенты.
2. **Версия сервиса** — версия кода и развёрнутого артефакта (`v1.2.3`). Один сервис может одновременно поддерживать несколько API-версий.
3. **Версия данных** — схема данных в БД, которая может меняться независимо от API. Рассматривается в статье о миграциях ([[47. Миграции архитектуры без downtime]]).

В этой статье мы фокусируемся на первых двух.

### Стратегии версионирования API

Существует несколько способов указать версию API, и у каждого есть свои плюсы, минусы и уместность в Go-экосистеме.

#### 1. Версионирование через URL

Самый распространённый и простой способ: `/api/v1/orders`, `/api/v2/orders`.

**Плюсы:**
- Наглядно и легко отлаживать (curl, браузер).
- Хорошо поддерживается API Gateway и балансировщиками.

**Минусы:**
- Нарушает принцип REST: URL представляет ресурс, а не его версию.
- Раздувает количество путей, усложняет маршрутизацию.

В Go это реализуется через префиксный роутинг:

```go
mux := http.NewServeMux()
mux.Handle("/api/v1/orders", v1.OrderHandler{})
mux.Handle("/api/v2/orders", v2.OrderHandler{})
```

#### 2. Версионирование через заголовок

Клиент отправляет заголовок `Accept-Version: v2` или кастомный `X-API-Version: 2024-01-01`. Сервер маршрутизирует на основе заголовка.

**Плюсы:**
- URL ресурса остаётся чистым.
- Гибкость: можно менять версию без смены URL.

**Минусы:**
- Сложнее отлаживать (нужно устанавливать заголовки в curl/Postman).
- Кэширование на уровне HTTP-прокси требует учёта заголовка `Vary`.

В Go версию извлекают в middleware:

```go
func VersionMiddleware(next map[string]http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        version := r.Header.Get("Accept-Version")
        if version == "" {
            version = "v1"
        }
        handler, ok := next[version]
        if !ok {
            http.Error(w, "unsupported version", http.StatusBadRequest)
            return
        }
        handler.ServeHTTP(w, r)
    })
}
```

#### 3. Content-Type версионирование

Клиент указывает версию в заголовке `Content-Type`: `application/vnd.myapp.order.v2+json`. Сервер парсит vendor media type.

**Плюсы:**
- Наиболее REST-идиоматичный подход (использование возможностей HTTP).
- Позволяет одновременно версионировать формат запроса и ответа.

**Минусы:**
- Сложность: многие клиенты и инструменты не поддерживают кастомные media types.
- Требует строгой дисциплины на стороне клиента.

В Go такой подход требует кастомного анмаршалинга и обработки заголовка `Content-Type`.

### SemVer для API

Стандарт **Semantic Versioning** (SemVer) предписывает формат `MAJOR.MINOR.PATCH` и чёткие правила:

- **PATCH** — обратно совместимые исправления (багфикс в логике).
- **MINOR** — добавление новой обратно совместимой функциональности (новый необязательный параметр, новый эндпоинт).
- **MAJOR** — нарушение обратной совместимости (удаление поля, изменение типа, новое обязательное поле).

В контексте API это означает: изменение `MAJOR` — это новая версия API (`v1` → `v2`). `MINOR` и `PATCH` не требуют новой версии.

> [!info] Под капотом
> В gRPC/Protobuf обратная совместимость частично обеспечивается правилами Protobuf: добавление полей с новыми номерами обратно совместимо, удаление — требует резервирования номеров. Но SemVer для API — это больше, чем синтаксис: изменение семантики (например, `CreateOrder` раньше списывал деньги, а теперь только резервирует) — это тоже MAJOR, хотя Protobuf схема не изменилась.

### Обратная совместимость: правила

Чтобы не плодить MAJOR-версии каждый месяц, проектируйте изменения так, чтобы они были обратно совместимы (backward-compatible):

- **Не удаляйте поля** — помечайте как `@Deprecated`, но продолжайте возвращать (возможно, с нулевыми значениями).
- **Новые поля делайте опциональными** — клиент, не знающий о поле, должен корректно работать без него.
- **Не меняйте тип существующих полей** — `string` → `int` ломает парсинг.
- **Не меняйте семантику существующих полей** — `price` в копейках → `price` в рублях.
- **Добавление новых эндпоинтов обратно совместимо** — старые продолжают работать.

В Go обратная совместимость структур достигается через тэги `json:",omitempty"` и осторожное добавление полей:

```go
type OrderV1 struct {
    ID    string `json:"id"`
    Total int    `json:"total"`
}

type OrderV2 struct {
    ID       string  `json:"id"`
    Total    int     `json:"total"`
    Currency string  `json:"currency,omitempty"` // новое поле, опционально
}
```

Если нарушить совместимость всё же необходимо — создаётся новая MAJOR-версия API, а старая поддерживается некоторое время.

### Версионирование gRPC и Protobuf

gRPC предлагает встроенные механизмы версионирования через Protobuf package и сервисы:

```protobuf
syntax = "proto3";

package order.v1;
// или
package order.v2;
```

Стратегии:
- **Разные пакеты Protobuf** (`order.v1.OrderService`, `order.v2.OrderService`) — полный аналог URL-версионирования. Требует отдельных gRPC-эндпоинтов.
- **В рамках одного сервиса** с добавлением новых методов: `CreateOrderV2()`. Старый метод `CreateOrder()` продолжает работать.
- **Совместимость на уровне Protobuf:** используйте зарезервированные номера полей для удалённых, никогда не меняйте номера существующих полей.

```protobuf
message CreateOrderRequest {
  reserved 2, 3; // удалённые поля
  string user_id = 1;
  repeated Item items = 4;
}
```

В Go-коде разные версии могут обслуживаться разными структурами или одной структурой, реализующей оба сгенерированных интерфейса.

### Механическая симпатия: цена версионирования

Версионирование добавляет накладные расходы в рантайме, и архитектор должен их минимизировать.

**Аллокации и сериализация.** Если для каждой версии API определена своя структура, маппинг между ними и доменной моделью ([[13. Hexagonal Architecture. Ports and Adapters]]) создаёт временные объекты. На высоком RPS это увеличивает давление на GC. Решение: там, где версии близки, использовать одну структуру с `omitempty` и минимальным маппингом, либо генерировать маппинг кодом, чтобы избежать рефлексии.

**Роутинг.** Проверка версии в middleware на каждом запросе — это доступ к карте (`map[string]handler`) и поиск строки. Стоимость ничтожна (десятки наносекунд), если не аллоцировать новые строки.

**Память и gRPC.** Протобаф-дескрипторы для нескольких версий увеличивают размер бинарника и могут дублировать код. Но на потребление памяти в рантайме это влияет минимально, так как дескрипторы загружаются один раз.

### Деплой без даунтайма и флаги версий

Одна из целей версионирования — дать возможность деплоить новую версию сервиса без остановки старой ([[47. Миграции архитектуры без downtime]]). Практики:
- **Rolling update в Kubernetes:** одновременно работают поды старой и новой версии. API должно быть обратно совместимо на переходный период.
- **Canary деплой:** новая версия получает малую долю трафика; метрики сравниваются со старой.
- **Feature Flags:** новая логика скрыта за флагом и может быть включена/выключена без передеплоя.

В Go это можно реализовать через конфигурацию и `context`:

```go
func (s *Service) CreateOrder(ctx context.Context, req CreateOrderRequest) (*Order, error) {
    if feature.IsEnabled(ctx, "new-pricing") {
        return s.createOrderV2(ctx, req)
    }
    return s.createOrderV1(ctx, req)
}
```

### Антипаттерны

- **Версия в названии сервиса.** `order-service-v2` — плохая практика. Сервис должен быть `order-service`, а версия — это его API.
- **Бесконечная поддержка старых версий.** Это раздувает код и увеличивает затраты на поддержку. Объявляйте сроки устаревания (deprecation) и отключайте старые версии по графику.
- **Слишком много MAJOR-версий.** Если вы выпускаете `v3` через месяц после `v2`, значит, вы не продумали дизайн. Проектируйте API так, чтобы прожить на одной MAJOR-версии минимум год.
- **Версионирование только для внешних API.** Внутренние сервисы тоже должны версионироваться, особенно если у них несколько потребителей в разных командах.

> [!tip] Собеседование
> **Вопрос:** У вас есть REST API, которым пользуются 10 внутренних команд. Вам нужно переименовать поле `user_id` в `customer_id` и изменить его формат со строки на число. Ваши действия?
> **Ответ:** Это нарушение обратной совместимости, значит, нужна новая MAJOR-версия API (`/v2/`). Я создаю новую версию эндпоинта с новым форматом, а старая продолжает работать. В старой версии поле `user_id` остаётся строкой. Рассылаю уведомление командам о сроке устаревания `v1` (например, 3 месяца). Мониторю использование `v1` по метрикам и отключаю её, когда трафик падает до нуля. В коде Go это будет два разных хендлера с разными структурами запросов, маппящихся на общий сервисный слой через адаптеры.

### Итог

Версионирование API — это не роскошь, а базовая гигиена распределённых систем. Оно позволяет командам независимо эволюционировать сервисы, избегая «распределённого монолита». В Go экосистеме URL-версионирование и Protobuf-пакеты — два основных механизма, дополняемые обратной совместимостью и дисциплиной объявления deprecation. Архитектор должен проектировать API, закладывая в него возможность изменений, и помнить, что цена версионирования — это продуманные структуры данных, чёткая маршрутизация и неизбежные затраты на поддержку старых версий.
