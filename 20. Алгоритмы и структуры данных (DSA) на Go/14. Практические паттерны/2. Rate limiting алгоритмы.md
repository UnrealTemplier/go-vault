## Введение: Зачем ограничивать поток?

Rate limiting (ограничение скорости) — это не просто «защита от спама». Это фундаментальный паттерн управления потоком данных, который обеспечивает стабильность микросервисов, защищает базы данных от cascade failure, контролирует стоимость облачных вызовов и реализует бизнес-логику тарификации. 

В отличие от простого `time.Sleep` или блокирующих каналов, корректный rate limiter должен отвечать на три вопроса мгновенно: «Сколько разрешено?», «Когда можно следующий?» и «Что делать с burst-трафиком?». При этом он обязан работать за `O(1)`, не создавать contention на мьютексах, минимизировать аллокации и предсказуемо масштабироваться на сотни тысяч RPS.

В этой статье мы разберём DSA-основу алгоритмов ограничения скорости, их реализацию в Go с учётом механики рантайма, таймеров и атомиков, а также архитектурные решения для распределённых кластеров.

> [!tip] Собеседование
> **Вопрос:** «Почему нельзя реализовать rate limiting через `time.Ticker` в каждой HTTP-горутине?»
> **Ответ:** `time.Ticker` создаёт объект таймера в куче, требует горутину-потребителя для освобождения ресурсов, и при миллионе одновременных запросов сгенерирует `timer heap` contention и OOM. Rate limiter должен быть stateless для вызова или шардированным stateful, с lazy-обновлением состояния и без блокирующих примитивов на каждый запрос.

## Классификация алгоритмов и их DSA-основа

Каждый алгоритм опирается на разные структуры данных и даёт разные гарантии.

| Алгоритм | DSA-основа | Бёрсты | Точность | Память на клиента |
|----------|------------|--------|----------|-------------------|
| **Fixed Window** | Счётчик + Timestamp | Жёсткие границы, spike x2 у границ | Низкая | O(1) |
| **Sliding Window Log** | Кольцевой буфер / Очередь | Плавные, точные | Высокая | O(N) за окно |
| **Sliding Window Counter** | 2 счётчика + вес | Сглаженные, аппроксимация | Средняя | O(1) |
| **Token Bucket** | Счётчик токенов + LastRefillTime | Разрешены до capacity | Высокая | O(1) |
| **Leaky Bucket** | FIFO очередь + постоянная скорость | Запрещены, трафик сглаживается | Высокая | O(N) в очереди |

В бэкенде **Token Bucket** де-факто стандарт для API-шлюзов и тарификации. **Sliding Window Counter** предпочтителен для аналитики и биллинга. **Leaky Bucket** применяется для защиты медленных downstream-сервисов.

## Production-реализация: Token Bucket в Go

Стандартный пакет `golang.org/x/time/rate` отлично подходит для внутренних задач, но на собеседованиях и в высоконагруженных шлюзах требуется понимать внутреннюю механику. Реализуем оптимизированный, потокобезопасный `TokenBucket`.

```go
package ratelimit

import (
	"sync"
	"time"
)

// TokenBucket реализует алгоритм корзины токенов.
// Потокобезопасен, O(1) на запрос, lazy refill.
type TokenBucket struct {
	mu            sync.Mutex
	capacity      float64
	rate          float64       // токенов в секунду
	tokens        float64
	lastRefillAt  time.Time
}

// NewTokenBucket создаёт лимитер с заданной пропускной способностью и burst.
func NewTokenBucket(rate float64, burst float64) *TokenBucket {
	return &TokenBucket{
		capacity:     burst,
		rate:         rate,
		tokens:       burst,
		lastRefillAt: time.Now(),
	}
}

// Allow проверяет и резервирует 1 токен. Возвращает true, если запрос разрешён.
func (tb *TokenBucket) Allow() bool {
	return tb.AllowN(1)
}

// AllowN проверяет и резервирует n токенов.
func (tb *TokenBucket) AllowN(n float64) bool {
	tb.mu.Lock()
	defer tb.mu.Unlock()

	now := time.Now()
	tb.refill(now)

	if tb.tokens >= n {
		tb.tokens -= n
		return true
	}
	return false
}

// WaitTime возвращает время ожидания до возможности выполнить запрос.
func (tb *TokenBucket) WaitTime() time.Duration {
	tb.mu.Lock()
	defer tb.mu.Unlock()

	now := time.Now()
	tb.refill(now)

	if tb.tokens >= 1 {
		return 0
	}
	// Сколько секунд нужно, чтобы набрать 1 - tb.tokens
	need := 1.0 - tb.tokens
	waitSec := need / tb.rate
	return time.Duration(waitSec * float64(time.Second))
}

// refill лениво пополняет токены на основе прошедшего времени.
// Вызывается ТОЛЬКО под tb.mu.
func (tb *TokenBucket) refill(now time.Time) {
	elapsed := now.Sub(tb.lastRefillAt).Seconds()
	if elapsed > 0 {
		tb.tokens += elapsed * tb.rate
		if tb.tokens > tb.capacity {
			tb.tokens = tb.capacity
		}
		tb.lastRefillAt = now
	}
}
```

Инженерные решения:
* **Lazy Refill**: Токены не обновляются в фоне таймером. Они вычисляются на лету при вызове `Allow()`. Это экономит CPU, устраняет `time.Ticker` аллокации и снимает нагрузку с планировщика.
* `float64` для токенов: Позволяет дробно начислять токены при малых `rate`, избегая целочисленного округления, которое может «съедать» пропускную способность на long-tail.
* `sync.Mutex` вместо атомиков: Математика с `float64` и `time` требует чтения/записи нескольких полей атомарно. `Mutex` здесь безопаснее и быстрее, чем CAS-циклы с гонками на плавающей точке.

## Механическая симпатия: таймеры, атомики и GC

Поведение rate limiter в Go напрямую зависит от работы рантайма и архитектуры CPU.

### Стоимость `time.Now()` и vDSO
Вызов `time.Now()` на Linux использует системный вызов `clock_gettime(CLOCK_MONOTONIC)`. Благодаря **vDSO** (Virtual Dynamically Shared Object) он не всегда вызывает переход в kernel space. Ядро маппит страницу времени в user space, и приложение читает её напрямую. Это ~5-10 нс. Однако при частых вызовах (`>1M RPS`) страница времени может быть вытеснена из кэша, что добавит ~50-100 нс. Оптимизация: кэшировать текущее время в `sync.Pool` или обновлять раз в 10 мс через одну фоновую горутину, но это добавляет complexity и редко оправдано.

### Конкурентность и False Sharing
Если вы создаёте `TokenBucket` для миллионов пользователей и храните их в `map[string]*TokenBucket`, каждый `mu sync.Mutex` занимает 16-32 байта. При высокой конкуренции разные ядра CPU будут постоянно инвалидировать кэш-линии, содержащие мьютексы (False Sharing). 
**Решение:** Шардировать лимитеры по хешу IP/ClientID. Каждый шард имеет свой мьютекс. Это распределяет contention по ядрам.

```go
type ShardedLimiter struct {
	shards []*TokenBucket
	mask   uint32
}
```

### Escape Analysis и аллокации
В коде выше `TokenBucket` не содержит указателей на внешние объекты, поэтому при использовании внутри функции может остаться на стеке. Но если он шарится между обработчиками, `Escape Analysis` выносит его в кучу. Давление на [[7. Глубокий Go (Внутреннее устройство)|сборщик мусора]] минимально, так как нет динамических структур. Для high-load систем рекомендуется предвыделять лимитеры в пуле, если клиенты эфемерны.

> [!info] Под капотом
> **Почему не `sync/atomic` для токенов?**
> Попытка реализовать `atomic.AddFloat64` приводит к гоночным условиям на операции `refill`. Два потока могут одновременно прочитать `tokens`, добавить разное количество и перезаписать друг друга. `atomic.CompareAndSwapFloat64` требует CAS-цикла, который при высокой конкуренции вызывает `livelock` и жрёт CPU. `sync.Mutex` в Go 1.18+ использует `semaphore` и `spinning`, что на современных CPU быстрее при средней конкуренции.

## Распределённый Rate Limiting

Локальный лимитер работает только на одном поде. В кластере Kubernetes нужно координировать счётчики между нодами. Архитектурно используется **Redis + Lua-скрипт**.

Атомарность в Redis гарантирует, что `GET`, расчёт токенов и `SET` выполняются без блокировок другими клиентами. Lua-скрипт выполняется в однопоточном event-loop Redis, что исключает race conditions.

```lua
-- Redis Lua script for distributed Token Bucket
local key = KEYS[1]
local rate = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])

local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens = tonumber(bucket[1])
local last_refill = tonumber(bucket[2])

if tokens == nil then
    tokens = capacity
    last_refill = now
end

local elapsed = now - last_refill
tokens = math.min(capacity, tokens + elapsed * rate)

if tokens >= requested then
    tokens = tokens - requested
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    redis.call('EXPIRE', key, math.ceil(capacity / rate) + 1)
    return 1 -- Allowed
end

return 0 -- Denied
```

**Механика:**
* Сетевой RTT + сериализация аргументов добавляют 0.5-2 мс к латентности.
* Для критичных к задержке сервисов используют **локальный кэш лимитов + асинхронную синхронизацию** или **Probabilistic Rate Limiting** через счётчики с TTL.
* Redis кластеризация: шардирование лимитов по хешу ключа через `HASH_SLOT` обеспечивает масштабирование до миллионов ключей.

## Ловушки production-разработки и вопросы с собеседований

> [!warning] Ловушка / Gotcha
> **Burst vs Sustained Rate**
> `rate = 10/sec, capacity = 20` означает: первые 2 секунды можно послать 20 запросов мгновенно, дальше — строго 10 в секунду. Если ваш downstream не выдерживает burst, установите `capacity == rate` или используйте **Leaky Bucket**, который выводит запросы строго равномерно.
> 
> **Утечка памяти в Sliding Window Log**
> Реализация на `[[6. Кольцевой буфер]]` или очереди требует очистки старых записей. Если вы забываете удалять timestamp старше окна, буфер растёт до OOM. Используйте кольцевой буфер фиксированного размера или фоновую очистку с TTL.
> 
> **Clock Skew в распределённых системах**
> Часы нод кластера могут расходиться на 10-50 мс. Если лимитер опирается на `time.Now()` из разных машин, один клиент может «перепрыгнуть» окно. Решение: использовать `Redis TIME` или синхронизировать часы через `chrony` с точностью до миллисекунд.

> [!tip] Собеседование
> **Вопрос 1:** «Как реализовать rate limiting для миллионов уникальных IP, не расходуя память на каждый?»
> **Ответ:** Использовать **Sharding + LRU eviction**. Хранить лимитеры только для активных клиентов. При превышении памяти или TTL удалять самые старые. Либо применить **Probabilistic Counting** (HyperLogLog, Count-Min Sketch) для аппроксимации, если допустима погрешность 1-2%.
> 
> **Вопрос 2:** «В чём разница между Token Bucket и Leaky Bucket с точки зрения DSA?»
> **Ответ:** Token Bucket использует счётчик (counter) и разрешает burst до ёмкости. Leaky Bucket использует FIFO очередь (queue) и выводит запросы с фиксированной скоростью, сглаживая пики. Для DDoS-защиты лучше Leaky Bucket, для тарификации API — Token Bucket.
> 
> **Вопрос 3:** «Почему нельзя блокировать горутину через `time.Sleep` внутри middleware при rate limit?»
> **Ответ:** `time.Sleep` блокирует горутину, но не возвращает клиенту `429 Too Many Requests`. Клиент держит соединение открытым, расходуя файловые дескрипторы сервера и память стека. Правильно: сразу отклонять с 429 и заголовком `Retry-After`, позволяя клиенту реализовать backoff.
> 
> **Вопрос 4:** «Как интегрировать rate limiter с [[1. Проектирование кэшей]] для защиты от cache stampede?»
> **Ответ:** Rate limiter ограничивает частоту запросов к БД, но не защищает от одновременных промахов кэша. Для stampede используйте `golang.org/x/sync/singleflight` или кэш с background refresh. Rate limiter должен стоять *перед* кэшем, чтобы отсекать явные аномалии трафика, а не защищать внутреннюю консистентность.

## Итог

* **Rate Limiting** — это не блокировка, а управление потоком. Выбор алгоритма диктуется бизнес-требованиями: Token Bucket для burst и тарификации, Leaky Bucket для сглаживания, Sliding Window для точной аналитики.
* В Go оптимальна **ленивая реализация** (`lazy refill`) на `sync.Mutex` и `time`. Это устраняет фоновые таймеры, снижает аллокации и даёт `O(1)` на запрос.
* **Механическая симпатия**: `time.Now()` дёшев благодаря vDSO, но `float64` атомарность невозможна. При масштабировании используйте шардирование лимитеров, чтобы избежать False Sharing и contention.
* **Распределённый лимитинг** требует Redis + Lua для атомарности. Сетевая латентность и clock skew — главные враги точности. Синхронизация NTP/chrony обязательна.
* **Production-паттерны**: Always return `429` + `Retry-After`, avoid `time.Sleep` in handlers, combine with [[1. Проектирование кэшей]] and circuit breakers for defense-in-depth.
* **Интервью фокус**: отличие burst от sustained rate, trade-offs memory vs accuracy, distributed consistency, lock-free attempts vs mutex reality.

Разобравшись с управлением потоком запросов на одном инстансе, мы переходим к архитектурной задаче маршрутизации в кластере: как направлять запросы к нужной ноде, минимизировать переезд данных при масштабировании и обеспечивать равномерную загрузку без централизованного координатора?
