## Почему триколорный маркинг — сердце Go GC

В предыдущей статье [[1. GC в Go. Обзор]] мы очертили общую архитектуру сборщика мусора: конкурентный mark-sweep с короткими паузами, нацеленный на низкую latency. Теперь мы спускаемся на уровень алгоритма, который делает возможной эту эффективную маркировку живых объектов без долгой остановки приложения. Этот алгоритм — **триколорный маркинг** (tri-color marking).

Триколорный маркинг — это абстракция для отслеживания состояния объектов в куче во время mark-фазы. Он лежит в основе любого конкурентного сборщика мусора, включая Go. Понимание триколорного алгоритма необходимо Senior-инженеру, чтобы разбираться в причинах GC-пауз ([[3. Stop the world]]), нагрузке от write barrier ([[5. Write barriers]]) и корректности работы приложения под конкурентной сборкой.

В этой статье мы разберём, как алгоритм раскрашивает объекты, почему он требует барьеров записи при конкурентной работе и как Go реализует триколорный обход, оглядываясь на исходный код рантайма.

## Основные понятия: три цвета

Алгоритм работает с абстрактными цветами, которые присваиваются объектам в куче:

- **Белый (White)** — объект не посещён сборщиком. По окончании маркировки белые объекты считаются мусором и подлежат удалению. Это начальный цвет всех объектов.
- **Серый (Grey)** — объект посещён, но не все его потомки (объекты, на которые он ссылается) обработаны. Серые объекты образуют «фронт волны» обхода.
- **Чёрный (Black)** — объект и все его потомки полностью обработаны. Чёрные объекты точно живые и останутся в куче после sweep.

Процесс маркировки перемещает объекты из белого множества в серое, а затем в чёрное. Когда серое множество опустеет, маркировка завершена. Все оставшиеся белые объекты — мусор.

```mermaid

    Grey -->|"Потомки обработаны"| Black["Чёрный объект<br>Живой, полностью обработан"]
    Black -.-|"Ссылается на"| White2["Белый<br>становится серым"]

```

> [!info] Под капотом
> В реальной реализации Go цвета кодируются битами в заголовке объекта (в структуре `mspan`, управляющей спанами). Конкретно, в `runtime/mbitmap.go` для каждого объекта в куче хранится двухбитовое поле: `gcBits` — это битовая карта, где 00 — белый, 01 — серый, 10 — чёрный. Алгоритм триколорного обхода реализован в `runtime/mgcmark.go`.

## Пошаговый алгоритм триколорной маркировки

1. **Инициализация.** Все объекты помечаются белыми (сброс битов). Стеки всех горутин и глобальные переменные — это «корни» (roots). GC помещает в серое множество непосредственно достижимые из корней объекты.
2. **Цикл маркировки.** Пока серое множество не пусто:
   - Извлекаем один серый объект.
   - Сканируем его поля: для каждого указателя, ведущего на другой объект, проверяем цвет целевого объекта.
   - Если целевой объект белый, красим его в серый и добавляем в серое множество.
   - После проверки всех указателей текущий объект красится в чёрный и удаляется из серого множества.
3. **Завершение.** Когда серое множество пусто, все живые объекты чёрные. Оставшиеся белые — недостижимы и будут собраны.

Этот алгоритм корректен в мире, где никто не меняет граф объектов. Но Go GC работает конкурентно с мутатором — приложение продолжает выполняться, меняя указатели. Поэтому возникает проблема «пропавшего объекта» (lost object problem), и требуются write barrier.

## Проблема пропавшего объекта и необходимость барьера

При конкурентной маркировке мутатор может «спрятать» белый объект от сборщика. Рассмотрим сценарий:

1. Чёрный объект A содержит указатель на белый объект C.
2. Серый объект B просматривается GC, но он больше не указывает на C (мутатор уже убрал ссылку).
3. До того, как A был сделан чёрным, мутатор записал в A указатель на C, а из B удалил ссылку на C.
4. Теперь C достижим только из чёрного A, но чёрные объекты больше не сканируются. GC считает C белым мусором и освобождает его — катастрофа.

Для предотвращения этого используется **write barrier** — небольшой код, выполняемый при каждой записи указателя. Он информирует GC о том, что объект мог измениться, и гарантирует, что белый объект не будет потерян.

## Write barrier в Go: стиль Dijkstra и Yuasa

Go использует гибридный write barrier, совмещающий идеи:

- **Барьер Дейкстры** (Dijkstra write barrier): при записи указателя в чёрный объект, если записываемый указатель ведёт на белый объект, этот белый объект сразу красится в серый. Это предотвращает появление белых объектов в чёрном.
- **Барьер Юасы** (Yuasa write barrier): при удалении указателя из серого или чёрного объекта, если удаляемый указатель указывает на белый объект, он также красится в серый. Это страховка от потери объектов через удаление ссылок.

В исходном коде Go (`runtime/mwbbuf.go`) реализован так называемый ** смешанный write barrier** для маркировки. На практике для каждой записи указателя в кучу компилятор вставляет вызов `gcWriteBarrier` — быструю inline-функцию, которая добавляет целевой объект в буфер барьера, если это нужно. Позже эти буферы обрабатываются GC.

> [!tip] Собеседование
> **Вопрос:** Объясните, почему триколорный алгоритм без write barrier неработоспособен при конкурентной маркировке.
> **Ответ:** Потому что мутатор может создать ссылку из чёрного объекта на белый, которая не будет замечена GC (чёрные объекты уже не сканируются). Write barrier предотвращает это, немедленно окрашивая белый объект в серый при записи ссылки.

## Практический пример триколорного обхода на Go

Представьте код:

```go
type Node struct {
    Value int
    Left  *Node
    Right *Node
}

func main() {
    root := &Node{Value: 1}
    root.Left = &Node{Value: 2}
    root.Right = &Node{Value: 3}

    // Пусть в этот момент начинается GC
    runtime.GC()
}
```

Перед GC объекты доступны из стека `main` (корень). GC:

- Помечает `root` как серый (из корней).
- Извлекает `root`, сканирует его поля: `Left` и `Right` пока белые → красят в серый.
- `root` становится чёрным.
- Далее поочерёдно обрабатывает `Left` и `Right`, они становятся чёрными (у них нет потомков).
- Серое множество опустело → mark завершён.

Если бы во время маркировки `root.Left` был изменён на новый узел `Node{Value: 4}`, write barrier обеспечил бы, что новый узел стал серым и не потерялся.

## Влияние на производительность и паузы

Триколорный обход сам по себе не требует остановки мира, если есть write barrier. Однако для запуска и завершения маркировки требуются короткие STW-паузы ([[3. Stop the world]]):

- **В начале:** чтобы инициализировать состояние, сбросить цвета, подготовить битовые карты.
- **В конце (mark termination):** чтобы убедиться, что серое множество опустело, обработать финальные буферы write barrier и завершить маркировку.

Длительность этих пауз зависит от размера кучи и количества горутин. В хорошо настроенном приложении они измеряются десятками микросекунд.

Стоимость write barrier — несколько дополнительных инструкций на каждую запись указателя. Это незаметно в большинстве приложений, но в коде, активно модифицирующем графы объектов (например, `sync.Map` с указателями), может накапливаться.

## Mechanical Sympathy: триколор и кэш

Обход графа объектов при маркировке неизбежно загружает множество кэш-линий. Это вымывает полезные данные приложения из L1/L2 кэша. Чем больше живых объектов (чёрных), тем больше памяти сканирует GC, и тем сильнее эффект. Поэтому минимизация количества долгоживущих объектов и ссылочных связей улучшает не только саму работу GC, но и пост-GC производительность ([[8. Cache friendliness]]).

Кроме того, write barrier пишет в глобальный буфер, который, скорее всего, находится в L3 или RAM, вызывая дополнительные cache miss. Поэтому в горячих циклах желательно избегать частых записей в указатели на куче.

## Итог

- **Триколорный маркинг** — алгоритм mark-фазы Go GC: объекты последовательно переходят из белого в серый и чёрный, что позволяет конкурентно обходить граф достижимости.
- Без write barrier алгоритм неконкурентоспособен из-за "пропавшего объекта". Go применяет гибридный барьер (Dijkstra + Yuasa), вставляемый компилятором при записи указателей.
- Алгоритм минимизирует паузы, но требует коротких STW для старта и завершения, и добавляет накладные расходы через барьеры и вымывание кэша.
- Понимание триколорного алгоритма необходимо для осознанного тюнинга GC и оптимизации структур данных (избегания избыточных указателей).
