## Введение: когда бенчмарк лжёт

В предыдущих статьях мы освоили синтаксис бенчмарков ([[2. Benchmarking в Go]]) и разобрали их внутреннее устройство ([[3. go test -bench под капотом]]). Мы знаем, как адаптируется `b.N`, как работают таймеры и параллельные прогоны. Но весь этот инструментарий обесценивается, если бенчмарк содержит логическую ошибку и выдаёт цифры, не имеющие отношения к реальной производительности.

Практика показывает: написать **правильный** бенчмарк намного сложнее, чем кажется. Компилятор Go становится умнее с каждым релизом и агрессивно удаляет «бесполезный» код. Рантайм подбрасывает GC в самый неподходящий момент. Процессорный кэш и планировщик создают иллюзию быстродействия, которая исчезает под нагрузкой. Принимая решения на основе искажённых данных, вы рискуете внедрить «оптимизацию», которая на проде ухудшит и latency, и throughput.

В этой статье собраны главные подводные камни бенчмарков Go: от компиляторной элиминации до скрытого влияния GC. Для каждой ловушки — пример, объяснение и способ защиты.

> [!warning] Ловушка / Gotcha
> Если вы запускаете бенчмарк без `-benchmem` и без проверки ассемблерного выхлопа, вы не измеряете код — вы меряете оптимизации компилятора, которые могут измениться в следующем минорном релизе Go.

## 1. Dead Code Elimination (удаление мёртвого кода)

Самая распространённая ловушка. Если результат вычислений не используется, компилятор имеет право удалить всю вычисляющую логику. Бенчмарк покажет 0.25 нс/оп, потому что измеряет только пустой цикл.

```go
// НЕВЕРНО: компилятор выбросит вызов expensiveFunc
func BenchmarkDeadCode(b *testing.B) {
    for i := 0; i < b.N; i++ {
        expensiveFunc()
    }
}
```

### Как защититься

- Присвоить результат переменной уровня пакета, недоступной для анализа компилятору:

```go
var result int32

func BenchmarkKeepAlive(b *testing.B) {
    for i := 0; i < b.N; i++ {
        result = expensiveFunc()
    }
}
```

- Использовать `runtime.KeepAlive` (Go 1.14+), хотя чаще применяют первый способ.
- В бенчмарках параллельных операций — использовать атомарное сохранение результатов в глобальную переменную.

> [!info] Под капотом
> Компилятор Go выполняет dead code elimination на уровне SSA-представления. Если инструкция не участвует ни в одной цепочке до выхода из функции (return, запись в память вне стека функции), она выкидывается. Глобальная переменная является видимой для всего пакета точкой выхода, поэтому операция сохраняется.

## 2. Неверная работа с таймерами

Часто дорогую подготовку или очистку оставляют внутри цикла, завышая `ns/op`. Или забывают `b.ResetTimer()` после инициализации.

```go
// НЕВЕРНО: инициализация входит в замер
func BenchmarkSliceAppend(b *testing.B) {
    for i := 0; i < b.N; i++ {
        s := make([]int, 1000000) // дорого!
        s = append(s, 1)
    }
}
```

Решение: использовать `b.ResetTimer()` или `b.StopTimer()`/`b.StartTimer()`.

```go
func BenchmarkSliceAppendFixed(b *testing.B) {
    s := make([]int, 1000000)
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        s = append(s, 1)
    }
}
```

> [!warning] Ловушка / Gotcha
> `b.ResetTimer()` обнуляет накопленное время, но **не обнуляет `b.N`**. Это значит, что первая итерация всё равно будет учтена в замере. Если первый вызов существенно медленнее (холодный кэш, ленивая инициализация), результат окажется смещённым. Для таких случаев делают «прогревочный» проход вне таймера.

## 3. Влияние GC и аллокаций

Даже если код написан правильно, сборщик мусора может сработать в середине прогона и добавить миллисекунды к измерению. Особенно сильно это проявляется в бенчмарках с интенсивными аллокациями.

```go
func BenchmarkStringBuilder(b *testing.B) {
    for i := 0; i < b.N; i++ {
        var sb strings.Builder
        for j := 0; j < 1000; j++ {
            sb.WriteString("hello")
        }
        _ = sb.String()
    }
}
```

На первый взгляд всё честно. Но `strings.Builder` внутри растёт, аллоцируя слайсы. При достаточно большом `b.N` кучa наполняется, запускается GC, и `ns/op` внезапно подскакивает. Среднее время оказывается не репрезентативным для короткоживущего запроса, где GC мог вовсе не сработать.

### Как защититься

- Всегда смотреть `-benchmem` — высокие B/op и allocs/op предупреждают о вмешательстве GC.
- Принудительно вызывать `runtime.GC()` перед циклом, чтобы состояние кучи было детерминированным.
- В реальных сценариях оптимизировать аллокации ([[1. Уменьшение аллокаций]], [[2. sync Pool]]), чтобы GC не влиял на хвосты ([[7. Tail latency и почему она важна]]).

## 4. Игнорирование прогрева кэша

Бенчмарк в Go стартует с «холодным» кэшем. Первая порция итераций тратит такты на прогрузку данных в L1/L2. Если `b.N` невелико (быстрая операция, `-benchtime=100ms`), кэш-промахи доминируют в результате. На реальном сервере, обрабатывающем тысячи запросов, структуры данных горячие, и такой эффект пропадает.

Обратная ситуация: бенчмарк настолько долог, что данные полностью прогреваются, а в production-коде между вызовами кэш вымывается другими операциями. Тогда бенчмарк покажет лучшее время, чем реальность.

**Защита:** использовать `-benchtime` не менее 1 секунды (по умолчанию), анализировать дисперсию между запусками (`-count`), и при необходимости добавлять ручной прогрев перед `b.ResetTimer()`.

## 5. Ловушки параллельных бенчмарков (`RunParallel`)

`b.RunParallel` запускает `GOMAXPROCS` горутин. Но разработчик может неверно интерпретировать результат:

- **Неверно:** считать, что `ns/op` — время одной операции одной горутины. На деле это общее время выполнения всех горутин, делённое на общее количество итераций. Contention и синхронизация уже зашиты в эту цифру.
- **Ожидать линейного ускорения.** Если бенчмарк упирается в контеншн на атомарной переменной или false sharing ([[8. False sharing]]), добавление горутин через `b.SetParallelism` может даже снизить пропускную способность.
- **Забывать о прогреве горутин.** Первые итерации после старта `RunParallel` включают разогрев локальных кэшей горутин и стартовые барьеры. Иногда стоит сделать холостой прогон перед измеряемым.

## 6. Однобокость: неправильный размер входных данных

Микробенчмарк, измеряющий конкатенацию трёх строк, не предсказывает поведение на 10000 строк. Бенчмарк хеш-таблицы на 10 ключах не показывает, что на 10 миллионах начнутся эвакуации бакетов (рост `map`, см. внутреннее устройство мап).

**Правило:** тестировать на нескольких реалистичных размерах (небольшие, средние, большие), а также при необходимости на искусственно вырожденных случаях (пустые, максимальные). Сравнивать тренды, а не одиночные точки.

## 7. Не-(рас)познавание статистической природы

Один прогон бенчмарка — это выборка из распределения возможных времён выполнения, зашумлённого ОС, GC и планировщиком. Принимать решение по одному запуску — грубая ошибка. Всегда использовать `-count=10` или больше и анализировать `benchstat` ([[6. Стабилизация результатов]]). Разброс может быть асимметричным: p50 стабильно, а p99 плавает из-за GC. Тогда среднее `ns/op` не ловит хвост.

## 8. Неправильное использование `-benchtime`

По умолчанию `-benchtime=1s`. Для очень быстрых операций (например, atomic.Add) этого хватает для миллиарда итераций, и бенчмарк может упереться в `b.N`=1e9. Для очень медленных (секунды на операцию) `b.N` останется 1, и погрешность `time.Now()` станет заметной. В таких случаях задают фиксированное число итераций: `-benchtime=100x`.

Но форсирование малого количества итераций чревато недостоверной статистикой и влиянием выбросов. Прежде чем фиксировать N, убедитесь, что понимаете, зачем это делаете.

## 9. Влияние инлайнинга и оптимизаций компилятора

Компилятор может заинлайнить функцию, и бенчмарк покажет производительность, которая в реальном коде (где инлайнинг не произошёл из-за размера функции или escape analysis) не достигается. И наоборот: флаг `-gcflags="-N -l"` отключает инлайнинг и делает замеры пессимистичными.

**Рекомендация:** бенчмарки запускать с оптимизациями по умолчанию (production-режим). При анализе смотреть ассемблер (`-gcflags="-S"`) или `go tool compile -S`, чтобы понять, что реально попало в замер.

## 10. Сокрытие аллокаций через пулы

Использование `sync.Pool` ([[2. sync Pool]]) внутри бенчмарка без его очистки между итерациями может симулировать несуществующую эффективность: если пул не опустошается, он возвращает объекты без аллокаций, что соответствует идеальному сценарию. На практике под нагрузкой пул может опустошаться или, наоборот, накапливать слишком много объектов.

**Защита:** между итерациями бенчмарка сбрасывать пул до реалистичного состояния, либо измерять и холодный, и горячий путь отдельно.

## Пример комплексного «честного» бенчмарка

```go
var globalResult []byte

func BenchmarkReadFull(b *testing.B) {
    data := make([]byte, 4096)
    reader := bytes.NewReader(bytes.Repeat([]byte("x"), 4096))
    
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        reader.Reset(bytes.Repeat([]byte("x"), 4096)) // Reset внутри, чтобы исключить подготовку
        n, _ := io.ReadFull(reader, data)
        globalResult = data[:n]
    }
}
```

Здесь мы:
- Используем глобальную переменную, чтобы избежать dead code elimination.
- Сбрасываем reader внутри цикла после `ResetTimer` (подготовка включена). Мораль: иногда подготовка неизбежна, тогда её включают в замер как часть реалистичного сценария.
- Всегда указываем `-benchmem` для контроля аллокаций.

## Итог

Подводные камни бенчмарков в Go — не экзотика, а повседневная реальность. Dead code elimination, GC, кэш-эффекты, contention, неверная обработка таймеров и статистическая небрежность превращают измерения в самообман. Senior-разработчик обязан знать каждую из этих ловушек и методично отрабатывать их:

- Сохранять результат глобально.
- Управлять таймерами осознанно.
- Контролировать аллокации (`-benchmem`).
- Прогревать кэш и стабилизировать окружение.
- Запускать с разными размерами входных данных и анализировать распределение.
- Сверяться с ассемблером, чтобы убедиться, что измеряется именно то, что задумано.

Только такой подход превращает бенчмарк из генератора цифр в надёжный фундамент для принятия инженерных решений.

Мы разобрали, как не наступить на грабли. Теперь предстоит понять, когда бенчмарк уместен, а когда нужны более широкие измерения. Об этом — следующая статья: [[5. microbenchmarks vs macrobenchmarks]].
