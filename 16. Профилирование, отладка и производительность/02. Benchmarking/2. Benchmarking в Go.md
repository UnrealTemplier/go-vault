## Введение: бенчмарк как первый шаг измерения

В [[1. Почему нельзя оптимизировать без измерений]] мы утвердили аксиому: никакой оптимизации без цифр. Стандартный инструмент для получения этих цифр в Go — встроенный фреймворк бенчмаркинга, запускаемый через `go test -bench`. Он не требует внешних библиотек, интегрирован с профилировщиками и поставляет разработчику точные, воспроизводимые метрики времени и памяти.

Эта статья — практическое руководство по написанию, запуску и интерпретации бенчмарков. Мы разберём синтаксис, таймеры, параллельное выполнение, метрики аллокаций и ключевые правила, отделяющие полезные измерения от бесполезных. Детали внутренней механики `go test -bench` (адаптивный подбор `b.N`, взаимодействие с ОС) вынесены в следующую статью: [[3. go test -bench под капотом]].

## Основы: функция `Benchmark`

Бенчмарк — это функция с сигнатурой `func BenchmarkXxx(b *testing.B)`, размещённая в файлах `*_test.go`. Имя должно начинаться с `Benchmark`, первая буква после префикса — заглавная. Параметр `*testing.B` управляет таймингом, числом итераций и отчётностью.

```go
import "testing"

func BenchmarkSprintf(b *testing.B) {
    for i := 0; i < b.N; i++ {
        _ = fmt.Sprintf("hello %d", i)
    }
}
```

Запуск: `go test -bench=. -benchmem`. Флаг `-bench` принимает регулярное выражение для фильтрации имён бенчмарков (`.` — все). `-benchmem` включает отчёт по аллокациям. Результат:

```
BenchmarkSprintf-8   	10000000	       112.3 ns/op	      16 B/op	       2 allocs/op
```

- `-8` — количество используемых ядер (GOMAXPROCS).
- `10000000` — значение `b.N`, при котором бенчмарк стабилизировался.
- `112.3 ns/op` — среднее время одной итерации.
- `16 B/op, 2 allocs/op` — средние байты и количество аллокаций на итерацию.

Если `b.N` не используется в цикле, бенчмарк не будет измерять полезную работу. Код внутри цикла должен быть достаточно «лёгким», чтобы `b.N` выросло до статистически значимого числа.

## Адаптивный подбор `b.N`

Фреймворк сам определяет, сколько раз выполнить цикл, чтобы получить стабильное время. Он запускает бенчмарк с малым `b.N` (например, 1), измеряет длительность, и если она меньше целевого времени (по умолчанию 1 секунда), увеличивает `b.N` кратно (обычно в 20-100 раз) и повторяет. Так продолжается до тех пор, пока измерение не достигнет заданной длительности (`-benchtime`). Это описано в [[3. go test -bench под капотом]].

В результате мы получаем амортизированное среднее время операции, которое устойчиво к шумам ОС и прогреву кэша.

> [!warning] Ловушка / Gotcha
> Если ваш код внутри цикла содержит «быструю» ветку, которая выполняется только при малых `b.N` (например, кэш, который не успевает заполниться), стабилизированное значение может не отражать реальную производительность под нагрузкой. Используйте `b.ResetTimer()` или прогревочные итерации вне измеряемого цикла.

## Таймеры: управление измеряемым интервалом

Часто настройка тестовых данных или очистка состояния не должны попадать в результаты. Для этого `testing.B` предоставляет три метода:

- `b.ResetTimer()` — сбрасывает накопленное время и счётчик итераций. Обычно вызывается после дорогой инициализации, непосредственно перед циклом.
- `b.StopTimer()` / `b.StartTimer()` — приостанавливают и возобновляют таймер для произвольных участков.

```go
func BenchmarkSliceAppend(b *testing.B) {
    data := make([]int, 0, 1000)
    // инициализация, которая не должна замеряться
    for i := 0; i < 1000; i++ {
        data = append(data, i)
    }
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        // целевая операция
        _ = append(data[:0], 1, 2, 3)
    }
}
```

Без `ResetTimer()` в измерения попала бы подготовка слайса, исказив картину. Аналогично, `StopTimer`/`StartTimer` применяются, когда внутри цикла нужно исключить часть работы (например, генерацию случайных входных данных, которая не должна учитываться).

> [!info] Под капотом
> Таймер бенчмарка в Go использует монотонные часы ОС (`CLOCK_MONOTONIC` в Linux), что гарантирует независимость от скачков системного времени. Однако переключения контекста, прерывания и работа GC добавляют дисперсию. Именно поэтому фреймворк усредняет множество итераций, а не замеряет одну.

## Параллельные бенчмарки: `RunParallel`

Когда мы хотим измерить конкурентное поведение (contention на мьютексах, пропускную способность каналов, масштабирование), используем `b.RunParallel`. Внутри `RunParallel` горутины выполняют тело цикла параллельно, а `b.N` распределяется между ними. Число горутин по умолчанию равно `GOMAXPROCS`, но может быть изменено через `b.SetParallelism`.

```go
func BenchmarkParallelAdd(b *testing.B) {
    var counter int64
    b.RunParallel(func(pb *testing.PB) {
        for pb.Next() {
            atomic.AddInt64(&counter, 1)
        }
    })
}
```

`pb.Next()` возвращает `true`, пока есть итерации. Здесь измеряется пропускная способность атомарной операции в условиях конкуренции. Важно: `b.N` — общее число итераций, а не на одну горутину.

`b.SetParallelism(p int)` устанавливает множитель: итоговое число горутин = `GOMAXPROCS * p`. Это позволяет симулировать более высокий уровень параллелизма, чем число ядер (полезно для IO-bound сценариев).

> [!tip] Собеседование
> **Вопрос:** Чем `b.RunParallel` отличается от ручного запуска горутин с `sync.WaitGroup` в бенчмарке?
> **Ответ:** `RunParallel` корректно распределяет `b.N` между горутинами, автоматически подстраивает итерации для стабилизации и включает в измерения накладные расходы на координацию (старт, ожидание). При ручном запуске горутин сложно гарантировать эквивалентный замер без искажений от `WaitGroup` и неравномерного распределения итераций.

## Суббенчмарки: `b.Run`

Для сравнения нескольких вариантов внутри одного теста удобно использовать `b.Run`. Каждый суббенчмарк получает собственное `b` и измеряется независимо.

```go
func BenchmarkConcat(b *testing.B) {
    parts := []string{"Hello", " ", "World", "!", " ", "Go", " ", "is", " ", "fast"}
    b.Run("Plus", func(b *testing.B) {
        for i := 0; i < b.N; i++ {
            var s string
            for _, p := range parts {
                s += p
            }
            _ = s
        }
    })
    b.Run("Builder", func(b *testing.B) {
        for i := 0; i < b.N; i++ {
            var sb strings.Builder
            for _, p := range parts {
                sb.WriteString(p)
            }
            _ = sb.String()
        }
    })
}
```

Вывод:
```
BenchmarkConcat/Plus-8      	 2000000	       600 ns/op
BenchmarkConcat/Builder-8   	10000000	       120 ns/op
```

Это позволяет не только сравнить реализации, но и организовать бенчмарки по категориям.

## Метрики аллокаций: `-benchmem` и `ReportAllocs`

Флаг `-benchmem` активирует встроенный в `testing.B` учёт выделений памяти. Для каждой итерации подсчитываются:
- **B/op** — байт, аллоцированных в куче.
- **allocs/op** — количество вызовов аллокатора.

Эти цифры выводятся автоматически. В коде можно принудительно включить отчёт через `b.ReportAllocs()` (актуально для тестов с `-bench` без `-benchmem`, но в реальности проще всегда добавлять `-benchmem`).

Снижение аллокаций напрямую снижает нагрузку на GC, что улучшает не только throughput, но и хвостовые задержки ([[7. Tail latency и почему она важна]]). Поэтому бенчмарк без `-benchmem` — полумера.

## Как избежать «мёртвого кода» и обмана измерений

Компилятор Go удаляет код, результат которого не используется. Если измеряемая функция возвращает значение, а мы его не сохраняем, компилятор может выбросить вызов. Решение — присвоить результат переменной уровня пакета или передать в `runtime.KeepAlive` (с версии Go 1.14 `KeepAlive` есть, но чаще используют присваивание глобальной переменной):

```go
var result int // глобальная переменная, чтобы избежать удаления

func BenchmarkSum(b *testing.B) {
    var s int
    for i := 0; i < b.N; i++ {
        s += sumRange(1, 100)
    }
    result = s // не даём компилятору выкинуть s
}
```

Подробнее ловушки разобраны в [[4. Подводные камни benchmark тестов]].

## Флаги `-bench` и окружение

Ключевые флаги, которые Senior должен знать как таблицу умножения:

| Флаг | Назначение |
|------|------------|
| `-bench <regexp>` | Запустить бенчмарки, соответствующие шаблону. `.` — все. `-bench=.` |
| `-benchtime <duration>` | Целевая длительность бенчмарка (или фиксированное число итераций `1000x`). По умолчанию 1 с. |
| `-count <n>` | Количество повторных запусков бенчмарка для оценки стабильности. |
| `-cpu <list>` | Список значений GOMAXPROCS, при которых выполнять бенчмарк (например, `-cpu=1,2,4`). |
| `-benchmem` | Включить отчёт по памяти. |
| `-cpuprofile <file>` | Записать CPU-профиль для последующего анализа. |
| `-memprofile <file>` | Записать профиль памяти. |
| `-blockprofile <file>` | Профиль блокировок. |
| `-mutexprofile <file>` | Профиль мьютексов. |
| `-timeout <duration>` | Таймаут на выполнение всех тестов. |

Пример комплексного запуска для снятия профилей:
```bash
go test -bench=. -benchmem -count=5 -cpuprofile=cpu.out -memprofile=mem.out > bench.txt
```
Это даёт и текстовый вывод, и профили для [[7. Профилирование внутри benchmark]].

## Стабилизация и повторяемость

Разовое измерение часто обманчиво. Разброс может достигать 10-20% из-за кэша, планировщика, энергосбережения CPU (DVFS), thermal throttling. Поэтому рекомендуется:

- Использовать `-count=10` (или больше) и анализировать распределение результатов с помощью утилиты `benchstat` (golang.org/x/perf/cmd/benchstat).
- Прогревать процессор (несколько секунд холостого выполнения) перед запуском, либо пользоваться изоляцией (cgroups, taskset).
- Отключать Turbo Boost и регулировать governor производительности (`cpupower frequency-set -g performance`) на выделенных машинах для стабильности.
- Не смешивать бенчмарки разных версий кода в одном запуске без явной изоляции.

Подробнее в [[6. Стабилизация результатов]].

## Mechanical Sympathy: влияние окружения на бенчмарк

Даже идеально написанный бенчмарк исполняется на реальном железе. Кэш-промахи, false sharing, TLB-промахи влияют на дисперсию. Например, если параллельный бенчмарк случайно размещает счётчики в одной кэш-линии ([[8. False sharing]]), результат окажется пессимистичным и нехарактерным для production-кода, где структуры выровнены. Инструменты уровня ОС (`perf stat`) помогают детектировать такие артефакты (см. [[5. Mechanical sympathy в backend разработке]]).

Кроме того, стандартный бенчмарк Go не изолирует GC. Если тестируемая функция активно аллоцирует, GC может включиться в середине измерения, увеличив среднее время и дисперсию. Флаг `-gcflags="-N -l"` отключает оптимизации, что может потребоваться для отладки, но для production-замеров оптимизации оставляют включёнными.

## Бенчмарк в CI/CD: защита от регрессий

Регулярный запуск бенчмарков в пайплайне с сохранением результатов и сравнением через `benchstat` позволяет автоматически отлавливать падения производительности ([[5. Performance regression detection]]). Для этого необходимо:

1. Хранить эталонные результаты в репозитории (например, `testdata/benchmarks.txt`).
2. В CI выполнять `go test -bench=. -count=10 > new.txt`.
3. Сравнивать `benchstat old.txt new.txt`.
4. Блокировать PR при статистически значимом ухудшении.

Это дисциплинирует разработчиков и не даёт производительности деградировать незаметно.

## Итог

- `go test -bench` — мощный стандартный инструмент с адаптивным подбором числа итераций, метриками памяти и параллельным выполнением.
- Таймеры (`ResetTimer`, `StopTimer/StartTimer`) позволяют исключить из измерений подготовительные операции.
- `b.RunParallel` корректно измеряет конкурентный код, а `b.Run` структурирует сравнение вариантов.
- Обязательно использовать `-benchmem` и избегать компиляторной элиминации кода.
- Стабильные измерения требуют `-count`, фиксированного окружения и анализа через `benchstat`.
- Бенчмарки должны стать частью CI/CD для предотвращения регрессий.

Вооружённые этим инструментом, мы можем перейти к его внутреннему устройству: [[3. go test -bench под капотом]], где разберём, как фреймворк адаптирует `b.N`, работает с таймерами и взаимодействует с планировщиком Go.
