## От «где память» к «почему она там оказалась»

В [[2. Heap vs stack]] мы разделили память на быстрый стек и управляемую GC кучу. В [[3. Escape analysis]] научились понимать, почему компилятор отправляет переменные в кучу. Но знать причины недостаточно — нужны точные цифры: сколько аллокаций совершается в конкретной функции, какого они размера, и как это влияет на производительность. Для этого существует **allocation profiling** — профилирование аллокаций.

Allocation profiling — это мост между теорией escape analysis и практической оптимизацией. Он не просто показывает, сколько памяти занято *сейчас* (это inuse profile), а фиксирует *каждый факт выделения* за время наблюдения. Это делает его незаменимым инструментом для поиска источников мусора, нагружающего GC, и для проверки гипотез после изменения кода.

Эта статья вооружит вас методикой сбора, чтения и интерпретации профиля аллокаций, свяжет с микроархитектурными последствиями и подготовит к детальному анализу через [[5. pprof memory profile]].

## Два взгляда на память: alloc_space против inuse_space

Пакет `runtime/pprof` предоставляет два принципиально разных среза памяти:

- **alloc_space** — суммарный объём байт, которые были выделены в куче с момента старта (или с начала профилирования). Учитывает даже те объекты, которые уже освобождены и собраны GC. Это «общий тоннаж» аллокаций.
- **inuse_space** — объём памяти, занятой живыми объектами в момент снятия профиля. Это «текущее население» кучи.

Для поиска горячих точек по аллокациям абсолютным приоритетом пользуется **alloc_space**. Именно он показывает, какие функции генерируют больше всего работы для аллокатора и GC. Inuse_space полезен для поиска утечек ([[6. Утечки памяти]]) и оценки фрагментации ([[7. Fragmentation]]).

> [!tip] Собеседование
> **Вопрос:** Чем отличаются alloc_space и inuse_space, и когда какой использовать?
> **Ответ:** alloc_space показывает все байты, выделенные за период, включая уже освобождённые. Он нужен для поиска функций, генерирующих максимальную нагрузку на аллокатор и GC. inuse_space показывает живые объекты — используется для диагностики утечек и оценки пикового потребления памяти.

## Как собрать профиль аллокаций

### 1. Через `net/http/pprof`

Самый простой способ для долгоживущего сервера. Импортируем `_ "net/http/pprof"` и выполняем HTTP-запрос:

```bash
curl -o alloc.prof "http://localhost:6060/debug/pprof/heap?seconds=30"
```

Параметр `seconds=30` предписывает профилировщику отслеживать аллокации в течение 30 секунд. Без него будет снят мгновенный inuse-профиль, что не подходит для анализа аллокационной активности.

Важно: при запросе с `seconds=N` перед стартом и после остановки принудительно вызывается `runtime.GC()`. Это помогает исключить шум от временных объектов, но всё же аллокации, сделанные за период, фиксируются полностью.

Эндпоинт `/debug/pprof/heap` поддерживает параметры:
- `?seconds=N` — сбор в течение N секунд (alloc_space).
- `?debug=1` — текстовый дамп.
- `?gc=1` — принудительный GC перед снятием.

### 2. Через `runtime/pprof` в коде

Для скриптов, CLI-утилит или тестов можно явно написать сохранение профиля:

```go
f, _ := os.Create("alloc.prof")
pprof.Lookup("heap").WriteTo(f, 0)
```

Но перед этим необходимо настроить частоту сэмплирования и, возможно, вызвать `runtime.GC()`. Такой подход даёт мгновенный inuse-профиль, а не накопленный за период. Для записи накопленных аллокаций за интервал нужно вручную сбрасывать и читать счётчики через `runtime.ReadMemStats`, что сложнее. Проще использовать HTTP-эндпоинт с `seconds`.

### 3. Через `go test -bench` и `-memprofile`

При запуске бенчмарков:

```bash
go test -bench=. -benchmem -memprofile=mem.prof
```

Профиль `mem.prof` будет содержать два сэмпла: `alloc_space` и `inuse_space`. Открыв его через `go tool pprof`, можно переключаться между ними.

## Параметры сбора: `MemProfileRate`

По умолчанию профилировщик памяти записывает информацию о каждом 512-м килобайте выделений (`runtime.MemProfileRate = 512 * 1024`). Это означает, что сэмплирование **не точное**, а статистическое: мелкие аллокации могут быть не учтены, если их суммарный объём не достигает порога.

Частоту можно изменить через переменную окружения:

```bash
GODEBUG=memprofilerate=1 go test -bench=.
```

Или в коде:

```go
runtime.MemProfileRate = 1
```

Установка в 1 означает «записывать каждую аллокацию». Это даёт максимальную точность, но резко замедляет программу и увеличивает размер профиля (профилирование может замедлить процесс в 10–100 раз). Используется только для локальной отладки.

Для production-сборок обычно оставляют значение по умолчанию или слегка снижают до 131072 (128 КБ), чтобы получить более плотное покрытие без фатального overhead.

> [!warning] Ловушка / Gotcha
> При стандартном `MemProfileRate=512KB` функции, выделяющие много мелких объектов (например, 100 тысяч объектов по 16 байт = 1.6 МБ), будут видны в профиле хорошо, потому что сумма превышает порог. Но функция, редко выделяющая единичные мелкие объекты, может быть пропущена. Профиль показывает статистически значимые источники.

## Анализ профиля: `go tool pprof`

Открываем профиль, полученный за 30 секунд:

```bash
go tool pprof -alloc_space alloc.prof
```

Ключевые команды:

- **`top`** — показывает функции, в которых было выделено больше всего памяти. По умолчанию сортировка по `flat` (память, выделенная непосредственно в теле функции). `top -cum` — по кумулятивной (включая потомков).
- **`list FuncName`** — аннотированный исходный код с объёмами аллокаций на каждой строке.
- **`tree`** — иерархическое представление, показывающее, кто кого вызывает и сколько памяти аллоцирует.
- **`web`** — граф вызовов.
- **`flamegraph`** через веб-интерфейс (`-http=:8080`) — визуализация alloc_space в виде пламени (аналогично [[3. Flamegraph]], но для памяти).

Пример вывода `top`:

```
(pprof) top
Showing nodes accounting for 1.50GB, 85% of 1.76GB total
      flat  flat%   sum%        cum   cum%
    0.60GB 34.09% 34.09%     0.80GB 45.45%  encoding/json.Unmarshal
    0.30GB 17.05% 51.14%     0.30GB 17.05%  runtime.makeslice
    0.20GB 11.36% 62.50%     0.20GB 11.36%  bytes.makeSlice
    0.15GB  8.52% 71.02%     0.15GB  8.52%  strings.Builder.grow
```

Здесь `encoding/json.Unmarshal` — главный «потребитель» аллокаций: 0.60 ГБ выделено непосредственно в ней, а с учётом вызываемых функций (например, `runtime.makeslice`) — 0.80 ГБ. Это прямое указание на то, что парсинг JSON — точка приложения усилий.

## Как ходить от профиля к коду

1. Найти функцию с наибольшим `flat` или `cum` в `alloc_space`.
2. Выполнить `list FuncName` и увидеть, на какой строке сколько памяти выделено.
3. Сопоставить с исходным кодом. Если строка содержит `make`, `append`, `new`, вызов интерфейса — именно здесь аллокация.
4. Проверить через `-gcflags="-m"`, какие переменные убегают ([[3. Escape analysis]]).
5. Применить соответствующий приём: предвыделение, переиспользование, `sync.Pool`, замена интерфейса на generic, value вместо pointer.
6. Снять профиль повторно и убедиться в снижении `alloc_space`.

> [!info] Под капотом
> Профиль памяти в Go — это не непрерывная запись, а серия сэмплов, сохраняемая в хеш-таблице внутри рантайма. Каждый сэмпл содержит стек вызовов и объём аллокаций. При записи профиля в protobuf эти сэмплы агрегируются. Именно поэтому в `top` мы видим суммы, а не отдельные события. Также важно: профиль записывает только аллокации, прошедшие через `mallocgc`. Стековые переменные в профиль не попадают.

## Типичные «горячие» источники аллокаций и стратегии борьбы

Повторяя классификацию из [[4. Top functions анализ]], но для памяти:

### 1. Сериализация / десериализация
`json.Unmarshal`, `xml.Decode`, `protobuf.Marshal` часто порождают множество временных слайсов и строк. Решения: альтернативные библиотеки (sonic, easyjson), переиспользование буферов, `sync.Pool` ([[2. sync Pool]]).

### 2. Конструирование строк
`strings.Builder`, конкатенация через `+` в цикле, `fmt.Sprintf`. `Builder` с `Grow` снижает аллокации до одной; `+` в цикле — множество.

### 3. Рост слайсов
`append` в слайс с неизвестной ёмкостью. Профиль покажет `runtime.growslice` и `runtime.makeslice`. Решение: предвыделение `make([]T, 0, cap)` ([[4. Предвыделение памяти]]).

### 4. Преобразования между типами
`[]byte(s)` для строки копирует данные в новый слайс. Это может быть неожиданным источником аллокаций. Использовать `unsafe` не всегда оправдано, но для горячих путей существуют подходы zero-copy ([[5. Zero copy подходы]]).

### 5. Конвертация в интерфейс
Передача конкретного типа в `interface{}` боксит значение. Если это происходит в цикле — аллокации накапливаются. Использовать generics или пересмотреть API.

### 6. Захват переменных в замыканиях
Горутины и отложенные функции с замыканиями могут уводить переменные в кучу. Профиль покажет аллокации в месте объявления замыкания.

## Практический пример: оптимизация JSON парсинга

Исходный код обработчика:

```go
func handle(w http.ResponseWriter, r *http.Request) {
    var req Request
    json.NewDecoder(r.Body).Decode(&req)
    // обработка req
    fmt.Fprint(w, "OK")
}
```

Профиль `alloc_space` за 30 секунд под нагрузкой показывает:

```
       flat  flat%   sum%        cum   cum%
   200.00MB 40.00% 40.00%   250.00MB 50.00%  encoding/json.(*Decoder).Decode
    50.00MB 10.00% 50.00%    50.00MB 10.00%  runtime.makeslice
```

Вывод: `Decode` генерирует 200 МБ аллокаций. Смотрим `list Decode`, видим, что внутри много `make` для строк и слайсов. Также видно, что `json.Decoder` использует буфер, который растёт с каждым запросом, если не переиспользовать `Decoder`.

Оптимизация: переиспользуем `Decoder` через `sync.Pool` и предвыделяем буфер:

```go
var decoderPool = sync.Pool{
    New: func() interface{} { return json.NewDecoder(nil) },
}

func handleOptimized(w http.ResponseWriter, r *http.Request) {
    dec := decoderPool.Get().(*json.Decoder)
    dec.Reset(r.Body)
    defer decoderPool.Put(dec)
    var req Request
    dec.Decode(&req)
    fmt.Fprint(w, "OK")
}
```

Повторный профиль показывает снижение `alloc_space` на 40%, а `benchmem` фиксирует падение `allocs/op` с 25 до 8. Это напрямую снижает нагрузку на GC и улучшает p99 latency ([[7. Tail latency и почему она важна]]).

## Ловушки и частые ошибки

> [!warning] Ловушка / Gotcha
> **Inuse_space вместо alloc_space.** Разработчик открывает `go tool pprof` без параметра `-alloc_space` и видит текущую живую память. Функция, которая генерирует много мусора, но быстро освобождает, будет почти невидима. Всегда начинайте с `-alloc_space`.

> [!warning] Ловушка / Gotcha
> **Слишком короткий профиль.** Как и с CPU ([[2. CPU profiling в Go]]), 5 секунд аллокационного профиля могут исказить картину из-за флуктуаций нагрузки. Минимум 30 секунд.

> [!warning] Ловушка / Gotcha
> **Игнорирование `MemProfileRate`.** На production-сервере с настройками по умолчанию вы можете не увидеть функции, которые аллоцируют много мелких объектов, но суммарно не дотягивают до 512 КБ на сэмпл. Это искажает картину. Если вы охотитесь на конкретный участок, временно уменьшайте `MemProfileRate`.

> [!warning] Ловушка / Gotcha
> **Скрытые аллокации в стандартной библиотеке.** Например, `time.Parse` или `net/http` аллоцируют внутренние структуры. Профиль честно покажет их, но не всегда очевидно, что вызов идёт из вашего кода. Используйте `tree` для трассировки.

## Mechanical Sympathy: как аллокации влияют на кэш и TLB

Каждая аллокация в куче — это не просто вызов `malloc`. Это:
- Поиск свободного спана (возможно, блокировка на `mcentral`).
- Обнуление памяти (через `memclr`, которая, правда, использует SIMD и быстра).
- Добавление объекта в граф GC.
- Потенциальное вымывание кэш-линий при последующем доступе к объекту.
- Рост давления на TLB, если объекты разбросаны по новым страницам.

Профиль аллокаций помогает не только снизить GC-паузы ([[6. GC pause и latency]]), но и улучшить cache friendliness ([[8. Cache friendliness]]), уменьшив количество холодных объектов.

## Итог

- **Allocation profiling** через `pprof` с `-alloc_space` — основной инструмент для поиска функций, генерирующих больше всего кучевых выделений.
- Он фиксирует суммарный объём аллокаций за период, в отличие от `inuse_space`, показывающего текущую живую память.
- Сбор профиля выполняется через HTTP (`/debug/pprof/heap?seconds=N`), флаги `go test` или прямой вызов `runtime/pprof`.
- Частота сэмплирования (`MemProfileRate`) по умолчанию 512 КБ; для точного анализа её можно снизить, но с осторожностью.
- Анализ `top -cum`, `list` и `tree` позволяет пройти от функции-лидера к конкретной строке, вызывающей аллокации.
- Типичные источники: JSON/XML, строки, рост слайсов, боксинг в интерфейсы, замыкания.
- Оптимизация, направляемая профилем аллокаций, снижает нагрузку на GC, улучшает latency и throughput.
