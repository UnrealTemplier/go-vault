## Почему компилятор решает, где жить переменной

В [[2. Heap vs stack]] мы установили фундаментальное различие: стек быстр и автоматически очищается, куча — медленна и нагружает GC. Но разработчик в Go не размещает переменные явно. Решение принимает **escape analysis** (анализ убегания) — статический анализ кода на этапе компиляции, который определяет, может ли переменная быть использована за пределами своего фрейма, и если да — отправляет её в кучу.

Понимание escape analysis критически важно для Senior Go-инженера: оно объясняет, почему совершенно безобидный на вид код генерирует гигабайты мусора, и даёт рычаг управления без отказа от идиоматичности. В этой статье мы разберём, как именно компилятор принимает решение, как читать его отчёты, как влиять на «убегание» и как проверять результаты с помощью профилировщика памяти ([[5. pprof memory profile]], [[4. Allocation profiling]]).

## Что такое Escape Analysis

Escape analysis — это статический анализ потока указателей, выполняемый компилятором Go. Его задача: для каждой переменной с типом, содержащим указатели, или для составных типов (слайсы, мапы, интерфейсы) определить, может ли указатель на неё (или она сама) «убежать» из текущей области видимости. Если да — переменная размещается в куче. Если нет — она остаётся на стеке.

Анализ консервативен: если компилятор не может строго доказать, что переменная не убегает, она отправляется в кучу. Поэтому часть переменных, которые фактически никогда не используются вне функции, всё равно могут быть размещены в куче из-за ограничений анализа.

Escape analysis работает на уровне SSA-представления внутри компилятора (после преобразования AST в SSA). Он проходит по всем функциям пакета и строит граф потока указателей. Подробнее «под капотом» будет ниже.

## Как компилятор решает: основные критерии убегания

Любая операция, которая делает указатель на переменную доступным «внешнему миру», вызывает её эвакуацию в кучу. Перечислим главные триггеры.

### 1. Возврат указателя из функции

Классика:

```go
func newInt() *int {
    x := 42
    return &x // &x уходит за пределы фрейма → x в куче
}
```

### 2. Отправка в канал

Значение, посланное в канал, может быть прочитано другой горутиной, которая живёт дольше текущей функции:

```go
ch := make(chan *int)
x := 42
ch <- &x // убегает
```

Даже если канал небуферизированный и приём происходит немедленно, компилятор не анализирует динамику, только факт передачи в канал.

### 3. Сохранение в глобальную переменную или поле структуры, которая переживёт функцию

```go
var global *int

func f() {
    x := 1
    global = &x // убегает, т.к. global переживёт f
}
```

Аналогично присваивание указателя в поле слайса, мапы или интерфейса, который возвращается или хранится глобально.

### 4. Вызов методов через интерфейс

Если переменная конкретного типа передаётся в функцию, принимающую интерфейс, происходит **боксинг** — значение копируется в кучу, чтобы быть представленным как интерфейс. Даже если функция не сохраняет значение, факт приведения к интерфейсу вызывает аллокацию (в ранних версиях Go, сейчас есть оптимизации, но общий принцип: `interface{}` может вызвать убегание).

```go
func print(v interface{}) { fmt.Println(v) }
x := 42
print(x) // x уходит в кучу? зависит от escape-анализа fmt.Println
```

`fmt.Println` принимает `...interface{}`, это заставляет аргументы «упаковываться» в интерфейсы. Компилятор может определить, что внутри `Println` значение не сохраняется, и всё же оставить его на стеке (поздние версии Go умеют такое), но часто всё равно происходит эвакуация.

### 5. Захват переменной замыканием

Горутина или анонимная функция, переживающая текущий фрейм, заставляет все захваченные переменные уйти в кучу:

```go
func start() {
    x := 0
    go func() { fmt.Println(x) }() // x убегает
}
```

Даже если горутина завершится до выхода из `start`, компилятор не знает порядок выполнения — он видит захват и эвакуирует.

### 6. Хранение в слайсе или мапе, которые могут пережить фрейм

Если слайс или мапа создаются локально, но затем возвращаются или присваиваются внешней структуре, все их элементы могут убежать (хотя компилятор может отслеживать, какие конкретно). Но в общем случае, если слайс «убегает», его backing array выделяется в куче.

### 7. Переменные в циклах, на которые берутся указатели

Компилятор анализирует, берётся ли указатель на переменную внутри цикла, и если да, то переменная может оказаться недоступной после итерации (раньше типичная ошибка с `for range`). Но сейчас итерационные переменные в `for range` с Go 1.22 имеют per-iteration, что изменило поведение. Однако для классического `for` с указателем может убегать:

```go
var ptrs []*int
for i := 0; i < 10; i++ {
    x := i
    ptrs = append(ptrs, &x) // x убегает, т.к. слайс ptrs переживёт цикл
}
```

## Как увидеть решения escape analysis

Компилятор Go сообщает о принятых решениях при флаге `-m` (и более детально с `-m -m`):

```bash
go build -gcflags="-m" .
```

Вывод:

```
./main.go:7:6: can inline newInt
./main.go:8:9: &x escapes to heap
./main.go:7:13: moved to heap: x
```

Первая строка говорит, что функция может быть инлайнена. Вторая — что указатель `&x` убегает. Третья — что сама переменная `x` перемещена в кучу.

Если переменная не убегает, мы увидим:

```
./main.go:15:3: x does not escape
```

Полезно применять к бенчмаркам или конкретным функциям, чтобы понять, почему аллокации высоки.

## Escape analysis под капотом

> [!info] Под капотом
> Escape analysis в Go реализован в пакете `cmd/compile/internal/escape`. Он работает на SSA-графе, но имеет собственный проход, который строит граф потока данных для указателей. Каждому выражению присваивается уровень "location" (stack или heap). Алгоритм использует абстрактную интерпретацию: он помечает узлы графа, если на них есть ссылки из "escape points" (точки убегания). Основные escape points: возвращаемое значение, аргументы функций с пометкой `//go:noescape` или без, глобальные переменные, каналы. Анализ консервативен и не отслеживает динамическое поведение (поток управления через горутины, динамические типы). С каждой новой версией Go анализ становится умнее: например, раньше `for range` циклы вызывали убегание переменной-итератора, сейчас (с 1.22) переменная на каждой итерации новая и не убегает. Также улучшилась обработка интерфейсов: если компилятор видит, что значение не покидает вызов, он может разместить его на стеке.

## Практические приёмы управления escape analysis

Senior может влиять на решения компилятора, проектируя API и структуры данных так, чтобы горячие переменные оставались на стеке.

### 1. Избегайте возврата указателей на локальные переменные, если это не необходимо

Если структура небольшая, возвращайте её по значению. Компилятор может разместить её в стеке вызывающей функции (frame pointer) и избежать кучи. Сравните:

```go
// убегает
func makeConfig() *Config { return &Config{Timeout: 10} }

// не убегает, если Config <= 64-128 байт
func makeConfig() Config { return Config{Timeout: 10} }
```

### 2. Передавайте по значению, а не по указателю, когда это уместно

Если функция не модифицирует структуру и структура невелика, передача по значению исключает необходимость в указателе и, возможно, позволит компилятору оставить её на стеке.

```go
func process(cfg Config) { ... }
```

### 3. Избегайте интерфейсов в горячих вызовах

Там, где нужна полиморфная обработка, и производительность критична, используйте generics (с Go 1.18+). Generics мономорфизируются на этапе компиляции, что позволяет избежать боксинга и кучи.

### 4. Не захватывайте переменные горутинами без необходимости

Передавайте значения через аргументы горутины — это может позволить переменной остаться на стеке:

```go
x := 42
go func(v int) { fmt.Println(v) }(x) // x не обязательно убежит
```

### 5. Используйте `//go:noinline` только для отладки

Иногда разработчики запрещают инлайнинг, чтобы заставить escape analysis держать переменную на стеке (инлайн мог бы раскрыть убегание). Но это хрупко и не рекомендуется для production.

### 6. Следите за размером структур

Компилятор может решить, что большую структуру дешевле скопировать в кучу, чем держать на стеке (из-за роста стека). Это эвристика, но общее правило: для структур > 1 КБ есть риск убегания; проверяйте.

## Проверка результата: профили памяти и бенчмарки

После изменения кода обязательно проверьте фактические аллокации через memory profile ([[5. pprof memory profile]]) или `go test -benchmem`. Например:

```go
func BenchmarkProcess(b *testing.B) {
    for i := 0; i < b.N; i++ {
        _ = process()
    }
}
```

Запуск:

```bash
go test -bench=. -benchmem
```

Если `allocs/op` и `B/op` снизились, вы на верном пути. Если нет — вернитесь к `-gcflags="-m"` и ищите неожиданные убегания.

## Типичные ловушки и неочевидные случаи

> [!warning] Ловушка / Gotcha
> **Сборка слайса с возвратом.** Если функция собирает слайс и возвращает его, слайс (заголовок) может быть на стеке, но его backing array всегда в куче. Это нормально. Но если вы внутри цикла добавляете элементы в слайс, который потом возвращается, все элементы в куче. Убегания элементов по отдельности не избежать, но можно снизить общее число аллокаций, предвыделяя ёмкость ([[4. Предвыделение памяти]]).

> [!warning] Ловушка / Gotcha
> **Передача `...interface{}`.** Функции вроде `fmt.Printf` принимают `interface{}`. Каждый аргумент, даже простой `int`, может быть упакован в интерфейс и аллоцирован. Если вы логируете в горячем пути, используйте structured logging без боксинга (например, `slog` с типизированными методами) или отложенную оценку.

> [!warning] Ловушка / Gotcha
> **Скрытые указатели.** Строки и слайсы содержат указатели (на начало данных). Если вы возвращаете структуру с полем `string`, а строка формируется из локального слайса, это может привести к аллокации строки в куче. Компилятор часто выделяет строку на стеке, но зависит от контекста.

> [!warning] Ловушка / Gotcha
> **Escape analysis не меж-пакетный.** Если функция пакета A возвращает указатель на локальную переменную, то в пакете B переменная уже в куче (при вызове A). Компилятор не может отменить это решение. Поэтому проектируйте API так, чтобы избегать возврата указателей на внутренние данные без нужды.

## Mechanical Sympathy: почему убегание вредит

Каждая переменная, ушедшая в кучу, не только требует аллокации (которая может быть быстрой), но и становится частью графа GC, что приводит к:
- Дополнительным затратам CPU на сканирование и маркировку ([[2. Tri color marking]]);
- Записям write barrier при каждом обновлении указателя на неё ([[5. Write barriers]]);
- Разбросу адресов, ведущему к кэш-промахам и TLB-промахам ([[1. Memory model Go]], [[8. Cache friendliness]]).

Стековые же данные линейны, плотно упакованы и автоматически освобождаются. Поэтому разумное управление escape analysis — это прямой путь к снижению latency и повышению throughput.

## Итог

- **Escape analysis** — статический анализ компилятора, решающий, может ли переменная быть использована вне текущего фрейма.
- Переменные убегают при возврате указателя, отправке в канал, сохранении в глобальные объекты, захвате замыканием, боксинге в интерфейс.
- Узнать о решениях можно с помощью `go build -gcflags="-m"`.
- Избегая ненужных указателей, интерфейсов в горячих путях и возврата указателей из функций, мы помогаем компилятору оставлять переменные на стеке.
- Убегание напрямую влияет на производительность через аллокации, GC-нагрузку и кэш-промахи.
- Проверять эффективность изменений следует через `-benchmem` и memory profile.

Теперь, понимая, как переменные распределяются между стеком и кучей, мы переходим к инструментальному анализу аллокаций: [[4. Allocation profiling]] — где детально рассмотрим, как снимать профили памяти и интерпретировать распределение объектов.
