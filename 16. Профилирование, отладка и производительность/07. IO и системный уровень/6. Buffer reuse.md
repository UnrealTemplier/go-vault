## Почему каждый `make([]byte, 4096)` — это будущая пауза GC

В [[5. Zero copy подходы]] мы увидели, как исключить копирование данных между ядром и userspace. Но даже если передача идёт через `sendfile` или `splice`, в реальном приложении всегда остаются буферы, которые приходится выделять: временные слайсы для парсинга, буферы чтения/записи, накопительные строки. Каждый вызов `make([]byte, size)` порождает аллокацию в куче, которая попадает под наблюдение GC. При интенсивном вводе-выводе — тысячи запросов в секунду — это превращается в поток мусора, который заставляет сборщик мусора работать чаще и дольше ([[9. Когда GC становится bottleneck]]).

**Buffer reuse** (переиспользование буферов) — это стратегия контроля аллокаций через возврат уже выделенной памяти в пул вместо освобождения и повторного выделения. Правильно реализованное переиспользование сокращает темп аллокаций на 50–90%, снижает нагрузку на GC на десятки процентов, а заодно улучшает пространственную локальность ([[8. Cache friendliness]]), потому что горячие буферы остаются в кэше процессора.

В этой статье мы систематизируем все техники переиспользования буферов, доступные в Go: от всем известного `sync.Pool` ([[2. sync Pool]]) до ручных локальных пулов и арен, разберём их стоимость, влияние на GC и механическую эмпатию. Мы свяжем это с [[1. Уменьшение аллокаций]], [[4. Предвыделение памяти]] и подготовимся к оптимизации файлового ввода-вывода ([[7. File IO оптимизация]]).

## Цена одного буфера: почему `make` — это дорого

Каждый `make([]byte, 4096)`:

1. Вызывает `runtime.mallocgc` — аллокацию в куче ([[4. Allocation profiling]]).
2. Обнуляет память (если не указан capacity с существующими данными) — `memclr`, который хоть и использует SIMD ([[7. SIMD и Go]]), всё равно стоит тактов.
3. Добавляет объект в граф GC ([[2. Tri color marking]]), который будет участвовать в mark-фазе и, возможно, вызовет mark assist ([[9. Когда GC становится bottleneck]]).
4. После освобождения — попадает в sweep, который должен вернуть спан в пул аллокатора. Если спад частых аллокаций, это ложится на scheduler GC.

При 10 000 запросов в секунду, где на каждый запрос выделяется 4 КБ буфер + ещё пара временных слайсов, общий темп аллокаций может составлять 100 МБ/с. Это полностью выматывает GC: он запускается каждые сотни миллисекунд, mark assist отъедает CPU, p99 latency штормит.

Переиспользование буферов через пулы решает эту проблему: буфер выделяется однажды, используется многократно, и GC больше не видит его (или видит значительно реже, в случае `sync.Pool`).

## sync.Pool: быстрый пул для временных объектов

`sync.Pool` — стандартный механизм Go для переиспользования временных объектов, которые могут совместно использоваться горутинами. Он предназначен для объектов, которые **не должны сохраняться между GC-циклами**: GC может очищать пул, удаляя неиспользуемые объекты.

### Внутреннее устройство sync.Pool

> [!info] Под капотом
> `sync.Pool` реализован в `sync/pool.go`. Он не содержит централизованного списка. Вместо этого объекты распределены по **локальным кэшам P** (логических процессоров). Это позволяет горутине, работающей на конкретном P, брать и класть объекты без блокировок — только атомарные операции. Если локальный кэш пуст, `sync.Pool` пытается «украсть» объект у другого P (аналогично work stealing, [[3. Work stealing]]). Если и это не удаётся, вызывается `New`-функция, создающая новый объект.
> 
> Важнейший аспект: при каждом GC-цикле все объекты из локальных кэшей `sync.Pool` могут быть удалены. Это сделано для предотвращения утечек памяти. Рантайм вызывает `poolCleanup`, который обнуляет все ссылки на объекты. Поэтому `sync.Pool` не подходит для долгоживущих объектов — только для переиспользования между запросами в пределах одного GC-цикла.

### Базовое применение sync.Pool для буфера

```go
var bufferPool = sync.Pool{
    New: func() interface{} {
        return make([]byte, 0, 4096)
    },
}

func handler(w http.ResponseWriter, r *http.Request) {
    buf := bufferPool.Get().([]byte)
    defer bufferPool.Put(buf[:0]) // сбрасываем длину, сохраняя capacity

    // использование buf как накопительного буфера
    buf = append(buf, "some data"...)
    ...
}
```

Важные нюансы:
- При возврате в пул сбрасываем длину (`buf[:0]`), сохраняя ёмкость. Это позволяет следующему пользователю сразу писать в существующий массив.
- Обязательно нужно привести тип после `Get()`, так как пул хранит `interface{}`.
- Нельзя изменять capacity слайса после `Put` (не дозаполнять), иначе данные могут утечь между запросами.

### Влияние sync.Pool на GC и latency

Преимущества:
- Радикально снижается количество аллокаций: `Get` часто обходится без вызова `make`.
- GC не видит объекты в пуле, пока они не вычищены. Это уменьшает размер живого набора и ускоряет mark-фазу.

Недостатки:
- Если буфер очень долго лежит в пуле, он может «протухнуть»: его память будет вытеснена из кэша процессора, и при `Get` будет cache miss. Это всё равно быстрее, чем аллокация, но не идеально.
- GC периодически очищает пул. Это может породить всплеск аллокаций сразу после GC, когда все пулы опустошены, и следующие запросы вынуждены вызывать `New`. Это явление называется **pool cache invalidation** и может создавать периодические всплески latency.

## Локальные пулы и arena-аллокаторы

Когда `sync.Pool` недостаточен или его поведение (очистка GC) неприемлемо, строят ручные пулы.

### Локальный пул на основе канала

```go
type Pool struct {
    ch chan []byte
}

func NewPool(size, cap int) *Pool {
    p := &Pool{ch: make(chan []byte, size)}
    for i := 0; i < size; i++ {
        p.ch <- make([]byte, 0, cap)
    }
    return p
}

func (p *Pool) Get() []byte {
    return <-p.ch
}

func (p *Pool) Put(b []byte) {
    select {
    case p.ch <- b:
    default:
        // пул заполнен, отбрасываем
    }
}
```

Этот подход:
- Не зависит от GC, буферы живут до тех пор, пока не будут вытеснены из пула.
- Позволяет контролировать максимальное количество буферов, предотвращая взрывной рост потребления памяти.
- Добавляет накладные расходы на работу с каналом ([[5. Sync primitives и их стоимость]]), но при нечастых операциях это приемлемо.

### Arena-аллокаторы (slab allocator)

Для очень горячих участков можно использовать технику «арены»: один крупный буфер выделяется заранее, и внутри него нарезаются куски для обработки запросов. В Go это можно реализовать через `sync.Pool`, хранящий большой слайс и атомарный счётчик смещения.

```go
type Arena struct {
    data []byte
    off  int64
}

func NewArena(size int) *Arena {
    return &Arena{data: make([]byte, size)}
}

func (a *Arena) Alloc(n int) []byte {
    off := atomic.AddInt64(&a.off, int64(n))
    if off > int64(len(a.data)) {
        return nil // арена исчерпана
    }
    return a.data[off-int64(n) : off]
}
```

Недостаток: сложно управлять возвратом памяти (она возвращается только при сбросе всей арены). Используется в системах, где пиковая нагрузка предсказуема, а GC должен быть минимизирован (высокочастотный трейдинг, хранилища ключ-значение).

## Когда переиспользование буферов необходимо, а когда — вредно

### Обязательно переиспользовать

- Буферы для ввода-вывода, размер которых известен и повторяется (чтение блоков, сетевые пакеты, буферы `io.Copy`).
- Накопительные строки (`bytes.Buffer`, `strings.Builder` — но `Builder` сам по себе не аллоцирует до финального вызова `String()`, однако его внутренний слайс можно переиспользовать).
- Временные структуры (декодеры JSON, буферы для gRPC).

### Осторожно или не надо

- Буферы, которые модифицируются и передаются дальше без копирования: риск sharing и гонок.
- Очень маленькие буферы (< 64 байт): аллокация в Go для таких объектов часто обслуживается маленькими спанами и может быть быстрее, чем оверхед пула (CAS, интерфейсы). Нужны бенчмарки.
- Буферы, чей размер непредсказуем: если растить слайс, переиспользование может привести к удержанию огромного capacity, потребляющего память. Лучше ставить лимит на максимальный размер буфера в пуле.

> [!warning] Ловушка / Gotcha
> **Забыли сбросить длину перед Put.** Если положить в пул слайс с `len > 0`, следующий `Get` может вернуть «непустой» буфер с чужими данными. Это может быть как утечкой конфиденциальных данных между запросами, так и логической ошибкой. Всегда `Put(buf[:0])`.

> [!warning] Ловушка / Gotcha
> **Захват буфера горутиной без возврата.** Если горутина берёт буфер из пула и уходит в долгий вызов (например, блокирующий ввод-вывод), а буфер не возвращает, пул может опустошиться. Для таких сценариев лучше использовать пул с блокировкой или арен.

## Buffer reuse и Mechanical Sympathy

Переиспользование буферов положительно влияет на процессор и память не только через GC, но и через кэш:

- **Горячие буферы.** Если буфер постоянно используется на одном и том же P, он остаётся в локальном L1/L2 кэше. `sync.Pool` специально устроен так, чтобы Get/Put выполнялись на одном P, сохраняя локальность.
- **TLB и страницы.** Однажды выделенный буфер занимает определённые страницы памяти. При переиспользовании эти страницы остаются в TLB, ускоряя доступ к ним. При постоянном пересоздании буферов страницы могут меняться, вызывая TLB-промахи.
- **Пространственная локальность.** Если в одном запросе используются несколько мелких буферов, их можно объединить в один большой (aggregate buffer), снижая количество аллокаций и улучшая локальность.

Но есть и отрицательный эффект: если буфер долго не используется, его память всё же вытесняется из кэша; при извлечении из пула мы получаем cold cache. Это всё равно дешевле, чем аллокация + обнуление, но разница может быть невелика для больших буферов, не влезающих в L3.

## Инструменты измерения и проверки

- **`go test -benchmem`** — покажет изменение `allocs/op` и `B/op` после внедрения пула.
- **pprof memory profile** ([[5. pprof memory profile]]) с `-alloc_space` — до и после внедрения: должно значительно снизиться количество аллокаций в горячей функции.
- **CPU-профиль** — уменьшится доля `runtime.mallocgc`, `runtime.gcAssistAlloc`.
- **GODEBUG=gctrace=1** — реже и короче GC-циклы.

## Практические примеры

### 1. Переиспользование буфера чтения из сокета

```go
var connBufPool = sync.Pool{
    New: func() interface{} {
        return make([]byte, 32*1024)
    },
}

func handleConn(conn net.Conn) {
    buf := connBufPool.Get().([]byte)
    defer connBufPool.Put(buf)

    for {
        n, err := conn.Read(buf)
        if err != nil {
            break
        }
        process(buf[:n])
    }
}
```

### 2. bytes.Buffer в HTTP-обработчике

```go
var bufPool = sync.Pool{
    New: func() interface{} { return bytes.NewBuffer(make([]byte, 0, 4096)) },
}

func handler(w http.ResponseWriter, r *http.Request) {
    buf := bufPool.Get().(*bytes.Buffer)
    buf.Reset()
    defer bufPool.Put(buf)

    buf.WriteString("Response: ")
    data, _ := io.ReadAll(r.Body)
    buf.Write(data)
    w.Write(buf.Bytes())
}
```

Этот подход убирает аллокацию `bytes.Buffer` и его внутреннего слайса для каждого запроса.

## Итог

- **Переиспользование буферов** (buffer reuse) — фундаментальная техника снижения темпа аллокаций и нагрузки на GC при интенсивном вводе-выводе.
- `sync.Pool` — основной инструмент: быстрый, lock-free для локального P, но подверженный очистке на каждом GC-цикле.
- Для контроля ёмкости или сохранения между циклами GC применяются ручные пулы (каналы) или арены.
- Практические правила: всегда сбрасывать длину, избегать удержания гигантских буферов, не забывать о конкурентном доступе.
- Buffer reuse улучшает не только аллокационную картину, но и пространственную локальность, снижая cache и TLB промахи.
- Измерения через benchmem и pprof позволяют точно оценить выигрыш.

Теперь, освоив техники переиспользования буферов, мы готовы целенаправленно применить их к конкретной области — оптимизации файлового ввода-вывода, где накладные расходы на системные вызовы и копирование особенно высоки. Следующая статья: [[7. File IO оптимизация]].
