## Block Profile: измерение невидимых ожиданий

CPU-профиль ([[2. CPU profiling в Go]]) показывает, на что процессор тратит такты. Execution tracer ([[3. execution tracer]]) визуализирует временную шкалу событий рантайма. Дамп горутин ([[4. goroutine dump]]) даёт мгновенный срез состояний. Но ни один из них не даёт прямого ответа на вопрос: «Сколько времени горутины тратят на ожидание каналов, мьютексов, сетевых операций?». Ответ на него даёт **block profile** — профиль блокировок.

Block profile — это агрегированная статистика событий блокировки, в отличие от сэмплирующего CPU-профиля. Он записывает каждое (или каждое N-е) событие, когда горутина переходит в состояние ожидания ([[2. Goroutines под капотом]]) по причинам: отправка/получение из канала, захват мьютекса, сетевое ожидание, ожидание на `sync.Cond`. Он показывает не только *где* блокируется программа, но и *насколько долго*.

Senior-инженер использует block profile для поиска бутылочных горлышек конкурентности, которые не видны в загрузке CPU: медленных потребителей каналов, контеншна на мьютексах без видимого spinning'а, перекосов в пайплайнах. Это третий по важности профиль после CPU и памяти.

## Что измеряет block profile

Block profile регистрирует **события перехода в состояние ожидания** — моменты, когда горутина вызывает `gopark` и планировщик снимает её с P. Ключевые категории:

- **Каналы**: `chan send`, `chan receive`.
- **Select**: ожидание в `select` (агрегируется как `select`).
- **Мьютексы**: `sync.Mutex.Lock`, `sync.RWMutex.Lock/RUnlock` (только время блокировки, не путать с mutex profile, который измеряет contention — см. [[6. mutex profile]]).
- **Сеть**: ожидание на сокете через netpoller ([[4. epoll kqueue и netpoller]]).
- **Системные вызовы**: блокирующие syscall'ы (файловый ввод-вывод, `futex`).
- **Cond**: ожидание на `sync.Cond.Wait`.

Для каждого события записывается:
- Стек вызовов, приведший к блокировке.
- Длительность блокировки (от `gopark` до `goready`).

В отличие от mutex profile, который фокусируется на частоте contention и показывает, кто держал мьютекс, block profile показывает **время ожидания**, не различая, кто был виновником. Это делает block profile более общим инструментом для поиска любых узких мест синхронизации.

## Как включить и настроить block profile

По умолчанию block profile **отключён**. Включение требует установки частоты через `runtime.SetBlockProfileRate(rate int)` или через HTTP-эндпоинт `pprof`.

### Установка rate

Параметр `rate` — это делитель: записывается каждое N-е событие блокировки. `rate=1` означает «записывать всё», что даёт полную картину, но создаёт значительный overhead (особенно на каналах и сети). `rate=1000` означает «записывать каждое 1000-е событие», снижая overhead до пренебрежимого, но уменьшая точность.

Рекомендации:
- **Локальная отладка:** `rate=1`.
- **Production:** `rate=10000` для долгосрочного мониторинга, либо включать `rate=1` точечно на время диагностики.

### Способы включения

1. **Через HTTP pprof:**
   Импорт `_ "net/http/pprof"` и установка `runtime.SetBlockProfileRate(1)` в `init()` или `main`.

2. **В тестах и бенчмарках:**
   ```bash
   go test -blockprofile=block.out -bench=.
   ```
   При этом рантайм временно устанавливает `rate=1` на время теста.

3. **Программно:**
   ```go
   runtime.SetBlockProfileRate(1)
   ```

> [!warning] Ловушка / Gotcha
> `SetBlockProfileRate` не накапливает статистику задним числом. Она начинает запись с момента вызова. Если установить её после того, как программа уже отработала значительную часть, профиль будет неполным.

## Сбор и анализ профиля

### Сбор файла

Через HTTP:

```bash
curl -o block.prof "http://localhost:6060/debug/pprof/block"
```

Через тесты:

```bash
go test -blockprofile=block.out -bench=.
```

### Анализ через `go tool pprof`

```bash
go tool pprof block.prof
```

Внутри интерактивного режима:

- **`top`** — показывает функции с наибольшим суммарным временем блокировки. Значения — в наносекундах (или микросекундах, если профиль долгий).
- **`list FuncName`** — построчный разбор, на какой строке кода произошла блокировка и сколько времени заняла.
- **`web`** / **`-http=:8080`** — визуализация графа вызовов или flamegraph.

Пример вывода `top`:

```
(pprof) top
Showing nodes accounting for 5.50s, 85% of 6.47s total
      flat  flat%   sum%        cum   cum%
     2.00s 30.92% 30.92%      2.00s 30.92%  runtime.chanrecv1
     1.50s 23.19% 54.11%      1.50s 23.19%  runtime.chansend1
     1.00s 15.46% 69.57%      1.00s 15.46%  runtime.selectgo
     0.80s 12.36% 81.93%      0.80s 12.36%  runtime.netpollblock
     0.20s  3.09% 85.02%      0.20s  3.09%  sync.(*Mutex).Lock
```

Здесь видно:
- 2 секунды (суммарно) горутины провели в ожидании чтения из канала (`chanrecv1`).
- 1.5 секунды — в ожидании записи в канал (`chansend1`).
- 1 секунда — в `select`.
- 0.8 секунды — в ожидании сети (`netpollblock`).
- 0.2 секунды — ожидание мьютекса.

Команда `list FuncName` покажет точные строки кода:

```go
func consumer(ch chan int) {
    for {
        val := <-ch   // здесь блокируется (chanrecv1)
    }
}
```

> [!info] Под капотом
> Внутри `runtime` каждое событие блокировки обрабатывается функцией `gopark`. Перед ней вызывается `traceBlockReason` и, если block profile активен, `saveBlockEvent`. Эта функция получает текущий стек, записывает временную метку и при разблокировке (`goready`) вычисляет дельту. Если `rate > 1`, срабатывает алгоритм сэмплирования: счётчик событий инкрементируется, и запись происходит только когда `count % rate == 0`. Таким образом, блокировки не замедляются постоянными аллокациями под профиль — оверхед в основном от учёта времени.

## Интерпретация результатов: диагностика проблем

### 1. Высокое время в `chanrecv1` или `chansend1`

Если в топе `chanrecv1` с большим временем, значит, горутины-читатели простаивают. Это нормально для потребителей, ожидающих данные. Но если время аномально велико при стабильном потоке данных, возможно:

- Продюсеры не успевают генерировать данные (bottleneck на входе).
- Канал не буферизирован, и продюсеры ждут, пока потребитель освободится (это будет видно по `chansend1`).
- Горутина зависла из-за утечки (см. пример ниже).

### 2. Высокое время в `selectgo`

Если много времени в `select`, скорее всего, горутины ожидают множество каналов, и ни один из них не готов. Например, `select` с `ctx.Done()` и рабочим каналом, где контекст никогда не отменяется, а данные приходят редко.

### 3. Время в `netpollblock`

Большое время ожидания сети — норма для серверных горутин, ожидающих запросы. Но если это время составляет секунды, а p99 latency должна быть микросекундами — возможно, сетевые вызовы не имеют тайм-аутов, что приводит к накоплению ожидающих горутин (утечка горутин на сети).

### 4. Время в `sync.(*Mutex).Lock`

Block profile показывает время **ожидания** мьютекса, но не причину contention. Он покажет, что горутина долго не могла захватить мьютекс. Для выявления того, кто держал мьютекс, нужно использовать **mutex profile** ([[6. mutex profile]]), который записывает стеки удерживающих горутин. Block profile дополняет его, показывая общее время задержки.

### 5. Каскадные блокировки

Если горутина А блокируется на канале, а горутина Б, которая должна разблокировать А, сама заблокирована на мьютексе — block profile покажет два вклада, но не причинно-следственную связь. Здесь на помощь приходит execution tracer ([[3. execution tracer]]).

## Пример: поиск утечки горутин через block profile

Сценарий: сервис обрабатывает 1000 RPS, но со временем p99 latency растёт, и количество горутин увеличивается.

1. Снимаем дамп горутин ([[4. goroutine dump]]) — видим тысячи горутин в `[chan receive]` на функции `processBatch`.
2. Включаем block profile: `runtime.SetBlockProfileRate(1)`.
3. Через 30 секунд снимаем профиль: `curl -o block.prof http://localhost:6060/debug/pprof/block`.
4. Анализируем: `top` показывает `runtime.chanrecv1` с временем в несколько секунд. `list processBatch` указывает на строку `<-ch`.
5. Смотрим код: канал `ch` создаётся для каждого запроса, и в него должен отправить результат воркер. Но при тайм-ауте запроса горутина-читатель уходит, а воркер продолжает висеть, пытаясь отправить в никем не читаемый канал (утечка горутины).
6. Решение: использовать буферизированный канал (`make(chan Result, 1)`) или `select` с `ctx.Done()` в воркере.

Block profile показал **точку блокировки**, а анализ кода выявил корневую причину.

## Механика записи и overhead

При `rate=1` block profile записывает стек вызовов для **каждой** блокировки. Это означает:

- Вызов `runtime.Stack` для получения стека (проход по фреймам, ~микросекунда на глубокий стек).
- Запись в хеш-таблицу профиля (с блокировкой, только одна горутина пишет).
- При высокой частоте блокировок (например, небуферизированные каналы в горячем цикле) overhead может достигать 5-10% CPU. Однако обычно блокировки происходят не настолько часто, чтобы стать доминирующим потребителем CPU.

При `rate > 1` overhead линейно уменьшается. Для production-мониторинга `rate=10000` практически незаметен, но даёт представительную выборку.

## Отличие от execution tracer и mutex profile

| Инструмент | Что измеряет | Гранулярность | Overhead |
|------------|-------------|---------------|----------|
| Block profile | Время ожидания по типам (каналы, мьютексы, сеть) | Агрегировано по стекам | Низкий-средний |
| Mutex profile | Contention на мьютексах, стеки удерживающих | Агрегировано по стекам | Низкий-средний |
| Execution tracer | Хронология всех событий, причины ожидания | Событийно, во времени | Высокий |

Block profile — компромисс между детализацией и overhead'ом. Он даёт точное время ожидания и стек, но не показывает, *почему* другая сторона не отвечает (для мьютексов — см. mutex profile, для каналов — нужен трейс).

## Mechanical Sympathy: как процессор видит блокировку

С точки зрения процессора, переход горутины в состояние ожидания — это:

1. Вызов `gopark` — сохранение регистров, вызов планировщика.
2. Планировщик снимает G с M и запускает другую G. Это контекстное переключение в userspace ([[4. Контекстные переключения]]).
3. Если новая G использует тот же M, переключение дёшево (десятки тактов). Но данные старой G могут быть вытеснены из кэша (особенно если M мигрирует на другое ядро).
4. Когда ожидаемая G разблокируется (`goready`), она встаёт в очередь раннабле. Её данные в кэше, скорее всего, уже холодны — доступ к ним будет с cache miss.

Block profile фиксирует именно время от `gopark` до `goready`. Это время включает не только ожидание, но и накладные расходы на миграцию и холодный старт. Поэтому большое время в block profile — это реальное время, которое запрос «потерял».

## Итог

- **Block profile** — агрегированная статистика времени ожидания горутин на каналах, мьютексах, сети и системных вызовах.
- Включается через `runtime.SetBlockProfileRate(rate)`; rate=1 — всё, rate>1 — сэмплирование.
- Анализируется через `go tool pprof`: `top`, `list`, `web` показывают точки, где горутины простаивают.
- Незаменим для поиска утечек горутин, перекосов в пайплайнах и неочевидных ожиданий, которые не видны в CPU-профиле.
- Дополняет mutex profile (contention) и execution tracer (хронология).
- С точки зрения «железа», время блокировки — это потерянные такты и холодный кэш, поэтому оптимизация ожиданий напрямую улучшает latency.

Теперь, освоив измерение времени ожидания, мы перейдём к профилю, который специализируется на конкретном типе синхронизации и отвечает на вопрос «кто виноват в contention?» — [[6. mutex profile]].
