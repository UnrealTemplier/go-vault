## Debug tools: арсенал отладчика и диагноста Go

Предыдущие статьи подраздела «Runtime и инструменты» были посвящены крупным специализированным механизмам: `runtime` пакету ([[1. runtime пакет]]), `go tool trace` ([[2. trace tool]]), execution tracer ([[3. execution tracer]]), дампу горутин ([[4. goroutine dump]]), профилям блокировок ([[5. block profile]]) и мьютексов ([[6. mutex profile]]), а также трассировке планировщика ([[7. scheduler trace]]). Но экосистема Go содержит ещё множество компактных, но мощных **инструментов отладки**, которые не вошли в отдельные статьи. Это переменные окружения (`GODEBUG`, `GOTRACEBACK`), утилиты компилятора, HTTP-эндпоинты `pprof`, а также средства статического анализа и низкоуровневой инспекции бинарника.

Senior-инженер знает их не как разрозненные утилиты, а как единый арсенал, в котором каждый инструмент занимает своё место: одни — для настройки рантайма в production без перекомпиляции, другие — для лабораторного анализа, третьи — для контроля корректности в CI. Эта статья выполняет роль справочного хаба по debug tools, связывая их с более глубокими статьями и механической эмпатией ([[5. Mechanical sympathy в backend разработке]]).

## Переменные окружения: GODEBUG, GOTRACEBACK и другие

Go предоставляет несколько переменных окружения, меняющих поведение рантайма и инструментов. Их можно задавать при запуске без изменения кода, что делает их незаменимыми для production-диагностики и тонкой настройки.

### GODEBUG — швейцарский нож отладки

`GODEBUG` — это строка с парами `ключ=значение`, разделёнными запятыми. Самые важные ключи:

- **`gctrace=1`** — выводит в stderr строку каждого GC-цикла. Мы активно использовали её в разделе [[05. Garbage Collector]]. Позволяет оценить частоту и длительность пауз, размер кучи, цель GOGC.
- **`memprofilerate=N`** — задаёт частоту сэмплирования memory profile (по умолчанию 512 КБ). Установка `1` заставляет записывать каждую аллокацию (огромный overhead), `0` отключает профилирование памяти. Подробно в [[5. pprof memory profile]].
- **`schedtrace=N`** — рассмотрена в [[7. scheduler trace]]: каждые N миллисекунд выводит состояние планировщика.
- **`scheddetail=1`** — вместе с `schedtrace` включает детальную информацию о каждом P и M.
- **`asyncpreemptoff=1`** — отключает асинхронную преемпцию (Go 1.14+). Может использоваться для отладки проблем, предположительно вызванных сигналами `SIGURG`.
- **`cgocheck=2`** — усиливает проверки при передаче указателей между Go и C (cgo). Замедляет выполнение, но выявляет нарушения правил.
- **`madvdontneed=1`** — меняет стратегию возврата памяти ОС на `MADV_DONTNEED` вместо `MADV_FREE` (Linux). Память немедленно возвращается ОС, но увеличивается CPU-нагрузка. Полезно в контейнерах с жёсткими лимитами (см. [[7. Fragmentation]]).
- **`inittrace=1`** — выводит время выполнения каждого `init()`-метода пакета. Помогает найти долгие инициализации при старте.

```bash
GODEBUG=gctrace=1,schedtrace=1000 ./my_server
```

Некоторые ключи GODEBUG специфичны для определённых пакетов (например, `http2debug=2` включает детальное логирование HTTP/2). Список постоянно пополняется, актуальную информацию можно найти в документации Go: `go doc runtime/debug`.

### GOTRACEBACK — управление стек-трейсами при панике

`GOTRACEBACK` определяет, насколько подробный стек будет выведен при панике (или при получении сигнала `SIGQUIT`):

- **`none`** — не выводить ничего.
- **`single`** (по умолчанию) — стек упавшей горутины.
- **`all`** — стеки всех горутин (аналог `debug=2` в pprof).
- **`system`** — все горутины, включая горутины самого рантайма (используется разработчиками Go).

Для production-диагностики рекомендуется `GOTRACEBACK=all`: при панике вы получите полный дамп всех горутин, что бесценно для расследования причин аварии. При отправке `SIGQUIT` процессу (`kill -QUIT <pid>`) текущие стеки выводятся без остановки программы.

```bash
GOTRACEBACK=all ./my_server
```

### Другие важные переменные

- **`GOGC`** / **`GOMEMLIMIT`** — управление GC, рассмотрены в [[7. GOGC и tuning]] и [[8. GOMEMLIMIT]].
- **`GOMAXPROCS`** — можно задать через переменную, эквивалентно `runtime.GOMAXPROCS`.
- **`GOPROXY`**, **`GONOSUMDB`**, **`GONOSUMCHECK`** — управление загрузкой модулей, больше про безопасность и CI, но Senior должен их знать для настройки сборочных пайплайнов.

## Инструменты компилятора: заглянуть в скомпилированный код

Когда бенчмарк показывает аномалию, а профилировщик не даёт однозначного ответа, необходимо спуститься на уровень ассемблера. Go предоставляет встроенные утилиты.

### `go tool compile -S` — дизассемблер компилятора

Выводит ассемблерный листинг для указанного пакета или файла. Показывает, во что превратился ваш код после всех оптимизаций, инлайнинга и escape analysis.

```bash
go tool compile -S main.go
```

В выводе видны:
- Вызовы других функций (CALL).
- Удалённые из-за инлайнинга тела (их нет как отдельных процедур).
- Проверки границ (bounds check).
- Метки и переходы, сгенерированные для `if`, `for`, `select`.

Комбинируйте с флагами `-m` (escape analysis) и `-l` (отключение инлайнинга). Подробно инлайнинг и его влияние на производительность рассмотрены в [[5. Inline и влияние на performance]].

### `go tool objdump` — дизассемблер готового бинарника

В отличие от `compile -S`, objdump работает уже со скомпилированным исполняемым файлом. Позволяет посмотреть ассемблерный код любой функции.

```bash
go tool objdump -s 'main\.process' ./my_server
```

Полезен, когда нужно проверить, что попало в финальный бинарник после линковки, и увидеть точные адреса инструкций.

### `go tool nm` — таблица символов

Выводит список всех символов (функций, глобальных переменных) в бинарнике с их адресами и размерами. Помогает понять, почему бинарник такой большой, какие пакеты тянут много кода.

```bash
go tool nm -size ./my_server | sort -k2 -rn | head -20
```

### `go tool size` — размер секций бинарника

Быстрый способ оценить вклад кода, данных и отладочной информации в размер исполняемого файла.

```bash
go tool size ./my_server
```

Пример вывода:
```
   text	   data	    bss	    dec	    hex	filename
 8240128	 152432	  45680	 8438240	  80c1e0	./my_server
```

`text` — исполняемый код, `data` — инициализированные данные, `bss` — неинициализированные. Значения позволяют судить о раздувании кода (code bloat) из-за инлайнинга или избыточных зависимостей.

## Статический анализ: go vet и его друзья

Не все проблемы производительности и корректности требуют рантайм-анализа. Многие из них можно обнаружить на этапе компиляции.

### `go vet` — встроенный набор анализаторов

`go vet` запускает серию проверок, встроенных в дистрибутив Go:
- **loopclosure** — захват переменной цикла замыканием (до Go 1.22).
- **fieldalignment** (с Go 1.19, опционально) — неоптимальный порядок полей структуры ([[9. Cache line и выравнивание]]).
- **copylocks** — копирование `sync.Mutex` по значению.
- **nilfunc** — вызов nil-функции.
- **printf** — несоответствие формата в `fmt.Printf`.
- **shadow** — затенение переменных (опционально).
- **unreachable** — недостижимый код.

Запуск:
```bash
go vet ./...
```

### staticcheck и golangci-lint

[staticcheck](https://staticcheck.io/) — мощный сторонний анализатор с сотнями проверок производительности, стиля и багов. Интегрируется с `golangci-lint`, который объединяет десятки линтеров и работает в CI.

Senior-инженер настраивает `golangci-lint` с жёстким конфигом, блокируя не только ошибки, но и неоптимальные паттерны (неиспользуемые параметры, пренебрежение обработкой ошибок, неэффективные конкатенации строк и т.д.).

## Delve: интерактивный отладчик

Delve (`dlv`) — де-факто стандарт для интерактивной отладки Go-программ. Позволяет ставить breakpoint'ы, шагать по коду, инспектировать переменные, запускать горутины под управлением отладчика. Детально его применение (включая удалённую отладку и core dump) рассматривается в отдельной статье [[2. Delve debugger]]. Здесь лишь отметим его место среди инструментов: Delve — это для пошаговой логической отладки, в отличие от pprof и trace, которые анализируют производительность.

## Core dump и postmortem

Когда программа падает с сигналом (SIGSEGV, SIGABRT), ОС может записать **core dump** — образ памяти процесса. Go поддерживает анализ core dump'ов через Delve. Этот процесс, а также методика postmortem-анализа аварийных завершений, описаны в [[5. Postmortem анализ]] и [[6. Core dumps]].

## Mechanical Sympathy: цена инструментов отладки

Каждый инструмент отладки добавляет overhead:

- **`GODEBUG=gctrace=1`** — незначительно, только строковый вывод.
- **pprof HTTP-эндпоинты** — при запросе создают нагрузку (особенно `profile` и `trace`).
- **`go tool compile -S`** — работает на этапе сборки, не влияет на рантайм.
- **Delve** — замедляет программу на порядки (breakpoint'ы реализованы через системные вызовы `ptrace`).
- **Core dump** — запись на диск может занять время, ядро ОС приостанавливает процесс.

Senior-инженер оценивает эту стоимость и выбирает комбинацию, дающую максимум информации при допустимом воздействии на production. Именно поэтому `schedtrace`, `gctrace` и эндпоинты pprof (с осторожностью) предпочтительнее, чем запуск под Delve на проде.

## Итог

- **Debug tools** Go — это не только pprof и trace, но и целый арсенал переменных окружения, утилит компилятора и статических анализаторов.
- **GODEBUG** и **GOTRACEBACK** управляют поведением рантайма и детализацией трейсов без перекомпиляции.
- **`go tool compile -S`**, **objdump**, **nm**, **size** предоставляют низкоуровневый доступ к скомпилированному коду.
- **HTTP-эндпоинты pprof** — основной интерфейс для получения профилей в production.
- **`go vet`** и **`golangci-lint`** предотвращают ошибки и неоптимальности на этапе написания кода.
- Каждый инструмент имеет свою цену, и мастерство Senior-инженера — в точном подборе инструмента под задачу без разрушения стабильности сервиса.

На этом мы завершаем подраздел «Runtime и инструменты». Следующий подраздел — «Практические оптимизации» — начнётся с фундаментальной техники, которая пронизывает все предыдущие разделы: [[1. Уменьшение аллокаций]].
