## Дамп горутин: мгновенный снимок конкурентного состояния приложения

CPU-профиль ([[2. CPU profiling в Go]]) сообщает *где* сжигаются такты, execution tracer ([[3. execution tracer]]) показывает *когда* и *почему* горутины переходили из состояния в состояние, а pprof памяти ([[5. pprof memory profile]]) — *сколько* памяти занято. Но часто для диагноза не хватает информации о том, чем занята каждая конкретная горутина **прямо сейчас**. Вопрос «Сколько у нас горутин и что они делают?» решает **goroutine dump** — мгновенный срез состояния всех горутин процесса.

Этот инструмент незаменим для поиска утечек горутин, обнаружения взаимных блокировок (deadlock) и анализа паразитной конкуренции, когда сотни горутин висят в ожидании одного ресурса. В отличие от `GODEBUG=schedtrace`, который показывает агрегированные цифры планировщика, дамп предоставляет индивидуальные стеки.

## Способы получения дампа

### 1. HTTP-эндпоинт `pprof`

Самый удобный способ — встроенный обработчик `/debug/pprof/goroutine`. Он поддерживает параметр `debug`:

- **`?debug=1`** — краткий список всех горутин, сгруппированных по состоянию и стеку. Подходит для быстрого подсчёта.
- **`?debug=2`** — полный стек-трейс каждой горутины, аналогичный тому, что выводится при панике, но без аварийного завершения.

```bash
# Краткий отчёт
curl http://localhost:6060/debug/pprof/goroutine?debug=1

# Полный дамп всех стеков
curl http://localhost:6060/debug/pprof/goroutine?debug=2
```

### 2. Программный вызов `pprof` и `runtime.Stack`

Тот же дамп можно записать в файл из кода, не открывая HTTP-порт:

```go
f, _ := os.Create("goroutines.txt")
pprof.Lookup("goroutine").WriteTo(f, 1) // debug=1
```

Либо получить срез байт с помощью `runtime.Stack(buf []byte, all bool)` и записать его куда нужно.

### 3. Сигнал `SIGQUIT` и `GOTRACEBACK`

По умолчанию при отправке процессу сигнала `SIGQUIT` (`kill -QUIT <pid>`) рантайм выводит в stderr полный дамп горутин без завершения программы. Поведением управляет переменная `GOTRACEBACK` (`none`, `single`, `all`, `system`). Значение `system` добавляет горутины самого рантайма.

Этот способ полезен в production-окружении без pprof-эндпоинта.

> [!warning] Ловушка / Gotcha
> Стек горутины в выводе `debug=2` — это **текстовое представление**, полученное обходом стека вызовов. Для десятков тысяч горутин такой дамп — огромная строка. Парсить её вручную сложно, лучше анализировать через `go tool pprof` или интерактивные скрипты.

## Профиль горутин в `go tool pprof`

Помимо текстового дампа, pprof предлагает профиль горутин как протобуф-файл (эндпоинт без `debug`):

```bash
curl -o goroutine.prof http://localhost:6060/debug/pprof/goroutine
go tool pprof goroutine.prof
```

Этот профиль содержит агрегированную информацию: **в каких функциях** и с какими стеками находятся горутины в данный момент. Команды `top`, `list`, `tree` работают привычным образом, показывая распределение горутин по точкам ожидания.

Например, `top` покажет:

```
(pprof) top
Showing nodes accounting for 1024 goroutines, 90% of 1138 total
      flat  flat%   sum%        cum   cum%
       500 43.94% 43.94%        500 43.94%  runtime.gopark (chan receive)
       200 17.57% 61.51%        200 17.57%  runtime.gopark (IO wait)
       150 13.18% 74.69%        150 13.18%  runtime.gopark (select)
        80  7.03% 81.72%         80  7.03%  sync.(*Mutex).Lock
```

Цифры — количество горутин, чей стек заканчивается данной функцией (или состоянием). Это прямой способ узнать, сколько горутин ожидают каналы, мьютексы, сеть или системные вызовы.

## Состояния горутин в дампе и их интерпретация

Каждая горутина в Go находится в одном из состояний, описанных в [[2. Goroutines под капотом]] (`_Gidle`, `_Grunnable`, `_Grunning`, `_Gwaiting`, `_Gdead`). В выводе `debug=2` состояние указывается в квадратных скобках перед стеком:

```
goroutine 123 [chan receive]:
main.consumer(0xc0000a4000)
    /app/main.go:45 +0x79
```

Возможные видимые состояния (скобки в выводе):

| Состояние | Значение | О чём говорит |
|-----------|----------|---------------|
| `[running]` | Горутина исполняется на M | Норма для активных горутин. Если в дампе одна и та же горутина всегда running — возможно, она зациклилась. |
| `[runnable]` | Готова к исполнению, ждёт P | Признак нехватки CPU или блокировки всех P другими горутинами. |
| `[chan receive]` | Ожидает чтения из канала (`<-ch`) | Горутина-читатель ждёт данные. Если канал никто не пишет, это навсегда. |
| `[chan send]` | Ожидает записи в канал (`ch <- val`) | Горутина-писатель ждёт получателя. Частая причина утечек. |
| `[IO wait]` | Ожидает завершения сетевого ввода-вывода | Горушка «запарковалась» на netpoller ([[4. epoll kqueue и netpoller]]). Нормально для серверных горутин. |
| `[select]` | Блокирована в операторе `select` | Ожидает готовности одного из каналов. |
| `[sync.Mutex.Lock]` | Заблокирована при захвате мьютекса | Contention на конкретном мьютексе. Можно проследить по стеку, кто его держит. |
| `[sleep]` | `time.Sleep` | Горушка спит. Может быть нормой (фоновый тикер) или признаком забытого sleep. |
| `[syscall]` | Выполняет блокирующий системный вызов | Файловый ввод-вывод, ожидание futex или другой syscall, блокирующий M. |
| `[finalizer wait]` | Ожидает выполнения финализатора GC | Редкое состояние, обычно говорит о проблемах с GC или финализаторами. |

Типичная диагностика:

- **Много горутин в `[chan send]`** и нет читателя — мёртвая блокировка (не путать с deadlock в планировщике, когда вообще нет активных горутин).
- **Растёт число горутин в `[chan receive]`** — продюсеров больше, чем потребителей.
- **Много `[runnable]` при низкой загрузке CPU** — возможно, все P застряли в syscall'ах, и планировщик не может их использовать.
- **`[sync.Mutex.Lock]` на одном и том же стеке** — contention на критической секции.

> [!tip] Собеседование
> **Вопрос:** Чем состояние `[IO wait]` отличается от `[syscall]`?
> **Ответ:** `[IO wait]` означает, что горутина ожидает сетевого события через netpoller и **не блокирует поток ОС** (M). `[syscall]` — горутина выполняет блокирующий системный вызов (например, чтение файла) и M непосредственно заморожен ядром до завершения.

## Типичные сценарии использования дампа горутин

### 1. Обнаружение утечки горутин

Симптом: метрика `go_goroutines` монотонно растёт. Снимаем дамп с интервалом в несколько минут, конвертируем в профиль горутин, смотрим `top`. Если количество горутин с определённым стеком растёт — утечка именно там.

```bash
curl http://localhost:6060/debug/pprof/goroutine > dump1.prof
sleep 300
curl http://localhost:6060/debug/pprof/goroutine > dump2.prof
go tool pprof -diff_base=dump1.prof dump2.prof
```

В выводе увидим функции, где число горутин увеличилось. Стеки укажут на забытый `close(ch)` или отсутствие `defer cancel()` для контекста.

### 2. Расследование deadlock

Если все горутины в состоянии `[chan send]` или `[chan receive]`, и ни одной в `[running]`, скорее всего, произошла взаимная блокировка. `debug=2` покажет точные номера строк, где горутины заблокированы, и можно проследить, ожидают ли они друг друга.

### 3. Оценка конкуренции

Если в дампе несколько тысяч горутин в `[runnable]` при `GOMAXPROCS=8`, процессор перегружен — либо нужно увеличивать число ядер, либо оптимизировать CPU-нагрузку. Если же `[runnable]` мало, но много в `[IO wait]`, процессор простаивает, и проблема в сети или бэкенде.

### 4. Проверка корректности shutdown

При завершении программы горутины должны корректно завершиться. Дамп после grace period покажет, остались ли «висящие» горутины, блокированные на каналах или mutex'ах без возможности завершения.

## Профиль горутин и автоматизация

Профиль можно анализировать скриптами через `go tool pprof -top -text`:

```bash
go tool pprof -top -text goroutine.prof
```

Или экспортировать в формат, пригодный для Prometheus (количество горутин в каждом состоянии). Пакет `runtime` предоставляет `NumGoroutine()`, но не даёт разбивку по состояниям. Однако можно периодически снимать дамп и парсить его для получения вектора состояний (готовые библиотеки типа `github.com/uber-go/gops` делают это через анализ `/debug/pprof/goroutine?debug=1`).

## Mechanical Sympathy: память, планировщик и дамп горутин

Каждая горутина потребляет как минимум **2 КБ стека** (при старте), а в процессе жизни может вырасти до гораздо больших размеров. 100 000 горутин — это минимум 200 МБ чисто на стеки. Плюс структура `g` в рантайме, плюс метаданные планировщика. Если дамп показывает необоснованно большое количество горутин, это не только логическая проблема, но и прямая угроза production-ресурсам.

С точки зрения планировщика, даже горутина в `[chan receive]` не занимает поток ОС, но участвует в очереди ожидания, и при массовом пробуждении (например, закрытие канала) планировщик получит лавину `goready`, что может вызвать кратковременную перегрузку. Дамп помогает спрогнозировать такие эффекты.

Снятие дампа — относительно лёгкая операция: `pprof` обходит все горутины и копирует их стеки. При десятках тысяч горутин это занимает миллисекунды и не останавливает мир.

## Связь с другими инструментами

- **Execution tracer** ([[3. execution tracer]]) показывает динамику переходов, а дамп — мгновенный снимок. Вместе они позволяют увидеть и статику, и динамику.
- **Block profile** ([[5. block profile]]) — следующая тема, фокусируется именно на времени блокировок, тогда как дамп показывает, где горутины застряли сейчас.
- **pprof-профиль горутин** — первое, что нужно заглянуть при подозрении на утечки, до снятия памяти.
- **`runtime.NumGoroutine()`** даёт только общее количество, дамп — качественный состав.

## Итог

- **Дамп горутин** — мгновенный срез состояния всех горутин, доступный через `/debug/pprof/goroutine`, `runtime.Stack` или `SIGQUIT`.
- Формат `debug=2` показывает полный стек и состояние, `debug=1` — агрегированный отчёт.
- Профиль горутин в `go tool pprof` позволяет анализировать распределение горутин по функциям и состояниям.
- Основные состояния: `running`, `runnable`, `chan receive/send`, `IO wait`, `syscall`, `sync.Mutex.Lock`, `sleep`. Интерпретация состояний — ключ к диагностике утечек, дедлоков и перегрузок.
- Дамп незаменим для поиска утечек горутин: дифференциальное сравнение профилей выявляет точки роста.
- С точки зрения механической эмпатии, избыток горутин тратит память и нагружает планировщик; дамп — первый шаг к их контролю.
- Этот инструмент, наряду с trace и профилями, составляет арсенал Senior-инженера для разбора сложных конкурентных проблем.

Теперь, умея получать и интерпретировать срез состояния горутин, мы готовы углубиться в анализ **причин** их ожиданий с помощью специализированного профиля — [[5. block profile]].
