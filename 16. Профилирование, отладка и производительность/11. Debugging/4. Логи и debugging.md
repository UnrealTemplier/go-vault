## Логи и debugging: почему `fmt.Println` — не стратегия

В предыдущих статьях мы рассмотрели интерактивную отладку ([[2. Delve debugger]]), которая требует остановки процесса, и детектор гонок ([[3. Race detector в проде]]), который меняет поведение программы. Оба инструмента либо неприменимы в production, либо искажают картину. Остаётся самый древний, универсальный и при этом самый недооценённый инструмент — **логи**. Правильно спроектированная система логирования — это не просто «записывать в файл», а полноценный источник данных для отладки, позволяющий восстановить состояние программы в момент ошибки без остановки мира.

Для Senior Go-инженера логи — это интерфейс наблюдаемости первого уровня. Когда метрики сигналят о росте ошибок, когда распределённая трассировка (из раздела [[18. Observability]]) показывает аномалию, вы идёте в логи и по Correlation ID восстанавливаете точную последовательность событий. Логи не заменяют профилировщик или трейс, но без них эти инструменты — микроскоп без подсветки.

В этой статье мы разберём, как превратить логи из потока текста в отладочный инструмент: структурированное логирование в Go (`slog`), контекст и Correlation ID, уровни и фильтрация, и — что критически важно — влияние логирования на производительность с точки зрения механической эмпатии ([[5. Mechanical sympathy в backend разработке]]). Мы свяжем это с аллокациями ([[4. Allocation profiling]]), сборщиком мусора ([[1. GC в Go. Обзор]]) и выбором правильного логгера.

## От `fmt.Println` к структурированному логированию

Традиционный подход — `fmt.Println("error:", err)` — имеет право на жизнь в скриптах, но в высоконагруженном сервисе он создаёт три проблемы:

1. **Нечитаемость для машин.** Парсить неструктурированный текст дорого и ненадёжно.
2. **Отсутствие контекста.** Непонятно, к какому запросу относится сообщение, если их перекрывают сотни горутин.
3. **Высокий overhead.** `fmt.Sprintf` порождает аллокации на каждое сообщение, создавая лавину работы для GC ([[9. Когда GC становится bottleneck]]).

Решение — **структурированное логирование** (structured logging). Вместо строки формируется запись с типизированными ключами и значениями: `{"level":"error","msg":"failed to fetch","user_id":42,"trace_id":"abc"}`. Такие записи легко индексируются системами агрегации (Loki, Elasticsearch), позволяя выполнять точные запросы: «все ошибки по пользователю 42 за последние 5 минут».

В Go стандартом стал пакет **`log/slog`**, появившийся в Go 1.21. Он предлагает:

- Уровни логирования (Debug, Info, Warn, Error).
- Структурированные атрибуты (`slog.String`, `slog.Int`, `slog.Any`).
- Поддержку контекста (`slog.InfoContext`, `slog.ErrorContext`).
- Архитектуру Handler'ов, позволяющую менять формат вывода (JSON, Text) и бэкенд без изменения кода логирования.

```go
import "log/slog"

func handler(w http.ResponseWriter, r *http.Request) {
    logger := slog.With("trace_id", r.Header.Get("X-Request-Id"))
    logger.InfoContext(r.Context(), "request started", "path", r.URL.Path)
    // ...
    if err != nil {
        logger.ErrorContext(r.Context(), "failed to process", "error", err)
    }
}
```

## Correlation ID в логах

В распределённой системе ([[8. Debugging distributed systems]]) каждый запрос проходит через несколько сервисов. Без общего идентификатора логи разных сервисов бесполезны для отладки. **Correlation ID** (или Trace ID) — это уникальный идентификатор, присваиваемый запросу на входе и передаваемый через все вызовы (HTTP-заголовки, gRPC metadata, атрибуты сообщений).

В Go ID помещается в `context.Context` через middleware и автоматически подхватывается логгером:

```go
type contextKey string
const traceIDKey contextKey = "traceID"

func TraceIDMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        id := r.Header.Get("X-Request-Id")
        if id == "" {
            id = uuid.New().String()
        }
        ctx := context.WithValue(r.Context(), traceIDKey, id)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

func LoggerFromContext(ctx context.Context, base *slog.Logger) *slog.Logger {
    if id, ok := ctx.Value(traceIDKey).(string); ok {
        return base.With("trace_id", id)
    }
    return base
}
```

Теперь каждая запись лога несёт trace_id, что позволяет в Kibana или Grafana Loki одним запросом вытащить полную цепочку событий по проблемному запросу. Детальнее о Correlation ID — в разделе «Распределённые системы» ([[14. Распределенные системы на Go/08. Практика/7. Correlation ID]]).

> [!info] Под капотом
> `context.WithValue` создаёт новый узел в цепочке контекстов (аллокация ~80-100 байт). Поиск по цепочке при каждом извлечении — O(N), но глубина редко превышает 3-5. Это пренебрежимо мало, если ID извлекается один раз на входе и передаётся логгеру явно, а не дёргается через `context.Value` в горячем цикле.

## Уровни логирования и динамическая фильтрация

Стандартные уровни: **Debug, Info, Warn, Error**. В production обычно устанавливают порог `Info`, чтобы не генерировать гигабайты отладочных данных. Однако при инциденте хочется поднять уровень до `Debug` для конкретного сервиса или даже для конкретного Correlation ID. Современные логгеры (slog, zerolog) позволяют это через динамическую настройку Handler'а.

`slog` поддерживает `LevelHandler` с атомарным уровнем:

```go
var level atomicLevel

func init() {
    level.Store(slog.LevelInfo)
    slog.SetDefault(slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
        Level: &level, // динамический уровень
    })))
}

// по HTTP-админке можно вызвать
func setLogLevel(w http.ResponseWriter, r *http.Request) {
    newLevel := r.URL.Query().Get("level")
    switch newLevel {
    case "debug":
        level.Store(slog.LevelDebug)
    case "info":
        level.Store(slog.LevelInfo)
    }
}
```

Это мощный инструмент: во время инцидента вы через админку включаете `Debug` на 30 секунд и получаете детальные логи без перезапуска.

## Производительность логирования: аллокации и GC

Логирование в горячем пути — один из главных источников скрытых аллокаций. Каждый вызов `slog.Info` с атрибутами создаёт:

- Слайс атрибутов (`[]slog.Attr`).
- Сами атрибуты (строки, значения).
- Итоговый буфер сообщения (если не async).

При 10 000 RPS и 5 строках лога на запрос это 50 000 аллокаций в секунду. Неоптимизированный логгер может стать главным драйвером GC и расширить p99 latency на десятки процентов.

### Облегчённые логгеры: zerolog и zap

До появления `slog` сообщество использовало **zerolog** (аллокации близки к нулю) и **zap** (аллокации минимизированы, высокая пропускная способность). Оба достигают zero-allocation или near-zero за счёт:

- Переиспользования буферов (`sync.Pool`, [[2. sync Pool]]).
- Сериализации JSON без рефлексии.
- Избегания `interface{}` для простых типов.

`slog` также имеет оптимизированный путь, но его стандартный JSON-обработчик пока уступает zerolog по zero-allocation-метрикам. В latency-критичных сервисах стоит бенчмаркать конкретный логгер под свою нагрузку.

### Асинхронное логирование

Запись в файл или отправка по сети через `syscall.Write` ([[1. Системные вызовы и их стоимость]]) может блокировать горутину на миллисекунды при заполнении буферов ядра. Асинхронные логгеры сбрасывают сообщения в кольцевой буфер, а отдельная горутина пишет их в вывод. Это изолирует hotspot, но добавляет задержку доставки и risk потери сообщений при падении.

> [!warning] Ловушка / Gotcha
> Асинхронное логирование маскирует проблему: если диск медленный, буфер будет расти, потребляя память. Необходимо мониторить размер буфера и устанавливать `RingBuffer` с политикой сброса (drop oldest) для защиты от OOM.

### Сэмплирование логов

Не каждый запрос достоин детального лога. Сэмплирование (логировать только 1% запросов или каждый N-й ошибочный) кратно снижает объём данных и накладные расходы. Реализуется через счётчик и `rand.Intn(100) == 0` на уровне Handler'а.

## Mechanical Sympathy: что происходит с процессором при записи лога

Каждая запись лога — это путь от строки в памяти до системного вызова `write` (или `sendto` для сети). С точки зрения процессора:

- **Форматирование.** Конкатенация строк, сериализация в JSON — это копирования памяти. Если буфер не переиспользуется, каждое копирование трогает новые кэш-линии, вытесняя полезные данные ([[8. Cache friendliness]]).
- **Системный вызов.** `write` в файл (даже если это `stdout`, перенаправленный в пайп) — это переход в ядро, копирование данных в буфер ядра, возможное ожидание при заполненном буфере ([[1. Системные вызовы и их стоимость]]). Это не только такты, но и загрязнение L1-кэша кодом ядра.
- **GC.** Если каждый лог создаёт временные слайсы, они быстро умирают, учащая GC-циклы и mark assist ([[9. Когда GC становится bottleneck]]).

Именно поэтому Senior-инженер, проектируя логирование, думает о:
- Минимизации аллокаций (пулы буферов).
- Батчинге записей (буферизация перед `Flush`).
- Ограничении объёма (сэмплирование, уровни).

## Конфигурация логирования в разных средах

- **Локальная разработка:** текстовый формат (`slog.NewTextHandler`), уровень `Debug`, вывод в stderr.
- **CI/тесты:** уровень `Warn` или `Error`, чтобы не зашумлять вывод, но ловить проблемы.
- **Production:** JSON-формат, уровень `Info`, отправка в агрегатор (Loki через HTTP или stdout с агентом), асинхронный или буферизированный вывод, обязательный Correlation ID.

## Практический пример: рефакторинг логирования для отладки инцидента

Сервис на Go падает с паникой раз в несколько часов. Логи — классический `fmt.Println` с кучей мусора. Что делаем:

1. Внедряем `slog` с JSON-форматом и Correlation ID.
2. Добавляем Middleware, проставляющий `trace_id` во все контексты.
3. Устанавливаем уровень `Info` по умолчанию, `Debug` — по сигналу из админки.
4. В момент следующего падения по логам (уже структурированным) находим `trace_id`, востанавливаем цепочку запросов, видим аномалию за 2 секунды до паники.
5. Гипотеза подтверждается, фикс деплоится. Логи стали не просто шумом, а диагностическим инструментом.

## Итог

- **Логи** — фундаментальный инструмент отладки в production, не заменяемый ни Delve, ни race detector.
- **Структурированное логирование** (`slog`) с Correlation ID превращает поток текста в запрашиваемый источник данных.
- **Уровни и динамическая фильтрация** позволяют менять детализацию без перезапуска.
- **Производительность:** логирование может быть дорогим; необходимо применять zero-allocation логгеры (zerolog, zap), переиспользование буферов, сэмплирование и асинхронный вывод.
- **Механическая эмпатия:** каждое записанное сообщение стоит тактов процессора, трогает кэш и создаёт давление на GC; Senior-инженер балансирует информативность и стоимость.
- В распределённых системах логи без Correlation ID бесполезны; их интеграция с трассировкой и метриками даёт полноту наблюдаемости.
