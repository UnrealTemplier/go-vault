## Delve: интерактивный отладчик Go

В предыдущей статье «[[1. Debugging в Go]]» мы определили отладку как искусство поиска логических ошибок и аномального поведения, и классифицировали инструменты. Теперь мы погружаемся в самый мощный из них — **Delve** (`dlv`). Это интерактивный отладчик, позволяющий ставить точки останова, шагать по коду, инспектировать переменные, переключаться между горутинами и анализировать core dump'ы. В отличие от pprof, который показывает статистическую картину, Delve даёт полный контроль над конкретным путём исполнения.

Для Senior Go-инженера Delve — не игрушка и не костыль на случай «непонятной ошибки». Это прецизионный инструмент, который применяется в лабораторных условиях: локально, на стейджинге, или для postmortem-анализа аварийного дампа. Умение работать с Delve входит в обязательный минимум компетенций, особенно при расследовании гонок данных (в связке с race detector, [[3. Race detector в проде]]) и сложных дедлоков.

## Delve и Go: архитектура и ограничения

Delve работает через системный вызов `ptrace` (Linux) или его macOS-аналог (`proc_*` API). Он «присоединяется» к запущенному процессу и перехватывает сигналы, чтобы останавливать исполнение, читать память и управлять потоком управления. Это означает, что Delve не внедряется в код компилятором — он работает на уровне уже скомпилированной программы, используя отладочную информацию DWARF, которую Go включает в бинарник по умолчанию (если не собран с `-ldflags="-s -w"`).

Из этого вытекают два важных следствия:

- **Полная изоляция от runtime:** Delve не требует перекомпиляции с флагами. Он работает с production-бинарником (если DWARF включён).
- **Значительный overhead:** `ptrace` останавливает **весь процесс** (или нить, в зависимости от режима). Это разрушает временные характеристики и может вызывать тайм-ауты в сетевых вызовах. Delve категорически не рекомендуется для продакшена.

## Установка и запуск

Современный способ установки — через `go install`:

```bash
go install github.com/go-delve/delve/cmd/dlv@latest
```

После этого `dlv` доступен в `$GOPATH/bin`. Рекомендуется добавить его в `PATH`.

Запуск Delve может происходить в двух режимах:

- **`dlv debug`** — скомпилировать и запустить программу под отладчиком (для main-пакетов).
- **`dlv exec`** — запустить уже скомпилированный бинарник.
- **`dlv attach`** — подключиться к работающему процессу по PID.
- **`dlv core`** — открыть core dump для postmortem-анализа ([[6. Core dumps]]).

```bash
dlv debug ./main.go
dlv exec ./my_server
dlv attach 12345
dlv core ./my_server core.dump
```

## Основные команды Delve

Интерфейс Delve — это REPL (Read-Eval-Print Loop) с командами. Ниже — основной набор, без которого невозможно работать.

### Точки останова (breakpoints)

Точка останова — это место в коде, где исполнение приостановится, и управление перейдёт к отладчику.

- **`break main.main`** — останов при входе в функцию `main` пакета `main`.
- **`break handler.go:45`** — останов на строке 45 файла `handler.go`.
- **`break -list`** — показать все точки останова.
- **`clear 1`** — удалить точку останова с индексом 1.
- **`condition 1 req.ID == 42`** — останов на точке 1 только когда `req.ID == 42`. Условные брейкпойнты замедляют выполнение, но незаменимы для анализа конкретных данных в цикле.

```go
(dlv) break processOrder
Breakpoint 1 set at 0x4a1b20 for main.processOrder() ./service.go:30
```

### Управление исполнением

- **`continue`** / **`c`** — продолжить выполнение до следующей точки останова.
- **`next`** / **`n`** — шаг на следующую строку в текущей функции (не заходя внутрь вызовов).
- **`step`** / **`s`** — шаг на следующую строку, заходя внутрь вызываемых функций.
- **`stepout`** / **`so`** — выполнить до выхода из текущей функции.
- **`restart`** — перезапустить программу под отладчиком (быстро, без перекомпиляции).

### Инспекция стека и переменных

- **`stack`** / **`bt`** — вывести стек вызовов текущей горутины.
- **`frame 2`** — переключиться на фрейм стека с индексом 2 (для просмотра локальных переменных вызывающей функции).
- **`print variable`** / **`p variable`** — вывести значение переменной.
- **`locals`** — показать все локальные переменные текущего фрейма.
- **`args`** — показать аргументы функции.

```go
(dlv) p user.Name
"Alice"
(dlv) p *user
main.User{ID: 42, Name: "Alice", Email: "alice@example.com"}
```

### Работа с горутинами

- **`goroutines`** / **`grs`** — список всех горутин с состояниями (running, runnable, waiting). Аналог дампа горутин ([[4. goroutine dump]]), но в отладчике.
- **`goroutine 15`** — переключиться на горутину с ID 15. После этого `stack`, `locals`, `frame` показывают контекст именно этой горутины.
- **`goroutines -t`** — вывести список горутин с трейсами (стеком вызовов каждой).

Это позволяет анализировать дедлоки прямо в отладчике: переключиться на каждую горутину, посмотреть, где она блокирована, и понять, кого ждёт.

### Прочие полезные команды

- **`disassemble`** / **`disass`** — показать ассемблерный листинг текущей функции (требует понимания ассемблера Go, но иногда необходимо).
- **`regs`** — содержимое регистров процессора.
- **`examinemem`** / **`x`** — дамп памяти по адресу.
- **`call`** — вызвать функцию из контекста отладчика (экспериментальная возможность).
- **`source`** — выполнить скрипт команд Delve из файла.

## Просмотр сложных типов и кастомные принтеры

Delve корректно отображает слайсы, мапы, интерфейсы и структуры. Для сложных типов можно использовать:

- **`p var`** — покажет значение с полями.
- **`p len(slice)`** — выведет длину слайса.
- **`p cap(slice)`** — ёмкость.
- **`p map[key]`** — значение по ключу (если ключ — простая константа).

Если стандартное представление неудобно, Delve поддерживает **pretty-print** и **функции для отображения** (через конфигурационный файл `dlv`). Например, можно добавить кастомную функцию для визуализации структур.

## Удалённая отладка (headless mode)

Для отладки в контейнере или на удалённом сервере Delve запускается в **headless-режиме**:

```bash
dlv debug --headless --listen=:2345 --api-version=2 ./main.go
```

К этому серверу можно подключиться:
- Из другого экземпляра `dlv connect localhost:2345`.
- Из IDE (GoLand, VSCode), настроив remote debug на этот порт.

Это позволяет отлаживать программу в окружении, идентичном production, но изолированном от внешнего трафика. Часто используется для отладки в Kubernetes: в pod внедряется Delve, и разработчик подключается к нему через `kubectl port-forward`.

> [!warning] Ловушка / Gotcha
> Headless-режим с открытым портом представляет угрозу безопасности: любой, кто может подключиться, получает полный контроль над процессом. Всегда используйте аутентификацию (ключ `--accept-multiclient` и туннели) и сетевую изоляцию.

## Анализ core dump

Core dump — это снимок памяти упавшей программы. Delve может открыть его и восстановить состояние всех горутин на момент краша. Это описано в [[6. Core dumps]], но подчеркнём базовый workflow:

1. Получить core dump (через `GOTRACEBACK=crash` или настройки ОС).
2. Запустить `dlv core ./my_binary core.dump`.
3. Выполнить `goroutines` — увидеть все горутины.
4. Переключиться на упавшую горутину, выполнить `stack` — увидеть trace до паники.
5. Посмотреть `locals` и `p variable` в каждом фрейме.

Delve также позволяет посмотреть `threads` (потоки ОС) и состояние регистров, если падение связано с некорректным системным вызовом.

## Интеграция с IDE

Хотя CLI Delve самодостаточен, для продуктивной работы используют интеграцию с IDE:

- **VSCode:** Расширение Go автоматически подключает Delve. Точки останова ставятся кликом в gutter. Запуск через `F5`. Возможна удалённая отладка через конфигурацию `launch.json`.
- **GoLand (IntelliJ IDEA):** Полноценная интеграция с Delve, удалённая отладка, отображение горутин в отдельной панели.

Интеграция с IDE не добавляет новых возможностей, но ускоряет навигацию. Senior-инженер должен уметь работать и в CLI (для удалённой отладки и скриптов), и в IDE.

## Mechanical Sympathy: цена интерактивной остановки

Когда Delve останавливает программу на точке останова:

- **Весь процесс замораживается** (или конкретный поток, если используются опции `--continue` и `--headless` с настройками). Планировщик горутин замирает, GC останавливается, netpoller не обрабатывает события.
- **Сетевые соединения разрываются** по таймаутам, если остановка длится дольше, чем TCP keep-alive.
- **Мониторинг** (Kubernetes liveness probes) может счесть сервис мёртвым и перезапустить контейнер.
- **Данные в кэшах** процессора остаются нетронутыми, но после возобновления программа, скорее всего, получит множество cache miss ([[8. Cache friendliness]]) из-за вытеснения данных за время остановки (если ядро переключило контекст на другой процесс). Хотя сама остановка не сбрасывает кэш, факт простоя увеличивает вероятность того, что ОС отберёт ядро.

Именно поэтому интерактивная отладка — инструмент лаборатории, а не продакшена. Для продакшена применяются логи, дампы горутин и трассировка.

## Типичные сценарии использования Delve

### 1. Расследование паники

Программа падает с `panic: runtime error: invalid memory address or nil pointer dereference`. Стек-трейс указывает на функцию `processData`. Запускаем Delve, ставим `break processData`, воспроизводим запрос. Проходим построчно (`next`), смотрим значения указателей. Находим, что `config.Cache` равен `nil` при определённых условиях.

### 2. Дедлок горутин

Тесты зависают навсегда. Запускаем тест под Delve (`dlv test`). Когда зависло, жмём Ctrl+C, получаем REPL. `goroutines` показывает, что горутина A ждёт канал `ch1`, горутина B ждёт `ch2`, и никто не пишет. Переключаемся между ними, смотрим стеки. Понимаем, что порядок операций перепутан.

### 3. Почему возвращается не то значение

Функция иногда возвращает пустую строку. Ставим условный брейкпойнт: `break getValue`, `condition 1 len(result) == 0`. Когда срабатывает, проходим по стеку (`stack`), смотрим, откуда пришёл вызов с пустым результатом.

### 4. Анализ core dump'а

Ночью упал production. Core dump сохранён. Открываем `dlv core`, переключаемся на упавшую горутину, видим панику из-за `send on closed channel`. Стек показывает, что канал был закрыт в `defer`, но отправитель ещё работал. Находим гонку по управлению каналом.

## Delve и CI/CD

Delve можно использовать в пайплайнах для автоматической отладки падающих тестов:

```bash
dlv test -- -test.run TestFlaky &
DLV_PID=$!
sleep 5
# отправить команды через API Delve
```

Но на практике это применяется редко: flaky-тесты обычно отлавливают через `-race` и `-count`.

## Итог

- **Delve** — интерактивный отладчик Go, работающий через `ptrace` и использующий DWARF-информацию бинарника.
- Основные возможности: точки останова, пошаговое исполнение, инспекция переменных и горутин, анализ core dump, удалённая отладка.
- Незаменим для поиска логических ошибок, расследования паник и дедлоков, когда проблему можно воспроизвести локально.
- Имеет значительный overhead и разрушает временные характеристики, поэтому не применяется на production.
- Интегрирован с VSCode и GoLand, поддерживает headless-режим для отладки в контейнерах.
- В комбинации с race detector, дампом горутин и трассировщиком образует полный арсенал для отладки конкурентных программ.
- Понимание ограничений Delve (влияние на планировщик, GC, сетевые тайм-ауты) — часть механической эмпатии Senior-инженера.
