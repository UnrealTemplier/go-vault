## Роль Nginx в архитектуре Go-сервиса

Go обладает встроенным HTTP-сервером (`net/http`), который является production-ready, производительным и конкурентно-безопасным. Однако в реальных высоконагруженных системах Go-сервис редко смотрит напрямую в интернет. Nginx выступает в роли «шлюза» (Edge Server), беря на себя тяжелую работу по обработке сетевых протоколов, безопасности и буферизации, освобождая Go для выполнения бизнес-логики.

### 1. Почему не просто net/http?

Если Go так хорош, зачем нам прокси? Ответ кроется в разделении ответственности и механике ОС.

*   **Защита от Slow Clients (Slowloris)**: Go-горутины дешевы, но не бесплатны. Если клиент отправляет заголовки по одному байту в минуту, горутина будет висеть в `Read`, потребляя память стека. Nginx полностью буферизует запрос в своей памяти и передает его в Go только когда запрос сформирован целиком.
*   **TLS Termination**: Рукопожатие TLS (RSA/ECDHE) требует значительных ресурсов CPU. Nginx оптимизирован для этого (использует OpenSSL напрямую), а Go реализует криптографию в чистом Go. Вынос TLS на Nginx снижает нагрузку на CPU воркеров Go.
*   **Обслуживание статики**: Nginx использует `sendfile` (zero-copy), отдавая файлы прямо из кэша диска в сетевой буфер ядра. Go вынужден копировать данные из ядра в User Space (Read) и обратно в ядро (Write), что нагружает CPU и шину памяти.

```mermaid
graph LR
    A["Клиент"] -->|"TLS / HTTP запрос"| B["Nginx"]
    B -->|"1 - TLS Handshake"| C["Расшифровка<br>в ядре или user space"]
    C -->|"2 - Буферизация тела"| D["Накопление запроса<br>в памяти Nginx"]
    D -->|"3 - Передача"| E["Go net/http Server"]
    E -->|"4 - Бизнес логика"| F["Обработка в горутине"]
```

> [!info] Под капотом
> Директива `sendfile` в Nginx позволяет избежать двойного копирования данных. Без `sendfile`: Disk -> Kernel Buffer -> User Buffer (Nginx) -> Kernel Socket Buffer -> NIC. С `sendfile`: Disk -> Kernel Buffer -> Kernel Socket Buffer -> NIC. Данные никогда не покидают пространство ядра, что экономит такты CPU и пропускную способность шины памяти.

### 2. Базовая конфигурация Reverse Proxy

Минимальная конфигурация для проксирования трафика на Go-приложение, запущенное на порту 8080.

```nginx
# Конфигурация upstream (пул бэкендов)
upstream go_backend {
    # Используем IP-адреса подов или контейнеров
    server 127.0.0.1:8080;
    
    # Важно: keepalive для переиспользования соединений между Nginx и Go
    keepalive 32; 
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://go_backend;
        
        # Передача заголовков для Go
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # HTTP/1.1 для работы keepalive с бэкендом
        proxy_http_version 1.1;
        proxy_set_header Connection "";
    }
}
```

> [!warning] Ловушка / Gotcha
> **Сброс заголовка Connection**: По умолчанию Nginx закрывает соединение с бэкендом после каждого запроса (HTTP/1.0 behavior). Установка `proxy_http_version 1.1` и пустого `Connection` заголовка включает постоянные соединения. Это критично для Go, так как создание нового TCP-соединения и рукопожатие дороже, чем обработка самого запроса. Без этого Go-сервер будет тратить ресурсы на `Accept` и аллокацию новых горутин вместо работы.

### 3. Буферизация и защита ресурсов

Nginx должен защищать Go от медленных клиентов и слишком больших запросов.

```nginx
server {
    # Максимальный размер тела запроса (например, для загрузки аватарок)
    client_max_body_size 10M;

    location /api/ {
        proxy_pass http://go_backend;

        # Буферизация запроса в файл или память
        # Если тело больше proxy_buffer_size, оно пишется во временный файл
        proxy_request_buffering on;
        proxy_buffer_size 4k;
        proxy_buffers 8 4k;

        # Буферизация ответа от Go
        # Nginx прочитает ответ от Go быстро и будет отдавать его клиенту
        # медленно, не блокируя горутину Go.
        proxy_buffering on;
    }
}
```

**Mechanical Sympathy**: Если `proxy_request_buffering on`, Nginx читает `RequestBody` полностью. Горутина в Go получает запрос уже целиком. Если `off`, Go начинает читать поток по мере поступления. Для Go это означает, что горутина активна все время, пока клиент шлет данные. Включение буферизации предпочтительно для защиты от медленных клиентов.

### 4. WebSocket и SSE (Server-Sent Events)

Go часто используется для Real-Time приложений (WebSockets). Nginx по умолчанию буферизирует ответы и закрывает соединения, что ломает WS.

```nginx
location /ws/ {
    proxy_pass http://go_backend;
    
    proxy_http_version 1.1;
    
    # Проброс заголовков обновления протокола
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    
    # Отключаем буферизацию для стриминга в реальном времени
    proxy_buffering off;
    proxy_cache off;
    
    # Таймауты для долгих соединений
    proxy_read_timeout 86400s;
    proxy_send_timeout 86400s;
}
```

> [!tip] Собеседование
> **Вопрос:** Почему WebSocket соединение может разрываться через 60 секунд за Nginx?
> **Ответ:** Nginx имеет дефолтный `proxy_read_timeout` (обычно 60с). Если WebSocket не шлет данные (ping/pong), Nginx считает соединение мертвым и рвет его. Нужно увеличивать таймауты или настраивать `proxy_read_timeout` специально для локации WebSocket.

### 5. Передача реального IP (Real IP)

Когда трафик идет через Nginx, Go видит `RemoteAddr` как `127.0.0.1` (или IP пода). Чтобы получить реальный IP клиента в Go, нужно настроить модуль `real_ip` и доверять заголовкам.

**Nginx:**
```nginx
# Доверяем заголовкам от балансировщика или CDN
set_real_ip_from 10.0.0.0/8;
set_real_ip_from 172.16.0.0/12;
real_ip_header X-Forwarded-For;
real_ip_recursive on;
```

**Go:**
В Go `r.RemoteAddr` все еще может содержать IP прокси, если вы не используете библиотеки типа `github.com/gorilla/handlers` или кастомный парсинг заголовка `X-Real-IP`.
> [!warning] Безопасность
> Никогда не доверяйте заголовку `X-Forwarded-For` без проверки `RemoteAddr`. Злоумышленник может подделать этот заголовок. `set_real_ip_from` в Nginx гарантирует, что IP заменен только если запрос пришел от доверенного прокси.

### 6. Производительность и Tuning

Настройки `nginx.conf`, влияющие на throughput:

1.  **worker_processes**: Обычно `auto` (по числу ядер CPU). Nginx использует неблокирующий I/O (`epoll`), поэтому один воркер может обрабатывать тысячи соединений.
2.  **worker_connections**: Лимит соединений на воркер. `worker_processes * worker_connections` = максимальная пропускная способность.
3.  **sendfile on**: Обязательно для статики.
4.  **tcp_nopush on** и **tcp_nodelay on**: Оптимизация отправки пакетов. `tcp_nopush` заставляет kernel ждать заполнения сегмента перед отправкой (уменьшает overhead), `tcp_nodelay` отключает алгоритм Нейгла для интерактивности.

### 7. Сравнение: Nginx vs Go net/http

| Функция | Nginx | Go net/http |
|---|---|---|
| **Статика** | `sendfile` (Zero-copy), очень быстро | Чтение в память -> Запись (User-space copy) |
| **Модель конкурентности** | Event-driven (Epoll), асинхронный | G-M-P Scheduler, горутины |
| **TLS** | OpenSSL (C), аппаратное ускорение | crypto/tls (Go), программное |
| **Буферизация** | Гибкая настройка буферов запроса/ответа | Потоковая обработка (Streaming) |
| **Динамический контент** | Проксирование (CGI/uWSGI/FastCGI) | Нативная поддержка |

### Итог

1. Nginx — это защита периметра. Он буферизует запросы, защищая Go от медленных клиентов (Slowloris).
2. Используйте `proxy_http_version 1.1` и `keepalive` для снижения overhead на создание соединений с Go.
3. Для WebSocket отключайте буферизацию (`proxy_buffering off`) и пробрасывайте заголовки `Upgrade`.
4. Nginx эффективнее Go в отдаче статики благодаря syscall `sendfile`.
5. Настройте `client_max_body_size`, чтобы избежать загрузки гигантских файлов в память Go-процесса.
6. Реализуйте корректную передачу IP через `real_ip` модуль.
