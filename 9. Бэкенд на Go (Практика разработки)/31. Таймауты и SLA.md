## Философия таймаутов и SLA

Таймаут в распределенной системе — это не просто "время ожидания ответа". Это архитектурный контракт, ограничивающий время жизни операции, объем потребляемых ресурсов и гарантирующий детерминированное поведение при деградации зависимостей. Без строгих таймаутов временная проблема во внешнем API превращается в каскадный отказ, пулы соединений истощаются, а горутины накапливаются до OOM.

SLA, SLO и SLI задают математические границы для этих таймаутов. Инженерная задача — перевести бизнес-требования надежности в конкретные миллисекунды на уровне кода, контекста и сетевых стеков.

1. Иерархия таймаутов и правило убывания

В микросервисной архитектуре таймауты образуют строгую иерархию. Критическое правило: **таймаут на каждом следующем уровне должен быть меньше предыдущего**, чтобы оставить запас времени на обработку ошибок, ретраи и логирование.

```mermaid
flowchart LR
    A["Мобильный клиент"] -->|"3.0 с"| B["API Gateway / LB"]
    B -->|"2.5 с"| C["Go Сервис"]
    C -->|"1.5 с"| D["База данных / Внешний API"]
    D -->|"Ответ или отмена"| C
    C -->|"Ответ или 504"| B
    B -->|"Ответ или 504"| A
```

Если таймаут базы данных равен таймауту шлюза, то при деградации БД сервис будет удерживать соединения до последнего момента. Шлюз завершит клиентскую сессию, а серверные горутины останутся заблокированными, потребляя память и файловые дескрипторы.

2. Под капотом. Механика таймеров в рантайме Go

Когда вы вызываете `context.WithTimeout` или `time.Sleep`, рантайм создает объект `runtime.timer`. Таймеры хранятся в глобальной минимальной куче (min-heap), отсортированной по времени срабатывания.

- Вставка таймера: `O(log n)` через `heap.Push`
- Проверка истекших таймеров: выделенная системная горутина `sysmon` периодически сканирует кучу, извлекает просроченные таймеры и пробуждает ожидающие горутины
- Отмена таймера: `timer.Stop()` удаляет элемент из кучи за `O(log n)`. Если горутина уже пробуждена, операция игнорируется
- Системные вызовы: `net.Conn.SetDeadline()` транслируется в `epoll_ctl(EPOLL_CTL_MOD)` с `EPOLLOUT`/`EPOLLIN` и таймером в ядре. Это позволяет ядру прерывать блокирующий `read()`/`write()` без участия Go-планировщика

> [!info] Под капотом
> Создание `context.WithTimeout` выделяет две структуры в куче: `cancelCtx` и `timerCtx`. При 100 000 RPS это генерирует ~1-2 МБ аллокаций в секунду. Для ultra-low-latency путей предпочтительнее `context.WithDeadline` с предварительно рассчитанным абсолютным временем `time.Now().Add(d)`, что позволяет компилятору применить escape analysis оптимизации и разместить часть контекста в стеке.

3. Идиоматичная настройка в Go

```go
package main

import (
    "context"
    "net/http"
    "time"
)

func setupClient() *http.Client {
    return &http.Client{
        // Timeout покрывает весь цикл: connect -> request -> response -> body read
        Timeout: 2 * time.Second,
        Transport: &http.Transport{
            MaxIdleConns:        100,
            IdleConnTimeout:     90 * time.Second,
            TLSHandshakeTimeout: 500 * time.Millisecond,
        },
    }
}

func handlerWithTimeouts(w http.ResponseWriter, r *http.Request) {
    // Контекст от родителя + явный таймаут для конкретной операции
    ctx, cancel := context.WithTimeout(r.Context(), 1*time.Second)
    defer cancel() // Обязательно для освобождения таймера

    req, err := http.NewRequestWithContext(ctx, http.MethodGet, "https://api.example.com/data", nil)
    if err != nil {
        http.Error(w, "request build failed", http.StatusInternalServerError)
        return
    }

    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        // Различаем отмену и таймаут
        if ctx.Err() == context.DeadlineExceeded {
            http.Error(w, "dependency timeout", http.StatusGatewayTimeout)
            return
        }
        http.Error(w, "request failed", http.StatusBadGateway)
        return
    }
    defer resp.Body.Close()

    // Дальнейшая обработка...
}
```

> [!warning] Ловушка / Gotcha
> **`http.Client.Timeout` vs `context.Context`**: `Timeout` в клиенте грубый. Он отменяет весь запрос, включая чтение тела. Если вам нужен точечный таймаут только на установку соединения или на запрос без чтения ответа, используйте `context.WithTimeout` + `http.RoundTripper`. Никогда не смешивайте оба механизма без четкого понимания приоритетов: контекст всегда имеет высший приоритет отмены.
> **Невызванный `cancel()`**: Если вы создаете таймаут-контекст и забываете `defer cancel()`, таймер останется в куче рантайма до срабатывания. При тысячах запросов в секунду это приводит к накоплению "зомби-таймеров", утечке памяти и лишним пробуждениям планировщика.

4. Расчет таймаутов через SLA и Error Budgets

Таймауты не берутся "с потолка". Они выводятся из SLO (Service Level Objectives) и latency-профиля зависимостей.

- **SLI**: Фактическая метрика (например, `http_request_duration_seconds{quantile="0.99"}`)
- **SLO**: Целевой уровень (например, 99.9% запросов должны отвечать быстрее 200 мс)
- **Error Budget**: `100% - SLO = 0.1%`. Это допустимое время сбоев в месяц (~43 минуты)
- **Таймаут сервиса**: Должен быть меньше SLI p99, чтобы оставить запас на ретраи. Формула: `Service Timeout = p99(dependency) + network jitter + retry backoff headroom`

Пример: Зависимость отдает данные за p99=80 мс. Сетевой джиттер внутри кластера ~10 мс. Планируется 1 ретрай с задержкой 50 мс. 
`Таймаут = 80 + 10 + 50 + 20 (буфер) ≈ 160 мс`.

> [!tip] Собеседование
> **Вопрос:** В чем разница между `Deadline` и `Timeout` в контексте?
> **Ответ:** `Timeout` — относительное время от текущего момента (`now + d`). `Deadline` — абсолютное время в будущем (`time.Time`). Под капотом `WithTimeout` вызывает `WithDeadline(time.Now().Add(d))`. Абсолютный дедлайн предпочтительнее при каскадных вызовах, так как он корректно сокращается при пробросе контекста через несколько сервисов, тогда как относительный таймаут будет суммироваться и может превысить общий бюджет времени.
> 
> **Вопрос:** Что произойдет, если установить `http.Client.Timeout: 1s`, а `context.WithTimeout` на 500 мс?
> **Ответ:** Контекст отменит операцию через 500 мс. Клиентский таймаут сработает только если контекст все еще активен. При отмене контекста `http.Client` закрывает соединение, возвращает `context.DeadlineExceeded`, а 1-секундный таймаут игнорируется.

5. Ловушки и антипаттерны

- **Блокировка на канале без таймаута**: `result := <-ch` будет висеть вечно, если канал никогда не получит данные. Всегда используйте `select { case <-ch: case <-ctx.Done(): }`.
- **Двойные таймауты на одном уровне**: Настройка `ReadTimeout` на сервере и `WriteTimeout` меньше `ReadTimeout` приведет к разрыву соединения во время обработки. `WriteTimeout` должен покрывать время генерации ответа + сетевую задержку.
- **Игнорирование `ctx.Err()` после блокирующей операции**: Если `db.QueryContext` вернул ошибку из-за таймаута, но вы продолжаете логику, сервис может работать с частичными данными. Всегда проверяйте `errors.Is(err, context.DeadlineExceeded)` и возвращайте `504 Gateway Timeout`.
- **Глобальные таймауты для фоновых задач**: Не используйте `http.Client.Timeout` для воркеров очередей. Там применяются стратегии `context.WithTimeout` с динамическим увеличением или асинхронные очереди с DLQ.

6. Производительность и Mechanical Sympathy

- **Аллокации таймеров**: `context.WithTimeout` создает `timerCtx` в куче. При использовании `sync.Pool` для переиспользования структур запросов и явном `timer.Reset()` можно сократить давление на GC на 20-30%.
- **Кэш-линии и планировщик**: При отмене контекста рантайм итерирует дерево дочерних `cancelCtx`. Глубокие вложенности разбрасывают указатели по памяти, вызывая cache misses. Держите глубину цепочки контекстов менее 10 уровней.
- **Системные вызовы**: `SetDeadline` вызывает `epoll_ctl` на каждое изменение. Для высоконагруженных пулов соединений (например, в gRPC или кастомных TCP-клиентах) используйте `netpoll` напрямую или пулируйте соединения с единым таймаутом на весь пул, а не на каждый `net.Conn`.

7. Итог

8. Таймауты — это архитектурные контракты, а не "костыли" для медленного кода.
9. Иерархия должна убывать: Клиент > Шлюз > Сервис > Зависимость.
10. `context.WithTimeout` + `defer cancel()` — обязательный паттерн для каждой внешней операции.
11. Рассчитывайте таймауты на основе p99 latency зависимостей, сетевого джиттера и бюджета на ретраи.
12. Никогда не блокируйте горутины на каналах или вызовах без проверки `ctx.Done()`.
13. Различайте `context.Canceled` (намеренная отмена) и `context.DeadlineExceeded` (истекло время) для точной маршрутизации ошибок.
14. Оставляйте error budget: SLA должно превышать SLO, а таймауты — учитывать вероятность каскадных деградаций.
