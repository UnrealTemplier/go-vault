## Логирование в production

В production-окружении логи перестают быть средством отладки для разработчика и становятся источником данных для SRE, системы мониторинга и машинного обучения. Главный сдвиг парадигмы: **лог — это структурированная запись события**, а не строка для чтения человеком. В Go это означает отказ от `fmt.Printf` в пользу строго типизированных библиотек, минимизирующих аллокации и обеспечивающих консистентность контекста.

### 1. Цена логирования: Механика и производительность

Запись лога кажется тривиальной операцией, но в высоконагруженном сервисе (10k+ RPS) она становится одним из главных потребителей CPU и генераторов мусора для GC.

Рассмотрим типичный вызов: `log.Printf("User %s logged in from %s", user, ip)`.
1.  **Форматирование**: Функция `fmt.Sprintf` (внутри `Printf`) аллоцирует буфер, форматирует строку, создает новую строку в куче.
2.  **Сериализация**: Если используется JSON-логгер, поля превращаются в байтовый срез.
3.  **Системный вызов**: Запись в файл или сокет требует переключения контекста в Kernel Space (`write` syscall).
4.  **Синхронизация**: Стандартный логгер использует мьютекс для защиты буфера записи. При конкуренции тысячи горутин ждут блокировку.

> [!info] Под капотом
> В Go 1.21+ библиотека `log/slog` оптимизирует путь `slog.Info`. Для примитивов (`int`, `string`) она избегает упаковки в `interface{}`. Однако, если вы передаете сложную структуру или используете `slog.Any`, включается рефлексия (`reflect`), что заставляет данные мигрировать в кучу (Escape Analysis). В production это создает "шум" для Garbage Collector. Каждый лишний байт в логе умножается на миллионы запросов, превращаясь в гигабайты аллокаций в секунду.

### 2. Идиоматичная структура лога

В production логи должны быть машиночитаемыми (обычно JSON). Поля должны быть предсказуемыми, чтобы система агрегации (Loki, Elasticsearch) могла их индексировать.

```json
{
  "level": "error",
  "ts": "2023-10-27T10:00:00.123Z",
  "msg": "failed to process payment",
  "service": "payment-svc",
  "version": "v1.2.3",
  "trace_id": "a1b2c3d4e5f6",
  "user_id": "u_123",
  "error": "timeout",
  "latency_ms": 5000
}
```

В коде Go это реализуется через `slog` с глобальными атрибутами:

```go
package main

import (
    "context"
    "log/slog"
    "os"
    "runtime"
)

func InitLogger() *slog.Logger {
    opts := &slog.HandlerOptions{
        Level:     slog.LevelInfo, // В продакшене Info, в dev Debug
        AddSource: true,           // Файл и строка (дорого в prod, лучше выключить или собирать редко)
        ReplaceAttr: func(groups []string, a slog.Attr) slog.Attr {
            // Переименование time в ts для совместимости
            if a.Key == slog.TimeKey {
                a.Key = "ts"
            }
            return a
        },
    }

    // JSON Handler быстрее читается машинами, чем Text
    handler := slog.NewJSONHandler(os.Stdout, opts)
    
    // Добавляем глобальные атрибуты, которые будут в КАЖДОМ логе
    return slog.New(handler).With(
        slog.String("service", os.Getenv("SERVICE_NAME")),
        slog.String("version", os.Getenv("APP_VERSION")),
        slog.String("env", os.Getenv("ENVIRONMENT")),
    )
}
```

### 3. Контекст и Request ID

Лог без идентификатора запроса бесполезен в микросервисах. Вы должны связывать логи разных сервисов в единую цепочку. Для этого используется `context.Context`.

```go
func WithRequestID(ctx context.Context, id string) context.Context {
    return context.WithValue(ctx, "request_id", id)
}

func RequestIDFromContext(ctx context.Context) string {
    if id, ok := ctx.Value("request_id").(string); ok {
        return id
    }
    return ""
}

// Middleware
func LoggingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        reqID := RequestIDFromContext(r.Context())
        
        // Создаем дочерний логгер для этого запроса
        logger := slog.Default().With(
            slog.String("request_id", reqID),
            slog.String("method", r.Method),
            slog.String("path", r.URL.Path),
            slog.String("remote_addr", r.RemoteAddr),
        )
        
        // Сохраняем в контекст, чтобы хендлеры использовали его
        ctx := context.WithValue(r.Context(), "logger", logger)
        r = r.WithContext(ctx)
        
        // ... вызов next
    })
}
```

> [!warning] Ловушка / Gotcha
> **Утечка чувствительных данных**: Никогда не логируйте тела запросов/ответов целиком (`req.Body`), заголовки `Authorization` или пароли в явном виде. В production это нарушение GDPR/PCI DSS. Используйте маскирование (например, `CreditCard: ****1234`).
> **Логирование внутри горячего цикла**: Если вы логируете каждый шаг итерации цикла с миллионом элементов, вы положите сервис. Используйте sampling (см. ниже) или логируйте только агрегированные результаты.

### 4. Производительность: Асинхронность и Буферизация

Синхронная запись в файл или сокет блокирует горутину. При всплеске трафика это создает backpressure. В Go для решения этой проблемы используют буферизацию на уровне пользователя или асинхронные логгеры.

**Вариант 1: Буферизация через bufio (Рекомендуемый для большинства)**
`os.Stdout` буферизован, но иногда нужен явный контроль. `slog` сам пишет в `io.Writer`. Если передать туда `bufio.Writer`, мы уменьшим количество `write` syscall.

**Вариант 2: Асинхронный логгер**
Существуют библиотеки (например, `uber/zap` в асинхронном режиме), которые кладут логи в канал, а отдельная горутина пишет их на диск.
*Плюс*: Не блокирует бизнес-логику.
*Минус*: При панике или резком падении процесса последние логи могут потеряться (они висят в канале).

Для Go `slog` достаточно быстр для синхронной работы, если использовать JSON и избегать сложных объектов.

### 5. Уровни логирования и Sampling

В production нельзя писать всё.
- **ERROR**: Сбои, требующие реакции. Пишем всегда.
- **WARN**: Аномалии, которые могут привести к сбою. Пишем всегда.
- **INFO**: Жизненный цикл запроса (Start, End). Пишем всегда (или сэмплируем).
- **DEBUG**: Детали отладки. **Отключено в production**.

**Сэмплирование (Sampling)**:
Если у вас 100k RPS, писать INFO лог на каждый запрос — дорого.
Реализуйте декоратор логгера, который пишет только каждый N-й лог, либо логи, содержащие определенные флаги (например, `is_slow=true`).

```go
type SamplingHandler struct {
    handler slog.Handler
    rate    int
    counter int64 // atomic
}

func (h *SamplingHandler) Handle(ctx context.Context, r slog.Record) error {
    // Всегда пишем ошибки
    if r.Level >= slog.LevelError {
        return h.handler.Handle(ctx, r)
    }
    
    // Для Info/Debug пропускаем часть
    // Реализация через atomic counter
    return nil 
}
```

### 6. Агрегация и Хранение

Логи с подов ephemeralны. Если под пересоздался, логи исчезли. Необходим централизованный стэк.
Обычно используется агент (Fluent Bit, Filebeat, Vector), который читает файлы логов (или stdout контейнера) и отправляет их в хранилище.

- **Loki**: Индексирует только метаданные (labels), содержимое логов сжимается. Дешевле Elasticsearch, идеален для Kubernetes.
- **Elasticsearch**: Полнотекстовый поиск по содержимому. Мощно, но дорого (RAM/Disk).
- **CloudWatch / Stackdriver**: Управляемые решения.

> [!tip] Собеседование
> **Вопрос:** Почему стандартный `log` пакет плох для микросервисов?
> **Ответ:** Он пишет в stdout/stderr без структуры (просто текст), не поддерживает уровни (info/warn/error) из коробки, не имеет контекста. Парсить такие логи в ELK/Loki сложно и дорого. В Go нужно использовать `log/slog` (Go 1.21+) или `zap`/`logrus`.
> 
> **Вопрос:** Как логировать панику, чтобы она попала в систему сбора логов?
> **Ответ:** Использовать `defer recover()`. Внутри recover извлечь стек (через `runtime/debug.Stack()`) и записать его через основной логгер. Важно не просто напечатать в stderr, а отправить структурированный JSON, иначе лог потеряется при ротации.

### 7. Итог

1. В production логи — это данные, а не текст. Используйте JSON и структурированное логирование.
2. Избегайте аллокаций в хот-пути: используйте `slog` с примитивами, избегайте `interface{}`.
3. Обязательно добавляйте `request_id` и `trace_id` в контекст каждого запроса.
4. Никогда не логируйте чувствительные данные (PII, Secrets).
5. Используйте буферизацию для снижения количества системных вызовов.
6. Настройте ротацию и агрегацию логов (Loki/ELK), так как диски контейнеров временны.
7. Применяйте sampling для высоконагруженных INFO логов, чтобы экономить ресурсы.
