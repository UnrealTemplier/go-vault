## Философия работы с Redis в Go

Redis — это не просто in-memory key-value хранилище. Это однопоточный сервер событий (event loop), управляющий сложными структурами данных (strings, hashes, lists, sets, streams, geospatial) с детерминированной латентностью. В экосистеме Go работа с Redis отличается от PHP или Java: здесь нет блокирующих тредов ОС на каждый запрос, а вся конкурентность управляется планировщиком Go и `netpoll`. Правильная интеграция с Redis требует понимания пулов соединений, протокола сериализации, атомарности операций и влияния парсинга ответов на сборщик мусора.

## 1. Архитектура сервера и протокол RESP

Redis использует текстовый бинарный протокол **RESP** (Redis Serialization Protocol). В версии 3 (RESP3) добавлены типизированные ответы, пуш-уведомления и нулевые указатели, что упрощает парсинг на стороне клиента.

Каждая команда отправляется как массив строк. Например, `GET user:1` превращается в:
```
*2\r\n$3\r\nGET\r\n$6\r\nuser:1\r\n
```
Ответ парсится посимвольно. В Go клиент `github.com/redis/go-redis/v9` оптимизировал этот процесс, избегая промежуточных аллокаций через буферы `[]byte` и прямой парсинг в структуры.

```mermaid
flowchart LR
    A["Горутина вызывает<br>команду Redis"] --> B["Запрос попадает<br>в пул соединений"]
    B --> C["Сериализация в RESP<br>буферизация в conn"]
    C --> D["syscall write<br>отправка в сокет"]
    D --> E["Ожидание через<br>epoll netpoll"]
    E --> F["syscall read<br>чтение ответа"]
    F --> G["Парсинг RESP3<br>десериализация в Go-типы"]
    G --> H["Возврат результата<br>горутине"]
```

> [!info] Под капотом
> Redis однопоточный. Это значит, что одна блокирующая операция (`KEYS *`, `HGETALL` на гигантский хеш, сложный Lua-скрипт) останавливает обработку всех остальных клиентов. В Go это особенно критично, так как тысячи горутин могут одновременно ждать ответа от одного зависшего сокета, потребляя память и создавая contention на `netpoll`.

## 2. Пул соединений и механика netpoll

`go-redis` использует собственный пул соединений, оптимизированный под высокую конкуренцию. В отличие от `database/sql`, пул Redis работает через каналы и атомарные счетчики, минимизируя блокировки.

```go
package cache

import (
    "context"
    "time"

    "github.com/redis/go-redis/v9"
)

func NewRedisClient(addr string) *redis.Client {
    return redis.NewClient(&redis.Options{
        Addr:         addr,
        PoolSize:     50,              // Максимум одновременных соединений
        MinIdleConns: 10,              // Предварительно открытые соединения
        ConnMaxIdleTime: 10 * time.Minute,
        ConnMaxLifetime: 30 * time.Minute,
        ReadTimeout:  200 * time.Millisecond,
        WriteTimeout: 200 * time.Millisecond,
    })
}

func GetUser(ctx context.Context, rdb *redis.Client, userID int64) (string, error) {
    key := fmt.Sprintf("user:%d:profile", userID)
    val, err := rdb.Get(ctx, key).Result()
    if err != nil {
        if err == redis.Nil { // Ключ не найден
            return "", nil
        }
        return "", fmt.Errorf("redis get: %w", err)
    }
    return val, nil
}
```

> [!info] Под капотом
> При вызове `rdb.Get()` горутина запрашивает соединение из пула. Если пул пуст, горутина блокируется на канале `pool.NewConn()`. После получения соединения вызывается `net.Conn.Read/Write`. Эти методы в Linux используют `netpoll` (обертка над `epoll`). Горутина переходит в состояние `_Gwaiting`, не потребляя тред ОС. Когда сокет готов к чтению/записи, `epoll` будит горутину, и планировщик перекидывает её в очередь P. Это позволяет одному треду ОС обрабатывать десятки тысяч одновременных Redis-запросов с латентностью < 1 мс.

## 3. Пайплайнинг и мультиплексирование

Главный инструмент оптимизации Redis в Go — **Pipeline**. Он группирует несколько команд в один системный вызов `write`, отправляет их пакетом, а затем читает все ответы через один `read`. Это сокращает RTT и количество syscall с `N` до `2`.

```go
func BatchUpdateUsers(ctx context.Context, rdb *redis.Client, users []*User) error {
    pipe := rdb.Pipeline()
    
    for _, u := range users {
        key := fmt.Sprintf("user:%d:profile", u.ID)
        data, err := json.Marshal(u)
        if err != nil {
            return fmt.Errorf("marshal user %d: %w", u.ID, err)
        }
        pipe.Set(ctx, key, data, 10*time.Minute)
    }
    
    // Выполнение всех команд одним пакетом
    _, err := pipe.Exec(ctx)
    if err != nil {
        return fmt.Errorf("pipeline exec: %w", err)
    }
    return nil
}
```

> [!warning] Ловушка / Gotcha
> `Pipeline` в `go-redis` **не гарантирует атомарность**. Команды отправляются пакетом, но Redis выполняет их последовательно, не блокируя других клиентов. Если между командами нужна атомарность, используйте `MULTI/EXEC` (транзакции Redis) или Lua-скрипты (`EVAL`). Транзакции Redis не откатываются при ошибках выполнения команд (например, `WRONGTYPE`), а продолжают выполнение, помечая команды как failed.

## 4. Сериализация, аллокации и GC-давление

`encoding/json` активно использует рефлексию. Каждый `json.Marshal`/`json.Unmarshal` создает временные структуры в куче. При тысячах RPS это вызывает всплеск аллокаций, попадающих в молодое поколение GC, что увеличивает микро-паузы.

**Оптимизация:**
1. Используйте `go-redis` методы `.Scan()` для прямых преобразований типов.
2. Для hot-path применяйте `msgpack` или `protobuf`, они быстрее JSON на 30-50% и требуют меньше аллокаций.
3. Предварительно выделяйте буферы через `sync.Pool`.

```go
func GetUserTyped(ctx context.Context, rdb *redis.Client, userID int64) (*User, error) {
    key := fmt.Sprintf("user:%d:profile", userID)
    
    // Прямое чтение в []byte, минуя строковые аллокации
    val, err := rdb.Get(ctx, key).Bytes()
    if err != nil {
        if err == redis.Nil {
            return nil, nil
        }
        return nil, err
    }

    var u User
    if err := json.Unmarshal(val, &u); err != nil {
        return nil, fmt.Errorf("unmarshal: %w", err)
    }
    return &u, nil
}
```

> [!info] Под капотом
> Метод `.Bytes()` возвращает `[]byte`, который ссылается на внутренний буфер парсера. Если вы используете его после завершения контекста или закрытия соединения, возможны data race. Всегда копируйте данные (`make([]byte, len(val)); copy(dst, val)`), если планируете хранить их дольше生命周期 обработчика. Escape Analysis в Go не может доказать безопасность возврата сырого буфера из драйвера, поэтому `.Bytes()` часто аллоцирует копию.

## 5. Ловушки production и антипаттерны

- **Блокирующие команды и пул**: `BLPOP`, `BRPOP`, `XREAD BLOCK` ждут данные. Они блокируют соединение на всё время ожидания. Если `PoolSize=50` и 50 горутин вызовут `BLPOP`, пул исчерпается. Решение: выделяйте отдельный клиент с большим `PoolSize` для pub/sub/streams или используйте асинхронные воркеры.
- **Контекст и отмена**: `go-redis/v9` полностью поддерживает `context.Context`. При отмене контекста клиент помечает соединение как `bad` и закрывает его. Команда на стороне Redis **не отменяется** (Redis однопоточный, отмена невозможна без `CLIENT KILL`). Это приводит к выполнению команды "в холостую".
- **`KEYS` и `SCAN`**: `KEYS pattern` блокирует Redis O(N). В production запрещен. Используйте `SCAN` с курсором и `COUNT`, но помните, что `SCAN` тоже может нагружать сервер при высокой дельте изменений.
- **Memory Fragmentation**: Redis использует `jemalloc` или системный `malloc`. При активном удалении ключей возникает фрагментация памяти (`mem_fragmentation_ratio > 1.5`). Это не лечится на стороне Go, но мониторинг метрик Redis обязателен.

> [!tip] Собеседование
> **Вопрос:** В чем разница между Redis Pipeline и Redis Transaction (MULTI/EXEC)?
> **Ответ:** Pipeline — это сетевая оптимизация. Команды отправляются пакетом для сокращения RTT, но выполняются Redis последовательно без гарантии атомарности. Transaction оборачивает команды в `MULTI/EXEC`, гарантируя, что они будут выполнены атомарно без вмешательства других клиентов. Однако транзакции Redis не поддерживают откат при ошибках выполнения команд. Для сложных атомарных операций предпочтительнее Lua-скрипты (`EVALSHA`), которые выполняются как одна команда.
> 
> **Вопрос:** Как безопасно реализовать distributed lock на Redis в Go?
> **Ответ:** Используйте `SET key value NX EX timeout` с уникальным значением (request ID) и проверяйте его перед `DEL`. Никогда не делайте `GET + SET` или `EXPIRE` отдельно — это race condition. В production используйте библиотеку `github.com/bsm/redislock` или `go-redsync`, которые реализуют корректную проверку владельца и автоматическое продление (watchdog).

## 6. Сравнение с другими экосистемами

| Аспект | PHP (phpredis) | Java (Jedis / Lettuce) | Python (redis-py) | Go (go-redis/v9) |
|---|---|---|---|---|
| Конкурентность | Синхронный, блокирует тред PHP-FPM | Jedis (sync, блокирует), Lettuce (async, Netty) | GIL блокирует, асинхронность через asyncio | Горутины + netpoll, неблокирующая асинхронность |
| Пул соединений | Отсутствует (одно соединение на процесс) | Встроенный (Apache Commons Pool) | ConnectionPool с thread-safety | Высокооптимизированный, lock-free частично |
| Пайплайнинг | `multi()` или `pipeline()` (sync) | `pipeline()` (async batch) | `pipeline()` (sync buffer) | `Pipeline()` (async, batch write/read) |
| Overhead | Минимальный (C-расширение), но блокирующий | JVM GC pressure, Netty overhead | GIL contention, object creation | Низкий, аллокации при парсинге, оптимизируемы |

## 7. Итог

1. Redis однопоточен: избегайте тяжелых команд, используйте `SCAN` вместо `KEYS`, мониторьте латентность.
2. `go-redis/v9` + `context` + `netpoll` обеспечивают максимальную пропускную способность без блокировок тредов ОС.
3. Пул соединений настраивайте под пиковую нагрузку: `PoolSize`, `MinIdleConns`, таймауты обязательны.
4. Пайплайнинг сокращает RTT и syscall, но не дает атомарности. Для атомарности используйте Lua-скрипты.
5. Сериализация через `json` создает GC-давление. Для highload применяйте бинарные форматы или прямые `.Bytes()`/`.Scan()`.
6. Контекст отменяет ожидание на клиенте, но не прерывает выполнение на стороне Redis. Учитывайте это при таймаутах.
7. Distributed locks требуют атомарного `SET NX EX` и строгой проверки владельца. Никогда не реализуйте кастомные замки без understanding edge cases.
