
Многие разработчики сразу тянутся к генераторам моков (`gomock`, `mockery`), как только видят интерфейс. Однако в Go-сообществе существует сильная культура **ручных моков (Hand-written Mocks)**. Для Senior-инженера умение написать мок руками — это не просто упражнение, а способ держать тесты чистыми, быстрыми и лишенными лишних зависимостей.

Ручной мок зачастую понятнее, чем сгенерированный код на 500 строк, и он не требует запуска кодогенерации при каждом изменении интерфейса.

## Структура идеального ручного мока

Хороший ручной мок должен выполнять две функции:
1. **Stubbing:** Возвращать предопределенные значения.
2. **Spying:** Сохранять информацию о вызовах для последующей проверки (ассертов).

Рассмотрим пример с интерфейсом почтового сервиса:

```go
type EmailSender interface {
    Send(ctx context.Context, to, subject, body string) error
}
```

### Паттерн "Функциональный мок"

Это самый гибкий способ реализации ручного мока. Вместо того чтобы хардкодить возвращаемые значения в поля, мы используем поля-функции. Это позволяет менять поведение мока прямо "на лету" внутри разных подтестов `t.Run`.

```go
// email_mock_test.go

type EmailSenderMock struct {
    // Поля-функции позволяют задавать поведение в каждом тесте
    SendFunc func(ctx context.Context, to, subject, body string) error

    // Поля для шпионажа (Spying)
    Calls []SendCall
}

type SendCall struct {
    To, Subject, Body string
}

func (m *EmailSenderMock) Send(ctx context.Context, to, subject, body string) error {
    // Записываем вызов для проверки в ассертах
    m.Calls = append(m.Calls, SendCall{to, subject, body})
    
    // Вызываем функциональное поле, если оно задано
    if m.SendFunc != nil {
        return m.SendFunc(ctx, to, subject, body)
    }
    return nil
}
```

## Использование в тестах

Благодаря такой структуре, мы можем легко тестировать как "счастливые пути", так и ошибки:

```go
func TestNotifyUser(t *testing.T) {
    t.Run("success", func(t *testing.T) {
        mock := &EmailSenderMock{
            SendFunc: func(ctx context.Context, to, subject, body string) error {
                return nil
            },
        }
        
        err := NotifyUser(context.Background(), mock, "test@test.com")
        
        if err != nil {
            t.Errorf("unexpected error: %v", err)
        }
        // Проверяем, что вызов был ровно один
        if len(mock.Calls) != 1 {
            t.Errorf("expected 1 call, got %d", len(mock.Calls))
        }
    })

    t.Run("gateway error", func(t *testing.T) {
        mock := &EmailSenderMock{
            SendFunc: func(ctx context.Context, to, subject, body string) error {
                return errors.New("network error")
            },
        }
        
        err := NotifyUser(context.Background(), mock, "test@test.com")
        
        if err == nil {
            t.Error("expected error, got nil")
        }
    })
}
```

## Mechanical Sympathy: Слайсы и конкурентность

При написании ручных моков для высоконагруженных систем или асинхронного кода нужно учитывать два нюанса:

1. **Memory Leaks:** Если ваш мок — это долгоживущий объект (например, в `TestMain` или глобальный фейк), слайс `Calls` будет расти бесконечно. В юнит-тестах это не проблема, но в интеграционных прогонах может привести к раздуванию памяти.
2. **Race Conditions:** Если тестируемый код вызывает мок из разных горутин параллельно, запись в слайс `m.Calls = append(...)` приведет к **Data Race**.

> [!warning] Ловушка / Gotcha
> Если вы ожидаете конкурентных вызовов мока, защитите поля шпионажа мьютексом:
> ```go
> func (m *EmailSenderMock) Send(...) error {
>     m.mu.Lock()
>     m.Calls = append(m.Calls, ...)
>     m.mu.Unlock()
>     // ...
> }
> ```

## Преимущества ручных моков перед генераторами

1. **Отсутствие магии:** Любой разработчик может нажать `F12` и увидеть реализацию мока. Сгенерированные файлы часто замусорены макросами и сложной логикой `reflect`.
2. **Быстрота компиляции:** Нет лишних зависимостей и этапов кодогенерации в Makefile/CI.
3. **Типобезопасность:** Вы работаете с чистым Go. Если вы измените сигнатуру интерфейса, компилятор сразу подсветит ошибку в вашем ручном моке. Генератор же промолчит, и тест упадет только в рантайме или после запуска `go generate`.



## Когда ручные моки НЕ нужны?

Если ваш интерфейс огромен (15+ методов), писать ручной мок — значит плодить сотни строк скучного кода. Это единственный случай, когда кодогенерация оправдана. Однако, как мы помним из статьи [[2. Интерфейсы как инструмент тестирования]], наличие такого интерфейса — повод для рефакторинга в сторону ISP.

## Итог

1. Ручной мок — это структура, реализующая интерфейс через поля-функции и слайсы для записи вызовов.
2. Используйте функциональные поля для изменения поведения мока в разных сценариях.
3. Не забывайте о потокобезопасности, если мок вызывается из горутин.
4. Ручные моки делают тесты "первоклассными" гражданами кодовой базы, а не побочным продуктом генераторов.

Если ручная работа становится слишком объемной, пора звать на помощь тяжелую артиллерию. Переходим к изучению стандарта кодогенерации моков в Go: [[4. gomock. Генерация моков]].
