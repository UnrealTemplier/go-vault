
В предыдущих статьях мы обсуждали, как изолировать код с помощью моков и почему избыточное мокирование (Overmocking) опасно. Однако есть одна критическая проблема, которую моки не решают: **согласованность**.

Если ваш микросервис-потребитель (Consumer) использует мок API другого сервиса (Provider), и этот Provider изменит формат ответа (например, переименует поле `user_id` в `uuid`), ваши Unit-тесты с моками останутся «зелеными», но в production система упадет. **Contract Testing (Контрактное тестирование)** — это способ гарантировать, что две независимые системы «говорят на одном языке» без необходимости запускать их одновременно.

## Что такое Контрактное тестирование?

Контрактное тестирование проверяет не логику работы сервиса, а **границу взаимодействия**. Это формализованное соглашение (контракт) между Потребителем и Поставщиком.

- **Consumer (Потребитель):** Сервис, который вызывает API.
- **Provider (Поставщик):** Сервис, который предоставляет API.
- **Contract (Pact):** JSON-файл, содержащий примеры запросов и ожидаемых ответов.

В отличие от интеграционных тестов (E2E), где нужно поднимать всю инфраструктуру, контрактные тесты проверяют совместимость сторон по отдельности.

## Consumer-Driven Contracts (CDC)

Самый популярный подход в Go — это **Consumer-Driven Contracts** (контракты, управляемые потребителем). Логика проста: тот, кто использует данные, диктует свои требования.

### Как это работает:

1. **На стороне Consumer:** Разработчик пишет тест, который описывает: «Если я отправлю `GET /user/1`, я ожидаю статус 200 и поле `name`». Библиотека (например, **Pact**) генерирует файл-контракт.
2. **Обмен:** Контракт передается Поставщику (через Pact Broker или просто Git).
3. **На стороне Provider:** Библиотека читает контракт, делает реальный запрос к локально запущенному сервису Поставщика и проверяет, соответствует ли ответ требованиям.

## Инструментарий для Go: Pact

**Pact** — это стандарт де-факто для контрактного тестирования в Go.

### Пример на стороне Consumer:

```go
func TestConsumer(t *testing.T) {
    pact := dsl.Pact{
        Consumer: "OrderService",
        Provider: "UserService",
    }

    // Описываем ожидаемое взаимодействие
    pact.
        AddInteraction().
        Given("User 1 exists").
        UponReceiving("A request for User 1").
        WithRequest(dsl.Request{
            Method: "GET",
            Path:   dsl.String("/user/1"),
        }).
        WillRespondWith(dsl.Response{
            Status: 200,
            Body:   dsl.Like(User{Name: "Alice"}), // "Like" проверяет структуру, а не точное значение
        })

    // Запускаем тест, который создаст файл-контракт
    err := pact.Verify(func() error {
        // Здесь вызывается реальный код клиента, который идет в мок-сервер Pact
        return client.GetUser("1")
    })
    assert.NoError(t, err)
}
```

## Почему это лучше интеграционных тестов?

|**Характеристика**|**Интеграционные тесты (E2E)**|**Контрактные тесты (Pact)**|
|---|---|---|
|**Скорость**|Медленные (нужно поднимать всё)|Быстрые (Unit-скорость)|
|**Стабильность**|Хрупкие (сеть, зависимости)|Стабильные (изолированные)|
|**Локализация ошибок**|Сложно понять, кто виноват|Сразу понятно, кто нарушил контракт|
|**Сложность**|Растет экспоненциально от кол-ва сервисов|Растет линейно|

## Mechanical Sympathy: Типизация в Go и контракты

В Go у нас есть строгая типизация структур, но она работает только внутри одного бинарника. При передаче данных по HTTP через JSON типизация теряется.

> [!info] Под капотом
> 
> Библиотеки контрактного тестирования в Go используют рефлексию для генерации JSON-схем из ваших структур. При верификации на стороне провайдера Pact не просто сравнивает строки, а проверяет соответствие типов (например, что поле, которое потребитель ждет как `string`, не пришло как `int`).

## Ловушки контрактного тестирования

1. **Проверка бизнес-логики:** Не пытайтесь проверить контрактом, правильно ли сервис считает скидку. Контракт должен проверять только «форму» данных (наличие полей, типы, статус-коды).
2. **Версионность:** Если вы изменили контракт, вы не можете просто задеплоить провайдера. Вам нужно убедиться, что _все_ потребители готовы к изменениям. Для этого используется инструмент **Can-I-Deploy**.
3. **Тестовые данные:** Поставщику часто нужно находиться в определенном состоянии (например, «пользователь с ID 1 существует»). Это настраивается через `State handlers` в Pact.

> [!tip] Собеседование
> 
> **Вопрос:** В чем разница между контрактным тестированием и использованием JSON Schema?
> 
> **Ответ:** JSON Schema проверяет только структуру документа. Контрактное тестирование (Pact) проверяет **взаимодействие**: конкретные эндпоинты, методы, параметры запроса и заголовки в контексте конкретных сценариев использования. Кроме того, Pact обеспечивает двустороннюю связь: провайдер обязан подтвердить, что он соблюдает требования потребителя.

## Итог

1. **Contract Testing** решает проблему «дрейфа моков», когда имитация API перестает соответствовать реальности.
2. Это фундамент для безопасного деплоя микросервисов независимо друг от друга.
3. Используйте **Pact** для Go, если у вас сложная распределенная архитектура.
4. Контракт — это не функциональный тест, это проверка совместимости интерфейсов.
