
Когда ваш проект разрастается, а интерфейсы становятся сложнее, ручное написание моков (см. [[3. Ручные моки]]) превращается в рутину. Для автоматизации этого процесса в экосистеме Go существует исторический стандарт — **Uber GoMock**.

`gomock` — это фреймворк для мокирования, который состоит из двух частей: кодогенератора `mockgen` и библиотеки времени выполнения для управления ожиданиями (expectations).

## Почему gomock?

В отличие от библиотек, работающих на чистой рефлексии в рантайме, `gomock` генерирует типизированный код. Это дает:
1. **Строгую проверку типов:** Вы не сможете вернуть `string` там, где метод ожидает `int`.
2. **Гибкое управление порядком:** Можно проверить, что `MethodA` был вызван строго *перед* `MethodB`.
3. **Интеграцию с IDE:** Вы видите методы мока в автодополнении.

## Установка и генерация

Для начала работы необходимо установить генератор:
```bash
go install go.uber.org/mock/mockgen@latest
```

Идиоматичный способ запуска генерации в Go — использование директивы `go:generate` прямо в файле, где определен интерфейс.

```go
// internal/storage/repo.go
package storage

import "context"

//go:generate mockgen -source=$GOFILE -destination=mocks/mock_repo.go -package=mocks
type Repository interface {
    Get(ctx context.Context, id int64) (string, error)
    Save(ctx context.Context, data string) error
}
```

Запуск команды `go generate ./...` создаст файл в папке `mocks`, готовый к использованию.

## Анатомия теста с gomock

Работа с `gomock` всегда строится на концепции **Контроллера**. Контроллер управляет жизненным циклом моков и проверяет, что все заданные ожидания были выполнены.

```go
func TestService_DataFlow(t *testing.T) {
    // 1. Создаем контроллер
    ctrl := gomock.NewController(t)
    // t.Cleanup(ctrl.Finish) в современных версиях gomock вызывается автоматически, 
    // если передать t в NewController
    
    // 2. Создаем экземпляр сгенерированного мока
    mockRepo := mocks.NewMockRepository(ctrl)
    
    // 3. Устанавливаем ожидания (Expectations)
    // Мы ожидаем, что метод Get будет вызван один раз с ID=42
    // и он должен вернуть "gold", nil.
    mockRepo.EXPECT().
        Get(gomock.Any(), int64(42)).
        Return("gold", nil).
        Times(1)

    // Мы ожидаем, что Save НЕ будет вызван
    mockRepo.EXPECT().
        Save(gomock.Any(), gomock.Any()).
        Times(0)

    // 4. Инжектим мок в тестируемый сервис
    svc := NewService(mockRepo)
    res, err := svc.FetchData(42)

    // 5. Стандартные ассерты
    require.NoError(t, err)
    assert.Equal(t, "gold", res)
}
```

> [!info] Под капотом
> При вызове `ctrl.Finish()` (или автоматически в `Cleanup`), gomock проверяет внутренний список "ожиданий". Если вы указали `.Times(1)`, но метод не был вызван, контроллер вызовет `t.Errorf`, и тест упадет. Это превращает ваш дублер из простого **Stub** в полноценный **Mock**.

## Продвинутые возможности

### Matchers (Матчеры)
Если вам не важны точные значения аргументов, используйте матчеры:
* `gomock.Any()` — любой аргумент.
* `gomock.Eq(x)` — аргумент, равный x.
* `gomock.Len(n)` — аргумент (слайс/мапа) длины n.
* `gomock.Not(x)` — всё, кроме x.

### Call Order (Порядок вызовов)
Для критичных сценариев (например, сначала транзакция, потом запись) можно использовать `gomock.InOrder`:

```go
gomock.InOrder(
    mockRepo.EXPECT().Get(gomock.Any(), int64(1)),
    mockRepo.EXPECT().Save(gomock.Any(), gomock.Any()),
)
```

### Do и DoAndReturn
Если нужно выполнить логику при вызове мока (например, изменить переданный по указателю объект):
```go
mockRepo.EXPECT().Get(gomock.Any(), gomock.Any()).
    Do(func(ctx context.Context, id int64) {
        fmt.Printf("Logging call for id: %d\n", id)
    }).
    Return("data", nil)
```

## Mechanical Sympathy: Кодогенерация vs Рефлексия

Почему `gomock` заставляет нас генерировать файлы, а не делает всё "на лету"?

> [!warning] Ловушка / Gotcha
> Использование рефлексии для создания прокси-объектов в рантайме (как это делает Mockito в Java) в Go сильно ограничено. Go — компилируемый язык со строгой типизацией. Чтобы объект удовлетворял интерфейсу, его методы должны существовать в бинарнике. 
> `mockgen` читает ваши исходники, анализирует дерево типов и пишет новый `.go` файл, который компилируется вместе с тестами. Это гарантирует, что вызов метода мока будет почти таким же быстрым, как вызов обычного метода, без оверхеда на `reflect.ValueOf` при каждом обращении.

## Минусы и критика

Несмотря на мощь, `gomock` часто критикуют за:
1. **Хрупкость (Fragility):** Если вы проверяете точное количество вызовов (`Times(1)`), малейший рефакторинг кода, не меняющий бизнес-результат, может уронить тесты.
2. **Читаемость:** Простыни настроек `EXPECT()` замусоривают тест, скрывая саму логику.
3. **Зависимость от генерации:** Нужно помнить о `go generate`, иначе тесты будут работать со старыми версиями интерфейсов.

> [!tip] Собеседование
> **Вопрос:** В чем разница между `gomock` и библиотекой `testify/mock`?
> **Ответ:** `gomock` требует кодогенерации и обеспечивает строгую типизацию ожиданий. `testify/mock` работает на рефлексии (хотя для него тоже есть генератор `mockery`) и использует строковые имена методов: `m.On("Get", 42)`. `gomock` считается более "безопасным" для больших проектов, так как ошибки в именах методов выявляются на этапе компиляции.

## Итог

1. `gomock` — мощный инструмент для создания полноценных моков с проверкой поведения.
2. Используйте `go:generate` для автоматизации сборки моков.
3. Контроллер (`gomock.Controller`) — центральный узел, проверяющий выполнение ожиданий.
4. Не злоупотребляйте проверкой количества вызовов и порядка, если это не критично для бизнес-логики, чтобы не делать тесты слишком хрупкими.

Если `gomock` кажется вам слишком тяжеловесным, стоит взглянуть на альтернативу, которая более органично вписывается в пакет `testify`. Переходим к: [[5. testify mock]].
