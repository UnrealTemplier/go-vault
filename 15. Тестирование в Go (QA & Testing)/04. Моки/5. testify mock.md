
Если `gomock` (см. [[4. gomock. Генерация моков]]) — это строгий фреймворк, ориентированный на кодогенерацию, то **testify/mock** — это более гибкий и лаконичный подход. Он является частью популярного семейства `testify`, что позволяет использовать его в связке с привычными ассертами.

Главное отличие: `testify/mock` не требует обязательной кодогенерации (хотя она возможна через `mockery`) и предоставляет более человекочитаемый API для описания ожиданий.

## Анатомия теста с testify/mock

В основе лежит структура `mock.Mock`, которую вы встраиваете в свой объект-заглушку. Вместо генерации сложных структур, вы просто описываете, что должен делать метод, используя строковые имена.

```go
package store_test

import (
    "testing"
    "[github.com/stretchr/testify/mock](https://github.com/stretchr/testify/mock)"
    "[github.com/stretchr/testify/assert](https://github.com/stretchr/testify/assert)"
)

// 1. Описываем структуру мока вручную (или генерируем через mockery)
type RepositoryMock struct {
    mock.Mock
}

func (m *RepositoryMock) Get(id int) (string, error) {
    // Вызываем метод Called, передавая аргументы.
    // Это позволяет библиотеке записать вызов и найти подходящее ожидание.
    args := m.Called(id)
    
    // Возвращаем значения, используя хелперы приведения типов
    return args.String(0), args.Error(1)
}

func TestService(t *testing.T) {
    repo := new(RepositoryMock)
    
    // 2. Установка ожиданий (On)
    repo.On("Get", 42).Return("success", nil).Once()
    
    // 3. Вызов тестируемого кода
    result, err := MyService(repo, 42)
    
    // 4. Проверка
    assert.NoError(t, err)
    assert.Equal(t, "success", result)
    
    // Проверяем, что все ожидания, описанные в On, были выполнены
    repo.AssertExpectations(t)
}
```

## Гибкость матчеров и аргументов

`testify/mock` позволяет игнорировать конкретные значения аргументов, если они не важны для логики теста.

* `mock.Anything`: эквивалент `gomock.Any()`.
* `mock.MatchedBy(func)` : позволяет написать кастомную логику проверки аргумента (например, проверить поле внутри структуры).

```go
repo.On("Save", mock.MatchedBy(func(u *User) bool {
    return u.Email != ""
})).Return(nil)
```

## Автоматизация через Mockery

Хотя `testify/mock` можно писать руками, для больших интерфейсов это становится утомительным. Сообщество создало стандартный инструмент для генерации таких моков — **mockery**.

**Преимущества mockery:**
1. Он генерирует моки в стиле `testify`, которые гораздо проще читать, чем "ассемблерный" выхлоп `gomock`.
2. Умеет автоматически находить все интерфейсы в проекте и генерировать для них заглушки одной командой: `mockery --all`.

> [!tip] Собеседование
> **Вопрос:** Что произойдет в `testify/mock`, если метод был вызван с аргументами, которые не были описаны в `On(...)`?
> **Ответ:** Библиотека выдаст `panic`. Это фундаментальное отличие от простых **Stubs**. В `testify/mock` любой вызов, который не был явно разрешен (expected), считается ошибкой. Если вам нужно разрешить любые вызовы, используйте `repo.On("Method", mock.Anything).Maybe()`.

## Mechanical Sympathy: Цена рефлексии в testify

В отличие от `gomock`, который генерирует строго типизированные вызовы, `testify/mock` работает на **интерфейсах и рефлексии** в рантайме.

> [!info] Под капотом
> Когда вы вызываете `m.Called(id)`, происходит следующее:
> 1. Библиотека берет `id`, упаковывает его в `interface{}` (аллокация в куче).
> 2. С помощью пакета `reflect` она ищет в списке зарегистрированных ожиданий (`ExpectedCalls`) то, которое соответствует строке "Get" и значению `id`.
> 3. Возвращаемые значения также хранятся как `[]interface{}` и кастуются к нужным типам через методы `String(0)`, `Int(1)` и т.д.
> 
> Это делает `testify/mock` чуть медленнее, чем `gomock`, но, как мы помним, в Unit-тестах это редко является бутылочным горлышком.



## Сравнение с gomock

| Характеристика | gomock | testify/mock |
| :--- | :--- | :--- |
| **Типизация** | Строгая (типизированные методы) | Слабая (строковые имена методов) |
| **Кодогенерация** | Обязательна (`mockgen`) | Опциональна (через `mockery`) |
| **Читаемость** | Сложная, много бойлерплейта | Высокая, лаконичный API |
| **Интеграция** | Отдельный фреймворк | Часть экосистемы `testify` |

## Итог

1.  `testify/mock` — отличный выбор для проектов, где уже используется `testify/assert`.
2.  Используйте `AssertExpectations(t)` в конце теста, чтобы убедиться, что мок был вызван именно так, как планировалось.
3.  Для автоматизации используйте `mockery`, чтобы не писать `Called()` и `Return()` руками.
4.  Помните о `require` vs `assert` при проверке результатов, полученных через моки.

В мире моков легко запутаться в терминологии. Когда мы используем `testify/mock`, мы создаем полноценный Mock, но иногда нам нужно что-то проще или, наоборот, сложнее. Чтобы раз и навсегда разобраться в "зоопарке" дублеров, переходим к статье: [[6. Fake vs mock vs stub]].
