
Интерфейсы в Go часто воспринимаются исключительно как инструмент полиморфизма. Однако для Senior-разработчика интерфейс — это прежде всего **граница (boundary)** и инструмент управления зависимостями. 

В этой статье мы разберем, как специфика интерфейсов Go (неявная реализация) позволяет строить сверхгибкие тесты, как использовать "интерфейсы-прослойки" для изоляции стандартной библиотеки и почему маленькие интерфейсы — это залог качественных моков.

## Неявная реализация как ключ к Mock-friendly дизайну

В Java или C# вы обязаны объявить, что класс реализует интерфейс (`implements`). Если вы используете стороннюю библиотеку, которая не предоставила интерфейс для своего клиента, вы не сможете его замокать без написания адаптера.

В Go всё иначе. Вы можете объявить интерфейс **в своем пакете**, и любой тип из любой сторонней библиотеки (или стандартной библиотеки Go), имеющий подходящие методы, автоматически его реализует.

> [!tip] Собеседование
> **Вопрос:** Где лучше объявлять интерфейс для мокирования: на стороне производителя (там, где структура определена) или на стороне потребителя (там, где она используется)?
> **Ответ:** На стороне **потребителя**. Это позволяет потребителю описывать только те методы, которые ему действительно нужны, упрощая создание моков и не создавая жесткой связи между пакетами.

## Изоляция "неудобных" зависимостей

Рассмотрим пакет `http.Client`. У него нет интерфейса в стандартной библиотеке, а его методы возвращают структуры, которые сложно конструировать вручную. Вместо того чтобы пытаться замокать весь HTTP-клиент, мы создаем минимальный интерфейс-прослойку.

```go
// В нашем пакете
type HTTPClient interface {
    Do(req *http.Request) (*http.Response, error)
}

type AuthService struct {
    client HTTPClient
}

func (s *AuthService) Verify(token string) bool {
    // Используем s.client.Do(...)
    return true
}
```

Теперь в тестах мы можем передать не только реальный `*http.Client`, но и простую структуру, которая вообще не ходит в сеть.



## Принцип интерфейсной сегрегации (ISP) в тестах

Чем больше методов в интерфейсе, тем сложнее его мокать. Если вы используете интерфейс с 20 методами (например, огромный `Storage`), но в конкретной функции вам нужен только `GetUser`, ваш мок всё равно должен будет реализовать все 20 методов (или "заглушить" их).

**Решение: Маленькие интерфейсы.**
В Go принято разделять интерфейсы на минимально возможные части.

```go
type UserReader interface {
    GetUser(id int) (*User, error)
}

type UserWriter interface {
    SaveUser(u *User) error
}

// В функции используем только то, что нужно
func UpdateName(repo UserReader, id int, name string) error {
    user, _ := repo.GetUser(id)
    // ...
    return nil
}
```

В тесте для `UpdateName` нам нужно замокать только один метод `GetUser`. Это делает тесты лаконичными и устойчивыми к изменениям в других частях системы.

## Mechanical Sympathy: Интерфейсы и аллокации

Как мы упоминали в [[1. Mocking. Основы]], интерфейсы — это структуры из двух указателей.

> [!info] Под капотом: Использование интерфейсов в тестах
> Когда вы заменяете конкретную структуру интерфейсом ради тестирования, вы создаете косвенность (indirection). 
> 1. **Динамический вызов:** Процессор не может предсказать адрес функции, что отключает оптимизации спекулятивного выполнения.
> 2. **Escape Analysis:** Данные, передаваемые в интерфейс, почти всегда попадают в кучу. 
> 
> **Инженерный вывод:** Не плодите интерфейсы там, где они не нужны для тестирования или расширяемости. Если логика чисто вычислительная (математика, парсинг строки), тестируйте её напрямую. Интерфейсы нужны для **границ IO и внешних систем**.

## Паттерн "Обертка над функцией" (Function Adapters)

Иногда даже целый интерфейс — это слишком много. Если вам нужно замокать всего один вызов (например, получение текущего времени или генерацию UUID), можно использовать тип-функцию.

```go
type TimeProvider func() time.Time

func ProcessOrder(now TimeProvider) {
    currentTime := now()
    // ...
}

func TestProcessOrder(t *testing.T) {
    // Мокаем время одной строчкой без создания структур и интерфейсов
    mockNow := func() time.Time {
        return time.Date(2026, 1, 1, 0, 0, 0, 0, time.UTC)
    }
    ProcessOrder(mockNow)
}
```

## Итог

1.  Интерфейсы — это "швы" в коде, через которые мы вставляем моки.
2.  Объявляйте интерфейсы на стороне потребителя, чтобы минимизировать зависимости.
3.  Следуйте ISP: маленькие интерфейсы (1-3 метода) гораздо легче мокать и поддерживать.
4.  Используйте функциональные типы для моканья одиночных вызовов.

Когда интерфейсы спроектированы правильно, написание моков становится механической задачей. В следующей статье мы разберем, как реализовать эти моки вручную без лишнего кода: [[3. Ручные моки]].
