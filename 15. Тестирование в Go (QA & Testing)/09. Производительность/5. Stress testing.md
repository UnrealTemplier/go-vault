## За пределом прочности: Разрушение во имя надежности

В прошлой статье [[4. Load testing]] мы говорили о нагрузочном тестировании. Мы симулировали *ожидаемый* пиковый трафик, чтобы убедиться, что система укладывается в заданный SLA (например, p99 < 50мс при 1000 RPS). 

Но что произойдет, если вас упомянет крупный блогер, и трафик вырастет не до 1000, а до 50 000 RPS за одну минуту? Что будет, если на вас начнется DDoS-атака? 

Нагрузочное тестирование отвечает на вопрос: *"Справимся ли мы с нашей работой?"*.
**Стресс-тестирование (Stress Testing)** отвечает на вопросы: *"При какой нагрузке мы умрем?"*, *"Как именно мы умрем?"* и *"Сможем ли мы ожить, когда атака закончится?"*.

В этой статье мы разберем, как Go-приложения ведут себя за гранью своих возможностей, и как тестировать сценарии катастрофического отказа.

## Анатомия смерти Go-приложения (Mechanical Sympathy)

Чтобы правильно написать стресс-тест, нужно понимать, как именно умирает рантайм Go. В отличие от языков с фиксированным пулом потоков (например, классические Java-серверы на Tomcat), HTTP-сервер в Go порождает **новую горутину на каждый входящий запрос**.

При экстремальном стрессе система обычно входит в один из трех сценариев отказа:

### 1. Goroutine Explosion (Взрыв горутин) и OOM
Представьте, что ваша база данных (PostgreSQL) начала "тормозить" и отвечает не за 10 мс, а за 2 секунды. 
Трафик продолжает поступать со скоростью 10 000 RPS. Поскольку запросы висят в ожидании БД, стандартный `net/http` сервер плодит новые горутины. За пару секунд у вас скапливаются сотни тысяч спящих горутин. Каждая горутина — это минимум 2 КБ стека плюс локальные переменные (структуры запросов, буферы). 
**Итог:** Потребление RAM пробивает потолок лимитов контейнера, и операционная система убивает процесс через **OOM Killer** (`SIGKILL`).

### 2. GC Death Spiral (Спираль смерти Garbage Collector'а)
Если ваше приложение выделяет много памяти (даже если оно быстро её освобождает), при экстремальном RPS частота аллокаций взлетает до небес. Сборщик мусора (GC) понимает, что куча (Heap) переполняется, и начинает работать агрессивнее. 
В Go GC забирает процессорное время (до 25% CPU в нормальном режиме, но в стрессе может забрать все 100%). Бизнес-логика перестает получать кванты времени процессора $\rightarrow$ запросы обрабатываются еще медленнее $\rightarrow$ накапливается еще больше объектов в памяти $\rightarrow$ GC работает еще активнее.
**Итог:** Приложение живо, но p99 latency улетает в десятки секунд. Запросы отваливаются по таймауту.

### 3. Исчерпание файловых дескрипторов (File Descriptor Starvation)
Если вы делаете исходящие запросы (в БД, в Redis, в другие микросервисы) и не настроили пулы соединений, или если на вас льется миллион входящих TCP-соединений, вы упретесь в лимиты ОС (`ulimit`). 
**Итог:** Любая попытка принять запрос или открыть файл будет возвращать ошибку `socket: too many open files`.

## Паттерны Стресс-тестирования

Если для Load Testing мы использовали стабильный профиль нагрузки (Constant Rate), то для Stress Testing мы используем другие профили. Лучшим инструментом для этого является **k6** от Grafana, так как он позволяет гибко программировать сценарии (Stages).

### Паттерн 1: Step Load (Ступенчатая нагрузка)
Мы ступенчато повышаем нагрузку, пока система не сломается, чтобы найти абсолютную пропускную способность (Capacity Point).

*Пример сценария k6 (script.js):*
```javascript
import http from 'k6/http';
import { check } from 'k6';

export const options = {
  stages: [
    { duration: '1m', target: 1000 }, // Разгон до 1k RPS
    { duration: '2m', target: 1000 }, // Держим
    { duration: '1m', target: 5000 }, // Разгон до 5k RPS
    { duration: '2m', target: 5000 }, // Держим
    { duration: '1m', target: 15000 }, // Экстремальный разгон
    { duration: '2m', target: 15000 },
  ],
};

export default function () {
  const res = http.get('[http://api.yourdomain.com/v1/heavy-endpoint](http://api.yourdomain.com/v1/heavy-endpoint)');
  check(res, { 'status is 200': (r) => r.status === 200 });
}
```
**Что мы ищем:** Момент, когда процент успешных ответов падает ниже 99%, или когда latency резко уходит вверх "клюшкой" (Hockey Stick Graph). Это ваш предел.

### Паттерн 2: Spike Testing (Тестирование всплесков)
Симулирует внезапную маркетинговую кампанию или DDoS. Мы держим фоновую нагрузку, затем мгновенно "бьем" гигантским трафиком, а затем резко убираем его.

```javascript
export const options = {
  stages: [
    { duration: '3m', target: 500 },    // Обычная нагрузка
    { duration: '10s', target: 20000 }, // Мгновенный всплеск!
    { duration: '3m', target: 500 },    // Возврат в норму
  ],
};
```
**Что мы ищем:** Сможет ли приложение восстановиться? Если после всплеска (когда нагрузка упала до 500) p99 latency не возвращается в норму, значит, у вас **утечка ресурсов** (например, зависли мьютексы, или пул соединений с базой остался в сломанном состоянии).

## Защита от стресса: Graceful Degradation

Стресс-тестирование не имеет смысла, если вы не встраиваете в код механизмы защиты. Приложение не должно умирать от OOM. Оно должно **Изящно деградировать (Graceful Degradation)** — отбрасывать лишний трафик, спасая ядро системы.

> [!tip] Собеседование
> **Вопрос:** Как защитить HTTP-сервер на Go от OOM при внезапном скачке трафика с 1 000 до 100 000 RPS?
> **Ответ:** Стандартный `http.ListenAndServe` не имеет лимита на количество горутин. Чтобы защитить сервер, нужно применить несколько паттернов:
> 1. **Timeouts:** Жесткие `ReadTimeout` и `WriteTimeout` на уровне `http.Server`.
> 2. **Context Cancellation:** Вся цепочка вызовов (включая SQL запросы) должна принимать `context.Context` от HTTP-запроса, чтобы немедленно прерывать работу, если клиент отвалился по таймауту.
> 3. **Rate Limiting:** Мидлварь с Token Bucket (например, `golang.org/x/time/rate`), которая будет мгновенно возвращать `429 Too Many Requests` для лишних запросов, не пуская их в тяжелую бизнес-логику.
> 4. **Circuit Breaker:** Защита нижестоящих систем (БД, микросервисов), чтобы не "добить" их шквалом запросов, если они начали тормозить.

### Пример защиты: Bounded Concurrency на уровне хэндлера

Если какой-то эндпоинт делает очень тяжелую работу (например, рендеринг видео или сложную криптографию), вы можете ограничить количество одновременно выполняемых запросов с помощью семафора на базе канала:

```go
package api

import (
	"net/http"
)

// Семафор: разрешаем максимум 100 тяжелых операций одновременно
var heavyLoadSemaphore = make(chan struct{}, 100)

func HeavyProcessingHandler(w http.ResponseWriter, r *http.Request) {
	// Пытаемся захватить слот
	select {
	case heavyLoadSemaphore <- struct{}{}:
		// Слот захвачен, гарантированно освобождаем его при выходе
		defer func() { <-heavyLoadSemaphore }() 
	default:
		// Все 100 слотов заняты! Спасаем сервер, отбрасывая запрос.
		// Это выполнится за микросекунды и спасет нас от OOM.
		http.Error(w, "Service Overloaded", http.StatusServiceUnavailable)
		return
	}

	// ... Здесь идет тяжелая бизнес-логика ...
}
```
Стресс-тест этого эндпоинта покажет идеальную картину: 100 запросов будут обрабатываться с нормальным временем, а остальные 99 900 мгновенно получат `503 Service Unavailable`. Сервер останется жив, GC не сойдет с ума, соседние легкие эндпоинты продолжат работать.

## Chaos Engineering: Тестирование отказов

Стресс системы — это не только избыток трафика. Это еще и отказ инфраструктуры. 

Современный подход (выходец из Netflix, так называемый Chaos Monkey) подразумевает умышленное внесение сбоев во время нагрузочного тестирования:
1. Запустите фоновую нагрузку `vegeta`.
2. В процессе умышленно "убейте" мастер-ноду базы данных, чтобы спровоцировать Failover на реплику.
3. Добавьте искусственную задержку в сеть (через `tc qdisc` или инструменты вроде Toxiproxy).
4. **Анализ:** Как ведет себя ваш Go-бэкенд? Зависают ли горутины в ожидании БД? Корректно ли отрабатывают `context.WithTimeout`? Не падает ли приложение в панику при потере коннекта к брокеру сообщений (Kafka/RabbitMQ)?
