## Анатомия задержек: Почему среднее значение вам врет

В предыдущей статье мы обсуждали нагрузочное тестирование и упоминали, что смотреть на среднее время ответа (Average Latency) — это путь к катастрофе в production. Пришло время разобрать математику производительности и понять метрики, на которых строится SLA (Service Level Agreement) любого крупного проекта.

Представьте, что вы владелец кофейни. У вас 100 клиентов. 99 из них получили свой кофе за 1 минуту. А один клиент, из-за сломавшейся кофемашины, ждал 100 минут.

Каково **среднее** время ожидания?

$(99 \times 1 + 100) / 100 = 1.99$ минуты.

Глядя на отчет "Среднее время: ~2 минуты", вы подумаете: _"Отлично, мы работаем быстро!"_. Но тот самый сотый клиент больше никогда к вам не вернется и напишет ужасный отзыв. В интернете этот сотый клиент — это 1% вашего трафика. Если у вас 10 000 RPS, вы теряете 100 клиентов каждую секунду, пока ваш дашборд показывает красивые "средние" цифры.

Чтобы видеть реальную картину, инженеры используют **Перцентили (Percentiles)**.

---

## Что такое p50, p95 и p99?

Перцентиль — это значение, ниже которого падает определенный процент наблюдений.

Чтобы найти перцентиль, нужно взять все времена ответов за период, **отсортировать их по возрастанию** и посмотреть на значение под нужным индексом.

- **p50 (Медиана):** Значение ровно посередине отсортированного списка. Половина запросов выполнилась быстрее, половина — медленнее. Это "типичный" запрос.
- **p95:** 95% запросов выполнились быстрее этого времени.
- **p99:** 99% запросов выполнились быстрее. Отражает "хвост" распределения (Tail Latency) — те самые запросы, которые попали на паузу Garbage Collector'а, блокировку мьютекса или сетевой таймаут.
- **p99.9:** "Три девятки". Метрика для критически важных систем (например, биржевых торгов).

### Tail Latency Amplification (Усиление хвостовых задержек)

Почему p99 так важен в микросервисной архитектуре? Из-за математического эффекта усиления.

Представьте, что страница вашего приложения делает запросы к 10 разным микросервисам для сборки ответа. Допустим, каждый микросервис работает отлично: его p99 = 1 секунда (только 1% запросов тормозит).

Какова вероятность, что пользовательский запрос к странице затормозит?

Для этого нужно вычислить вероятность того, что _хотя бы один_ из 10 микросервисов попадет в свой 1-процентный хвост:

$P(slow) = 1 - (0.99)^{10} \approx 0.0956$

Почти **10% пользователей** получат медленный ответ! Если страница делает 50 запросов, медленный ответ получат уже $39\%$ пользователей. Именно поэтому Google и Amazon тратят миллионы долларов на оптимизацию метрик p99 и p99.9.

## Как считать перцентили в Go?

### 1. Наивный подход (Сортировка в памяти)

Если вы пишете собственный скрипт для локального бенчмарка, посчитать перцентиль очень просто:

```go
package metrics_test

import (
	"fmt"
	"slices"
	"time"
)

func ExampleCalculatePercentiles() {
	// Допустим, мы собрали задержки 100 запросов
	latencies := []time.Duration{
		10 * time.Millisecond, 12 * time.Millisecond, 11 * time.Millisecond,
		// ... еще 96 нормальных запросов ...
		800 * time.Millisecond, // GC пауза
	}

	// 1. Обязательно сортируем!
	slices.Sort(latencies)

	// 2. Находим индексы
	total := len(latencies)
	p50 := latencies[int(float64(total)*0.50)]
	p95 := latencies[int(float64(total)*0.95)]
	p99 := latencies[int(float64(total)*0.99)]

	fmt.Printf("p50: %v, p95: %v, p99: %v\n", p50, p95, p99)
}
```

### 2. Проблема Production (Streaming Percentiles)

Наивный подход отлично работает в тестах, где у нас 10 000 элементов в массиве.

Но что делать на production-сервере, который обрабатывает 20 000 RPS? Вы не можете сохранять каждую `time.Duration` в массив и сортировать миллионы элементов каждую секунду. У вас закончится оперативная память (OOM), а сортировка убьет процессор.

Для подсчета перцентилей "на лету" (Streaming) в потоке данных используют математические аппроксимации:

1. **HDR Histogram (High Dynamic Range Histogram):** Сохраняет задержки в заранее созданные "корзины" (buckets) с логарифмической точностью. Потребляет фиксированное количество памяти.
2. **T-Digest:** Алгоритм кластеризации данных "на лету", который дает невероятно точные значения на крайних перцентилях (p99.9).

## Метрики в реальном мире: Prometheus

Де-факто стандартом мониторинга Go-приложений является Prometheus. На собеседованиях уровня Middle/Senior часто задают хитрый вопрос про типы метрик в Prometheus.

> [!tip] Собеседование
> 
> **Вопрос:** В Prometheus есть два типа метрик для задержек: `Summary` и `Histogram`. Что вы выберете для мониторинга p99 на кластере из 10 микросервисов и почему?
> 
> **Ответ:** Всегда выбираем `Histogram`.
> 
> Тип `Summary` считает точные перцентили (через стриминговые алгоритмы) прямо на стороне Go-клиента. Проблема в том, что перцентили **нельзя усреднять**. Если у вас 10 серверов, и каждый отдает свой p99, вы не можете сложить их и поделить на 10 — это математическая ошибка.
> 
> Тип `Histogram` просто раскладывает запросы по корзинам (например, `bucket_10ms`, `bucket_50ms`, `bucket_100ms`). Prometheus Server собирает эти корзины со всех 10 серверов, суммирует _сами корзины_, и уже на своей стороне вычисляет аппроксимированный p99 с помощью функции `histogram_quantile()`. Это единственный способ получить достоверный p99 для всего кластера.

### Идиоматичный код для Prometheus Histogram

```go
package monitoring

import (
	"github.com/prometheus/client_golang/prometheus"
	"github.com/prometheus/client_golang/prometheus/promauto"
)

// Создаем гистограмму на уровне пакета
var requestLatency = promauto.NewHistogramVec(
	prometheus.HistogramOpts{
		Name: "http_request_duration_seconds",
		Help: "Задержка HTTP запросов",
		// КРИТИЧЕСКИ ВАЖНО: Правильно подобрать корзины под ваш SLA
		Buckets: []float64{0.01, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10}, 
	},
	[]string{"endpoint", "method", "status"},
)

// Использование в Middleware
func MetricsMiddleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()
		
		// Оборачиваем ResponseWriter, чтобы перехватить status code
		// ...
		
		next.ServeHTTP(w, r)
		
		duration := time.Since(start).Seconds()
		requestLatency.WithLabelValues(r.URL.Path, r.Method, "200").Observe(duration)
	})
}
```

В Grafana (интерфейсе для Prometheus) вы напишете следующий PromQL запрос, чтобы увидеть p99 по всему кластеру:

```
histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, endpoint))
```

## Итог

1. **Забудьте про среднее.** Оптимизируйте медиану (p50) для счастья основной массы пользователей и p99 для стабильности системы в целом.
2. **Tail Latency Amplification:** В микросервисах медленный 1% запросов на нижнем уровне превращается в 10-30% медленных запросов для конечного клиента.
3. **Сортировка решает:** Перцентили считаются через сортировку. Для потоковых (streaming) данных в production используются аппроксимации (Prometheus Histograms).
4. **Мониторинг:** Используйте `Histogram` в Prometheus, чтобы иметь возможность агрегировать p99 с нескольких инстансов вашего приложения.
