## Битва за время: Как заставить CI работать на вас

В прошлой статье [[3. Тестирование в CI CD]] мы построили надежный пайплайн, работающий по принципу Fail-Fast. Мы добавили линтеры, юнит-тесты, интеграционные проверки и гейты покрытия.

Но с ростом проекта возникает новая проблема: пайплайн начинает "толстеть". Если ваши тесты выполняются 15–20 минут, разработчики начинают контекстно переключаться: они идут пить кофе, открывают YouTube или берут другую задачу. Когда CI наконец "падает", возвращение к исходной задаче занимает массу когнитивных усилий.

**Время обратной связи (Feedback Loop) в CI должно составлять не более 3–5 минут.** Единственный способ достичь этого на крупном монолите — агрессивный параллелизм.

В Go-экосистеме параллелизм тестов существует на трех совершенно разных архитектурных уровнях. Давайте разберем их от микроуровня кода до макроуровня инфраструктуры.

---

## Уровень 1: Внутрипроцессный параллелизм (t.Parallel)

Самый базовый уровень — это распараллеливание тестов внутри одного Go-пакета. Вы добавляете вызов `t.Parallel()` в начало тестовой функции или внутрь сабтеста (`t.Run`).

```go
func TestProcessData(t *testing.T) {
	t.Parallel() // Маркируем тест как параллельный
	// ...
}
```

> [!info] Под капотом: Как работает t.Parallel()
> 
> Когда раннер `go test` встречает вызов `t.Parallel()`, он **приостанавливает** выполнение этого теста и помещает его в специальную очередь.
> 
> Сначала Go выполняет все последовательные (не параллельные) тесты в пакете один за другим. Как только они завершаются, Go берет очередь параллельных тестов и запускает их одновременно в разных горутинах.
> 
> Количество одновременно работающих параллельных тестов ограничено флагом `-parallel` (по умолчанию равно `GOMAXPROCS`, то есть количеству ядер процессора).

### Ловушки t.Parallel()

Казалось бы, почему не добавить `t.Parallel()` вообще во все тесты? Потому что параллелизм обнажает архитектурные недостатки вашего кода.

1. **Глобальное состояние (Global State):** Если ваши тесты модифицируют глобальные переменные (например, подменяют глобальный `logger` или меняют `os.Setenv`), `t.Parallel()` мгновенно вызовет Data Race или логические ошибки. Параллельные тесты обязаны быть **абсолютно изолированными** (Pure Tests).
2. **Общие ресурсы (Shared Resources):** Если два параллельных теста пытаются поднять `httptest.Server` на жестко захардкоженном порту `:8080`, второй тест упадет с ошибкой `bind: address already in use`. (Используйте порт `:0`, чтобы ОС сама выдала свободный порт).

## Уровень 2: Пакетный параллелизм (go test ./...)

Мало кто задумывается об этом, но команда `go test ./...` **уже параллельна по своей природе**.

Когда вы запускаете эту команду, утилита `go` компилирует каждый пакет в отдельный бинарный файл теста, а затем запускает эти бинарники параллельно!

> [!warning] Ловушка / Gotcha: Mechanical Sympathy и лимиты CI
> 
> По умолчанию `go test` запускает параллельно `GOMAXPROCS` пакетов.
> 
> Стандартный бесплатный раннер в GitHub Actions имеет **2 vCPU** (виртуальных ядра) и 7 ГБ RAM.
> 
> Если ваш проект состоит из 50 пакетов, и многие из них содержат тяжелые тесты (поднимают In-Memory базы или активно используют горутины), запуск на 2-ядерном CI-раннере приведет к дикому **Lock Contention** (борьбе за процессор) и троттлингу. Тесты будут выполняться _медленнее_, чем если бы вы запускали их последовательно.
> 
> **Решение:** Явно управляйте флагом `-p` (количество параллельно собираемых и тестируемых пакетов). Иногда `go test -p=2 ./...` или даже `-p=1` на слабых CI-машинах отрабатывает быстрее и стабильнее, избавляя вас от Flaky-тестов, падающих по таймауту.

## Уровень 3: Инфраструктурный параллелизм (CI Matrix & Sharding)

Если `go test ./...` все равно занимает 10 минут, значит, вы уперлись в потолок одного CI-раннера. Пора масштабироваться горизонтально — распределять работу между несколькими виртуальными машинами.

### Паттерн 1: Матрица задач (Matrix Strategy)

Современные CI-системы позволяют размножить одну задачу на несколько раннеров в один клик. Чаще всего матрицу используют для проверки разных версий Go, но мы можем использовать её для изоляции разных _типов_ тестов.

Пример для **GitHub Actions**:

```yaml
test:
  runs-on: ubuntu-latest
  strategy:
    # Fail-fast: если падает один job из матрицы, остальные отменяются
    fail-fast: true 
    matrix:
      # Разбиваем тесты на логические блоки, каждый запустится на СВОЕЙ виртуалке
      test-suite: 
        - name: "Unit Tests"
          command: "go test -short -race ./..."
        - name: "Integration DB"
          command: "go test -tags=integration ./internal/repository/postgres/..."
        - name: "Integration Redis"
          command: "go test -tags=integration ./internal/repository/redis/..."

  name: ${{ matrix.test-suite.name }}
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-go@v5
      with:
        go-version: '1.22'
    
    # Запускаем команду из матрицы
    - name: Run Tests
      run: ${{ matrix.test-suite.command }}
```

**Результат:** Вы получаете три параллельных джоба. Интеграционные тесты с Postgres не мешают тестам с Redis забирать процессор, и общее время пайплайна равно времени самого долгого джоба (а не их сумме).

### Паттерн 2: Test Sharding (Шардирование тестов)

Что делать, если у вас один гигантский пакет E2E-тестов, который выполняется 15 минут?

Вы можете разделить запуск _внутри_ одного пакета по разным CI-машинам.

Для этого используется встроенная библиотека `testing`, которая умеет фильтровать тесты по регулярному выражению (`-run`). Но вручную писать регулярки неудобно. В крупных проектах используют утилиты-обертки (например, `gotestsum` или самописные скрипты), которые делят список тестов на N корзин (Buckets).

В самом Go есть малоизвестный, но мощный флаг **`-shard`** (часто реализуемый через сторонние раннеры или скрипты), который хэширует имена тестов и выполняет только те, которые принадлежат текущему шарду.

- Job 1 выполняет: `go test -run=^Test[A-M] ./tests/e2e/...`
- Job 2 выполняет: `go test -run=^Test[N-Z] ./tests/e2e/...`

## Финальный босс: Параллелизм интеграционных тестов

Юнит-тесты распараллелить легко. Настоящая боль начинается, когда вы пытаетесь запустить с `t.Parallel()` интеграционные тесты, которые ходят в реальную базу данных (PostgreSQL/MySQL).

Если Тест А делает `INSERT user`, а параллельный Тест Б в эту же секунду делает `DELETE FROM users`, оба теста упадут с непредсказуемыми результатами.

У Senior-инженеров есть 3 стратегии решения этой проблемы (от быстрой к медленной):

### Стратегия 1: Транзакционный откат (Savepoints)

Перед каждым тестом открывается транзакция базы данных. В конце теста вызывается `tx.Rollback()`. База данных откатывает изменения, оставляя состояние чистым.

- **Плюс:** Молниеносно быстро. Десятки тестов могут работать параллельно на одной БД.
- **Минус:** Не работает, если ваша бизнес-логика сама управляет транзакциями (коммитит их). Тест не сможет использовать транзакцию поверх другой транзакции без сложных хаков (например, PostgreSQL SAVEPOINT).

### Стратегия 2: Уникальные схемы/таблицы (Schema-per-Test)

Внутри функции `setupTestDB()` вы генерируете случайный суффикс (например, UUID) и создаете для теста отдельную `SCHEMA` в Postgres (или отдельную `DATABASE` в MySQL), накатываете туда миграции, а в `t.Cleanup` удаляете её (DROP).

- **Плюс:** Идеальная изоляция. Тесты можно писать с `t.Parallel()`. Полная имитация production-поведения (с коммитами транзакций).
- **Минус:** Накатывание миграций на каждую новую схему занимает время. (Хак: можно создать одну "шаблонную" базу (Template DB) с накатанными миграциями и использовать оператор `CREATE DATABASE db_test_xyz TEMPLATE db_master`, что происходит за миллисекунды).

### Стратегия 3: Изолированные контейнеры (Testcontainers)

Каждый тестовый пакет поднимает свой собственный Docker-контейнер с базой данных через библиотеку `testcontainers-go`.

- **Плюс:** Невероятная надежность. Вы тестируете точную копию production-инфраструктуры.
- **Минус:** Поднятие контейнера занимает 2–5 секунд. Если каждый пакет будет поднимать свой контейнер, CI-сервер может задохнуться от нехватки RAM. Рекомендуется использовать этот паттерн на уровне пакета (в функции `TestMain`), а внутри пакета тесты гонять последовательно.

## Итог

1. **Локальный параллелизм (`t.Parallel`):** Требует безупречной архитектуры без глобального состояния. Отлично подходит для юнит-тестов и алгоритмики.
2. **Пакетный параллелизм:** Включен в Go по умолчанию (`go test ./...`). Внимательно следите за тем, сколько пакетов вы тестируете одновременно на слабых CI-раннерах.
3. **Инфраструктурный параллелизм (Matrix):** Разделяйте интеграционные, E2E и юнит-тесты на разные "Job'ы" в CI, чтобы они выполнялись на разных виртуальных машинах одновременно.
4. **Базы данных:** Интеграционные тесты требуют строгой изоляции (Транзакции, Уникальные Схемы или Testcontainers), иначе параллелизм превратится в генератор плавающих (flaky) ошибок.

Выстроить быстрый пайплайн — это инженерное искусство. Когда ваши тесты проходят за 2 минуты вместо 20, скорость доставки фич в production возрастает кратно, а разработчики перестают ненавидеть процесс написания тестов. Это завершающий аккорд в построении профессионального процесса Quality Assurance в Go.
