## Сигнал сквозь шум: Зачем отдельное логирование в тестах?

В прошлой статье [[8. Debugging тестов]] мы вскользь упомянули, что `fmt.Println` — это зло для параллельных тестов, и посоветовали использовать `t.Log`. Однако логирование в тестировании — это не просто инструмент для отладки сломанного кода. Это полноценная система наблюдаемости (Observability) для ваших тестов.

Когда ваш CI-пайплайн падает, у вас нет возможности подключить отладчик Delve. Логи, которые оставил после себя упавший тест — это единственный способ понять, что произошло. Но если тесты "спамят" в консоль, найти полезную информацию становится невозможно.

В этой статье мы разберем, как правильно управлять выводом в тестах, как "подружить" production-логгеры (Zap, slog) с тестовым фреймворком Go, и какие артефакты стоит логировать при падениях.

---

## 1. Фундамент: Механика t.Log и t.Logf

Главное свойство `t.Log` — это **буферизация**.

Когда вы вызываете `fmt.Println`, текст мгновенно пишется в стандартный поток вывода (`os.Stdout`). Если у вас параллельно работают 10 тестов, их строки перемешаются в нечитаемую кашу.

Когда вы вызываете `t.Log`, фреймворк `testing` сохраняет эту строку в специальный внутренний буфер, привязанный к конкретному экземпляру `*testing.T` (к конкретному тесту или сабтесту `t.Run`).

> [!info] Под капотом: Правила вывода буфера
> Раннер `go test` покажет содержимое буфера `t.Log` в консоли **только в двух случаях**:
> 1. Тест завершился с ошибкой (через `t.Fail`, `t.Error`, `require.NoError` и т.д.).
> 2. Тесты были запущены с флагом "verbose": `go test -v`.
> 
> Если тест прошел успешно и флаг `-v` не передан, все логи `t.Log` просто молча удаляются из памяти. Это гениальное решение, которое сохраняет консоль кристально чистой (Zero Noise), пока не произойдет проблема.

## 2. Интеграция Production-логгеров

Здесь возникает главная архитектурная проблема. Ваша бизнес-логика ничего не знает о `*testing.T`. Она ожидает интерфейс логгера (например, `*zap.Logger` или `*slog.Logger`).

```go
// Бизнес-логика
func ProcessPayment(amount int, logger *zap.Logger) error {
    logger.Info("starting payment processing", zap.Int("amount", amount))
    // ...
    if err != nil {
        logger.Error("payment failed", zap.Error(err))
        return err
    }
    return nil
}
```

Если в тесте вы передадите обычный production-логгер (`zap.NewProduction()`), он будет писать логи прямо в `os.Stdout` или `os.Stderr`. Вы потеряете буферизацию `go test`, и ваша консоль снова превратится в свалку.

### Решение для Uber Zap: zaptest

Авторы библиотеки `go.uber.org/zap` предусмотрели эту проблему и создали пакет `zaptest`. Он создает специальный логгер, который перехватывает все вызовы `Info/Error` и перенаправляет их под капот в `t.Log`.

```go
package payment_test

import (
	"testing"
	"go.uber.org/zap/zaptest"
	"yourproject/internal/payment"
)

func TestProcessPayment(t *testing.T) {
	// Создаем логгер, привязанный к текущему тесту
	logger := zaptest.NewLogger(t)

	// Передаем его в бизнес-логику
	err := payment.ProcessPayment(100, logger)

	// Если ProcessPayment упадет, мы увидим все внутренние Info/Error логи,
	// аккуратно отформатированные и привязанные именно к этому тесту!
}
```

### Решение для slog (Go 1.21+)

Если вы используете встроенный пакет `log/slog`, вам нужно перенаправить его вывод (который ожидает `io.Writer`) в `t.Log`.

Проблема в том, что `*testing.T` не реализует интерфейс `io.Writer`. Нам нужно написать простую обертку (Adapter):

```go
package testutil

import (
	"log/slog"
	"testing"
)

// testingWriter адаптирует *testing.T под интерфейс io.Writer
type testingWriter struct {
	t *testing.T
}

func (w testingWriter) Write(p []byte) (n int, err error) {
	// TrimSpace нужен, чтобы t.Log не добавлял лишние переносы строк
	w.t.Logf("%s", p)
	return len(p), nil
}

// NewSlogLogger создает slog логгер для тестов
func NewSlogLogger(t *testing.T) *slog.Logger {
	t.Helper()
	writer := testingWriter{t: t}
	// Используем TextHandler или JSONHandler по вкусу
	handler := slog.NewTextHandler(writer, &slog.HandlerOptions{
		Level: slog.LevelDebug, // В тестах нам нужны все детали
	})
	return slog.New(handler)
}
```

Теперь в тесте вы просто вызываете `logger := testutil.NewSlogLogger(t)` и передаете его в свои сервисы.

## 3. Что логировать при падении (Test Observability)

Обычная ошибка выглядит так:
`Error: expected 200, got 500`.

Это абсолютно бесполезно. Почему 500? Какой был запрос? Какое тело ответа? 
Senior-инженеры добавляют в тесты механизм **Dump on Failure** (Сброс контекста при падении).

### Паттерн: HTTP Dump

Если вы пишете E2E или интеграционные API тесты (с использованием `httptest`), всегда логируйте полные HTTP-запросы и ответы при возникновении ошибки. Для этого в Go есть пакет `net/http/httputil`.

```go
func TestAPI_CreateUser(t *testing.T) {
	// ... Подготовка httptest.Server ...

	req, _ := http.NewRequest(http.MethodPost, ts.URL+"/users", body)
	resp, err := http.DefaultClient.Do(req)
	require.NoError(t, err)

	// Читаем тело ответа для ассертов
	respBody, _ := io.ReadAll(resp.Body)
	resp.Body.Close()

	if resp.StatusCode != http.StatusCreated {
		// ТЕСТ УПАЛ!
		// Сбрасываем весь контекст запроса и ответа для дебага в CI
		reqDump, _ := httputil.DumpRequestOut(req, true)
		t.Logf("--- DUMP REQUEST ---\n%s\n--------------------", string(reqDump))
		
		t.Logf("--- DUMP RESPONSE BODY ---\n%s\n-----------------------", string(respBody))

		t.Fatalf("Expected 201 Created, got %d", resp.StatusCode)
	}
}
```

### Паттерн: State Dump (Сброс состояния БД)

Если интеграционный тест БД падает, часто полезно знать, что вообще было в таблице в этот момент.

```go
	if err != nil {
		// Дополнительный SQL-запрос только для вывода в лог при падении
		rows, _ := db.Query("SELECT id, status FROM users")
		// ... форматируем в строку ...
		t.Logf("DB State at failure:\n%s", dbStateString)
		t.Fatal(err)
	}
```

> [!warning] Ловушка / Gotcha: Логирование секретов
> Помните, что логи тестов (если они падают в CI) сохраняются в артефактах GitHub Actions или GitLab. Если ваш интеграционный тест использует реальные токены, пароли или Stripe ключи (даже тестового окружения), прогоняйте дампы через маскировщик (Sanitizer) перед вызовом `t.Log`, иначе вы сольете секреты в логи CI.

## 4. Этапы жизненного цикла теста

Длинные интеграционные тесты (которые выполняются по 10-20 секунд) бывает сложно читать в логах. Рекомендуется явно логировать этапы **Arrange-Act-Assert**.

```go
func TestComplexIntegration(t *testing.T) {
	t.Log("[SETUP] Initializing test database...")
	db := setupDB(t)

	t.Log("[SETUP] Seeding 10,000 test users...")
	seedUsers(t, db)

	t.Log("[ACT] Running migration background worker...")
	runWorker()

	t.Log("[ASSERT] Verifying user statuses...")
	// ... проверки ...
}
```
Если тест упадет по таймауту или панике, последний `t.Log` в буфере точно скажет вам, на каком именно этапе тест завис.

## Итог

1.  **Никакого `fmt.Println` в тестах.** Только `t.Log` и `t.Logf` для поддержания порядка в консоли и работы с параллельностью.
2.  **Адаптеры логгеров:** Никогда не оставляйте production-логгер писать напрямую в `Stdout` во время тестов. Оборачивайте `zap` через `zaptest`, а `slog` через кастомный `io.Writer`.
3.  **Богатый контекст падений:** Логируйте дампы HTTP-запросов (`httputil.DumpRequest`), тела ответов и состояния структур при падении ассертов.
4.  **Разметка этапов:** В длинных (E2E) тестах логируйте переходы между фазами Setup, Act и Assert.

Этим материалом мы закрываем практическую часть организации тестовых стендов и CI-пайплайнов. Качественное логирование — это тот самый механизм, который превращает "красный крестик" в CI из повода для паники в четкую инструкцию по исправлению бага.
