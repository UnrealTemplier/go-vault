## Иллюзия безопасности: Вся правда о Test Coverage

Вы внедрили линтеры, настроили параллельный CI, написали фаззинг и бенчмарки. И вот менеджер или техлид просит вас повесить в Readme репозитория красивый бейдж: **Coverage 100%**.

Метрика покрытия кода тестами (Code Coverage) — это самый популярный и одновременно самый опасный инструмент в арсенале QA. Опасен он тем, что дает ложное чувство безопасности (False Sense of Security). 

В этой, завершающей статье модуля "Тестирование в Go", мы разберем, как именно Go считает покрытие, почему 100% — это часто признак плохой архитектуры, и как превратить Coverage из инструмента давления в инструмент инсайтов.

## Mechanical Sympathy: Как Go считает покрытие

В языках вроде Java (с инструментом JaCoCo) покрытие часто считается путем модификации скомпилированного байт-кода на лету. В C/C++ используются флаги компилятора вроде `gcov`.

В Go используется другой, невероятно элегантный подход: **Модификация исходного кода (AST Rewriting)**.

Когда вы запускаете `go test -cover`, происходит следующее:
1. Инструментарий Go парсит ваши `.go` файлы в Абстрактное Синтаксическое Дерево (AST).
2. Он находит все базовые блоки кода (Basic Blocks) — участки кода без ветвлений.
3. **Перед** компиляцией Go автоматически вставляет в ваш код скрытые счетчики.

Ваш красивый код:
```go
func IsValid(age int) bool {
    if age >= 18 {
        return true
    }
    return false
}
```

То, что реально компилируется при `go test -cover`:
```go
var GoCover = struct { Count [2]uint32 }{}

func IsValid(age int) bool {
    if age >= 18 {
        GoCover.Count[0]++ // Скрытый счетчик ветки A
        return true
    }
    GoCover.Count[1]++     // Скрытый счетчик ветки B
    return false
}
```
Благодаря этому подход работает очень быстро, а выгрузка результатов (флаг `-coverprofile=coverage.out`) представляет собой просто дамп этих счетчиков в текстовый файл.

## Ловушка 1: Assertion-Free Tests (Тесты-пустышки)

Это самое страшное преступление в тестировании, которое порождается требованиями "покрыть код на 100%".

Покрытие кода измеряет только одно: **Был ли этот код выполнен (вызван) во время прогона тестов?**
Оно **НЕ измеряет**, проверили ли вы результат работы этого кода!

```go
// Бизнес-логика (с багом: всегда возвращает 0)
func CalculateTax(amount int) int {
    tax := amount * 0.20
    return 0 // Oops!
}

// Тест-преступник
func TestCalculateTax(t *testing.T) {
    CalculateTax(100) // Мы вызвали функцию!
    // Нет require.Equal(t, 20, got). Нет ассертов.
}
```
При запуске `go test -cover` вы получите **100% покрытия** для функции `CalculateTax`. Строки выполнились, счетчики кликнули. Но тест абсолютно бесполезен, а баг поедет в production.

**Как бороться:** Использовать линтеры (например, `testifylint`), которые ругаются на тестовые функции без вызовов `assert`/`require`. И, конечно же, строгий Code Review.

## Ловушка 2: Фетиш 100% и Закон убывающей доходности

Стремление к 100% покрытию в микросервисной архитектуре — это пустая трата денег бизнеса. График полезности тестов подчиняется закону убывающей доходности (Diminishing Returns).

* **0% -> 60%:** Легко достигается. Вы покрываете "счастливые пути" (Happy Paths) и базовые ошибки бизнес-логики. Огромный ROI (Return on Investment).
* **60% -> 80%:** Требует усилий. Вы мокаете зависимости, пишите табличные тесты на граничные условия (Edge Cases). Высокий ROI.
* **80% -> 100%:** Ад. Вы тратите часы, чтобы написать тест, который имитирует ошибку `rows.Scan()` из базы данных или сбой маршалинга JSON (который никогда не упадет на валидных структурах). ROI стремится к нулю.

> [!warning] Ловушка / Gotcha: Тестирование `if err != nil`
> В Go половина строк кода — это обработка ошибок.
> ```go
> data, err := json.Marshal(user)
> if err != nil { // Как покрыть эту строку, если user всегда валиден?
>     return fmt.Errorf("failed to marshal: %w", err)
> }
> ```
> Пытаться покрыть эту строку — значит писать сложный мок или передавать сломанную структуру с бесконечным циклом ссылок (чтобы сломать `json.Marshal`). Это делает тесты хрупкими и нечитаемыми. Оставьте эту строку непокрытой.

**Золотой стандарт индустрии:** 70–80% покрытия для бэкенда. Для библиотек и критического ядра (финансы, криптография) — 90%+.

## Ловушка 3: Statement vs Branch Coverage

На собеседованиях часто спрашивают про разницу между Statement Coverage (Покрытие выражений/строк) и Branch Coverage (Покрытие ветвей).

**Go поддерживает только Statement Coverage.**

Посмотрите на этот код:
```go
func Process(a, b bool) {
    if a && b {
        fmt.Println("Success")
    }
}
```

Если вы напишете тест `Process(true, true)`, `go test -cover` покажет **100% покрытия**. Обе строки были выполнены.
Но с точки зрения логики (Branch Coverage), мы не протестировали 3 других состояния автомата (true/false, false/true, false/false). Go не покажет вам, что эти логические пути не проверены.

**Как бороться:** Фаззинг (Fuzzing) и Table-Driven тесты с матрицей состояний. Не доверяйте цифре 100% слепо.

## Эволюция: Интеграционное покрытие (Go 1.20+)

Долгое время в Go была огромная проблема: `go test -cover` работал только для юнит-тестов (запускаемых через функцию `TestXxx`). 
Если вы собирали бинарник вашего сервера, запускали его, и стреляли по нему HTTP-запросами из Postman или E2E-тестов (написанных на Python или JS), вы получали **0% покрытия**, потому что бинарник не умел экспортировать метрики покрытия.

**Революция в Go 1.20:** Теперь можно собирать *само приложение* с инструментированием!

```bash
# Собираем бинарник со встроенными счетчиками покрытия!
go build -cover -o myapp ./cmd/api

# Указываем папку, куда бинарник скинет результаты при выключении
export GOCOVERDIR=/tmp/covdata

# Запускаем приложение
./myapp &
PID=$!

# ... Запускаем внешние E2E тесты (Postman, k6, Cypress) ...

# Останавливаем приложение (Graceful Shutdown обязателен!)
kill -SIGINT $PID 

# Формируем человекочитаемый отчет из сырых данных
go tool covdata textfmt -i=/tmp/covdata -o profile.out
go tool cover -html=profile.out
```

Это меняет правила игры. Теперь ваши E2E-тесты и интеграционные тесты дают реальный вклад в общую метрику покрытия продукта.

## Как использовать Coverage как профессионал

Если проценты врут, зачем нам Coverage? 
Senior-разработчик использует покрытие не как метрику для KPI, а как **тепловую карту (Heatmap) для поиска слепых зон**.

Для этого используется команда:
```bash
go test -coverprofile=c.out ./...
go tool cover -html=c.out
```

Она открывает браузер с исходным кодом, где:
* **Зеленый цвет:** Код покрыт.
* **Красный цвет:** Код не покрыт.
* **Серый цвет:** Неисполняемый код (типы, интерфейсы).

**Алгоритм действий на Code Review:**
1. Вы открываете `go tool cover -html`.
2. Вы ищете **красные блоки** внутри *бизнес-логики* (а не в `if err != nil` от системных вызовов).
3. Если вы видите, что целый `switch/case` или алгоритм скидки светится красным — вы отклоняете Pull Request и просите дописать тесты.

### Исключение сгенерированного кода

Красный цвет в сгенерированных файлах (Proto-файлы, Mocks, Swagger) портит общую статистику. Стандартный `go test` не умеет игнорировать файлы.

В CI-пайплайне (о котором мы говорили в [[3. Тестирование в CI CD]]) используют утилиты (например `grep`) для очистки отчета:
```bash
go test -coverprofile=c.out ./...
# Удаляем из отчета все моки и proto-файлы
grep -v -E "mock_.*\.go|\.pb\.go" c.out > c_clean.out
go tool cover -func=c_clean.out
```
