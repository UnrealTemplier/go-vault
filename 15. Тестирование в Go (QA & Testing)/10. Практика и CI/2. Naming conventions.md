## Анатомия имен: Как называть тесты, чтобы вас понимали

В прошлой статье [[1. Организация тестов в проекте]] мы разобрались с глобальной архитектурой: пакетами, тегами сборки и директориями `testdata`. Мы научились правильно раскладывать файлы по полочкам. 

Но когда вы открываете конкретный файл `*_test.go`, вы сталкиваетесь с новой проблемой: как назвать тестовую функцию, локальную переменную или сабтест, чтобы другой разработчик (или вы сами спустя полгода) мгновенно понял, что именно здесь проверяется?

В Go нет строгих фреймворков вроде RSpec (Ruby) с их `describe` и `it`. Вся семантика строится на **соглашениях об именовании (Naming Conventions)**. Следование этим соглашениям — это маркер, который отличает Senior Go-разработчика от новичка, пишущего на Go с акцентом Java или Python.

## 1. Именование файлов: Зеркальное отражение

Базовое правило, заложенное в инструментарий Go: файлы с тестами должны заканчиваться на `_test.go`. Но хорошим тоном является правило **зеркального именования**.

Если у вас есть файл с бизнес-логикой `user.go`, его тесты должны лежать строго в `user_test.go`. 
Не сваливайте все тесты пакета в один гигантский `repository_test.go`. 

> [!tip] Собеседование
> **Вопрос:** У нас есть файл `auth.go`. Мы написали для него Black-box тесты в `auth_test.go` (в пакете `auth_test`). Теперь нам нужно протестировать сложную приватную функцию хэширования из `auth.go` (White-box). Как назвать файл для этих тестов?
> **Ответ:** Идиоматичный подход — создать файл `auth_internal_test.go` (с пакетом `auth`). Суффикс `_internal_test` (или `_whitebox_test`) сразу дает понять команде, что внутри лежат тесты, завязанные на неэкспортируемую реализацию.

## 2. Именование функций: Строгий контракт

Инструмент `go test` использует рефлексию (reflection) и анализ AST для поиска тестов. Он ищет функции, которые соответствуют строгой сигнатуре: `func TestXxx(*testing.T)`.

> [!warning] Ловушка / Gotcha: Регистр первой буквы
> Это классическая ошибка новичков. Если вы назовете функцию `TestcalculateAge(t *testing.T)` (с маленькой буквы `c`), **тест не запустится**. 
> Правило `go test` гласит: буква, следующая за словом `Test`, **не может быть строчной буквой**. Функция `Testcalculate` будет проигнорирована раннером как обычная (не тестовая) функция. Правильно: `TestCalculateAge`.

### Тестирование обычных функций
Если вы тестируете функцию `ParseURL`, тестовая функция должна называться `TestParseURL`.

### Тестирование методов структур (Method Convention)
В Go нет классов, но есть структуры и методы. Как показать, что мы тестируем метод `Save` у структуры `UserRepository`? 
Стандарт индустрии — использование символа подчеркивания `_` для разделения типа и метода:

```go
// Тестируем метод GetByID у структуры UserRepo
func TestUserRepo_GetByID(t *testing.T) { ... }

// Тестируем метод Validate у структуры Order
func TestOrder_Validate(t *testing.T) { ... }
```
Такой формат невероятно удобен при запуске конкретных тестов из консоли:
`go test -run=^TestUserRepo_` запустит тесты всех методов структуры `UserRepo`.

## 3. Именование Table-Driven сабтестов (t.Run)

В статье о [[4. Table driven tests]] мы использовали `t.Run(tc.name, ...)` для запуска подтестов. Именование этих сценариев — это то, где многие ломают копья.

В `tc.name` **не нужно** использовать camelCase, snake_case или повторять имя тестируемой функции. Пишите **человекочитаемые предложения с пробелами**, описывающие суть сценария. 

Раннер `go test` автоматически заменит пробелы на символы подчеркивания при выводе в консоль, чтобы вы могли легко скопировать имя для флага `-run`.

**Плохо:**
```go
name: "TestUserValidation_EmptyEmail"
name: "err_not_found"
name: "case1"
```

**Отлично (Идиоматично):**
```go
name: "successful creation of premium user"
name: "returns error when email is empty"
name: "conflict when user already exists"
```

Если тест упадет, вы увидите понятный лог:
`FAIL: TestUserRepo_Create/returns_error_when_email_is_empty`

## 4. Переменные want и got: Священная корова Go

В экосистемах Java (JUnit), C# (NUnit) или JS (Jest) принято использовать термины `expected` (ожидаемое) и `actual` (фактическое).

`assertEquals(expected, actual)`

В мире Go сложилась своя, уникальная традиция, продиктованная исходным кодом стандартной библиотеки (Standard Library). Авторы языка повсеместно используют короткие слова **`want`** (то, что мы хотим получить) и **`got`** (то, что мы получили по факту).

```go
// НЕ ИДИОМАТИЧНО (Стиль Java/C#)
func TestCalculate(t *testing.T) {
    expected := 42
    actual := Calculate()
    require.Equal(t, expected, actual)
}

// ИДИОМАТИЧНО (Стиль Go)
func TestCalculate(t *testing.T) {
    want := 42
    got := Calculate()
    require.Equal(t, want, got)
}
```

Почему `want` и `got`?
1. Они короче (4 буквы против 8 и 6), что делает код визуально чище.
2. Они выравниваются по табуляции:
   ```text
   want: 42
   got:  43
   ```
3. Это мгновенно выдает в вас разработчика, глубоко погруженного в культуру языка. Всегда используйте эти имена в ваших Table-Driven структурах:

```go
type testCase struct {
    name    string
    input   string
    want    int   // вместо expected
    wantErr bool  // вместо expectsError
}
```

## 5. Именование Benchmarks, Fuzz и Examples

Соглашения для тестов производительности и фаззинга зеркально повторяют правила для `TestXxx`.

* **Benchmarks:** `Benchmark<Type>_<Method>`. Пример: `BenchmarkRouter_ServeHTTP(b *testing.B)`
* **Fuzzing:** `Fuzz<Type>_<Method>`. Пример: `FuzzParseTLV(f *testing.F)`

**Особый случай: Примеры (Examples)**
В Go есть уникальный механизм тестов, которые одновременно являются документацией — функции `ExampleXxx()`. Если вы назовете функцию правильно, утилита `go doc` автоматически привяжет этот код к документации вашей функции на сайте `pkg.go.dev`.

* `func ExampleParse()` — привяжется к функции `Parse`.
* `func ExampleUser_Save()` — привяжется к методу `Save` типа `User`.
* `func ExampleUser_Save_withTimeout()` — создаст отдельный подраздел примера (с суффиксом `withTimeout`).

```go
func ExampleReverse() {
    fmt.Println(Reverse("hello"))
    // Output: olleh
}
```

## 6. Именование тестовых двойников (Mocks/Stubs)

При создании фейковых реализаций интерфейсов придерживайтесь следующих префиксов:

* **Mock:** (Например, `MockUserRepository`). Означает, что это умный объект (обычно сгенерированный через `gomock` или `mockery`), который проверяет, *какие методы были вызваны* и с какими аргументами.
* **Stub:** (Например, `StubUserRepository`). Означает, что это "тупая" заглушка, которая всегда возвращает жестко захардкоженный ответ и не содержит логики проверки вызовов.
* **Fake:** (Например, `FakeUserRepository` или `InMemoryUserRepository`). Означает, что это легковесная, но рабочая реализация (например, хранящая данные в `map` вместо PostgreSQL).

## Итог

1.  **Файлы:** Зеркально отражают тестируемые файлы (`router_test.go`).
2.  **Функции:** `Test<FuncName>` для обычных функций и `Test<Type>_<Method>` для методов. Первая буква после `Test` всегда заглавная.
3.  **Сабтесты:** Понятные, короткие предложения на английском языке с пробелами.
4.  **Переменные:** Оставьте `expected/actual` в прошлом. Ваш выбор — **`want` и `got`**.
5.  **Документация:** Используйте `ExampleXxx`, чтобы убить двух зайцев: покрыть код тестом и сгенерировать красивую документацию.

Соблюдение этих простых, но строгих правил превращает репозиторий из свалки кода в профессиональный инженерный продукт. 

Теперь, когда наши тесты идеально организованы, названы по стандартам индустрии и работают как часы, пришло время поручить их выполнение роботам. В следующей статье мы переходим к автоматизации: [[3. Настройка CI (GitHub Actions, GitLab CI)]].
