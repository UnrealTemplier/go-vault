## Архитектура порядка: Как не утонуть в собственных тестах

Вы прошли огромный путь: от базовых ассертов до фаззинга, профилирования и стресс-тестирования. У вас есть десятки файлов `*_test.go`, моки, стагбы, золотые файлы и бенчмарки. 

Но когда проект вырастает из микросервиса в солидный монолит, возникает проблема: `go test ./...` начинает выполняться по 5 минут, потому что юнит-тесты перемешались с интеграционными, поднимающими базы данных в Docker. Разработчики перестают запускать тесты локально, CI/CD пайплайн "краснеет", а тестовые утилиты дублируются в пяти разных пакетах.

В этой статье мы разберем стандарты индустрии по организации тестовой базы в Go-проектах: как разделять тесты, где хранить данные и как использовать теги сборки.

## 1. Пакетная принадлежность: Black-box vs White-box

В Go тесты всегда лежат рядом с тестируемым файлом (например, `handler.go` и `handler_test.go`). Но у вас есть выбор, как назвать пакет внутри файла `handler_test.go`.

### White-box тестирование (package mypkg)
Если вы используете то же имя пакета, что и в основном коде, тест получает доступ ко всем неэкспортируемым (приватным) переменным, структурам и функциям.

```go
// Файл: user.go
package domain
func calculateAge(birthDate time.Time) int { ... } // Приватная функция

// Файл: user_test.go
package domain
func TestCalculateAge(t *testing.T) {
    // Тест имеет доступ к приватной функции
    age := calculateAge(dob) 
}
```
**Когда использовать:** Для сложной алгоритмики внутри пакета, где необходимо изолированно протестировать приватные хелперы (например, математические расчеты, парсинг специфичных форматов).

### Black-box тестирование (package mypkg_test)
Добавление суффикса `_test` к имени пакета создает виртуальный внешний пакет. Тест сможет обращаться только к экспортируемому (публичному) API вашего кода.

```go
// Файл: api_test.go
package api_test // <-- Суффикс _test

import (
    "testing"
    "yourproject/internal/api" // Приходится импортировать тестируемый пакет!
)

func TestUserAPI(t *testing.T) {
    // Доступ только к публичным методам: api.NewHandler(), api.UserResponse
}
```
**Когда использовать:** Это **золотой стандарт** для тестирования слоев (Handlers, Services, Repositories). 
Black-box тесты защищают вас от хрупкости (Fragile Tests). Вы тестируете контракт (что пакет делает), а не его реализацию (как он это делает). Если вы отрефакторите приватные методы пакета, Black-box тест не сломается.

> [!tip] Собеседование
> **Вопрос:** Если тесты лежат в `mypkg_test`, как протестировать приватный метод или замокать приватную переменную пакета `mypkg`?
> **Ответ:** Вы не должны этого делать в Black-box тесте. Если приватная функция настолько сложна, что требует отдельного мокирования, это сигнал архитектурной проблемы — нарушение принципа единственной ответственности (SRP). Эту логику нужно вынести в отдельный пакет (или интерфейс), сделать публичной и мокировать/тестировать отдельно. 
> Как "хак" (Backdoor), можно создать в `mypkg` файл `export_test.go` с пакетом `package mypkg`, который публично экспортирует приватные переменные *только для тестов*, но в enterprise-разработке этого стараются избегать.

## 2. Изоляция интеграционных тестов

Самая частая проблема CI/CD: медленные тесты (ходящие в БД или сеть) выполняются вместе с быстрыми юнит-тестами. 

Существует три идиоматичных способа их разделить.

### Способ A: Флаг t.Short()
В стандартной библиотеке есть механизм пропуска долгих тестов.
Запускаем тесты с флагом: `go test -short ./...`

```go
func TestUserRepository_Integration(t *testing.T) {
	if testing.Short() {
		t.Skip("Пропуск интеграционного теста в коротком режиме")
	}
	
	// Поднимаем базу данных, делаем запросы...
}
```
**Плюсы:** Работает "из коробки".
**Минусы:** Не подходит, если видов тестов больше двух (например: Unit, Integration, E2E, Contract).

### Способ B: Переменные окружения (Environment Variables)
Часто тесты разделяют через `os.Getenv`.

```go
func TestPaymentGateway(t *testing.T) {
	if os.Getenv("RUN_E2E") == "" {
		t.Skip("Установите RUN_E2E=1 для запуска E2E тестов")
	}
}
```

### Способ C: Теги сборки (Build Tags) — Стандарт индустрии
Лучший способ — полностью исключить интеграционные тесты из процесса компиляции при обычном прогоне. Для этого в начале файла (до `package`) пишется специальный комментарий (Build Constraint).

```go
//go:build integration

package repository_test

import "testing"

func TestDatabaseConn(t *testing.T) {
    // Тест скомпилируется и запустится ТОЛЬКО если указан тег integration
}
```

**Как запускать:**
* Юнит-тесты (игнорируют файлы с тегом `integration`): `go test ./...`
* Только интеграционные (игнорируют файлы с тегом `unit`, если он есть): `go test -tags=integration ./...`

Это делает локальную разработку (TDD) молниеносной, а тяжелые тесты делегируются отдельному шагу в CI-пайплайне.

## 3. Директория testdata: Магия компилятора

Вам часто нужно хранить тестовые данные (fixtures): JSON-файлы с ответами API, SQL-скрипты для миграций, SSL-сертификаты или "золотые файлы" (Golden Files).

В Go для этого существует зарезервированное имя директории — `testdata`.

> [!info] Под капотом
> Директория `testdata` магическая. Инструментарий Go (включая компилятор `go build`, сборщик `go run` и утилиты `go vet` / `go get`) **полностью игнорирует** эту папку. Содержимое `testdata` никогда не попадет в итоговый бинарник и не вызовет ошибок линтера, даже если вы положите туда сломанный `.go` файл или гигабайтный дамп базы.

```text
internal/
  parser/
    parser.go
    parser_test.go
    testdata/
      valid_payload.json
      malformed.json
      expected_output.golden
```

Чтение из неё в тестах максимально простое, так как `go test` всегда устанавливает рабочую директорию (Current Working Directory) в ту папку, где лежит сам файл теста:

```go
func TestParseJSON(t *testing.T) {
	// Путь относительный от папки parser/
	data, err := os.ReadFile("testdata/valid_payload.json")
	require.NoError(t, err)
	// ...
}
```

## 4. Глобальная структура папок в проекте

Где должны лежать сами тесты? 

1. **Unit и компонентные тесты:** Всегда лежат рядом с кодом в папках `/internal/...` и `/pkg/...`.
2. **Integration тесты (Repository):** Обычно лежат рядом с реализацией репозитория (например, `internal/repository/user_pg_test.go`) и закрыты тегом `//go:build integration`.
3. **E2E и API тесты:** Так как эти тесты проверяют систему целиком (импортируют множество пакетов или поднимают `httptest.Server` на уровне `main`), их выносят в отдельную папку в корне проекта.

Типичный Layout энтерпрайз-проекта:

```text
my-backend/
├── cmd/
│   └── api/main.go
├── internal/
│   ├── handlers/
│   │   ├── auth.go
│   │   └── auth_test.go        # Unit (Black-box: package handlers_test)
│   ├── repository/
│   │   ├── pg.go
│   │   └── pg_test.go          # Integration (go:build integration)
│   └── testutil/               # Пакет с хелперами (генераторы данных, моки)
│       └── db.go
├── pkg/                        # Публичные утилиты
├── tests/                      # Внешние E2E тесты
│   ├── e2e/
│   │   └── checkout_test.go    # Поднимает приложение целиком (go:build e2e)
│   └── contract/               # Pact/Contract тесты (go:build contract)
└── testdata/                   # Глобальные тестовые данные (сертификаты и т.д.)
```

## 5. Тестовые утилиты и t.Helper()

Если у вас есть повторяющаяся логика настройки в тестах (например, инициализация базы данных, генерация JWT-токенов или создание HTTP-запроса), выносите её во вспомогательные функции.

Их можно хранить в пакете `internal/testutil` или прямо в файлах `*_test.go`.

> [!warning] Ловушка / Gotcha: Стек-трейс хелперов
> Если ваша вспомогательная функция делает `require.NoError(t, err)`, и тест падает, компилятор укажет строку ошибки **внутри хелпера**. 
> Когда этот хелпер используется в 50 разных тестах, вы не поймете, *какой именно* тест упал, пока не прочтете весь лог.

**Обязательное правило:** Всегда используйте `t.Helper()` в начале вспомогательных тестовых функций. Эта команда говорит фреймворку `testing` исключить эту функцию из стек-трейса при падении.

```go
package testutil

import (
	"testing"
	"[github.com/stretchr/testify/require](https://github.com/stretchr/testify/require)"
	"yourproject/internal/domain"
)

// GenerateTestUser создает пользователя для тестов
func GenerateTestUser(t *testing.T, email string) domain.User {
	t.Helper() // КРИТИЧЕСКИ ВАЖНО!

	u, err := domain.NewUser(email)
	
	// Если err != nil, ошибка в консоли укажет на ту строку в вашем TestXXX, 
	// где была вызвана функция GenerateTestUser, а не на строку ниже.
	require.NoError(t, err) 
	
	return u
}
```

## Итог

1.  Используйте **Black-box тестирование** (`package mypkg_test`) для проверки бизнес-слоев и контрактов, и White-box для изолированной сложной алгоритмики.
2.  Разделяйте быстрые и медленные тесты с помощью тегов компилятора (`//go:build integration`).
3.  Храните моки, JSON-дампы и сертификаты в директории `testdata`, чтобы не засорять бинарник.
4.  Выносите общие функции в `testutil` и обязательно маркируйте их вызовом `t.Helper()`.
5.  Глобальные E2E тесты храните в выделенной папке `/tests` в корне репозитория.
