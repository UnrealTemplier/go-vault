## Конец копипасты: Идиоматичный подход

В языках вроде Java или PHP (с PHPUnit) для тестирования функции с разными входными данными разработчики часто используют механизмы вроде `@DataProvider`. В других экосистемах пишут десятки отдельных функций: `TestAddPositiveNumbers`, `TestAddNegativeNumbers`, `TestAddZero`. Это порождает гигантское количество бойлерплейта — повторяющегося кода инициализации, вызова функции и проверки результата.

Go, как язык с акцентом на минимализм и явность, предлагает другой стандарт де-факто — **Table Driven Tests (Табличные тесты)**.

Табличный тест — это паттерн, при котором все тестовые сценарии (кейсы) определяются как элементы единого массива (или слайса) структур. Затем вы используете обычный цикл `for` для итерации по этой "таблице", выполняя тестируемую функцию для каждой строки и сравнивая результат с ожидаемым.

## Анатомия табличного теста

Рассмотрим типичный бэкенд-сценарий: валидация статуса платежа.

```go
package payment

import (
	"errors"
	"testing"
)

var ErrInvalidTransition = errors.New("invalid state transition")

func Transition(current, next string) error {
	if current == "PENDING" && next == "SUCCESS" {
		return nil
	}
	if current == "SUCCESS" && next == "REFUNDED" {
		return nil
	}
	return ErrInvalidTransition
}
```

А теперь напишем для него production-ready табличный тест:

```go
func TestTransition(t *testing.T) {
	// 1. Определение таблицы через анонимную структуру
	tests := []struct {
		name    string // Обязательное поле для имени кейса
		current string
		next    string
		wantErr error
	}{
		// 2. Заполнение "строк" таблицы
		{
			name:    "valid pending to success",
			current: "PENDING",
			next:    "SUCCESS",
			wantErr: nil,
		},
		{
			name:    "valid success to refund",
			current: "SUCCESS",
			next:    "REFUNDED",
			wantErr: nil,
		},
		{
			name:    "invalid success to pending",
			current: "SUCCESS",
			next:    "PENDING",
			wantErr: ErrInvalidTransition,
		},
	}

	// 3. Итерация и выполнение
	for _, tt := range tests {
		// Используем t.Run для изоляции каждого кейса в подтест
		t.Run(tt.name, func(t *testing.T) {
			err := Transition(tt.current, tt.next)
			
			// Проверка ошибок: идиоматичный подход с errors.Is
			if !errors.Is(err, tt.wantErr) {
				t.Errorf("Transition() error = %v, wantErr %v", err, tt.wantErr)
			}
		})
	}
}
```

### Разбор компонентов

1. **Анонимная структура (`[]struct{...}`):** В Go не принято создавать именованные структуры вроде `TestCase` на уровне пакета для тестов. Анонимная структура позволяет определить поля прямо по месту использования, не засоряя пространство имен.
2. **Поле `name`:** Критически важный элемент. При падении теста вы должны мгновенно понять, *какой именно* сценарий сломался, без необходимости считать индексы массива.
3. **Поля `want` и `wantErr`:** Негласный стандарт именования ожидаемых результатов. Использование префикса `want` позволяет легко читать код ассертов (`got %v, want %v`).
4. **`t.Run(name, func)`:** Метод, который регистрирует выполнение функции как отдельного подтеста.

> [!info] Под капотом
> Почему паттерн табличных тестов в Go так эффективен с точки зрения "железа" (Mechanical Sympathy)?
> Слайс анонимных структур под капотом — это структура `slice` (указатель на массив, длина, вместимость). Сам массив располагается в памяти как **единый непрерывный блок байтов** (Contiguous memory block).
> 
> ```mermaid
> flowchart LR
>     SliceHeader["Slice Header<br>Data Pointer"] -->|"Ссылается на"| Heap["Непрерывный блок в куче"]
>     subgraph Heap ["Memory Heap"]
>         direction LR
>         Struct1["Case 1<br>name, current, next"] --- Struct2["Case 2<br>name, current, next"] --- StructN["Case N<br>name, current, next"]
>     end
> ```
> Когда процессор (CPU) начинает выполнять цикл `for _, tt := range tests`, аппаратный Prefetcher (предсказатель выборок) мгновенно распознает линейный паттерн доступа к памяти. Он заранее подгружает следующие элементы `tt` в сверхбыстрые кэши L1/L2 целыми кэш-линиями (Cache Lines, обычно по 64 байта). 
> В ООП-языках, где каждый тестовый кейс — это отдельный класс-объект, раскиданный по куче, процессор страдает от кэш-промахов (Cache Misses). В Go итерация по тестовой таблице работает со скоростью света.

## Ловушка замыкания: `tt := tt`

В бэкенде мы часто делаем тесты параллельными для ускорения CI-пайплайна. И здесь кроется самый знаменитый баг в истории Go-разработчиков, который стал классическим вопросом на собеседованиях.

Рассмотрим код (версия Go до 1.22):

```go
func TestParallelTable(t *testing.T) {
	tests := []struct {
		name string
		val  int
	}{
		{"case 1", 1},
		{"case 2", 2},
		{"case 3", 3},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			t.Parallel() // Говорим рантайму запустить подтест конкурентно
			
			// Ловушка! Какое значение val здесь будет?
			if tt.val < 0 { 
				t.Fail()
			}
		})
	}
}
```

> [!warning] Ловушка / Gotcha
> **До Go 1.22** переменная цикла `tt` переиспользовалась. Она имела один и тот же адрес в памяти на протяжении всего цикла, а её значение просто перезаписывалось на каждой итерации.
> Когда вы вызываете `t.Parallel()`, рантайм `testing` ставит вызов функции на паузу и продолжает цикл. Цикл мгновенно доходит до конца, переменная `tt` получает значение последнего кейса (`{"case 3", 3}`).
> Затем планировщик будит все три горутины подтестов, и *все они* читают по замыканию (Closure) одно и то же последнее значение `tt`. У вас три раза выполнится тест для "case 3", а "case 1" и "case 2" будут проигнорированы!

**Решение (Idiomatic Go < 1.22):**
Необходимо было создавать локальную копию переменной внутри цикла (Shadowing):
```go
for _, tt := range tests {
	tt := tt // Создаем новую переменную tt в области видимости итерации
	t.Run(tt.name, func(t *testing.T) {
		t.Parallel()
		// Теперь безопасно
	})
}
```

**Современный Go (1.22+):**
В Go 1.22 семантика цикла `for` была изменена на уровне языка. Теперь переменные, объявленные в `range`, автоматически создаются заново для каждой итерации. Костыль `tt := tt` больше не нужен, но знать о нем необходимо для поддержки старых кодовых баз и прохождения собеседований.

## Продвинутые техники табличных тестов

### 1. Использование `map` вместо `slice`

Иногда вместо слайса структур разработчики используют мапу (hash-table), где ключом выступает имя теста.

```go
func TestWithMap(t *testing.T) {
	tests := map[string]struct {
		input int
		want  int
	}{
		"positive": {input: 5, want: 10},
		"negative": {input: -5, want: -10},
	}

	for name, tt := range tests {
		t.Run(name, func(t *testing.T) {
			// ...
		})
	}
}
```

> [!tip] Собеседование
> **Вопрос:** В чем архитектурный минус использования `map` для табличных тестов вместо `[]struct`?
> **Ответ:** Как мы обсуждали в статье о детерминизме, итерация по `map` в Go **рандомизирована**. Тесты каждый раз будут запускаться в разном порядке. Если между тестами есть скрытая зависимость (которой быть не должно, но в legacy бывает), баг станет "плавающим". Кроме того, слайс сохраняет визуальный порядок: тесты выполняются ровно так, как вы их читаете в коде сверху вниз. `[]struct` — это выбор Senior-инженера по умолчанию.

### 2. Динамическая генерация таблиц

Таблицу не обязательно хардкодить. Вы можете парсить JSON/YAML файлы из директории `testdata/` и динамически формировать слайс со структурами прямо перед циклом `for`. Это позволяет тестировщикам или аналитикам добавлять новые тест-кейсы без перекомпиляции и изменения исходного кода.

## Итог

1. **Table Driven Tests** — это идиоматичный стандарт Go для параметризованного тестирования, снижающий дублирование кода.
2. Под капотом используется слайс анонимных структур, что дает **максимальную утилизацию кэша CPU** (L1/L2) за счет линейного чтения памяти.
3. Помните про **loop closure bug** при использовании `t.Parallel()` в проектах со старыми версиями Go (< 1.22).
4. Избегайте использования `map` для таблиц, если вам важен детерминированный порядок выполнения.

Мы уже вскользь упомянули важнейший метод `t.Run()`, который делает табличные тесты читаемыми в логах консоли и позволяет запускать их выборочно (например, через `go test -run TestTransition/valid_success_to_refund`). Пора заглянуть под капот этого механизма в следующей статье: [[5. Subtests и t.Run]].
