
В микросервисной архитектуре, когда команды работают над разными сервисами на разных репозиториях, самым хрупким местом становится точка их соприкосновения. Если сервис `A` ожидает, что поле в JSON называется `user_id`, а сервис `B` после обновления начал присылать `uuid`, вся система развалится на продакшене.

**API Contract (API-контракт)** — это формальное, машиночитаемое соглашение о том, как сервисы взаимодействуют друг с другом. Это "конституция" вашего взаимодействия, которая должна быть стабильной и защищенной от случайных изменений.

## 1. Типы контрактов

Существует два основных подхода к определению контрактов в Go-экосистеме:

### А. Схемные контракты (gRPC / Protobuf)
Это самый надежный способ. Вы описываете структуру данных и методы в `.proto` файлах. 
* **Single Source of Truth:** Файл контракта отделен от реализации.
* **Генерация кода:** Из одной схемы генерируется типизированный Go-код для клиента и сервера.
* **Обратная совместимость:** Protobuf спроектирован так, чтобы добавление новых полей не ломало старых клиентов.



### Б. Документарные контракты (OpenAPI / Swagger)
Используются для REST API. Обычно это YAML или JSON файл, описывающий эндпоинты, методы и схемы запросов/ответов.
* **Для внешних клиентов:** Идеально, если ваш API используют мобильные приложения или сторонние интеграции.
* **Инструментарий:** Позволяет генерировать интерактивную документацию (Swagger UI) и клиентские SDK.

## 2. Контрактное тестирование (Consumer-Driven Contracts)

Даже если у вас есть схема, вы можете случайно нарушить логику. Например, начать возвращать пустую строку там, где раньше всегда был ID. Чтобы этого избежать, применяется **Pact-тестирование**.

Суть **Consumer-Driven Contracts (CDC)**:
1. Потребитель (Consumer) пишет тест, который фиксирует его ожидания от API.
2. Эти ожидания упаковываются в JSON-файл (Pact).
3. Поставщик (Provider) запускает этот файл против своего кода. Если код поставщика не соответствует ожиданиях потребителя, билд в CI/CD падает.

## 3. Mechanical Sympathy: Контракты и рантайм Go

В Go работа с контрактами напрямую связана с эффективностью памяти.

> [!info] Под капотом: Опциональность полей
> В JSON (и старых версиях Protobuf) было сложно отличить "нулевое значение" (например, 0 или "") от "отсутствующего значения".
> В современном Go и Protobuf v3 для этого используются **указатели** или специальные типы-обертки (например, `google.protobuf.Int64Value`). 
> Это позволяет избежать логических ошибок, когда вы случайно затираете данные в базе нулями просто потому, что клиент не прислал поле в запросе.

## 4. Золотые правила работы с контрактами

1. **Закон Постела (Принцип робастности):** "Будь консервативен в том, что отправляешь, и либерален в том, что принимаешь". Если вам прислали лишнее поле, которого нет в контракте — просто игнорируйте его, не выдавайте ошибку.
2. **Никогда не удаляйте поля:** Если поле больше не нужно, пометьте его как `deprecated`, но оставьте в схеме. Удаление — это всегда breaking change.
3. **Версионирование в URL/Заголовках:** Всегда закладывайте возможность существования `v1` и `v2` одновременно.

```go
// Пример идиоматичного описания контракта через Protobuf
// service.proto
syntax = "proto3";

package orders.v1;

message CreateOrderRequest {
  string user_id = 1;
  repeated string item_ids = 2;
  // Используем обертку для опционального поля
  optional string promo_code = 3; 
}

service OrderService {
  rpc CreateOrder(CreateOrderRequest) returns (CreateOrderResponse);
}
```

> [!tip] Собеседование
> **Вопрос:** Как вы обновите API контракт, если нужно внести ломающее изменение (Breaking Change)?
> **Ответ:** Я запущу параллельную версию API. Сначала деплоится сервер, поддерживающий и старый, и новый контракт. Затем постепенно переводятся все клиенты на новый контракт (Canary или Blue-Green). Только когда трафик на старый эндпоинт упадет до нуля, его можно будет удалить.

## Итог

API-контракт — это единственный способ сохранить рассудок в системе из десятков микросервисов. В Go лучше всего использовать **gRPC/Protobuf** для внутренних вызовов, так как это дает гарантии компилятора еще до запуска кода.

Но даже самый лучший контракт бесполезен, если вы не можете обновить один сервис, не сломав другой. О том, как управлять версиями и обеспечивать плавную эволюцию системы, поговорим в следующей статье: [[6. Versioning сервисов]].
