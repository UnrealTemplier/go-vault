
Разработка микросервисов — это марафон, а не спринт. Ваши API неизбежно будут меняться по мере развития бизнеса. Однако в распределенной системе вы не можете просто обновить все сервисы одновременно. В любой момент времени в вашем кластере будут сосуществовать разные версии потребителей и поставщиков.

**Versioning (Версионирование)** — это стратегия управления жизненным циклом API, которая позволяет системе эволюционировать, не вызывая каскадных сбоев у зависимых сервисов.

## 1. Два пути эволюции API

### Обратимая совместимость (Backward Compatibility)
Новая версия сервера понимает запросы от старых клиентов. 
* **Пример:** Добавление нового необязательного поля в ответ или поддержку нового опционального параметра запроса.
* **Правило Go/Protobuf:** Не меняйте номера тегов у существующих полей и не переименовывайте их.

### Прямая совместимость (Forward Compatibility)
Старые клиенты могут работать с новыми серверами.
* **Пример:** Клиент версии `v1` получает ответ от сервера `v2`, который содержит лишние поля. Клиент просто игнорирует неизвестные данные.
* **Закон Постела в Go:** При десериализации JSON или Protobuf в структуры, рантайм Go по умолчанию игнорирует поля, которых нет в `struct`. Это и обеспечивает прямую совместимость.

## 2. Стратегии версионирования

### Версионирование в пути (Path Versioning)
Самый распространенный способ для REST API.
`https://api.service.com/v1/orders`
`https://api.service.com/v2/orders`
* **Плюсы:** Полная изоляция логики, легко маршрутизировать на уровне Nginx/Ingress.
* **Минусы:** Раздувание кодовой базы (приходится поддерживать два набора хендлеров).

### Версионирование через заголовки (Header Versioning)
`Accept: application/vnd.myapi.v2+json`
* **Плюсы:** URL остается чистым, ресурс остается тем же самым.
* **Минусы:** Сложнее тестировать через браузер, требует сложной логики диспетчеризации внутри Go-кода.

### Пакетное версионирование в gRPC
В Protobuf версионирование закладывается на уровне `package`.
```proto
package orders.v2;

service OrderService { ... }
```
В Go это превращается в разные импорты: `import "github.com/org/gen/orders/v2"`.

## 3. Семантическое версионирование (SemVer)

Для библиотек и микросервисов используется формат `MAJOR.MINOR.PATCH` (например, `2.3.1`):
1.  **MAJOR:** Ломающие изменения (Breaking Changes). Требует смены префикса API (`/v1` -> `/v2`).
2.  **MINOR:** Новая функциональность без потери совместимости.
3.  **PATCH:** Багфиксы.

---

## 4. Процесс вывода из эксплуатации (Deprecation)

Вы не можете вечно поддерживать `v1`. Идиоматичный процесс удаления старой версии в микросервисах:

1.  **Announce:** Пометить методы как `Deprecated` в документации и коде.
2.  **Monitor:** Использовать метрики (Prometheus), чтобы увидеть, кто всё еще ходит в `v1`.
3.  **Sunset Header:** Добавить в HTTP-ответы заголовки `Sunset` или `Warning`, уведомляющие клиентов о скором отключении.
4.  **Brownout:** Кратковременные (на несколько минут) искусственные отключения старой версии, чтобы "взбодрить" тех, кто проигнорировал уведомления.
5.  **Shutdown:** Окончательное удаление кода.

## Mechanical Sympathy: Версионирование и типизация Go

В Go при работе с разными версиями API критически важно избегать дублирования логики при сохранении строгой типизации.

> [!info] Под капотом: Композиция и адаптеры
> Вместо того чтобы копировать всю бизнес-логику для `v2`, идиоматично создать слой адаптеров.
> 1. Внутренняя логика работает с `InternalOrder`.
> 2. Хендлер `v1` мапит `v1.Request` -> `InternalOrder`.
> 3. Хендлер `v2` мапит `v2.Request` -> `InternalOrder`.
> Это позволяет обновлять контракты, не трогая ядро системы.

---

> [!tip] Собеседование
> **Вопрос:** Что считается "ломающим изменением" (Breaking Change) в Protobuf/gRPC?
> **Ответ:** > 1. Изменение номера тега у поля.
> 2. Изменение типа поля (например, с `int32` на `string`).
> 3. Переименование сервиса или метода.
> 4. Удаление поля (если клиент всё еще его ожидает).
> 5. Изменение правил валидации (например, поле стало обязательным).

## Итог

Версионирование — это способ сделать изменения безопасными. Помните, что **лучшее изменение — это то, которое не ломает совместимость**. Если вы можете добавить новое поле вместо изменения старого — сделайте это.

Даже если вы идеально версионируете API, вы всё равно столкнетесь с ситуациями, когда старые клиенты будут присылать данные, которые сервер `v2` должен корректно обрабатывать. О том, как обеспечить выживаемость системы при постоянных обновлениях, поговорим в следующей статье: [[7. Backward compatibility]].
