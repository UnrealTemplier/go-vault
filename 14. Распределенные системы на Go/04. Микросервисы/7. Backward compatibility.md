
В распределенной системе, где сотни микросервисов деплоятся независимо, **обратная совместимость (Backward Compatibility)** — это не просто вежливость, это механизм выживания. Вы никогда не сможете обновить сервер и всех его клиентов одновременно.

Обратная совместимость означает, что **новый сервер по-прежнему понимает запросы от старых клиентов**. Если вы выкатили `v2` сервиса, а мобильное приложение пользователя, не обновлявшееся три месяца, внезапно получило `HTTP 400` или `500` — вы нарушили контракт.

---

## 1. Золотые правила Protobuf

Поскольку в Go-микросервисах gRPC является стандартом де-факто, обратная совместимость здесь завязана на механику **Protocol Buffers**.

### Поля и Теги

В бинарном потоке Protobuf не передает имена полей (вроде `user_id`). Он передает только **номера тегов**.

- **Правило 1:** Никогда не меняйте номера тегов у существующих полей.
- **Правило 2:** Если вы удаляете поле, зарезервируйте его номер и имя через `reserved`, чтобы никто случайно не использовал их в будущем.

### Новые поля

Добавление нового поля в Protobuf — это всегда безопасно. Старые клиенты, получая ответ от нового сервера, просто проигнорируют неизвестный тег. Новый сервер, получая запрос от старого клиента, увидит отсутствие поля и использует **Zero Value** (по умолчанию) или проверит наличие через `has` (если используется `optional`).

## 2. Ловушки JSON в Go

С JSON всё сложнее. В Go стандартный `encoding/json` по умолчанию игнорирует неизвестные поля при десериализации (unmarshal) в структуру, что дает нам "бесплатную" совместимость. Но есть нюансы:

> [!warning] Ловушка / Gotcha: Проблема Zero Values
> 
> В Go `int` по умолчанию равен `0`, а `bool` — `false`.
> 
> Представьте, что в `v1` поле `is_active` всегда было `true`. В `v2` вы решили его удалить. Старый клиент отправляет JSON без этого поля. Ваш Go-сервер `v2` десериализует это в структуру, где `IsActive` станет `false`. Логика сломается.
> 
> **Решение:** Используйте указатели (`*bool`, `*int`) для опциональных полей. Если в JSON поля нет — указатель будет `nil`.

## 3. Паттерн "Expand and Contract" (Расширение и Сужение)

Если вам нужно внести "ломающее" изменение (например, переименовать поле или изменить его тип), вы не можете сделать это за один деплой. Используется четырехэтапный танец:

1. **Expand (Расширение):** Добавьте новое поле в схему, сохранив старое. Сервер должен уметь писать в оба поля, но читать преимущественно из старого.
2. **Migrate (Миграция):** Обновите всех клиентов, чтобы они начали использовать новое поле.
3. **Sync (Синхронизация):** Переключите логику сервера на чтение только из нового поля. Убедитесь, что старое поле больше не используется (мониторинг метрик).
4. **Contract (Сужение):** Удалите старое поле из контракта и кода.

## 4. Mechanical Sympathy: Unknown Fields в Go

Когда Go-сервис получает Protobuf-сообщение с полями, которых он "не знает" (пришедшими из более новой версии клиента), он не просто выбрасывает их в корзину.

> [!info] Под капотом: Поле `XXX_unrecognized`
> 
> Сгенерированные Go-структуры для Protobuf (особенно в старых версиях `github.com/golang/protobuf`) содержали поле `XXX_unrecognized []byte`.
> 
> Современная библиотека `google.golang.org/protobuf` хранит неизвестные поля внутри скрытых механизмов рефлексии.
> 
> **Зачем это нужно?** Это позволяет реализовать прокси-серверы. Сервис может получить сообщение, не знать часть его полей, но при пересылке этого сообщения дальше (например, в БД или другому сервису) — **сохранить** эти неизвестные поля. Это критично для Forward Compatibility.

## 5. Валидация и обратная совместимость

Самый частый способ случайно сломать совместимость — это **ужесточение валидации**.

- **v1:** Поле `description` необязательно.
- **v2:** Вы добавили проверку `if req.Description == "" { return Error }`.
- **Результат:** Все старые клиенты, не присылавшие описание, внезапно "отыквятся".

**Совет:** Любая новая валидация должна сначала выкатываться в режиме `soft-launch` (логирование ошибок без блокировки запроса), чтобы вы увидели, сколько клиентов она затронет.

---

> [!tip] Собеседование
> 
> **Вопрос:** Как проверить обратную совместимость API автоматически?
> 
> **Ответ:** Кроме `Pact-тестов`, можно использовать линтеры схем. Для Protobuf есть инструмент `buf breaking`, который сравнивает текущую версию `.proto` файлов с предыдущей (из git) и выдает ошибку, если вы удалили поле или изменили его номер/тип.

## Итог

1. **Теги в Protobuf священны.** Никогда не меняйте их.
2. **JSON — коварен.** Используйте указатели для различения "нуля" и "отсутствия".
3. **Ломайте плавно.** Используйте "Expand and Contract".
4. **Слушайте Постела.** Будьте снисходительны к тому, что вам присылают (игнорируйте лишнее).

Мы научились поддерживать стабильность при обновлении кода. Но что, если сервис обновляется слишком часто или их становится слишком много? Как клиенту найти нужный экземпляр сервиса в динамической среде облака? Переходим к теме: [[8. Service Discovery]].
