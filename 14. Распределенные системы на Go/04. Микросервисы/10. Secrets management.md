
В мире микросервисов грань между «конфигурацией» и «секретом» часто размыта. Запомните простое правило: **если коммит этого значения в публичный репозиторий заставит вас менять все пароли — это секрет.**

**Secrets Management** — это дисциплина безопасного хранения, передачи и ротации чувствительных данных (паролей, токенов, сертификатов).

---
### 1. Почему Environment Variables — это «на троечку»?

Многие следуют методологии 12-Factor App и пихают секреты в переменные окружения. Это лучше, чем хардкод, но у этого подхода есть «дыры»:
* **Логирование:** Команда `env` или случайный дамп состояния процесса при падении выведет все ваши секреты в логи.
* **Дочерние процессы:** Переменные окружения наследуются всеми форками и сторонними утилитами, которые вы запускаете из кода.
* **Видимость в UI:** В консолях управления (AWS, Kubernetes) переменные часто видны всем, у кого есть доступ к просмотру подов.

---
### 2. HashiCorp Vault: Золотой стандарт

Для серьезных систем используется **HashiCorp Vault** или облачные аналоги (AWS Secrets Manager, GCP Secret Manager).

**Ключевые фичи:**
* **Dynamic Secrets:** Vault может генерировать временный пароль к базе данных специально для вашего сервиса, который сам удалится через час.
* **Leasing & Revocation:** Вы можете мгновенно отозвать доступ у скомпрометированного сервиса.
* **Audit Log:** Вы точно знаете, кто, когда и какой секрет запрашивал.

### 3. Секреты в Kubernetes

В K8s есть объект `Secret`. По умолчанию они хранятся в `etcd` в формате **base64** (это НЕ шифрование, это просто кодирование!).

**Как сделать их безопаснее:**
1. **Encryption at rest:** Включите шифрование `etcd` на уровне кластера.
2. **External Secrets Operator:** Позволяет синхронизировать секреты из внешнего Vault напрямую в K8s Secrets.
3. **Vault Agent Injector:** Самый безопасный путь. Vault прокидывает секреты прямо в `tmpfs` (оперативную память) пода в виде файла. Они никогда не касаются диска и не светятся в `env`.

### 4. Mechanical Sympathy: Безопасность в рантайме Go

Даже если вы безопасно доставили секрет в Go-приложение, его можно случайно «засветить».

> [!warning] **Опасность: String Immortality**
> Строки в Go неизменяемы. Если вы прочитали пароль в `string`, он может остаться в памяти до следующего цикла GC. А если он попадет в лог через `fmt.Sprintf("%v", cfg)`, он останется там навсегда.

**Идиоматичные советы:**
* **Используйте `[]byte` вместо `string`:** Вы можете занулить массив байт (`copy(buf, []byte{0})`) сразу после того, как установили соединение с базой.
* **Кастомные маршалеры:** Реализуйте интерфейс `Stringer` или `MarshalJSON` для структур с секретами, чтобы они всегда выводили `****`.

```go
type DBConfig struct {
    Host     string
    Password string `json:"-"` // Скрываем при обычном маршалинге
}

// Защита от случайного логирования всей структуры
func (c DBConfig) String() string {
    return fmt.Sprintf("DBConfig(Host: %s, Password: [REDACTED])", c.Host)
}
````

### 5. Итог

1. **Git — это не место для секретов.** Используйте `git-crypt` или `sops`, если всё же нужно хранить их в репо, но лучше этого избегать.
2. **Vault / Cloud KMS:** Используйте специализированные хранилища с политиками доступа.
3. **Краткосрочность:** Стремитесь к тому, чтобы ваши секреты имели короткий срок жизни (TTL) и автоматически ротировались.
4. **Гигиена кода:** Не выводите структуры конфигурации в логи целиком.
