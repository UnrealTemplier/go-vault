## Антипаттерны микросервисов: Как не надо строить распределенные системы

Микросервисы часто воспринимаются как "серебряная пуля": разбил монолит на кусочки — и вот у вас Netflix, масштабирование и скорость разработки. Реальность же такова, что микросервисная архитектура — это компромисс. Вы меняете сложность кода на сложность эксплуатации и сети.

Если перейти грань между "правильными" микросервисами и хаосом, вы получите **Распределенный Монолит** — худшее из двух миров: сложность распределенных систем плюс жесткая связность монолита.

В этой статье разберем самые частые и болезненные антипаттерны, которые убивают производительность Go-сервисов и ментальное здоровье разработчиков.

---

## 1. Распределенный Монолит (Distributed Monolith)

Это самый страшный антипаттерн. Вы разрезали монолит на сервисы, но не разорвали связи между ними.

**Признаки:**
*   Чтобы задеплоить сервис A, нужно обязательно задеплоить сервис B той же версии.
*   Сервисы шарят одну базу данных (об этом ниже).
*   Изменение контракта в одном сервисе ломает рантайм другого.

### Почему это плохо?
Вы получаете все минусы микросервисов (сетевые задержки, сложность деплоя, распределенные транзакции), но не получаете их плюсов (независимый деплой, масштабирование отдельных частей). Система становится хрупкой: падение одного маленького сервиса роняет всю цепочку.

> [!warning] Ловушка / Gotcha
> В Go часто пытаются вынести общую логику в отдельный репозиторий (Shared Library), который подключают ко всем сервисам.
> Если вы меняете функцию в библиотеке и вынуждены обновлять и пересобирать 10 сервисов, чтобы починить один — у вас распределенный монолит.
> **Решение**: Используйте Versioning для библиотек и строго следите за обратной совместимостью.

## 2. Наносервисы (Nanoservices)

Это болезнь "разрезания по классам". Выносить каждый класс или функцию в отдельный микросервис.

**Пример**: В интернет-магазине выносим "Сервис скидок", "Сервис расчета НДС", "Сервис округления цен" в три разных деплоя.

### Почему это плохо?
*   **Network Latency**: Вместо вызова функции в памяти (наносекунды), вы идете по сети (миллисекунды). Суммарная latency API растет кратно.
*   **Chatty Interface**: Количество сетевых вызовов explodes. Для формирования одной корзины нужно сделать 20 HTTP-запросов.

> [!info] Под капотом
> Сетевой вызов (RPC) на порядки дороже вызова функции.
> *   **Function call**: Push registers, Call instruction (такты CPU).
> *   **HTTP call**: Syscall `write`, Marshal JSON, TCP Handshake (если нет keep-alive), Network RTT, Unmarshal JSON, Syscall `read`.
> В Go маршалинг JSON — дорогая операция (рефлексия). Если ваш сервис делает цепочку из 5 наносервисов, CPU тратит 50% времени просто на упаковку/распаковку JSON.

**Решение**: Группируйте функциональность по бизнес-поддоменам (DDD). "Сервис ценообразования" должен включать в себя и скидки, и НДС.

## 3. Общая база данных (Shared Database)

Самый частый антипаттерн при миграции с монолита. Сервисы разделены по коду, но все ходят в одну и ту же PostgreSQL базу.

**Проблема**: База данных становится де-факто API.
1.  Сервис `Orders` меняет схему таблицы `users` (добавляет колонку).
2.  Сервис `Auth`, который делал `SELECT * FROM users`, падает с ошибкой десериализации.

### Почему это плохо?
*   **Coupling**: Сервисы связаны через схему данных. Невозможно менять схему независимо.
*   **Lock contention**: Два сервиса могут блокировать одни и те же строки таблицы, вызывая каскадные задержки.

**Решение**: Паттерн **Database per Service**. Если сервисам нужны чужие данные, они должны запрашивать их через API, а не лезть в таблицу напрямую. Для синхронизации данных использовать [[1. Saga pattern]] или Event Carrying State Transfer.

## 4. Синхронный ад (Synchronous Hell)

Сервисы вызывают друг друга по цепочке синхронно.
`Client -> Service A -> Service B -> Service C`.

Если сервис C отвечает 1 секунду, то пользователь ждет минимум 3 секунды (если нет параллелизма), а нагрузка висит на всех сервисах в цепочке.

### Почему это плохо?
*   **Каскадные сбои**: Если C падает по таймауту, потоки/горутины в B висят и ждут, накапливая нагрузку. B падает, затем A.
*   **Слабая отказоустойчивость**: Система не может работать, если недоступен хотя бы один участник цепочки.

**Решение**: Асинхронное взаимодействие через Message Broker (RabbitMQ, Kafka) или паттерн [[2. CQRS]] с Event-driven подходом.
`Service A -> Publish Event -> Broker -> Service B`.

## 5. Умные пайпы и глупые эндпоинты (Smart Pipes, Dumb Endpoints)

В попытке оптимизировать запросы разработчики часто начинают добавлять бизнес-логику в инфраструктурные слои.

**Пример**: В [[4. API Gateway]] или в Middleware (например, в gRPC Interceptor) пишут логику проверки "Можно ли этому юзеру давать скидку?".

```go
// Антипаттерн: Бизнес-логика в middleware
func DiscountMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // Проверка бизнес-правила в слое маршрутизации!
        if user.IsPremium && time.Now().Weekday() == time.Friday {
            // ... меняем заголовки ...
        }
        next.ServeHTTP(w, r)
    })
}
```

### Почему это плохо?
Инфраструктура должна быть "глупой" и быстрой. Она должна только маршрутизировать и трансформировать протоколы. Бизнес-логика должна жить в Domain Service. Иначе вы получаете "черный ящик", который невозможно тестировать и отлаживать.

## 6. Отсутствие идемпотентности (Lack of Idempotency)

В распределенных системах сеть ненадежна. Timeouts случаются.
Клиент отправил запрос `POST /pay`. Сервис обработал платеж, но сеть упала перед отправкой ответа. Клиент (или Retry-механизм) отправляет запрос снова. Деньги списываются дважды.

> [!warning] Ловушка / Gotcha
> В Go `http.Client` по умолчанию не делает ретраи для POST запросов (в отличие от некоторых библиотек в других языках). Но если вы используете Ribbone, Istio Retries или свой цикл retry, вы легко столкнетесь с двойным выполнением.

**Решение**: Каждая операция изменения состояния (Mutation) должна быть идемпотентной.
Используйте **Idempotency Key** (уникальный ID запроса), который передается в заголовке.

```go
func (s *PaymentService) Charge(ctx context.Context, req ChargeRequest, idempotencyKey string) error {
    // 1. Проверяем, был ли уже такой ключ
    exists, _ := s.cache.Get(idempotencyKey)
    if exists {
        return nil // Уже обработали, просто возвращаем успех (или предыдущий результат)
    }
    
    // 2. Выполняем транзакцию
    err := s.db.Withdraw(req.UserID, req.Amount)
    if err != nil {
        return err
    }
    
    // 3. Сохраняем ключ (TTL должен быть разумным)
    s.cache.Set(idempotencyKey, "processed", 24*time.Hour)
    return nil
}
```

## 7. God Service (Божественный сервис)

Обратная сторона наносервисов. Один сервис делает всё: и авторизацию, и рассылку email, и генерацию отчетов.

**Признаки**:
*   Огромное количество зависимостей в конструкторе (20+ интерфейсов).
*   Изменения в совершенно разных фичах требуют правок в одном репозитории.
*   Команда боится трогать код, потому что "а вдруг сломается то, о чем мы не знаем".

**Решение**: Декомпозиция по бизнес-границам (Bounded Contexts). Выделение отдельного сервиса для Email, отдельного для Billing.

---

## Итог

Микросервисы — это сложный инструмент. Чтобы не выстрелить себе в ногу, избегайте:

1.  **Distributed Monolith**: Не делайте сервисы зависимыми друг от друга при деплое.
2.  **Nanoservices**: Не дробите слишком мелко, Network Latency убьет производительность.
3.  **Shared Database**: Каждому сервису — свою базу.
4.  **Synchronous Chains**: Используйте асинхронность, где это возможно.
5.  **Business Logic in Middleware**: Оставьте Gateway и Middleware "глупыми".
6.  **Non-idempotent APIs**: Всегда проектируйте обработку ретраев.
