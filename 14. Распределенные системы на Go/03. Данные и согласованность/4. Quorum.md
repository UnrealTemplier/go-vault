
В прошлых статьях мы увидели два полюса распределенных систем: абсолютную консистентность (Strong Consistency), требующую синхронизации, и абсолютную доступность (Eventual Consistency), позволяющую узлам отставать. Мы также рассмотрели [[3. Read repair]], который помогает лениво чинить данные.

Но в реальном мире, особенно в AP-базах данных (наследниках Amazon Dynamo, таких как Cassandra или Riak), консистентность — это не бинарный выбор. Это ползунок (tunable consistency), который ты, как Go-разработчик, можешь двигать для каждого отдельного запроса к БД. 

Математический фундамент, позволяющий управлять этим ползунком, называется **Кворумом (Quorum)**.

## Уравнение Кворума: R + W > N

В основе кворумных систем лежит простая переменная `N` — **Replication Factor (Фактор репликации)**. 
Это количество узлов, на которых физически хранится копия одной и той же записи. 

> [!warning] Ловушка / Gotcha: Что такое N?
> Частая ошибка на собеседованиях — путать `N` с общим размером кластера. Если в твоем кластере Cassandra 100 серверов, а Replication Factor равен 3, то `N = 3`. Каждая строчка данных лежит только на трех конкретных серверах.

Чтобы система работала надежно, нам нужно определить два параметра:
* **W (Write Quorum):** От скольких узлов координатор должен получить подтверждение об успешной записи (fsync на диск или в RAM), прежде чем вернуть клиенту HTTP 200 OK.
* **R (Read Quorum):** Сколько узлов координатор должен опросить при чтении, чтобы сравнить их версии и вернуть самую свежую (отбросив отставшие реплики).

**Золотое правило Кворума:**
Чтобы гарантировать, что чтение *всегда* вернет самую свежую запись (избежать Stale Reads), система должна удовлетворять неравенству:

**R + W > N**

### Принцип Дирихле (Механика пересечения)

Почему это уравнение работает? Это математическое применение принципа голубиных гнезд (или пересечения множеств). Если сумма узлов для чтения и узлов для записи строго больше общего количества реплик, то **множество узлов чтения гарантированно пересечется с множеством узлов записи как минимум в одном узле**.

Этот один общий узел (Intersection Node) и принесет самую свежую версию данных (с максимальным Timestamp), которую координатор отдаст клиенту.

## Настройка ползунка (Use Cases)

В зависимости от профиля нагрузки (Write-heavy или Read-heavy), ты настраиваешь `R` и `W` в драйвере базы данных в твоем Go-коде. Рассмотрим для `N = 3`.

### 1. Сбалансированный (W = 2, R = 2)
Самый популярный подход (Quorum). 
`2 + 2 > 3` (условие выполняется). 
Система переживет падение одного узла. И чтение, и запись имеют умеренную задержку (Latency), так как ждут ответа от 2 узлов из 3.

### 2. Оптимизация записи (W = 1, R = 3)
`1 + 3 > 3` (условие выполняется).
Запись происходит невероятно быстро. Координатор ждет ответа только от одного узла (ближайшего) и сразу отвечает клиенту. 
Но чтение страдает: координатор обязан опросить все 3 реплики. Если хотя бы один из 3 узлов "моргнет" в сети, чтение завершится ошибкой таймаута (потеряна Availability для чтения). 
*Применение:* Системы сбора метрик, логов или IoT-телеметрии, где данные пишутся потоком, а читаются редко.

### 3. Оптимизация чтения (W = 3, R = 1)
`3 + 1 > 3` (условие выполняется).
Идеально для кэшей или каталогов товаров. Чтение молниеносное (читаем с одной, любой ближайшей реплики). 
Запись медленная и хрупкая (нужно дождаться подтверждения от всех 3 узлов). Если один узел лежит, мы не сможем обновить данные.

### 4. Полный Eventual Consistency (W = 1, R = 1)
`1 + 1 < 3` (условие НЕ выполняется).
Это режим "педаль в пол". Быстрая запись, быстрое чтение. Но если ты прочитаешь с Узла 3 данные, которые только что записал на Узел 1 (и они еще не успели синхронизироваться через Gossip), ты получишь старую версию (Stale Read).
Мы осознанно жертвуем консистентностью ради максимальной пропускной способности и доступности.

## Mechanical Sympathy: Реализация кворума в Go

Если ты пишешь собственный координатор для распределенной системы (или просто агрегируешь данные из микросервисов), тебе придется реализовать сбор кворума самостоятельно. 

Это классический паттерн **Fan-Out / Fan-In**, который в Go элегантно реализуется через горутины и каналы с обязательной отменой контекста (Cancellation), чтобы не плодить "сиротские" (orphan) горутины, когда кворум уже собран.

```go
// WriteWithQuorum реализует запись с ожиданием W подтверждений
func WriteWithQuorum(ctx context.Context, data []byte, W int, nodes []Node) error {
	if len(nodes) < W {
		return errors.New("недостаточно узлов для кворума")
	}

	// Создаем контекст для отмены оставшихся запросов, когда кворум собран
	quorumCtx, cancel := context.WithCancel(ctx)
	defer cancel() // Обязательно отменяем при выходе

	ackCh := make(chan struct{}, len(nodes))
	errCh := make(chan error, len(nodes))

	// Fan-Out
	for _, n := range nodes {
		go func(node Node) {
			if err := node.SendWrite(quorumCtx, data); err != nil {
				errCh <- err
				return
			}
			ackCh <- struct{}{}
		}(n)
	}

	// Fan-In
	acks := 0
	errs := 0
	maxErrors := len(nodes) - W

	for {
		select {
		case <-ackCh:
			acks++
			if acks >= W {
				// Кворум собран! defer cancel() убьет оставшиеся медленные горутины
				return nil 
			}
		case <-errCh:
			errs++
			// Fast-fail: если ошибок больше, чем допустимо, мы математически не соберем W
			if errs > maxErrors {
				return errors.New("кворум не может быть достигнут из-за отказов узлов")
			}
		case <-ctx.Done():
			// Родительский таймаут истек (например, от http.Request)
			return fmt.Errorf("таймаут сбора кворума: %w", ctx.Err())
		}
	}
}
```

> [!info] Под капотом: Накладные расходы (Overhead)
> В коде выше мы видим механику задержки (Latency). Запрос завершится со скоростью **W-го самого быстрого узла**. Если W=2, и узлы ответили за 5мс, 10мс и 200мс, твой клиент получит ответ через 10мс. Запрос к самому меделенному узлу (200мс) будет прерван через `quorumCtx`. Это отличный пример того, как кворум "срезает" длинные хвосты задержек (Tail Latency), о которых мы говорили в [[3. Latency и network fallacies]].

## Sloppy Quorum и Hinted Handoff

Что делать, если `N = 3`, `W = 2`, но два целевых узла для ключа `X` лежат (отключилось питание в стойке)? Формально, запись должна упасть, так как мы не можем собрать `W=2`. Система теряет Доступность (Availability).

Dynamo-подобные БД (Cassandra, Riak) решают это механизмом **Нестрогого кворума (Sloppy Quorum)**. 
Координатор понимает, что целевые узлы мертвы. Чтобы не отказывать клиенту (AP-система!), он временно пишет данные на *соседние* узлы в кластере (которые не должны хранить этот ключ), прикрепляя к ним "записку": *"Эй, это для Узла 2, отдай ему, когда он проснется"*. 

Эта "записка" называется **Hinted Handoff**. 

Как только Узел 2 восстанавливается, соседний узел (по Gossip-протоколу) видит это и пересылает ему накопленные данные. 
*Опасность:* В момент Sloppy Quorum чтение `R=2` не найдет эти данные (так как они лежат на "чужом" узле). Уравнение `R + W > N` временно нарушается, и система деградирует до Eventual Consistency, но клиент не получает HTTP 500.

> [!tip] Собеседование
> **Вопрос:** Гарантирует ли строгое выполнение `R + W > N` абсолютную линеаризуемость (Linearizability), как в Raft?
> **Ответ:** Нет! Это одна из главных ловушек. `R + W > N` гарантирует только чтение последней записи *при условии последовательных операций*. Если два клиента параллельно (concurrently) пишут в один и тот же ключ без распределенной блокировки, их записи могут дойти до реплик в разном порядке из-за дрейфа часов ([[5. Time и clock drift]]). Координаторы применят стратегию LWW (Last Write Wins) на основе неточных таймстемпов, и одна из записей будет молча стерта. Dynamo-базы без консенсуса (Raft/Paxos) **не** подходят для финансовых транзакций, даже с максимальным кворумом.

## Итог

1. **Гибкость:** Кворум (R+W>N) позволяет находить компромисс между консистентностью, производительностью записи и производительностью чтения, не меняя базу данных.
2. **Пересечение:** Математика кворума базируется на обязательном пересечении множеств узлов чтения и записи.
3. **Go-паттерны:** Реализация кворумных запросов в бэкенде требует виртуозного владения Fan-Out, `select` и `context.Context` для "срезания хвостов" задержек.
4. **Слабые места:** Кворум отлично справляется с отказами серверов, но не решает проблему конкурентных (одновременных) записей от разных клиентов.

Если кворум не решает проблему параллельных записей, а физические часы серверов врут, как нам понять, какое событие в распределенной системе произошло раньше, а какое позже? Нам нужен механизм, фиксирующий причинно-следственные связи без оглядки на физическое время. Об этом математическом фокусе мы поговорим в следующей статье: [[5. Vector clocks]].
