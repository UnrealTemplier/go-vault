
В прошлой статье [[6. Conflict resolution]] мы обсуждали, что в системах с Eventual Consistency конфликты неизбежны. Мы можем решать их грубо (LWW), вручную на стороне клиента или использовать математически выверенный подход — **CRDT (Conflict-free Replicated Data Types)**.

Если Векторные Часы ([[5. Vector clocks]]) — это способ *обнаружить* конфликт, то CRDT — это способ *дизайнить* данные так, чтобы конфликты разрешались сами собой детерминированным образом.

## Что такое CRDT?

**CRDT** — это структуры данных, которые могут обновляться независимо на разных узлах без центральной координации. При этом гарантируется, что если все узлы получили один и тот же набор обновлений (даже в разном порядке), они придут к **абсолютно идентичному состоянию**.

Математически это достигается за счет того, что операция слияния (Merge) обладает тремя свойствами:
1. **Коммутативность:** Порядок слияния не важен ($A \cup B = B \cup A$).
2. **Ассоциативность:** Группировка не важна ($(A \cup B) \cup C = A \cup (B \cup C)$).
3. **Идемпотентность:** Повторное слияние того же самого не меняет результат ($A \cup A = A$).

## Типы CRDT

Существует два основных способа реализации CRDT:
* **State-based (CvRDT):** Узлы обмениваются своим полным состоянием. Слияние происходит через функцию `join()`. Требует больше трафика, но устойчиво к потере сообщений.
* **Operation-based (CmRDT):** Узлы обмениваются только операциями (например, "добавь 5"). Требует надежного канала доставки (сообщения не должны теряться и должны приходить хотя бы один раз).

Рассмотрим самые популярные структуры, которые ты можешь встретить в распределенных базах данных или совместных редакторах кода.

### 1. Counters (Счетчики)

Обычный `int64` в Go не является CRDT. Если два узла одновременно сделают `inc`, а потом обменяются состояниями, один `inc` потеряется.

* **G-Counter (Grow-only Counter):**
  Хранит вектор счетчиков — по одному на каждый узел. Узел `i` может инкрементировать только свою ячейку `V[i]`. Функция слияния берет `max()` для каждой ячейки.
  
* **PN-Counter (Positive-Negative Counter):**
  Позволяет и увеличивать, и уменьшать. Состоит из двух G-Counter: один для прибавлений (`P`), другой для вычитаний (`N`). Итоговое значение — `sum(P) - sum(N)`.

### 2. Sets (Множества)

* **G-Set (Grow-only Set):**
  Элементы можно только добавлять. Слияние — обычное объединение множеств.
* **2P-Set (Two-Phase Set):**
  Использует два внутренних G-Set: один для добавленных элементов, другой для удаленных ("tombstones"). Если элемент попал в набор удаленных, он никогда не вернется.
* **OR-Set (Observed-Remove Set):**
  Самый продвинутый вариант. Позволяет добавлять и удалять элементы многократно. Каждому добавлению присваивается уникальный тег (UUID/Timestamp). Удаление помечает конкретные теги как "мертвые".

## Mechanical Sympathy: CRDT в Go

Реализация CRDT в Go требует внимания к аллокациям, так как структуры часто сериализуются и пересылаются по сети.

> [!info] Под капотом: Карта или Слайс?
> Для G-Counter идиоматично использовать `map[string]int64`, где ключ — ID узла. Однако в высокопроизводительных системах с фиксированным числом узлов лучше использовать `[]int64`, где индекс соответствует порядковому номеру узла. Это избавляет от хэширования и значительно ускоряет `join()`, так как итерация по слайсу дружелюбна к L1-кэшу процессора.

Пример реализации простейшего **G-Counter** на Go:

```go
type GCounter struct {
	mu     sync.RWMutex
	nodeID string
	state  map[string]int64
}

func NewGCounter(nodeID string) *GCounter {
	return &GCounter{
		nodeID: nodeID,
		state:  make(map[string]int64),
	}
}

// Inc увеличивает значение локального счетчика
func (c *GCounter) Inc() {
	c.mu.Lock()
	defer c.mu.Unlock()
	c.state[c.nodeID]++
}

// Value возвращает текущую сумму всех узлов
func (c *GCounter) Value() int64 {
	c.mu.RLock()
	defer c.mu.RUnlock()
	var total int64
	for _, v := range c.state {
		total += v
	}
	return total
}

// Merge выполняет CvRDT операцию слияния (Join)
func (c *GCounter) Merge(other *GCounter) {
	c.mu.Lock()
	defer c.mu.Unlock()
	other.mu.RLock()
	defer other.mu.RUnlock()

	for node, val := range other.state {
		// Главное правило CvRDT: функция слияния должна быть LUB (Least Upper Bound)
		// В случае счетчика это всегда MAX
		if current, ok := c.state[node]; !ok || val > current {
			c.state[node] = val
		}
	}
}
```

## Где это применяется?

1. **Redis CRDT (Redis Enterprise):** Позволяет делать активную-активную репликацию между географически распределенными дата-центрами. Ты пишешь в локальный Redis, он сам мерджит изменения с другими регионами.
2. **Shared Editors (Google Docs / Figma):** Используют либо CRDT, либо OT (Operational Transformation) для одновременного редактирования текста.
3. **Распределенные чаты:** Позволяют видеть сообщения в правильном порядке и восстанавливать историю после дисконнекта.

---

> [!tip] Собеседование
> **Вопрос:** В чем главный минус CRDT по сравнению с обычной записью в БД?
> **Ответ:** Метаданные (Overhead). Чтобы гарантировать бесконфликтное слияние, CRDT нужно хранить историю или вектор версий. Например, в OR-Set удаление элемента не стирает его физически, а создает "надгробие" (Tombstone). Если элементов миллионы, объем метаданных может в разы превысить объем полезной нагрузки, что потребует сложной логики очистки (Garbage Collection) старых метаданных.

## Итог

1. **CRDT** превращают разрешение конфликтов из вероятностного процесса в математически гарантированный.
2. **Сходимость (Strong Eventual Consistency):** Узлы могут расходиться, но при получении всех сообщений они гарантированно придут к одному состоянию без ручного вмешательства.
3. **Go-инженерия:** При реализации CRDT важно помнить об ассоциативности и коммутативности операций и стараться минимизировать размер пересылаемого стейта.

Как обеспечить целостность, если "все или ничего" нужно применить к распределенной системе? Переходим к сложной теме: [[8. Distributed transactions]].
