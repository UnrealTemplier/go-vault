
В прошлой статье [[5. Vector clocks]] мы научились с помощью математики выявлять конфликты. Мы поняли, что если два вектора версий конкурентны, то ни одна запись не имеет приоритета над другой. Система в этот момент сохраняет оба варианта (так называемые **Siblings** или «сиблинги»).

Но база данных не может вечно находиться в состоянии неопределенности. Рано или поздно эти две (или более) ветки реальности должны сойтись в одну. Процесс превращения набора конфликтующих версий в одно финальное состояние называется **Conflict Resolution (Разрешение конфликтов)**.

В распределенных системах существует два принципиально разных подхода к этой задаче: автоматический и ручной.

## 1. Автоматическое разрешение: Last Write Wins (LWW)

Самый простой, но и самый опасный способ. Система просто смотрит на физический `timestamp` записи. Тот, чье время больше — победил. Остальные данные просто удаляются.

> [!warning] Ловушка / Gotcha: Потеря данных
> Как мы знаем из статьи [[5. Time и clock drift]], часы на серверах никогда не синхронизированы идеально. 
> Если Клиент А записал `x = 10` в 10:00:01 (по часам сервера А), а Клиент Б записал `x = 20` в 10:00:02 (по часам сервера Б, которые спешат), то запись Б победит, даже если фактически она была сделана раньше. LWW — это сознательный отказ от корректности ради простоты.

## 2. Разрешение на стороне клиента (Application-side)

Этот подход используется в по-настоящему надежных AP-системах (например, в Riak или при определенных настройках Cassandra). База данных не пытается быть «умной». При чтении (GET) она возвращает клиенту **массив** всех конфликтующих значений вместе с их векторными часами.

Твой Go-бэкенд получает этот массив и применяет бизнес-логику для слияния.

### Пример: Корзина интернет-магазина
Если в одной версии корзины лежит `[Apple, Banana]`, а в другой `[Apple, Orange]`, логичным разрешением конфликта будет **объединение (Union)** множеств: `[Apple, Banana, Orange]`.

```go
type Cart struct {
    Items []string
    VClock vclock.VectorClock
}

// ResolveConflicts — пример логики слияния сиблингов в Go
func ResolveConflicts(siblings []Cart) Cart {
    if len(siblings) == 0 {
        return Cart{}
    }
    
    finalItems := make(map[string]struct{})
    mergedVClock := vclock.New()

    for _, s := range siblings {
        // Объединяем товары
        for _, item := range s.Items {
            finalItems[item] = struct{}{}
        }
        // Сливаем векторные часы (беря максимумы)
        mergedVClock.Merge(s.VClock)
    }

    result := Cart{VClock: mergedVClock}
    for item := range finalItems {
        result.Items = append(result.Items, item)
    }
    return result
}
```

После слияния Go-клиент обязан записать результат обратно в БД (Write-back), чтобы «схлопнуть» конфликт в базе данных.

## 3. Магия слияния: CRDT

Что если бы мы могли проектировать структуры данных так, чтобы они **всегда** сходились к одному результату автоматически, без участия клиента и без риска потери данных?

Такие структуры существуют и называются **CRDT (Conflict-free Replicated Data Types)**.



### Принцип работы CRDT
CRDT гарантируют, что если операции над данными обладают свойствами **коммутативности** (порядок не важен), **ассоциативности** (группировка не важна) и **идемпотентности** (повтор не важен), то итоговое состояние будет идентичным на всех узлах.

Рассмотрим два популярных типа CRDT:

#### G-Counter (Grow-only Counter)
Обычный инкремент в распределенной системе — это ад (из-за потерь обновлений). G-Counter решает это, храня вектор из счетчиков для каждого узла.
* Узел А инкрементирует свою ячейку: `[A:1, B:0]`.
* Узел Б инкрементирует свою: `[A:0, B:1]`.
* Слияние — это всегда `max()` по каждой ячейке: `[A:1, B:1]`.
* Сумма всех ячеек — это итоговое значение счетчика (2).

#### PN-Counter (Positive-Negative Counter)
Позволяет и увеличивать, и уменьшать значение. Внутри это два G-Counter: один для инкрементов, другой для декрементов. Итоговое значение — разница между ними.

## Mechanical Sympathy: Реализация слияния в Go

В высоконагруженных Go-сервисах разрешение конфликтов может стать узким местом из-за аллокаций. Если у тебя миллионы ключей с сиблингами, постоянное создание `map` для дедупликации (как в примере выше) нагрузит Garbage Collector.

> [!info] Под капотом: Оптимизация слияния
> Для эффективного слияния списков в Go идиоматично использовать заранее аллоцированные слайсы и сортировку вместо мап, если данных много:
> 1. Собрать все элементы из всех сиблингов в один слайс.
> 2. Отсортировать слайс (`sort.Strings`).
> 3. Пройтись одним циклом и удалить дубликаты (используя два указателя).
> Это гораздо бережнее относится к памяти (Zero-allocation слияние возможно при использовании `sync.Pool`).

## Итог

1. **Conflict Resolution** — неизбежный этап в системах с Eventual Consistency.
2. **LWW (Last Write Wins)** — просто, быстро, но ведет к непредсказуемой потере данных.
3. **Application-side Merge** — надежно, позволяет реализовать сложную бизнес-логику (например, слияние корзины покупок), но требует от Go-клиента умения обрабатывать массивы значений.
4. **CRDT** — «золотой стандарт» автоматического слияния, математически гарантирующий сходимость без участия клиента.
