## Анатомия задержек: Поиск узких мест в распределенной системе

В статье [[31. API observability.md]] мы связали логи и метрики единым `Trace ID`. Теперь, когда пользователь жалуется на медленную загрузку страницы, вы можете найти ошибку в логах. 

Но что если ошибки нет? Логи чистые, метрики показывают статус 200 OK, но время ответа (Duration) составляет 5 секунд вместо положенных 100 миллисекунд. Где именно запрос провел эти 5 секунд? 
* Висел в очереди пула соединений PostgreSQL?
* Ждал ответа от стороннего API (например, Stripe)?
* Медленно десериализовал гигантский JSON из Redis?

В монолите можно было использовать встроенный профайлер (pprof). В распределенной микросервисной архитектуре запрос перепрыгивает между серверами, базами данных и брокерами сообщений. Для визуализации этого пути используется **Распределенная трассировка (Distributed Tracing)**.

## Анатомия трассы: Дерево времени

Трассировка оперирует тремя фундаментальными понятиями, которые стандартизированы проектом **OpenTelemetry (OTel)**:

1. **Trace (Трасса):** Весь жизненный цикл запроса от клика в браузере до финального ответа базы данных. Имеет уникальный `Trace ID`.
2. **Span (Спа́н / Отрезок):** Единица работы в рамках трассы. Например, "Обработка HTTP-запроса", "SQL-запрос к таблице users" или "gRPC-вызов". Спан содержит время начала, время завершения, название и набор атрибутов.
3. **Span Context:** Механизм иерархии. Каждый Спан (кроме корневого) имеет `Parent Span ID`, что позволяет сборщику (например, Jaeger) выстроить из них дерево (каскадную диаграмму).

## Идиоматичный Go: Контекст как кровеносная система

В Go реализация распределенной трассировки невозможна без жесткой дисциплины использования `context.Context`. Именно через контекст библиотека OpenTelemetry прокидывает активный Спан вглубь вашего приложения.

> [!info] Под капотом: Mechanical Sympathy
> В отличие от логов, создание Спана — это относительно дорогая операция. 
> Когда вы вызываете `tracer.Start()`, рантайм Go делает системный вызов (или обращается к vDSO) для получения точного времени `time.Now()`. Затем он аллоцирует структуру Спана в куче (Heap). Наконец, при вызове `span.End()`, Спан помещается в буфер, где фоновая горутина сериализует его (обычно в Protobuf) и отправляет по сети (gRPC) в OTel Collector.
> Если вы начнете создавать Спаны на каждую итерацию цикла из 10 000 элементов, ваш сервер потратит больше процессорного времени на трассировку, чем на бизнес-логику! Спанами оборачивают только значимые I/O операции или тяжелые алгоритмические блоки.

### Пример интеграции OpenTelemetry в бизнес-логику:

```go
import (
	"context"
	"go.opentelemetry.io/otel"
	"go.opentelemetry.io/otel/attribute"
	"go.opentelemetry.io/otel/trace"
)

// Инициализация глобального трейсера для пакета
var tracer = otel.Tracer("order-service/billing")

func ProcessPayment(ctx context.Context, orderID string, amount float64) error {
	// 1. Создаем дочерний Спан. Контекст ОБЯЗАН прийти сверху (из HTTP-хендлера)
	ctx, span := tracer.Start(ctx, "ProcessPayment", trace.WithAttributes(
		attribute.String("order.id", orderID),
		attribute.Float64("order.amount", amount),
	))
	// 2. Гарантируем закрытие Спана (зафиксирует время End)
	defer span.End()

	// 3. Вызываем БД, обязательно передавая обогащенный ctx
	err := db.Charge(ctx, orderID, amount)
	if err != nil {
		// 4. Логируем ошибку прямо в Спан
		span.RecordError(err)
		span.SetStatus(codes.Error, "failed to charge database")
		return err
	}

	return nil
}
```

## Границы сети: W3C Trace Context

Как Спан из "Order Service" понимает, что он является дочерним для Спана из "API Gateway", если они запущены на разных серверах?

Проблема передачи метаданных между сервисами (Context Propagation) долгое время была зоопарком форматов (B3 Headers, Jaeger Headers). Сейчас индустрия пришла к единому стандарту — **W3C Trace Context**.

Когда ваш Go-клиент делает исходящий HTTP-запрос (или gRPC вызов), OpenTelemetry автоматически инжектирует в заголовки запроса строку `traceparent`.

Формат заголовка: `traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01`
* `00` — Версия формата.
* `4bf92f35...` — **Trace ID** (уникальный идентификатор всей транзакции).
* `00f067aa...` — **Span ID** (идентификатор Спана-родителя, то есть вызывающей стороны).
* `01` — **Trace Flags** (флаг сэмплирования, говорит о том, нужно ли сохранять эту трассу).

Принимающий Go-сервис (через OTel Middleware) читает этот заголовок, создает корневой контекст для своего хендлера и объявляет себя дочерним по отношению к `00f067aa...`. Так замыкается распределенная цепь.

> [!warning] Ловушка / Gotcha: Потеря контекста в фоновых задачах
> Самая частая ошибка разработчиков в Go: запуск горутины и передача в нее `r.Context()`. 
> ```go
> go func(ctx context.Context) {
>     db.Save(ctx, data) // ОШИБКА!
> }(r.Context())
> ```
> Если HTTP-запрос завершится до того, как отработает горутина, контекст будет отменен (Canceled), и запрос в БД упадет. 
> Если же вы создадите `context.Background()`, вы **разорвете цепь трассировки** (Trace ID потеряется).
> **Решение:** Использовать функцию `trace.ContextWithSpanContext(context.Background(), trace.SpanContextFromContext(r.Context()))`. Это создаст "чистый" контекст без таймаутов, но скопирует в него Trace ID и Span ID из HTTP-запроса.

## Проблема масштаба: Сэмплирование (Sampling)

Выкатывая трассировку в production с нагрузкой 5000 RPS, вы быстро обнаружите, что сеть дата-центра забита отправкой OTel-спанов, а кластер Jaeger падает от нехватки диска.
Вы не можете (и не должны) сохранять 100% трасс. Вам нужно **Сэмплирование**.

> [!tip] Собеседование
> **Вопрос:** В чем разница между Head-based и Tail-based сэмплированием?
> **Ответ:** > * **Head-based sampling:** Решение "сохранять или выбросить" принимается в самом начале (на балансировщике или первом API Gateway), например, случайным образом сохраняя 1% запросов. Флаг "сохранить" записывается в заголовок `traceparent`, и все последующие микросервисы послушно отправляют свои спаны. Плюс: нулевая нагрузка на сеть. Минус: если редкая и страшная ошибка происходит в 0.01% случаев, вы скорее всего её не поймаете, так как этот запрос не попал в счастливый 1%.
> * **Tail-based sampling:** Решение принимается в самом конце. Все микросервисы всегда отправляют 100% спанов в промежуточный буфер (OTel Collector). Коллектор ждет завершения трассы, анализирует её, и если видит, что внутри была ошибка или задержка > 2 секунд — сохраняет трассу в БД. Если это был обычный быстрый запрос 200 OK — отбрасывает. Плюс: вы не пропустите ни одной аномалии. Минус: требует гигантских вычислительных мощностей и памяти для самого коллектора.

## Архитектура: OTel Collector

Ваш Go-бэкенд не должен напрямую отправлять данные в базу данных (например, Jaeger, Zipkin или Datadog). 
Идиоматичная инфраструктура подразумевает деплой агента — **OpenTelemetry Collector** — на каждом физическом узле (или DaemonSet в Kubernetes).

1. Go-приложение по быстрому локальному gRPC-каналу скидывает спаны в Collector (занимает микросекунды).
2. Collector буферизует их, осуществляет Tail-based сэмплирование, фильтрует PII (скрывает номера кредитных карт в тегах) и уже большими батчами асинхронно отправляет в SaaS-провайдера или вашу БД. 

Это паттерн защищает ваше приложение от падения, если база данных трассировок временно недоступна.

## Итог

1. **Распределенная трассировка** визуализирует дерево задержек, показывая узкие места (Slow Queries) в сложной микросервисной среде.
2. В Go **`context.Context`** является абсолютным фундаментом. Без его передачи от функции к функции трассировка развалится на куски.
3. Соблюдайте механическую симпатию: не создавайте спаны внутри горячих циклов, используйте **W3C Trace Context** для границ сети.
4. В Highload-системах обязательно применяйте стратегии **сэмплирования** (Tail-based для критичных систем, Head-based для экономии) и делегируйте отправку данных локальному агенту (OTel Collector).
