## За пределами Запрос-Ответ: Зачем нужны стримы

В статье [[16. gRPC. Основы.md]] мы рассмотрели Unary RPC — классический паттерн взаимодействия, где клиент отправляет один запрос и получает один ответ. Для подавляющего большинства бизнес-задач (получить профиль, создать транзакцию) этого достаточно. 

Но что делать, если нам нужно выгрузить из базы отчет на 10 Гигабайт? В классическом REST мы бы уперлись в лимиты оперативной памяти (OOM), пытаясь сериализовать весь JSON целиком, либо были бы вынуждены реализовывать сложную пагинацию ([[10. Pagination, filtering, sorting.md]]). Что если нам нужно получать поток котировок с биржи в реальном времени, не устанавливая новое TCP-соединение на каждый тик?

Для этих задач gRPC предоставляет механизм **Streaming (Потоковой передачи)**, построенный поверх нативных возможностей HTTP/2.

## Виды Streaming в gRPC

В контрактах Protobuf ([[17. Protocol Buffers.md]]) стриминг декларируется ключевым словом `stream`.

1. **Server Streaming:** Клиент отправляет один запрос, а сервер отвечает потоком сообщений. (Например: скачивание большого файла по кускам, подписка на поток логов).
2. **Client Streaming:** Клиент отправляет поток сообщений, а сервер возвращает один ответ после того, как клиент завершит передачу. (Например: загрузка тяжелого видеофайла на сервер, потоковая отправка телеметрии/метрик).
3. **Bidirectional Streaming (Двунаправленный):** Обе стороны отправляют и читают сообщения в произвольном порядке. Стримы полностью независимы. (Например: чат в реальном времени, голосовой звонок, сложный процесс синхронизации состояния).

```protobuf
service DataService {
  // 1. Server Streaming
  rpc DownloadFile (FileRequest) returns (stream Chunk);
  
  // 2. Client Streaming
  rpc UploadTelemetry (stream Metric) returns (UploadStatus);
  
  // 3. Bidirectional Streaming
  rpc SyncChat (stream ChatMsg) returns (stream ChatMsg);
}
```

## Mechanical Sympathy: HTTP/2 Фреймы и Backpressure

Что происходит под капотом, когда мы открываем gRPC-стрим?
Мы помним, что между клиентом и сервером есть только **одно TCP-соединение**. Стрим — это просто логическая абстракция. Все сообщения (`Chunk`, `Metric`, `ChatMsg`) прозрачно сериализуются в Protobuf, разбиваются на бинарные фреймы `DATA` и отправляются в сокет с определенным `Stream ID`.

Но здесь возникает фундаментальная физическая проблема распределенных систем: **Разница скоростей**.
Что произойдет, если мощный сервер (Go) читает файл с NVMe SSD со скоростью 3 ГБ/с и отправляет его в Server Stream, а на другой стороне сидит мобильный клиент с медленным 3G-интернетом? 

Если бы gRPC просто бесконтрольно писал в сокет, буферы сетевой карты переполнились бы, операционная система начала бы отбрасывать пакеты, а TCP-окно схлопнулось. Память сервера была бы забита неотправленными горутинами.

Здесь вступает в игру магия HTTP/2 — **Flow Control (Управление потоком)**.

> [!info] Под капотом: WINDOW-UPDATE и блокировки в Go
> В HTTP/2 каждый стрим имеет свое окно (квоту) байт, которые он может отправить без подтверждения (по умолчанию 65KB). 
> Когда сервер вызывает метод `stream.Send(msg)`, рантайм `grpc-go` проверяет это окно. Если квота исчерпана (клиент не успевает читать), метод `Send()` **БЛОКИРУЕТ текущую горутину**.
> Она "засыпает" в планировщике Go, освобождая поток ядра (M). 
> Когда мобильный клиент вычитывает часть данных, он отправляет служебный фрейм `WINDOW_UPDATE`. Netpoller на сервере получает его, планировщик "будит" вашу горутину, и `Send()` продолжает работу.
> 
> Этот механизм называется **Backpressure (Обратное давление)**. Благодаря ему ваш Go-сервер никогда не упадет от OOM из-за того, что клиент слишком медленный. Вы просто не сможете читать с диска быстрее, чем клиент может принять по сети.

```mermaid
flowchart LR
    Server["Go Server<br>stream.Send"] -->|"DATA Frame"| Network["TCP Соединение"]
    Network -->|"Буфер заполнен"| Client["Медленный Клиент"]
    
    Client -.->|"WINDOW-UPDATE<br>Увеличить окно"| Network
    Network -.->|"Пробуждение горутины"| Server
```

## Идиоматичный Go: Реализация Streaming

Рассмотрим самую сложную и мощную концепцию — двунаправленный стрим. 

Сгенерированный `protoc` интерфейс для `SyncChat` будет выглядеть так:
```go
type DataService_SyncChatServer interface {
	Send(*ChatMsg) error
	Recv() (*ChatMsg, error)
	grpc.ServerStream
}
```

Для обработки Bidirectional Stream мы ВСЕГДА используем две горутины. Одну для чтения (`Recv`), другую для записи (`Send`).

```go
func (s *server) SyncChat(stream pb.DataService_SyncChatServer) error {
    // Канал для синхронизации завершения
    errCh := make(chan error)

    // 1. Читающая горутина
    go func() {
        for {
            // Ресивер блокируется до получения нового фрейма
            in, err := stream.Recv()
            if err == io.EOF {
                // Клиент закрыл стрим на отправку (полуокрытое соединение)
                close(errCh)
                return
            }
            if err != nil {
                errCh <- err
                return
            }
            // Бизнес-логика обработки входящего сообщения
            log.Printf("Получено: %s", in.Message)
        }
    }()

    // 2. Пишущий цикл (в основной горутине хендлера)
    for {
        select {
        case <-stream.Context().Done():
            // Клиент оборвал TCP-соединение или сработал таймаут
            return stream.Context().Err()
            
        case err := <-errCh:
            // Читающая горутина завершилась (с ошибкой или io.EOF)
            return err
            
        case outMsg := <-s.businessMessageQueue:
            // Нам нужно отправить сообщение клиенту
            if err := stream.Send(outMsg); err != nil {
                return err // Ошибка сети (например, клиент отвалился)
            }
        }
    }
}
```

> [!warning] Ловушка / Gotcha: Потокобезопасность (Thread Safety) `Send` и `Recv`
> Это самая частая ошибка, которая приводит к панике (panic) в production: **Concurrent stream writes**.
> В библиотеке `grpc-go` методы стрима НЕ являются полностью потокобезопасными.
> * Вы **МОЖЕТЕ** вызывать `Send` и `Recv` одновременно из двух РАЗНЫХ горутин (как в примере выше).
> * Вы **НЕ МОЖЕТЕ** вызывать `Send` из нескольких горутин одновременно. И вы не можете вызывать `Recv` из нескольких горутин одновременно.
> 
> Если 10 горутин попытаются сделать `stream.Send()` в один стрим, рантайм запаникует с ошибкой `concurrent write to grpc stream`. Если вам нужно пушить в стрим из разных мест, вы обязаны использовать единый канал (как `s.businessMessageQueue` выше) и единственную горутину-писателя (Writer-Loop), либо защищать `Send` через `sync.Mutex` (что менее идиоматично).

## Client Streaming и io.EOF

В клиентском стриме (когда клиент загружает файл частями), сервер вызывает `Recv()` в цикле. Как понять, что файл закончился?

В мире Go для этого используется стандартная парадигма пакета `io`. Когда клиент на своей стороне вызывает `stream.CloseAndRecv()`, он отправляет HTTP/2 фрейм с флагом `END_STREAM`. 
На сервере метод `stream.Recv()` разблокируется и возвращает ошибку `io.EOF`.

```go
func (s *server) UploadTelemetry(stream pb.DataService_UploadTelemetryServer) error {
    var totalCount int32
    for {
        metric, err := stream.Recv()
        if err == io.EOF {
            // Клиент закончил отправку. Мы возвращаем финальный ответ.
            return stream.SendAndClose(&pb.UploadStatus{
                Success: true,
                Count:   totalCount,
            })
        }
        if err != nil {
            return err // Сетевая ошибка или отмена
        }
        
        totalCount++
        // Сохраняем metric в БД...
    }
}
```

> [!tip] Собеседование
> **Вопрос:** Могут ли gRPC стримы заменить брокеры сообщений (Kafka/RabbitMQ) для асинхронного взаимодействия микросервисов?
> **Ответ:** Нет. gRPC стрим — это синхронный RPC-вызов, привязанный к конкретному TCP-соединению между двумя конкретными подами. Если сервер или клиент упадет, стрим разорвется, и все данные в памяти (буферах) будут потеряны. gRPC не обеспечивает гарантий доставки (At-Least-Once), персистентности на диске и Consumer Groups. gRPC стримы идеально подходят для синхронной передачи больших объемов данных (файлы, видео) или Signaling-серверов, но не заменят настоящие очереди сообщений.

## Настройка стримов: Тюнинг под Highload

Стандартные настройки gRPC-стримов в Go слишком консервативны для высоконагруженных систем. Если вы перегоняете огромные файлы по внутренней гигабитной сети дата-центра, вы можете столкнуться с тем, что скорость упирается в 10-20 МБ/с.

Это происходит из-за маленьких окон HTTP/2 Flow Control (по умолчанию 65 КБ). Сервер отправляет 65 КБ и ждет ACK (подтверждения) от клиента, простаивая.

**Идиоматичный тюнинг gRPC-сервера (в `main.go`):**

```go
server := grpc.NewServer(
    // Увеличиваем размер начального окна для стрима (по умолчанию 64KB -> 1MB)
    grpc.InitialWindowSize(1 << 20),
    // Увеличиваем размер окна для всего TCP-соединения
    grpc.InitialConnWindowSize(1 << 20),
    // Увеличиваем максимальный размер одного входящего сообщения (по умолчанию 4MB)
    grpc.MaxRecvMsgSize(10 << 20), // 10MB
    // Настраиваем Keepalive, чтобы мертвые TCP-соединения вовремя разрывались
    grpc.KeepaliveParams(keepalive.ServerParameters{
        Time:    5 * time.Minute, // Пинговать клиента, если нет активности
        Timeout: 20 * time.Second, // Ждать PONG ответа
    }),
)
```

## Итог

1. **gRPC Streaming** решает проблему передачи огромных файлов и Real-time данных, недоступных для классического REST с пагинацией.
2. Благодаря встроенному в HTTP/2 механизму **Flow Control**, gRPC нативно поддерживает **Backpressure**, защищая ваш Go-бэкенд от переполнения памяти при работе с медленными клиентами.
3. Двунаправленные стримы строятся на базе двух горутин (Read-Loop и Write-Loop).
4. Запомните наизусть: `Send` и `Recv` **нельзя** вызывать конкурентно из множества горутин для одной и той же операции. Для записи нужен либо мьютекс, либо единый канал-писатель.

gRPC обладает феноменальной производительностью, строгими бинарными контрактами и мощными стримами. Означает ли это, что мы должны полностью отказаться от REST в пользу gRPC? В следующей статье мы проведем финальное, честное архитектурное сравнение двух этих подходов: [[19. gRPC vs REST.md]].
