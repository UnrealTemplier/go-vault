## Ваш код мертв, если им не умеют пользоваться

Мы прошли огромный путь: от архитектуры базы данных и высокоскоростного gRPC до сложной аутентификации и Graceful Shutdown. Наш Go-сервис — это шедевр инженерной мысли, способный выдержать 100 000 RPS. 

Но в реальном бизнесе есть суровый закон: **Если вашего API нет в документации (или она непонятна), вашего API не существует.** Для Senior-инженера документация — это не "задача для технического писателя". Это **Developer Experience (DX) — Пользовательский интерфейс (UX) для других программистов**. Если интеграция с вашим сервисом вызывает боль, клиенты уйдут к конкурентам, а соседние команды внутри компании будут саботировать переход на ваш сервис.

В этой статье мы разберем индустриальные стандарты создания документации, которая не устаревает и вызывает восторг у потребителей.

## Три кита Developer Experience (DX)

Типичная ошибка Backend-разработчика — сгенерировать Swagger UI из кода и считать работу выполненной. Swagger — это лишь справочник. Идеальная документация API состоит из трех слоев:

### 1. Reference (Справочник API)
Это "Словарь". Жестко структурированное описание эндпоинтов, методов, заголовков, параметров и JSON-схем.
* **Инструмент:** OpenAPI (Swagger), GraphQL Schema, Protobuf Definitions.
* **Цель:** Дать точный ответ на вопрос "Какой тип у поля `createdAt`?".

### 2. Guides & Concepts (Руководства и Концепции)
Это "Учебник". Справочник не объяснит, *почему* нужно вызывать эндпоинты в определенном порядке.
* **Примеры:** "Как работает наша пагинация", "Жизненный цикл заказа", "Как получить и обновить OAuth2 токен".
* **Инструмент:** Markdown-статьи рядом с Reference.

### 3. Recipes & SDKs (Рецепты и Клиенты)
Разработчики ленивы. Никто не хочет собирать HTTP-запросы вручную через `curl`. 
* Дайте им готовые коллекции Postman / Insomnia.
* Предоставьте сгенерированные SDK для Go, TypeScript, Python (см. [[15. Генерация клиентов и серверов.md]]).
* Напишите готовые сниппеты (Code Snippets) для типичных задач ("Скопируй и вставь, чтобы отправить SMS").

## Главная метрика API: TTFC (Time to First Call)

**Time to First Call (Время до первого успешного вызова)** — это главная продуктовая метрика вашего API. Это время, которое проходит с момента, как сторонний разработчик открыл вашу документацию, до момента, когда он получил `200 OK` в своей консоли.

В идеальном API (например, у Stripe или Twilio) TTFC составляет **менее 5 минут**.
Как это достигается?
1. **Песочница (Sandbox):** Наличие тестового окружения, где можно "подергать" API без страха списать реальные деньги.
2. **Токены в один клик:** Разработчик не должен писать письмо в саппорт, чтобы получить тестовый `API_KEY`. Он должен генерироваться в личном кабинете мгновенно.
3. **Автоподстановка:** Современные порталы (например, ReadMe.com) автоматически подставляют сгенерированный тестовый токен авторизованного разработчика прямо в сниппеты кода `curl` в документации.

## Анатомия идеального OpenAPI Справочника

Если вы используете подход Design-First ([[14. OpenAPI и Swagger.md]]), ваш `openapi.yaml` должен быть произведением искусства.

> [!warning] Ловушка / Gotcha: Справочник без примеров
> Самая частая ошибка — описать типы данных (`string`, `integer`), но не дать примеров. Фронтендер видит: `amount: integer`. Что это? Рубли? Копейки? Доллары? 
> **Решение:** OpenAPI поддерживает поле `example`. Вы **обязаны** заполнять его для каждого поля. Сгенерированные на основе `example` мок-ответы позволят клиентам писать свой код, не дожидаясь готовности вашего бэкенда.

**Что отличает Senior OpenAPI от Junior OpenAPI:**

1. **Описания (Descriptions):** Используйте Markdown в полях `description`. Не пишите "ID пользователя" для поля `userId`. Пишите: *"Уникальный UUIDv4 пользователя. Формируется при регистрации. Необходим для всех мутирующих операций"*.
2. **Документирование Ошибок (RFC 7807):** Описать структуру `200 OK` недостаточно. Вы должны задокументировать `400 Bad Request`. Перечислите все возможные бизнес-ошибки (`error_code: INSUFFICIENT_FUNDS`, `error_code: USER_BLOCKED`), чтобы клиент мог написать правильные `switch/case` в своем коде.
3. **Безопасность (Security Schemes):** Явно укажите, какие эндпоинты требуют `Bearer` токен, а какие публичны. 

## Инфраструктура: Где живут документы

Swagger UI (классический зеленый интерфейс) морально устарел. Он тяжелый и плохо справляется с файлами на тысячи строк. Индустрия перешла на более мощные инструменты.

### Для внешних публичных API (B2B/B2C)
* **Redocly / Stoplight / ReadMe:** Платформы, которые берут ваш `openapi.yaml` и генерируют из него невероятно красивые, быстрые и удобные порталы с трехколоночным дизайном (Навигация -> Описание -> Код/Пример).
* Они поддерживают версионирование, темные темы и интеграцию с Markdown-руководствами.

### Для внутренних микросервисов (Enterprise)
В компании из 50 микросервисов разработчики тратят часы, чтобы просто найти, где лежит документация нужного сервиса.
* **Spotify Backstage:** Индустриальный стандарт для Developer Portals внутри компаний. Это централизованный каталог. Вы добавляете в репозиторий вашего Go-сервиса файл `catalog-info.yaml`, и Backstage сам находит сервис, показывает, кто им владеет (On-call команда), где его логи, метрики, и рендерит его API документацию.

## Ловушка: Как не дать документации "протухнуть"

Документация, которая не совпадает с кодом (рассинхрон) — это хуже, чем отсутствие документации. Она заставляет клиентов писать неработающий код, тратить часы на дебаг и терять доверие к вашей команде.

Как гарантировать синхронизацию?

1. **Linter в CI/CD (Spectral):** Как мы упоминали ранее, `spectral` проверит, что вы не забыли добавить `description` и `example` к новому полю перед тем, как слить Pull Request.
2. **Contract Testing (Pact):** Описан в [[34. Contract testing.md]]. Если код возвращает `int`, а в документации (и контрактах клиентов) указан `string` — пайплайн упадет.
3. **Отказ от ручного роутинга:** Если вы используете Code-First, генерируйте Swagger из кода (`swaggo`). Если Design-First — генерируйте интерфейсы Go из OpenAPI (`oapi-codegen`). **Никогда не пишите роутеры и DTO вручную параллельно с YAML-схемой.**

> [!tip] Собеседование
> **Вопрос:** Мы выпускаем новую мажорную версию API (v2). Как правильно отобразить это в документации для текущих клиентов v1?
> **Ответ:** Документация должна поддерживать историю версий (Version Dropdown). Мы не удаляем документацию v1, мы помечаем её как `Deprecated` (устаревшую) и вешаем плашку с датой полного отключения (Sunset date). В документации v2 мы обязаны выпустить **Migration Guide (Руководство по миграции)** — статью с таблицей "Как было в v1 -> Как стало в v2" и примерами рефакторинга для клиентов.

## Документирование асинхронных API

Если с REST всё понятно (OpenAPI), то как документировать WebSockets ([[22. WebSocket.md]]), Server-Sent Events ([[23. Server Sent Events.md]]) или Kafka-сообщения?

Для управляемых событиями архитектур (Event-Driven Architecture) индустрия создала **AsyncAPI Specification**.
Она синтаксически очень похожа на OpenAPI, но вместо путей (Paths) там каналы (Channels) и топики (Topics), а вместо методов `GET/POST` — операции `Publish` и `Subscribe`. Обязательно используйте AsyncAPI для описания ваших очередей сообщений и WS-стримов.

## Итог

1. **Документация — это ваш продукт.** Она определяет Developer Experience (DX) и напрямую влияет на успех вашего API.
2. Измеряйте и оптимизируйте метрику **Time to First Call (TTFC)** через песочницы, понятные туториалы и готовые сниппеты кода.
3. **OpenAPI Справочник** должен быть исчерпывающим: обязательные примеры (`example`), подробные текстовые описания и исчерпывающий список возможных ошибок (`4xx`, `5xx`).
4. Используйте современные порталы (**Redoc, Backstage**) вместо голого Swagger UI.
5. Автоматизируйте защиту от "протухания" через генерацию кода, CI-линтеры и Contract Testing.
