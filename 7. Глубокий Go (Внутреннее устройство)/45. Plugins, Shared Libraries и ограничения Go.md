
В прошлой статье ([[44. Linker и Build Modes.md]]) мы узнали, что Go умеет собирать динамические библиотеки (`-buildmode=c-shared`), чтобы отдавать свой код в Python, C++ или Ruby. Это работает отлично: рантайм Go аккуратно прячется внутри `.so` файла и обслуживает ваши функции.

Но что, если мы хотим обратного? Что, если мы пишем расширяемую систему (например, API-шлюз, игровой сервер или систему мониторинга) на Go, и хотим подгружать кастомную логику от пользователей или других команд "на лету", не перекомпилируя основное ядро?

В C++ или C# это делается элементарно через динамическую загрузку (`dlopen` / `LoadLibrary`). В Go для этого был создан пакет `plugin`. На бумаге это звучит как идеальное решение. На практике — это **самая проблемная и наименее используемая фича во всем языке**.

Давайте разберем, как работают плагины в Go, почему они причиняют столько боли, и как Senior-инженеры решают задачу расширяемости на самом деле.

## 1. Иллюзия простоты: пакет plugin

В версии Go 1.8 разработчики добавили режим сборки `-buildmode=plugin`.

**Как это выглядит в идеальном мире:**
Вы пишете отдельный пакет `main`, но вместо функции `main()` объявляете экспортируемые функции или переменные:

```go
// plugin.go
package main

import "fmt"

// Экспортируемая функция
func Hello(name string) {
    fmt.Printf("Привет, %s! Я загружен из плагина.\n", name)
}
````

Собираете его как плагин:

Bash

```bash
go build -buildmode=plugin -o myplugin.so plugin.go
```

А в основном приложении динамически загружаете этот файл:

```go
// main.go
package main

import "plugin"

func main() {
    // 1. Открываем .so файл (под капотом вызывается dlopen)
    p, err := plugin.Open("myplugin.so")
    
    // 2. Ищем символ (функцию или переменную) по имени
    sym, err := p.Lookup("Hello")
    
    // 3. Делаем Type Assertion (кастуем к нужной сигнатуре)
    helloFunc := sym.(func(string))
    
    // 4. Вызываем!
    helloFunc("Gopher")
}
```

Код выглядит элегантно. Типы проверяются в рантайме (через `eface/iface`, которые мы разбирали в [[35. iface и eface. Как устроены интерфейсы.md]]).

Но как только вы попытаетесь использовать это в Production, вы столкнетесь со стеной ограничений.

## 2. Жестокая реальность: Почему плагины не работают

Механизм плагинов в Go имеет настолько строгие ограничения, что их использование в распределенных командах становится практически невозможным.

### Ограничение 1: Отсутствие кроссплатформенности

Плагины работают **только на Linux, FreeBSD и macOS**. Официальной поддержки Windows нет и, скорее всего, никогда не будет из-за архитектурных различий в обработке DLL и особенностей рантайма Go.

### Ограничение 2: Ад версий зависимостей

Основное приложение и плагин **обязаны** иметь абсолютно идентичные версии всех общих зависимостей.

Если ваше ядро использует `github.com/google/uuid` версии `v1.2.0`, а команда, пишущая плагин, импортировала `v1.3.0` — функция `plugin.Open()` выдаст фатальную ошибку: `plugin was built with a different version of package`.

Вам придется синхронизировать `go.mod` во всех репозиториях плагинов.

### Ограничение 3: Идентичность компилятора и флагов

Вы не можете скомпилировать ядро на Go 1.21.0, а плагин на Go 1.21.1. Версии компилятора должны совпадать до патча.

Более того, любые флаги сборки (например, `-tags`, параметры CGO или даже `GOPATH` в старых версиях) должны быть строго одинаковыми.

### Ограничение 4: Отсутствие выгрузки (Memory Leak)

В пакете `plugin` есть функция `Open()`, но **нет функции `Close()` или `Unload()`**.

Если вы загрузили `myplugin.so`, он останется в оперативной памяти и адресном пространстве вашего процесса до тех пор, пока приложение не будет перезапущено. Вы не можете реализовать механизм "Горячей перезагрузки" (Hot Reload), просто подменяя `.so` файлы — память будет утекать с каждым новым загруженным плагином.

## 3. Под капотом: Почему рантайм такой параноик?

Почему разработчики языка не могут просто ослабить эти ограничения? Ответ кроется в архитектуре рантайма, Сборщика Мусора (GC) и системы типов.

- **Единый GC:** Когда загружается плагин, рантайм должен "слить" метаданные плагина с метаданными основного приложения. GC должен начать сканировать глобальные переменные плагина в поисках указателей. Если плагин собран с другими флагами или другой версией компилятора, `layout` памяти (смещения структур) может отличаться. GC пойдет по неверным адресам и разрушит память.
- **Слияние itab:** Вспомните структуру `itab` (мост между типом и интерфейсом). Если в ядре есть интерфейс `Logger`, а в плагине тип `MyLogger`, рантайму нужно динамически сгенерировать `itab` для них. Для этого хэши типов должны совпадать байт в байт.
- **Один рантайм на процесс:** Go не умеет запускать два изолированных рантайма в одном адресном пространстве. Плагин грузится в тот же рантайм. Если плагин паникует (`panic`) — падает всё приложение.

Линкер генерирует уникальный криптографический хэш (Build ID) для каждого пакета. При вызове `plugin.Open()` рантайм жестко сверяет эти хэши. Малейшее несовпадение — отказ.

## 4. Mechanical Sympathy: Как делать плагины правильно?

Senior-инженеры давно смирились с тем, что нативные плагины Go (`.so`) мертвы. Для построения расширяемых систем сегодня используются три других архитектурных паттерна.

### Паттерн 1: Out-of-Process Plugins (gRPC / RPC)

Это золотой стандарт, который использует компания HashiCorp (в Terraform, Vault и Consul).

Они написали библиотеку `hashicorp/go-plugin`.

**Как это работает:**

1. Плагин — это обычный, независимый исполняемый `.exe` (или бинарник Linux) файл.
2. Основное приложение запускает плагин как дочерний процесс (`os.Exec`).
3. Они общаются друг с другом через `gRPC` или `net/rpc` по локальным сокетам (или `stdin/stdout`).

**Плюсы:** 100% изоляция. Если плагин упал (паника, segfault), ядро продолжает работать. Разные версии Go? Разные языки (плагин на Python)? Без проблем! Можно безопасно убивать процессы (Hot Reload).

**Минусы:** Накладные расходы на сериализацию (Protobuf/JSON) и передачу данных между процессами (Context Switch).

### Паттерн 2: WebAssembly (Wasm)

Современный тренд. Вы компилируете пользовательский код в формат `.wasm`. В основное Go-приложение встраивается легковесная Wasm-виртуальная машина (например, библиотека `tetratelabs/wazero`).

**Плюсы:** Wasm-модули выполняются _внутри_ процесса (быстрее, чем gRPC), но в абсолютно изолированной "песочнице" (Sandbox). Плагин не может сломать рантайм Go, не имеет доступа к сети и диску (если вы не разрешите). `wazero` не требует CGO.

**Минусы:** Wasm всё еще накладывает ограничения на работу с горутинами и системными вызовами внутри плагина.

### Паттерн 3: Встраиваемые скриптовые языки

Если плагины нужны для описания простой бизнес-логики (например, правил маршрутизации), в ядро Go встраивают интерпретатор.

Популярные варианты:

- **Lua:** `yuin/gopher-lua`
- **JavaScript:** `dop251/goja`
- **Expr:** Библиотека `antonmedv/expr` для безопасного вычисления выражений.

## Итог

1. **`buildmode=plugin`** — это попытка принести в Go нативные динамические библиотеки (по аналогии с `dlopen`).
2. Ограничения делают этот механизм непригодным для реальных задач: нет Windows, жесткая привязка к версиям зависимостей и компилятора, невозможность выгрузить плагин из памяти.
3. Рантайм Go вынужден быть параноиком при загрузке плагинов, чтобы защитить Сборщик мусора и систему типов от повреждения памяти.
4. В реальном мире (Production) для плагинов используют изоляцию: **Out-of-Process (gRPC)** или **WebAssembly (Wasm)**. Это дает безопасность, возможность перезагрузки и независимость стеков.

Мы детально разобрались с процессом компиляции, сборки и загрузки кода. До сих пор мы рассматривали код так, будто он выполняется предсказуемо и детерминированно. Но в конкурентной среде, где миллионы горутин обращаются к общей памяти без должной синхронизации (мьютексов или атомиков), возникает **Состояние Гонки (Data Race)**. Это самые неуловимые баги, которые могут проявляться раз в неделю на Production.

К счастью, в инструментарий Go встроен мощнейший механизм для их поиска.
