
В прошлых статьях мы изучили рефлексию ([[37. Reflect под капотом.md]]), которая позволяет динамически работать с типами, сохраняя при этом безопасность рантайма. Пакет `unsafe`, как следует из названия, — это территория, где правила языка Go перестают действовать.

Использование `unsafe` — это сознательный отказ от безопасности ради экстремальной производительности или интеграции с низкоуровневым кодом. Senior-разработчик должен понимать, что `unsafe` — это не просто «быстрый способ», это инструмент, который может превратить ваш код в мину замедленного действия при каждом обновлении версии Go или смене архитектуры процессора.

## 1. Три кита unsafe

Пакет `unsafe` строится вокруг трех фундаментальных сущностей:

1.  **`unsafe.Pointer`**: Специальный тип указателя, который может хранить адрес любого объекта. Компилятор позволяет конвертировать любой обычный указатель (`*T`) в `unsafe.Pointer` и обратно.
2.  **`uintptr`**: Это целое число, размер которого достаточен для хранения адреса памяти. В отличие от указателей, с `uintptr` можно выполнять арифметические операции (сложение, вычитание).
3.  **`unsafe.Sizeof`, `Offsetof`, `Alignof`**: Функции, которые на этапе компиляции возвращают метаданные о расположении структур в памяти.

## 2. Механика обмана: Арифметика указателей

В обычном Go вы не можете прибавить 8 к указателю, чтобы перейти к следующему полю структуры. В `unsafe` это стандартная практика.

Представим структуру:
```go
type User struct {
    ID     uint32
    Active bool
    Name   string
}
````

Чтобы прочитать поле `Name`, не используя имя поля, мы можем сделать так:

1. Берем адрес структуры `*User`.
2. Конвертируем в `unsafe.Pointer`.
3. Конвертируем в `uintptr` для расчетов.
4. Прибавляем смещение поля `Name` (используя `unsafe.Offsetof`).
5. Конвертируем обратно в `unsafe.Pointer`, а затем в `*string`.

## 3. Смертельная ловушка: Сборщик мусора и uintptr

Это самая опасная ошибка при работе с `unsafe`.

**Запомните: Сборщик мусора (GC) понимает `unsafe.Pointer`, но игнорирует `uintptr`.**

```go
// СМЕРТЕЛЬНО ОПАСНО
ptr := uintptr(unsafe.Pointer(new(User)))
// ... здесь может запуститься GC ...
u := (*User)(unsafe.Pointer(ptr))
```

Если между первой и второй строкой запустится GC, он увидит, что на объект `User` не осталось «живых» указателей (ведь `uintptr` — это просто число для него). GC удалит объект или переместит его (при росте стека), а ваш `ptr` станет указывать на мусор. Результат — непредсказуемый краш.

**Правило:** Переход `Pointer -> uintptr -> Pointer` должен быть **атомарным выражением** (в одну строку), чтобы компилятор видел связь и не позволил GC удалить объект в середине расчета.

## 4. Zero-copy конвертации: []byte в string

Мы упоминали это в [[34. Внутреннее устройство string.md]]. Стандартная конвертация `string([]byte)` копирует память. С `unsafe` мы можем просто «подменить» заголовок.

С версии Go 1.20 для этого есть официальные функции:

```go
// Без копирования!
s := unsafe.String(unsafe.SliceData(b), len(b))
```

Рантайм берет указатель на данные слайса и создает на их основе заголовок строки. Это экономит гигабайты аллокаций в высоконагруженных парсерах (например, JSON или Protobuf).

## 5. Нарушение инкапсуляции

`unsafe` позволяет читать и писать **приватные поля** чужих структур. Если вы знаете структуру `hmap` из стандартной библиотеки (которую мы разбирали в [[31. Внутреннее устройство map. hmap, bmap, эвакуация.md]]), вы можете через `unsafe` залезть в её внутренности и узнать количество переполненных бакетов, даже если эти поля не экспортированы.

Многие популярные библиотеки (например, `fasthttp` или высокопроизводительные драйверы БД) используют это, чтобы выжать максимум из стандартных типов Go.

## Mechanical Sympathy: Цена риска

1. **Несовместимость**: Ваш код может сломаться в Go 1.27, если разработчики изменят внутренний `layout` структур в рантайме.
2. **Портабельность**: Смещения полей зависят от архитектуры (x86 vs ARM) и выравнивания (`Alignof`).
3. **Оптимизации компилятора**: Использование `unsafe` может помешать компилятору выполнять некоторые оптимизации, так как он больше не может гарантировать безопасность графа объектов.

> [!tip] Собеседование. Зачем нужен unsafe?
> 
> **Вопрос:** Если это так опасно, почему бы не запретить это вовсе?
> 
> **Ответ:** Без `unsafe` невозможно реализовать сам рантайм Go (аллокатор памяти, планировщик, рефлексию). Также он необходим для эффективного взаимодействия с ОС и C-библиотеками (CGO), где данные часто приходят в виде «сырых» указателей.

## Итог

1. **`unsafe.Pointer`** — это мост между строгой системой типов Go и сырой памятью.
2. **`uintptr`** позволяет делать арифметику, но он «невидим» для Сборщика Мусора.
3. Основное применение: **Zero-copy** оптимизации и доступ к низкоуровневым структурам рантайма/ОС.
4. Использование `unsafe` требует понимания правил выравнивания (`Alignment`) и поведения GC.
