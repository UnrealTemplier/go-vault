
В арсенале бэкенд-разработчика мьютексы и `WaitGroup` решают подавляющее большинство задач синхронизации. Однако, когда вы начинаете писать инфраструктурный код, высоконагруженные кэши, парсеры или сетевые фреймворки (подобные `fasthttp` или `gin`), базовых примитивов становится недостаточно. Вам нужны инструменты точечной оптимизации аллокаций памяти и аппаратные инструкции без обращения к ядру ОС.

В этой статье мы разберем продвинутую часть пакета `sync`, которая часто становится решающей на собеседованиях уровня Middle+/Senior.

---

## 1. sync.Once: Ленивая инициализация и Синглтоны

Классическая проблема: у вас есть ресурсоемкая операция (например, чтение конфига с диска или установка соединения с БД), которую нужно выполнить **строго один раз**, даже если тысяча горутин одновременно попытается её вызвать.

Использование обычного `Mutex` здесь избыточно, так как после первой инициализации все последующие чтения будут неоправданно блокироваться (убивая производительность). `sync.Once` решает это элегантно.

```go
package main

import (
	"fmt"
	"sync"
)

var (
	once sync.Once
	cfg  *Config
)

type Config struct { /* ... */ }

func loadConfig() {
	fmt.Println("Инициализация конфига... (вызовется лишь однажды)")
	cfg = &Config{}
}

func GetConfig() *Config {
	// Допустимо вызывать из тысяч горутин одновременно
	once.Do(loadConfig)
	return cfg
}
```

### Под капотом: Fast Path и Slow Path

Как `sync.Once` умудряется быть потокобезопасным и при этом не тормозить последующие вызовы?
Внутри структуры есть флаг `done` (атомарный счетчик) и `m` (мьютекс).

1. **Fast Path (Быстрый путь):** При вызове `Do()`, рантайм атомарно читает флаг `done`. Если он равен `1`, функция мгновенно возвращает управление (операция занимает доли наносекунд).
2. **Slow Path (Медленный путь):** Если флаг равен `0`, горутина захватывает мьютекс. *Внутри* критической секции она **еще раз** проверяет флаг (Double-Checked Locking). Если он все еще `0`, выполняется переданная функция, и флаг атомарно выставляется в `1`.

> [!warning] Ловушка / Gotcha: Deadlock внутри sync.Once
> Если функция, переданная в `once.Do`, внутри себя (или по цепочке вызовов) попытается снова вызвать `once.Do` на том же экземпляре, вы получите мгновенный **Deadlock** (взаимную блокировку). Первая горутина захватила мьютекс и ждет завершения функции, а функция пытается снова захватить этот же мьютекс.

## 2. sync.Cond: Сигналы и широковещание

`sync.Cond` (Condition Variable) — это примитив синхронизации, позволяющий горутине заснуть (отвязаться от процессора) до тех пор, пока не наступит определенное условие, о котором ей сообщит другая горутина.

Во многих случаях `Cond` можно заменить каналами (и в Idiomatic Go так и принято). Но у `Cond` есть одна уникальная суперспособность — метод `Broadcast()`, который может **одновременно разбудить сотни спящих горутин**, ничего им не передавая. С каналами для этого пришлось бы закрыть канал (но тогда его нельзя использовать повторно) или отправлять `N` сообщений.

### Как это работает

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	mu := &sync.Mutex{}
	cond := sync.NewCond(mu)
	ready := false

	for i := 0; i < 3; i++ {
		go func(id int) {
			mu.Lock()
			// ВАЖНО: всегда проверяем условие в цикле for!
			for !ready {
				// Wait атомарно отпускает мьютекс и усыпляет горутину.
				// При пробуждении - снова захватывает мьютекс.
				cond.Wait()
			}
			fmt.Printf("Горутина %d начала работу\n", id)
			mu.Unlock()
		}(i)
	}

	time.Sleep(1 * time.Second)
	
	mu.Lock()
	ready = true
	mu.Unlock()
	
	fmt.Println("Рассылаем сигнал всем!")
	cond.Broadcast() // Будит все горутины, зависшие на Wait()

	time.Sleep(1 * time.Second) // Даем горутинам время на вывод
}
```

> [!tip] Собеседование
> **Вопрос:** Почему `cond.Wait()` **всегда** нужно оборачивать в цикл `for condition == false`, а не писать `if condition == false`?
> **Ответ:** Из-за феномена, известного на уровне ОС как **Spurious Wakeups (Ложные пробуждения)**. Планировщик ОС или рантайм Go может разбудить горутину даже без явного вызова `Signal()` или `Broadcast()`. Кроме того, между моментом пробуждения горутины и моментом, когда она реально захватит мьютекс, другая горутина может успеть изменить состояние обратно на `false`. Цикл `for` гарантирует, что горутина не продолжит работу, пока условие не будет истинным железобетонно.

## 3. sync.Pool: Уничтожение работы для Garbage Collector

В высоконагруженном бэкенде мы часто аллоцируем краткосрочные объекты: буферы для JSON, структуры контекстов запросов, слайсы байт. Постоянное выделение памяти в куче (Heap Allocation) приводит к двум проблемам:
1. Затраты CPU на саму аллокацию.
2. Взрывной рост пауз сборщика мусора (GC), которому нужно сканировать и удалять этот мусор.

`sync.Pool` — это потокобезопасный пул объектов. Вместо создания нового объекта мы просим его у пула (`Get`). Поработали — вернули обратно (`Put`). Если пул пуст, он сам создаст новый объект через функцию `New`.

```go
var bufferPool = sync.Pool{
	New: func() any {
		return make([]byte, 0, 4096) // Выделяем 4KB памяти
	},
}

func ProcessRequest() {
	// Берем буфер из пула (без аллокации!)
	buf := bufferPool.Get().([]byte)
	
	// Важно! Сбрасываем длину перед использованием, но сохраняем capacity
	buf = buf[:0] 
	
	// ...работаем с buf...
	
	// Возвращаем в пул
	bufferPool.Put(buf)
}
```

### Под капотом и Mechanical Sympathy

Архитектура `sync.Pool` тесно связана с планировщиком Go (моделью G-M-P). 
Если бы `Pool` был единым списком объектов с глобальным мьютексом, на 64-ядерном сервере горутины бы убили производительность, стоя в очереди за буфером.

Вместо этого рантайм создает **свой собственный локальный пул для каждого логического процессора (P)** (`poolLocal`).
Когда горутина вызывает `Get()`, она привязывается к текущему `P` (прерывания отключаются) и берет объект из **приватной** переменной своего `P` вообще без блокировок! Если там пусто, она берет из **общего** списка этого `P` с минимальным спин-локом. И только если и там пусто, она ворует объекты у других `P` (Work Stealing).

> [!warning] Ловушка / Gotcha: Жизненный цикл sync.Pool
> 1. **Не используйте Pool для stateful-объектов.** Например, для сетевых соединений (TCP/DB). Объекты в пуле могут быть уничтожены сборщиком мусора в любой момент! (До Go 1.13 пул полностью очищался при каждом цикле GC. После 1.13 добавили Victim Cache, поэтому объекты живут 2 цикла GC).
> 2. **Утечка конфиденциальных данных.** Всегда сбрасывайте состояние объекта перед возвратом в пул (`buf = buf[:0]`). Если вы вернете структуру со старыми данными (например, паролем пользователя A), следующий запрос (от пользователя B) может получить эту структуру и прочитать чужой пароль.

## 4. sync/atomic: Железобетонная синхронизация

Иногда нам нужно просто увеличить счетчик просмотров или атомарно обновить указатель на кэш. Захватывать ради этого `sync.Mutex` — значит заставлять горутину проходить сложный путь (Fast Path + потенциальная парковка в ядре).

Пакет `sync/atomic` предоставляет доступ к низкоуровневым аппаратным инструкциям процессора.

На архитектуре x86-64 операция вроде `atomic.AddUint64` компилируется в одну-единственную ассемблерную инструкцию с префиксом `LOCK` (например, `LOCK XADD`). 

**Как это работает на уровне железа:**
Префикс `LOCK` заставляет процессор захватить кэш-линию L1 эксклюзивно для себя (используя протокол когерентности кэшей MESI) или заблокировать системную шину памяти на долю наносекунды. Другие ядра процессора аппаратно не смогут прочитать или изменить эту ячейку памяти, пока операция не завершится. Никаких планировщиков, никаких очередей — чистая физика.

### Новые типы в Go 1.19+

Исторически атомарные операции в Go делались через функции (например, `atomic.AddInt64(&counter, 1)`). Это было чревато ошибками: программист мог случайно прочитать `counter` напрямую (вызвав Data Race).

В Go 1.19 завезли строго типизированные атомики:

```go
package main

import (
	"fmt"
	"sync/atomic"
)

func main() {
	var views atomic.Int64 // Безопасная структура
	
	views.Add(1) // Увеличиваем
	views.Store(100) // Явно устанавливаем
	
	// Читаем безопасно
	fmt.Printf("Просмотры: %d\n", views.Load()) 
}
```

### Compare-And-Swap (CAS)

Основа Lock-Free программирования — операция `CompareAndSwap` (CAS).
Она делает следующее: "Запиши в переменную значение `new`, **только если** прямо сейчас там лежит значение `old`". 

Это позволяет строить сложные неблокирующие алгоритмы:
```go
var state atomic.Uint32

// Пытаемся перевести статус из 0 в 1.
// Вернет true, если мы были первыми, и false, если кто-то другой уже изменил состояние.
if state.CompareAndSwap(0, 1) {
	fmt.Println("Я захватил ресурс Lock-Free способом!")
}
```

> [!info] Под капотом
> Помните Fast Path в `sync.Mutex` из прошлой статьи? Внутри исходников Go (`src/sync/mutex.go`) быстрое получение мьютекса реализовано ровно через эту же инструкцию:
> `atomic.CompareAndSwapInt32(&m.state, 0, mutexLocked)`

## Итог

1. **`sync.Once`**: Используем для синглтонов. Супер-быстро благодаря атомарной проверке флага перед мьютексом. Остерегайтесь рекурсивных вызовов (Deadlock).
2. **`sync.Cond`**: Мощный механизм `Broadcast()` для пробуждения сразу всех ждущих горутин. Метод `Wait` всегда оборачиваем в цикл `for` (защита от Spurious Wakeups).
3. **`sync.Pool`**: Маст-хэв для высоконагруженных серверов (API, парсеры). Убирает мусор в куче, спасает от долгих пауз GC. Не забывайте очищать данные (сбрасывать длину слайсов) перед возвратом в пул.
4. **`sync/atomic`**: Работает на уровне инструкций CPU. Быстрее мьютексов, но подходит только для простых примитивов (числа, указатели, флаги).
