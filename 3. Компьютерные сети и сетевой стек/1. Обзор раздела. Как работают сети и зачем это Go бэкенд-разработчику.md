## Зачем бэкенд-разработчику знать сети?

Бэкенд-разработка в современном понимании — это проектирование систем, которые общаются. Даже если вы пишете пакетный обработчик или CLI-инструмент, вы работаете с файлами, сокетами, контекстом и каналами, которые в конечном итоге сводятся к передаче данных между процессами или узлами. 

Для Go-разработчика знание сетевого стека — это не просто «умение настроить nginx». Это фундамент для понимания:
*   Как `net/http` обрабатывает 100k+ одновременных подключений без OOM.
*   Почему блокирующий вызов в горутине может «повесить» тред ОС.
*   Как работают задержки (latency) и потеря пакетов на уровне приложения.
*   Как проектировать отказоустойчивые микросервисы и балансировать нагрузку.

Go не изобрёл сеть, но его модель конкурентности была спроектирована вокруг эффективной работы с сетевым I/O. Понимание этой связи отделяет инженера, который «пишет код», от архитектора, который «проектирует системы».

## Архитектура сетевого взаимодействия: от приложения до провода

Любое сетевое взаимодействие проходит через строгую иерархию абстракций. В Go эта иерархия частично скрыта стандартной библиотекой, но на уровне Senior/Lead вы должны чётко понимать границы каждой прослойки:

1.  **Прикладной уровень (L7):** HTTP, gRPC, WebSocket. Здесь формируются бизнес-логики, сериализация и маршрутизация.
2.  **Транспортный уровень (L4):** TCP или UDP. Гарантии доставки, упорядочивание, контроль потока.
3.  **Сетевой уровень (L3):** IP (IPv4/IPv6). Маршрутизация пакетов между подсетями.
4.  **Канальный уровень (L2):** Ethernet, MAC-адресация. Передача фреймов между устройствами в одном сегменте.
5.  **Физический уровень (L1):** Медиа, сигналы, оптоволокно.

> [!info] Под капотом
> Go-рантайм не оперирует «пакетами» напрямую. Вы работаете с `net.Conn`, который инкапсулирует файловый дескриптор (fd). Задача рантайма — эффективно мапить этот fd на горутины, избегая блокировок на уровне ОС.

## Go и сетевой стек: Netpoller и event-loop

В отличие от PHP или старых версий Java, где на каждое подключение создавался отдельный системный тред (thread-per-connection), Go использует модель **event-loop с неблокирующим I/O**. 

Когда вы вызываете `listener.Accept()` или `conn.Read()`, горутина не блокирует системный тред навсегда. Вместо этого:
1.  Горутина вызывает `syscall.Read` на fd.
2.  Если данные ещё не пришли, ядро Linux возвращает `EAGAIN`/`EWOULDBLOCK`.
3.  Go-рантайм перехватывает это состояние через `netpoller` и **паркует** (park) горутину, возвращая системный тред в пул планировщика.
4.  Тред уходит выполнять другие горутины.
5.  Когда данные появляются, `netpoller` пробуждает горутину и ставит её в очередь на выполнение.

```go
// Идиоматичный сервер. Одна горутина на подключение не блокирует тред ОС.
func main() {
    ln, err := net.Listen("tcp", ":8080")
    if err != nil {
        log.Fatal(err)
    }
    defer ln.Close()

    for {
        conn, err := ln.Accept()
        if err != nil {
            log.Println("accept error:", err)
            continue
        }
        // Горутина будет парковаться на conn.Read() 
        // и не занимать тред ОС, пока ждёт данные.
        go handleConn(conn)
    }
}
```

## Mechanical Sympathy: Почему event-loop в Go быстрее тредов?

С точки зрения архитектуры процессора и ОС, контекстное переключение (context switch) между системными тредами стоит дорого:
*   **TLB flush:** Кэш трансляции адресов памяти инвалидируется.
*   **Кэш-линии L1/L2:** Данные текущего треда вытесняются, происходит cache miss.
*   **Смена стека:** Необходимо сохранить/восстановить регистры CPU, RSP, RFLAGS.
*   **Планировщик ОС:** Переход в Ring 0, проверка приоритетов, балансировка нагрузки между ядрами.

Переключение между горутинами в Go происходит в User Space. Рантайм просто меняет указатель стека (2 КБ по умолчанию) и регистры `PC` и `SP`. Это занимает десятки тактов CPU против тысяч для системного context switch. 

`netpoller` минимизирует количество системных вызовов `epoll_wait`/`kqueue`, группируя их и используя `epoll_pwait` для безопасной работы с сигналами. Это прямое влияние на throughput и latency вашего сервиса.

## Под капотом: Как устроен `netpoller`

В исходниках Go `netpoller` реализован в `src/runtime/netpoll.go`. Он не является отдельным потоком, а интегрирован в планировщик:

*   **Linux:** `epoll` (epoll_ctl, epoll_wait). Использует `epoll_pwait` для обработки сигналов без race-conditions.
*   **BSD/macOS:** `kqueue`. Аналогичный API для event notification.
*   **Windows:** `IOCP` (I/O Completion Ports).
*   **WASM:** `webPoller` (асинхронные Web APIs).

При старте приложения `netpollInit` регистрирует fd в OS-реестре. Когда горутина блокируется на I/O, `netpollBlock` добавляет fd в список ожидания и вызывает `runtime.stopm`. Когда fd становится готовым, `netpollReady` ставит `g` в очередь `runq` через `gopark`/`goready`. 

> [!tip] Собеседование
> **Вопрос:** Что произойдёт, если внутри горутины вызвать `time.Sleep` или блокирующий syscall, который не поддерживает `netpoller`?
> **Ответ:** Горутина будет парковаться, но **системный тред (M) останется заблокированным** на этом syscall до его завершения. Если таких тредов станет больше `GOMAXPROCS`, новые горутины не смогут исполняться, и throughput упадёт. Именно поэтому критично использовать неблокирующие обёртки или выносить такие операции в отдельные треды.

## Ловушки и собеседования

*   **Gotcha 1: `net.Conn` не потокобезопасна.** Одновременные вызовы `Read()`/`Write()` из разных горутин на одном соединении приведут к race-conditions на уровне fd и повреждению буферов. Используйте `sync.Mutex` или изолируйте соединение одной горотиной.
*   **Gotcha 2: Keep-Alive и утечка соединений.** Если клиент не закрывает соединение, а сервер не читает `EOF`, `netpoller` продолжит держать fd открытым. Нагрузочные тесты должны явно проверять состояние `tcpdump` и лимиты `ulimit -n`.
*   **Gotcha 3: DNS-блокировки.** `net.ResolveTCPAddr` или первый вызов `net.Dial` может заблокировать тред на DNS-резолвер. В Go 1.21+ это стало менее критично благодаря `netgo` и фону, но в высоконагруженных сервисах рекомендуется кэшировать DNS или использовать `resolvconf` с `ndots:0`.

```go
// Правильный handling ошибок чтения в цикле
for {
    n, err := conn.Read(buf)
    if err != nil {
        if errors.Is(err, io.EOF) {
            break // Клиент закрыл соединение
        }
        // Логика обработки сетевых ошибок (timeout, reset)
        return err
    }
    // обработка n байт
}
```

## Сравнение с другими языками

| Язык/Стек | Модель I/O | Управление тредами | Скалирование до 100k+ коннектов |
|-----------|------------|---------------------|----------------------------------|
| **PHP-FPM** | Blocking (Fork/Thread) | Один процесс на запрос | ❌ Требует nginx proxy + fastcgi |
| **Java (pre-21)** | Blocking (Thread-per-req) | OS Threads (2MB+ стека) | ⚠️ Высокий overhead, требует виртуальных тредов (Loom) |
| **Node.js** | Event-loop (libuv) | Single-threaded JS + worker threads | ✅ Отлично, но блокировка JS-цикла убивает CPU-bound задачи |
| **Go** | Async I/O (`netpoller`) | Multi-threaded event-loop (M-P-G) | ✅ Оптимален: нет блокировок тредов + нативный параллелизм |

Go выигрывает за счёт гибридной модели: неблокирующий I/O через `netpoller` + легковесные горутины + многопоточный планировщик. Это даёт лучшую утилизацию CPU и предсказуемую latency.

## Итог и навигация

Сетевой стек — это не абстракция, а набор жёстких ограничений и гарантий, диктуемых железом и ОС. В Go вы платите за удобство `net.Conn` прозрачностью работы с `netpoller` и планировщиком. Понимание того, как fd мапится на горутины и как `epoll` управляет готовностью, позволяет проектировать сервисы, которые не падают под нагрузкой и не утекают памятью.
