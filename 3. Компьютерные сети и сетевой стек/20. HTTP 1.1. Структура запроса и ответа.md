## Введение

HTTP/1.1 остаётся фундаментом, на котором строятся все современные веб-сервисы, прокси-уровни и отладочные утилиты. Даже при переходе на HTTP/2 или HTTP/3, понимание бинарного формата запросов и ответов критически важно для диагностики производительности, настройки балансировщиков, работы с legacy-системами и понимания того, как `net/http` в Go трансформирует сырой TCP-пайплайн в структурно-типизированные объекты.

В этой статье мы разберём синтаксис и семантику HTTP/1.1, а также то, как Go-рантайм парсит, хранит и обрабатывает эти структуры с точки зрения производительности и управления памятью.

## Структура HTTP-запроса

Каждый HTTP-запрос состоит из трёх обязательных частей, разделённых двойным переводом строки `\r\n\r\n`:

1. **Start Line (Строка запроса)**: `METHOD SP Request-URI SP HTTP-Version CRLF`
   - `METHOD`: `GET`, `POST`, `PUT`, `DELETE`, `PATCH`, `HEAD`, `OPTIONS`, `TRACE`, `CONNECT`.
   - `Request-URI`: Путь, query-параметры и фрагмент (например, `/api/v1/users?id=123`).
   - `HTTP-Version`: Строка `HTTP/1.1`.
2. **Headers (Заголовки)**: Пары `Key: Value CRLF`. Заголовки чувствительны к регистру ключей, но спецификация требует от реализаций регистронезависимого сравнения.
3. **Body (Тело)**: Опциональный блок данных. Его наличие и размер определяются заголовком `Content-Length` или `Transfer-Encoding: chunked`.

## Структура HTTP-ответа

Ответ сервера имеет аналогичную структуру, но начинается со строки статуса:

1. **Start Line (Строка статуса)**: `HTTP-Version SP Status-Code SP Reason-Phrase CRLF`
   - `Status-Code`: Трёхзначный код (например, `200`, `404`, `500`).
   - `Reason-Phrase`: Человекочитаемое описание (например, `OK`, `Not Found`). В современных реализациях часто игнорируется машинами.
2. **Headers**: Метаданные ответа (`Content-Type`, `Set-Cookie`, `Cache-Control`, `Server` и др.).
3. **Body**: Тело ответа (JSON, HTML, бинарные данные). Аналогично запросу, может отсутствовать или передаваться chunked-encoding.

## Устройство в Go: `http.Request` и `http.Response`

В Go структура запроса и ответа представлена типами `http.Request` и `http.Response` из пакета `net/http`.

```go
// http.Request (упрощённо)
type Request struct {
    Method       string            // Метод запроса
    URL          *url.URL          // Парсинг URI, query-параметров
    Proto        string            // "HTTP/1.1"
    ProtoMajor   int               // 1
    ProtoMinor   int               // 1
    Header       Header            // map[string][]string
    Body         io.ReadCloser     // Stream тела запроса
    ContentLength  int64           // -1 если chunked или неизвестно
    TransferEncoding []string      // ["chunked"]
    Host         string            // Заголовок Host
    Context      context.Context   // Контекст с таймаутами и отменой
    // ... другие поля
}

// http.Header (типизированная обёртка над map)
type Header map[string][]string
```

> [!info] Под капотом
> В Go 1.21+ `http.Header` остаётся `map[string][]string`. Ключи всегда приводятся к lower-case при парсинге для обеспечения регистронезависимого доступа. Значения хранятся как slice строк, что позволяет корректно обрабатывать дублирующиеся заголовки (например, `Set-Cookie`, `Cache-Control`).

## Mechanical Sympathy и работа с памятью

Парсинг HTTP/1.1 в Go происходит в hot-path-е сервера. Каждая аллокация на критическом пути влияет на GC pressure и CPU cache locality.

1. **Escape Analysis**: Заголовки HTTP-запроса почти всегда уходят в кучу (`heap`). Это происходит потому, что `http.Header` хранит `map[string][]string`, где строки живут дольше, чем стек функции-парсера. `http.Request` также аллоцируется в куче при создании обработчика.
2. **CPU Cache & String Interning**: Go не делает автоматического string interning для заголовков. При каждом запросе создаются новые `string` объекты. Для оптимизации в высоконагруженных сервисах используют `[]byte` на этапе парсинга и избегают `string()` преобразований до последнего момента.
3. **`bufio` и `bytes`**: `net/http` использует `bufio.Reader` для буферизации сырых байтов из сокета. Парсер работает по конечному автомату (state machine), последовательно распознавая `\r\n` и извлекая токены. Это минимизирует количество системных вызовов `read()` на уровне ОС.

```go
// Идиоматичная обработка запроса с учётом контекста и ошибок
func HandleRequest(w http.ResponseWriter, r *http.Request) {
    // 1. Проверка контекста (отмена запроса, таймаут)
    select {
    case <-r.Context().Done():
        http.Error(w, "request cancelled", http.StatusServiceUnavailable)
        return
    default:
    }

    // 2. Безопасное чтение тела (ограничение размера)
    r.Body = http.MaxBytesReader(w, r.Body, 10<<20) // 10 MB limit
    
    // 3. Парсинг тела (JSON пример)
    var payload struct {
        ID string `json:"id"`
    }
    if err := json.NewDecoder(r.Body).Decode(&payload); err != nil {
        http.Error(w, "invalid payload", http.StatusBadRequest)
        return
    }
    
    // 4. Работа с заголовками
    userAgent := r.Header.Get("User-Agent") // Возвращает "" если нет
    cookies := r.Cookies()                   // Парсит Cookie заголовок

    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusOK)
    fmt.Fprintf(w, `{"id": "%s"}`, payload.ID)
}
```

> [!warning] Ловушка / Gotcha
> **Дублирующиеся заголовники**: `r.Header.Get("X-Custom")` вернёт только первое значение, склеенное через запятую (если оно не является специфичным, например `Set-Cookie`). Для получения всех значений используйте `r.Header.Values("X-Custom")`.
> **Блокирующее чтение**: `r.Body.Read()` блокирует горутину до появления данных или закрытия соединения. Всегда оборачивайте чтение в `context.WithTimeout` или используйте `http.MaxBytesReader`, чтобы избежать DoS-атак через медленные соединения (Slowloris).

## Сравнение с другими языками

| Аспект | Go (`net/http`) | PHP (ZTS) / Python (WSGI) | C# (ASP.NET Core) |
|--------|-----------------|---------------------------|-------------------|
| **Жизненный цикл** | `http.Server` пулит горутины, объекты `Request/Response` создаются на каждый connection, но `http.Transport` переиспользует TCP-сокеты. | Перезапуск PHP-скрипта или использование ZTS-модуля. Каждый запрос = новый процесс/потоки, ручная парсинг заголовков. | Middleware pipeline. `HttpContext` создаётся на запрос, но `HttpClient` пулит соединения через `IHttpClientFactory`. |
| **Работа с памятью** | `[]byte` → `string` конвертация вызывает аллокацию. `http.Header` мапа на каждый запрос. | Зависит от движка (Zend, PHP-FPM). Частые аллокации строк из-за динамической типизации. | `Span<T>` и `Memory<T>` для zero-allocation парсинга в .NET 8+. Go отстаёт в этом, но `bytes` пакет оптимизирован. |
| **Конкурентность** | Горутина на соединение (или `netpoller` для неблокирующего I/O). | Многопоточность (потоки ОС) или event-loop (Swoole/ReactPHP). | `async/await` + `Task` pool. |

> [!tip] Собеседование
> **Вопрос:** Почему `r.Header.Get()` работает быстрее, чем прямой доступ к `map`?
> **Ответ:** `Header.Get()` использует `CanonicalMIMEHeaderKey` для нормализации ключа в lower-case и проверяет дубликаты. Прямой доступ к `r.Header["key"]` может вернуть `nil` slice или пропустить заголовок, если он пришёл в другом регистре. Кроме того, `Get` возвращает `""` при отсутствии, что предотвращает панику при разыменовании.
> 
> **Вопрос:** Как Go обрабатывает `Transfer-Encoding: chunked`?
> **Ответ:** `r.Body` автоматически оборачивается в `chunkedReader`. При вызове `Read()` он считывает размер чанка, читает данные, пропускает `\r\n` и повторяет до нулевого чанка. Это прозрачно для разработчика, но добавляет накладные расходы на парсинг размера. В Go 1.17+ добавлена оптимизация для chunked-encoding в `net/http`.

## Итог

1. **HTTP/1.1** состоит из строки, заголовков и тела, разделённых `\r\n`.
2. **В Go** запросы и ответы представлены типами `http.Request` и `http.Response`, где `http.Header` — это `map[string][]string` с регистронезависимым доступом.
3. **Memory & Performance**: Парсинг аллоцирует строки и мапы в куче. `bufio` минимизирует syscalls, но escape-analysis неизбежен в hot-path. Используйте `[]byte` и `bytes.Buffer` для критичных к производительности участков.
4. **Gotchas**: Дублирующиеся заголовки, блокирующее чтение без контекста, `Content-Length` vs chunked, регистрозависимость ключей.
5. **Архитектура**: `net/http` использует `netpoller` (epoll/kqueue) для неблокирующего I/O, превращая потоковые сокеты в горутины. Это отличает Go от PHP (per-process) и приближает к C++/Rust event-loop моделям, но с синтаксической простотой.
