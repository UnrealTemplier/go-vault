## Маршрутизация: Таблица маршрутов, шлюзы и default route

Для Go-разработчика, работающего в облачных средах, Kubernetes или на многоинтерфейсных серверах, маршрутизация перестает быть абстракцией. Понимание того, как ядро Linux выбирает исходящий IP-адрес, куда уходят пакеты при `net.Dial` и как работают fallback-правила, критично для отладки сетевых проблем, настройки балансировщиков и написания предсказуемого сетевого кода.

## Что такое маршрутизация?

**Маршрутизация (Routing)** — это процесс определения пути для передачи сетевого пакета от источника к получателю на основе адреса назначения. На уровне операционной системы это решается **подсистемой маршрутизации ядра**.

Важно разделять два понятия:
*   **Routing (Маршрутизация)** — интеллектуальная часть. Принятие решения «куда отправить пакет» на основе таблицы маршрутов.
*   **Forwarding (Переадресация)** — механическая часть. Физическая отправка кадра на следующий узел (шлюз) или в локальную сеть.

В Go-приложениях мы редко работаем с таблицей маршрутов напрямую. Мы вызываем `net.Dial` или `net.Listen`, а ядро Linux автоматически выполняет lookup при создании сокета. Но когда сервер имеет несколько сетевых интерфейсов (например, `eth0` для трафика клиентов и `eth1` для межсервисного общения), неправильное понимание маршрутизации приводит к «обратным путям» (asymmetric routing) и разрыву соединений.

## Таблица маршрутов (Routing Table)

Таблица маршрутов — это структура данных ядра, которая сопоставляет диапазоны IP-адресов назначения с параметрами доставки. В Linux её можно посмотреть командой `ip route show` (или `route -n`).

Каждая запись содержит следующие ключевые поля:
*   **Destination (dst)**: Сеть или хост назначения (например, `192.168.1.0/24` или `10.0.0.1`).
*   **Prefix Length (mask)**: Маска подсети, определяющая длину префикса.
*   **Gateway (gw)**: Адрес следующего узла (Next Hop). Пустое значение означает прямую доставку.
*   **Device (dev)**: Исходящий сетевой интерфейс (`eth0`, `br-abc123` и т.д.).
*   **Metric**: Приоритет маршрута при наличии нескольких вариантов. Чем меньше число, тем выше приоритет.
*   **Type**: `local` (адрес принадлежит хосту), `unicast` (обычный маршрут), `blackhole` (пакеты отбрасываются), `prohibit` (отбрасываются с ICMP).

### Алгоритм выбора: Longest Prefix Match (LPM)

При получении пакета ядро не ищет маршрут линейно. Оно использует **алгоритм наивысшего префикса (Longest Prefix Match)**. Ядро ищет запись в таблице, которая максимально точно совпадает с IP-адресом назначения.

```text
Таблица маршрутов:
10.0.0.0/8      via 192.168.1.1   dev eth0   metric 100
10.0.1.0/24     via 172.16.0.1    dev eth1   metric 50
10.0.1.50/32    via 127.0.0.1     dev lo     metric 0

Запрос: net.Dial("tcp", "10.0.1.50:8080")
Результат:
1. 10.0.0.0/8 совпадает на 8 бит
2. 10.0.1.0/24 совпадает на 24 бита
3. 10.0.1.50/32 совпадает на 32 бита -> ВЫБРАН
```

> [!tip] Собеседование
> **Вопрос:** Как ядро Linux выбирает маршрут, если есть несколько записей с одинаковой длиной префикса?
> **Ответ:** При равной длине префикса ядро опирается на **metric**. Если метрики равны, срабатывает механизм ECMP (Equal-Cost Multi-Path) — трафик распределяется по нескольким интерфейсам (обычно по хэшу из src/dst IP и портов). Если метрик нет, выбирается первая найденная запись в памяти.

## Шлюзы (Gateways) и локальная доставка

**Шлюз (Gateway / Next Hop)** — это IP-адрес устройства, которое принимает пакет, если адрес назначения не находится в локальной подсети.

*   **Локальная доставка (Direct Delivery):** Если маска подсети хоста и маски подсети назначения совпадают, пакет не уходит к шлюзу. Ядро проверяет, принадлежит ли IP интерфейсу (`type local`), или выполняет ARP/Neighbor Discovery для поиска MAC-адреса получателя в L2.
*   **Удаленная доставка:** Если адреса в разных подсетях, пакет отправляется на `Gateway`. Ядро должно знать MAC-адрес шлюза для инкапсуляции в Ethernet-кадр (через ARP для IPv4 или NDP для IPv6).

> [!warning] Ловушка / Gotcha
> Частая ошибка при настройке многоинтерфейсных серверов: `metric` путают с длиной префикса.
> `192.168.0.0/16` (маска 16) имеет **больший** префикс, чем `192.168.0.0/24` (маска 24)? Нет! 24 > 16. Поэтому `10.0.1.50/32` всегда победит `10.0.0.0/8`, независимо от метрик. Метрика учитывается ТОЛЬКО при равенстве длины префикса.

## Default Route (0.0.0.0/0)

**Default route (шлюз по умолчанию)** — это маршрут с префиксом `0.0.0.0/0` (или `::/0` для IPv6). Он работает как «catch-all» (сетевой подхват). Если ни одна конкретная запись в таблице не совпала с IP назначения, пакет отправляется по default route.

```bash
# ip route show default
default via 172.17.0.1 dev eth0 proto dhcp metric 100
```

В облачных провайдерах (AWS, GCP, Yandex Cloud) default route часто указывает на виртуальный маршрутизатор VPC. В Kubernetes он указывает на CNI-плагин (например, Calico или Cilium), который обеспечивает overlay-сеть.

При наличии нескольких default route с одинаковым префиксом `0.0.0.0/0` ядро использует `metric` для выбора приоритетного интерфейса. Это часто применяется при настройке failover или балансировки WAN.

## Под капотом: Linux FIB и Lookup

В ядре Linux таблица маршрутов называется **FIB (Forwarding Information Base)**. Она построена на базе **радикального дерева (Radix Tree / Trie)**, что обеспечивает время поиска O(k), где k — длина префикса (32 для IPv4, 128 для IPv6). Это константное время по практическим меркам, что критично для высоконагруженных сетевых сервисов.

При вызове `connect()` или `bind()` в Go:
1.  Пользовательское пространство передаёт dst IP в ядро через syscall `sys_connect`.
2.  Ядро вызывает `fib_lookup()`.
3.  Алгоритм LPM находит лучшую запись в FIB.
4.  Если маршрут указывает на шлюз, ядро выполняет ARP-запрос (если нет в кэше `neighbour table`).
5.  Ядро выбирает **исходящий IP-адрес (source IP)** на основе интерфейса из маршрута. Именно этот IP возвращается в Go при вызове `conn.LocalAddr().(*net.TCPAddr).IP`.

> [!info] Под капотом
> В старых ядрах (до 2.6) использовалась кэшируемая таблица маршрутов (`ip_route_cache`), которая была подвержена состояниям гонки (RACE conditions) при динамических обновлениях. В современных ядрах (4.19+) кэш убран, lookup выполняется напрямую в FIB с использованием RCU (Read-Copy-Update) для безопасного параллельного чтения без блокировок. Это снижает latency сетевых операций в Go-сервисах под высокой нагрузкой.

## Go и маршрутизация: Как net выбирает IP

Пакет `net` в Go не содержит собственной таблицы маршрутов. Он делегирует выбор ядру. Однако есть важные нюансы, о которых часто забывают на собеседованиях:

1.  **Выбор source IP:** При `net.Dial("tcp", "dst_ip:port")` Go вызывает `connect()`. Ядро смотрит в таблицу маршрутов, находит `dev` и выбирает IP этого интерфейса как source. Если интерфейсов несколько, Go **не** выбирает IP самостоятельно. Он доверяет ядру.
2.  **Принудительная привязка к интерфейсу:** Если нужно отправить трафик через конкретный интерфейс, несмотря на таблицу маршрутов, используется `SO_BINDTODEVICE`. В Go это реализуется через `syscall.SetsockoptString` или `net.ListenConfig.Control`:
    ```go
    cfg := net.ListenConfig{
        Control: func(network, address string, c syscall.RawConn) error {
            return c.Control(func(fd uintptr) {
                syscall.SetsockoptString(int(fd), syscall.SOL_SOCKET, syscall.SO_BINDTODEVICE, "eth1")
            })
        },
    }
    listener, _ := cfg.Listen(context.Background(), "tcp", ":8080")
    ```
3.  **IP_FREEBIND / IP_TRANSPARENT:** В Go-балансировщиках (например, `haproxy` или самописных прокси на Go) часто нужно слушать IP, который физически не принадлежит серверу. Это достигается флагом `IP_FREEBIND` (Linux) или `IP_TRANSPARENT`. В Go 1.21+ это настраивается через `net.ListenConfig` или `sysfd` напрямую.
4.  **Multi-homed и DNS:** Go по умолчанию резолвит DNS последовательно. Если у сервера несколько интерфейсов, `net.Resolver` может вернуть IPs в порядке, зависящем от конфигурации `/etc/resolv.conf` и `nsswitch.conf`. В распределенных системах это требует явного контроля через `net.Dialer.Resolver`.

> [!tip] Собеседование
> **Вопрос:** Как заставить Go-сервис отправлять исходящий трафик через конкретный IP, если в таблице маршрутов default route указывает на другой интерфейс?
> **Ответ:** Либо менять метрику default route в ядре (не всегда возможно в контейнере), либо использовать `SO_BINDTODEVICE` на уровне сокета. Для точного контроля source IP в Go 1.21+ можно использовать `IP_TRANSPARENT` + `IP_FREEBIND` + `bind()` к нужному IP, либо использовать `net.Dialer.LocalAddr` для указания исходящего адреса, хотя финальное решение всё равно зависит от ядра и policy routing (`ip rule`).

## Итог

1.  **Маршрутизация** — это функция ядра Linux (FIB), а не уровня приложения.
2.  **Longest Prefix Match** определяет маршрут. При равенстве префикса используется `metric`.
3.  **Default route (`0.0.0.0/0`)** — fallback для всех несекционированных адресов.
4.  **Go не выбирает source IP самостоятельно** — он делегирует это ядру через `connect()`. Для обхода используются `SO_BINDTODEVICE` или `IP_FREEBIND`.
5.  В облаках и Kubernetes маршрутизация усложняется CNI-плагинами, overlay-сетями (VXLAN) и service mesh, что требует понимания `ip rule` и policy routing.
