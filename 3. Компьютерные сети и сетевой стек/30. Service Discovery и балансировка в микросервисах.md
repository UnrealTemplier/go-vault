## Динамическая природа микросервисов и проблема маршрутизации

В монолитной архитектуре адреса сервисов статичны: `db.internal:5432` или `cache.internal:6379`. В микросервисной парадигме экземпляры постоянно масштабируются, перезапускаются и меняют сетевые адреса. Захардкоженные конфиги или DNS-записи с долгим TTL приводят к `connection refused` и `timeout` при каждом деплое.

**Service Discovery (SD)** решает задачу динамического обнаружения доступных эндпоинтов. Для Go-разработчика это не просто абстракция, а критический слой, напрямую влияющий на `net/http.Transport`, пулы соединений и время отклика.

## Архитектурные паттерны: Client-Side vs Server-Side

Существует два фундаментальных подхода к балансировке и обнаружению:

1. **Client-Side Discovery**: Клиент сам опрашивает реестр (Consul, etcd, Kubernetes DNS) и выбирает бэкенд.
   * *Плюсы*: Полный контроль над алгоритмом, отсутствие единой точки отказа в маршрутизации, возможность кастомных стратегий (consistent hashing, locality-aware routing).
   * *Минусы*: Клиент должен знать протокол и уметь обрабатывать ошибки, усложняется код, дублирование логики между сервисами.
   * *В Go*: Прямой вызов `consul` или `etcd` API, кастомный `RoundTripper`, использование `net.Dialer` с кастомным `DialContext`.

2. **Server-Side Discovery**: Клиент обращается к виртуальному адресу (VIP), а балансировщик (L4/L7) маршрутизирует трафик.
   * *Плюсы*: Прозрачность для клиента, централизованное управление, легкое внедрение mTLS и rate limiting.
   * *Минусы*: Дополнительный сетевой hop, latency overhead, сложность отладки.
   * *В Go*: Работа через `[[27. Прокси, Reverse Proxy и API Gateway]]` (Envoy, HAProxy, K8s Ingress). Клиент общается с `localhost:8080` (sidecar) или внешним VIP.

> [!tip] Собеседование
> **Вопрос:** Когда в Go-экосистеме предпочтительнее клиентская балансировка, а когда серверная?
> **Ответ:** Клиентская LB нужна, когда алгоритм маршрутизации зависит от состояния сервиса (например, consistent hashing для сессий, least-connections для CPU-bound задач, или awareness кэша). Серверная LB доминирует в стандартных REST/gRPC кластерах, где критичны observability, mTLS и отключение токсичных инстансов. В Go мы часто комбинируем подходы: sidecar (Envoy) для L4/L7, а внутри сервиса — легковесный client-side LB для кросс-зональной маршрутизации.

## DNS как базовый механизм discovery в Go

Kubernetes и многие облачные провайдеры реализуют SD через DNS. При запросе `svc.default.svc.cluster.local` CoreDNS возвращает список IP подов. Go использует встроенный `net.Resolver` для преобразования имени в IP.

> [!info] Под капотом
> В Go 1.11+ встроенный DNS-клиент кэширует ответы. Кэш работает на уровне `net` пакета и использует TTL из ответа сервера, но имеет внутреннюю минимальную задержку инвалидации. Если DNS-сервер возвращает TTL = 0 или очень маленькое значение, Go может всё равно удержать запись в памяти до 2 секунд (или до исчерпания лимита кэша, который по умолчанию ~1000 записей). Это приводит к **stale DNS** эффекту: клиент продолжает использовать старый IP, хотя инстанс уже уничтожен.

```go
// Идиоматичное создание resolver с кастомным dialer для контроля TTL
import "net"

func NewResolver() *net.Resolver {
    return &net.Resolver{
        PreferGo: true, // Использовать встроенный Go-парсер вместо системного cgo
        Dial: func(ctx context.Context, network, address string) (net.Conn, error) {
            // Кастомный dial к конкретному DNS-серверу (напр. CoreDNS)
            d := &net.Dialer{Timeout: 2 * time.Second}
            return d.DialContext(ctx, "udp", "127.0.0.1:53")
        },
    }
}
```

## Балансировка нагрузки: алгоритмы и реализация

В Go нет встроенного LB-модуля для списка бэкендов. Мы реализуем его через `net/http.RoundTripper` или кастомный `DialContext`.

Основные алгоритмы:
* **Round Robin**: Простой, но уязвим к `cache stampede` и неравномерной загрузке.
* **Least Connections**: Требует метрик состояния, сложно реализовать в stateless-среде без sidecar.
* **Consistent Hashing**: Сохраняет аффинность сессий, критично для кэшей и очередей.
* **Random + Weighted**: Наивный, но эффективный при большом количестве инстансов (закон больших чисел).

```go
package main

import (
    "context"
    "net"
    "net/http"
    "sync"
    "sync/atomic"
    "time"
)

// ServiceBalancer реализует client-side LB с health-check и round-robin
type ServiceBalancer struct {
    endpoints  []string
    currentIndex int64
    clients    map[string]*http.Client
    mu         sync.RWMutex
    resolver   *net.Resolver
}

func NewBalancer(endpoints []string, timeout time.Duration) *ServiceBalancer {
    // Инициализация клиентов с настроенным Transport
    // Важно: MaxIdleConnsPerHost по умолчанию = 2. 
    // Для LB это критично: при ротации IP пул быстро исчерпается.
    tr := &http.Transport{
        MaxIdleConns:        100,
        MaxIdleConnsPerHost: 50, 
        IdleConnTimeout:     90 * time.Second,
        DialContext: (&net.Dialer{
            Timeout:   5 * time.Second,
            KeepAlive: 30 * time.Second,
        }).DialContext,
        ForceAttemptHTTP2: true,
    }

    c := &http.Client{
        Timeout:   timeout,
        Transport: tr,
    }

    clients := make(map[string]*http.Client)
    for _, ep := range endpoints {
        clients[ep] = c
    }

    return &ServiceBalancer{
        endpoints: endpoints,
        clients:   clients,
        resolver:  net.DefaultResolver,
    }
}

// NextEndpoint выбирает следующий доступный эндпоинт
func (b *ServiceBalancer) NextEndpoint() (string, error) {
    // Копируем срезы для избежания race conditions при чтении
    b.mu.RLock()
    eps := make([]string, len(b.endpoints))
    copy(eps, b.endpoints)
    b.mu.RUnlock()

    if len(eps) == 0 {
        return "", fmt.Errorf("no endpoints available")
    }

    // Atomic инкремент для lock-free round-robin
    idx := atomic.AddInt64(&b.currentIndex, 1)
    return eps[idx%int64(len(eps))], nil
}
```

> [!tip] Собеседование
> **Вопрос:** Почему `atomic.AddInt64` лучше `sync.Mutex` для round-robin в Go?
> **Ответ:** В высоконагруженном LB мы избегаем блокировок на hot-path. Атомарные операции используют аппаратные инструкции (CMPXCHG на x86), которые работают за несколько тактов CPU и не вызывают переключения контекста или блокировку кэш-линий (cache line bouncing) при низкой конкуренции. Mutex включается только при реальном конфликте, что делает атомики предпочтительными для простых счетчиков.

## Health Checks и graceful degradation

Discovery бесполезен без валидации состояния бэкенда. Пассивные health checks (ошибки на уровне TCP/HTTP) приводят к отправке трафика на "больные" инстансы до таймаута соединения. Активные checks (polling `/health`) требуют собственного цикла.

В Go это реализуется через `http.Client` с кастомным `CheckRedirect` и `Transport.DialContext`, либо через отдельный `goroutine` с `time.Ticker`. При падении check'а инстанс временно исключается из пула.

```go
// Пример интеграции health check в dialer
func (b *ServiceBalancer) dialWithHealth(ctx context.Context, network, addr string) (net.Conn, error) {
    // 1. Проверка статуса инстанса в локальном кэше
    status := b.getHealthStatus(addr)
    if status == Down {
        return nil, fmt.Errorf("endpoint %s is unhealthy", addr)
    }
    
    // 2. Реальный dial
    d := &net.Dialer{Timeout: 3 * time.Second}
    conn, err := d.DialContext(ctx, network, addr)
    if err != nil {
        b.markAsDown(addr) // Пассивная маркировка при fail
    }
    return conn, err
}
```

## Go-специфика: Transport, connection pooling и сетевой стек

`net/http.Transport` — это ядро сетевого взаимодействия в Go. Он управляет пулом соединений через `netpoller` (epoll на Linux, kqueue на macOS/BSD). 

Когда DNS меняет IP, старое соединение (`net.Conn`) не инвалидируется автоматически. Go продолжает использовать его до `IdleConnTimeout` или пока оно не упадет. Это вызывает **TCP state mismatch**: клиент держит `ESTABLISHED` соединение, а сервер уже закрыл его (или переназначил порт), что приводит к `use of closed network connection` или `broken pipe`.

> [!warning] Ловушка / Gotcha
> **DNS TTL vs IdleConnTimeout**: Если TTL DNS = 10с, а `IdleConnTimeout` = 90с, Go будет держать 80 секунд тупое соединение. При деплое это вызывает `connection refused` для 80% запросов.
> **Решение**: 
> 1. Устанавливать `IdleConnTimeout < DNS_TTL`.
> 2. Использовать `Transport.DialContext` с кастомным resolver, который форсирует re-resolution при `err == ErrConnectionClosed`.
> 3. В Kubernetes использовать `ExternalIPs` или `Headless Services` с `publishNotReadyAddresses: true`.

Механика работы `Transport`:
1. При запросе ищется свободный `net.Conn` в пуле (`idleConn`).
2. Если нет — создается новое через `DialContext`.
3. Соединение регистрируется в `netpoll` для асинхронного чтения/записи.
4. При `IdleConnTimeout` или `MaxIdleConns` соединение закрывается через `conn.Close()`, что освобождает FD и кэш-линии.

## Итог

1. **Service Discovery** в Go чаще реализуется через DNS или легковесные клиенты (Consul/etcd), а не тяжелые фреймворки.
2. **Client-Side LB** дает контроль, но требует ручной реализации health checks и инвалидации кэша. **Server-Side** (Envoy/K8s) снимает нагрузку с кода, но добавляет latency.
3. **DNS TTL и `http.Transport`** должны быть согласованы. `MaxIdleConnsPerHost` по умолчанию = 2, что убивает производительность при частой ротации IP.
4. **netpoller и epoll** делают Go высокопроизводительным, но старые соединения с изменившимся DNS требуют явной инвалидации или таймаутов.
5. На собеседованиях ожидают понимания trade-offs между атомарными счетчиками и mutex, влияния TTL на TCP-пулы и стратегии graceful degradation при падении бэкендов.
