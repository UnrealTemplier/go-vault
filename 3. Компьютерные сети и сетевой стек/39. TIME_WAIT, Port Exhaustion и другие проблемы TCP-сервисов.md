## Введение: Сетевой ресурс как узкое место

В высоконагруженных Go-сервисах TCP-соединения — это не просто абстракция API, а ограниченный системный ресурс, управляемый ядром Linux. Когда вы пишете HTTP-сервер, gRPC-клиент или брокер, вы постоянно создаете и уничтожаете сокетные дескрипторы. Одной из самых частых и коварных проблем при масштабировании становится **Port Exhaustion** (исчерпание локальных портов), вызванное состоянием `TIME_WAIT`.

Разберем, почему оно возникает, как влияет на `conntrack` и планировщик Go, и как правильно тюнить стек, чтобы сервис не падал под нагрузкой.

## TIME_WAIT: Зачем ядру нужно «держать мертвецов»

Состояние `TIME_WAIT` возникает после закрытия соединения по инициативе стороны (чаще всего сервера). Согласно RFC 793 и современным реализациям, соединение остается в этом состоянии `2 * MSL` (Maximum Segment Lifetime). В Linux по умолчанию `tcp_fin_timeout = 60`, то есть **60 секунд**.

Зачем ядро «держит» закрытый сокет, если данные уже ушли?
1. **Надежность финального ACK:** Если финальный `ACK` от `FIN` теряется в сети, peer-сторона повторит `FIN`. Если сокет будет удален сразу, он ответит `RST`, что приведет к ошибке `Connection reset by peer` на стороне клиента. `TIME_WAIT` гарантирует, что повторный `FIN` будет обработан и сброшен корректно.
2. **Очистка «застывших» пакетов:** В медленных или перегруженных сетях пакет с тем же `seq` и `port` может прийти с опозданием. Если порт будет занят новым соединением, он интерпретирует старый пакет как часть новой сессии. `TIME_WAIT` предотвращает эту коллизию.

> [!info] Под капотом
> В Linux ядро не хранит полноценный `struct sock` для `TIME_WAIT`. Вместо этого используется легковесная структура `struct tcp_timewait_sock`. Она помещается в хеш-таблицу `tcp_time_wait_hash`. Это экономит память, но создает нагрузку на `conntrack` (таблицу отслеживания состояний), которая мониторит все активные и недавние соединения для NAT, брандмауэра и QoS.

## Port Exhaustion: Механика исчерпания

Каждое TCP-соединение идентифицируется кортежем: `SrcIP:SrcPort:DstIP:DstPort`. Для исходящих соединений клиент использует **эфемерные порты** (ephemeral ports). В Linux диапазон обычно `32768-60999` (около 28K портов).

Если ваш Go-сервис или клиент делает слишком много короткоживущих соединений (например, 10k RPS к внешнему API или базе данных с `pool_size = 1`), порт исчерпывается. Состояние `TIME_WAIT` блокирует порт для создания новых исходящих соединений.

Симптомы:
- `dial tcp: bind: address already in use`
- `connect: cannot assign requested address`
- В `dmesg`: `nf_conntrack: table full, dropping packet`

> [!warning] Ловушка / Gotcha
> На серверной стороне `TIME_WAIT` накапливается, если вы агрессивно закрываете соединения без пулинга. На клиентской стороне порт исчерпывается быстрее всего, если вы не переиспользуете соединения (`Keep-Alive`). В Go это часто случается при небрежной настройке `net/http.Transport`.

## Стратегии борьбы: SO_REUSE, тюнинг ядра и пулинг

### 1. SO_REUSEADDR и SO_REUSEPORT
- `SO_REUSEADDR` позволяет привязать сокет к адресу, даже если есть сокеты в `TIME_WAIT` или `CLOSE_WAIT`. **Не создает новых соединений**, а лишь ускоряет повторное использование адреса при перезапуске процесса.
- `SO_REUSEPORT` (доступен в Linux 3.9+) позволяет нескольким процессам/нитям привязаться к одному `IP:port`. Ядро распределяет входящие `accept` между ними на уровне mmap, что снимает contention на `accept` lock и улучшает локальность кэш-линий CPU.

### 2. Тюнинг ядра Linux
```bash
# Разрешить повторное использование TIME_WAIT для исходящих соединений (безопасно в современных ядрах)
net.ipv4.tcp_tw_reuse = 1

# Расширить диапазон эфемерных портов (для клиентов)
net.ipv4.ip_local_port_range = 1024 65535

# Лимит хеш-корзин TIME_WAIT (по умолчанию ~262144)
net.ipv4.tcp_max_tw_buckets = 524288
```
> [!tip] Собеседование
> **Вопрос:** Почему `SO_REUSEADDR` не спасает от `TIME_WAIT` на сервере при перезапуске?
> **Ответ:** `SO_REUSEADDR` лишь разрешает `bind()` к занятому адресу. Он не удаляет активные состояния `TIME_WAIT` из таблицы conntrack. Для мгновенного освобождения нужно ждать завершения таймера или использовать `SO_REUSEPORT` с несколькими воркерами, либо настраивать `tcp_tw_reuse` (только для исходящих соединений).

### 3. Connection Pooling
В Go пулинг встроен в `net/http.Transport`. Если `IdleConnTimeout` слишком мал (< 60 сек на Linux), соединения будут закрываться до завершения `TIME_WAIT`, вызывая постоянный цикл `SYN -> FIN`. Это убивает производительность и быстро исчерпывает порты.

## Go-специфика: net/http, netpoll и системные вызовы

### Как Go управляет соединениями
- `net.Listen` использует `sysSocket` -> `bind` -> `listen`. При включенном `SO_REUSEPORT` ядро балансирует `accept` между горутинными трэдами.
- `net/http` по умолчанию использует `Transport` с пулом. Критичные поля: `MaxIdleConns`, `MaxIdleConnsPerHost`, `IdleConnTimeout`.
- `netpoll` (epoll/kqueue) мониторит файловые дескрипторы. Каждый сокет — это FD. При исчерпании лимита FD (`ulimit -n`) или портов, `accept` начинает падать с `EMFILE` или `ENFILE`.

> [!info] Под капотом
> В Go 1.21+ планировщик использует `netpoll` для асинхронного приема соединений. При высокой нагрузке на `TIME_WAIT` сокеты не потребляют CPU, но занимают память в ядре. Когда таблица `conntrack` переполняется, ядро начинает `drop` пакеты, что вызывает silent retransmissions и рост latency.

### Пример правильной настройки для production
```go
transport := &http.Transport{
    MaxIdleConns:        100,
    MaxIdleConnsPerHost: 100, // Важно! По умолчанию равен MaxIdleConns
    IdleConnTimeout:     90 * time.Second, // Должен быть > 2 * MSL (60s)
    DisableKeepAlives:   false,
    TLSHandshakeTimeout: 10 * time.Second,
}

// Для высоконагруженных серверов: включаем SO_REUSEPORT
lc := net.ListenConfig{
    Control: func(network, address string, c syscall.RawConn) error {
        return c.Control(func(fd uintptr) {
            if err := syscall.SetsockoptInt(int(fd), syscall.SOL_SOCKET, syscall.SO_REUSEPORT, 1); err != nil {
                return err
            }
            // Также полезно включить SO_REUSEADDR для быстрого перезапуска
            if err := syscall.SetsockoptInt(int(fd), syscall.SOL_SOCKET, syscall.SO_REUSEADDR, 1); err != nil {
                return err
            }
        })
    },
}

ln, err := lc.Listen(context.Background(), "tcp", ":8080")
if err != nil {
    log.Fatalf("listen failed: %v", err)
}
```

## Сравнение подходов: Go vs C++/Java/PHP
| Аспект | Go | C++ / Java | PHP |
|--------|----|------------|-----|
| **Пулинг соединений** | Встроен в `net/http.Transport`, настраивается через `IdleConnTimeout` | Требует внешних библиотек (OkHttp, Boost.Asio) или ручной настройки | Отсутствует по умолчанию, FPM убивает процессы, соединения не переиспользуются |
| **Управление FD** | `netpoll` (epoll/kqueue), автоматическое управление дескрипторами | Ручное управление `epoll`/`kqueue` или libevent | Сигнал-Driven или select/poll (устаревшие модели) |
| **TIME_WAIT влияние** | Критично при неверном тюнинге `Transport` | Критично, но разработчики часто пишут свои pooling-слои | Нивелируется перезапуском PHP-FPM, но вызывает утечку памяти/FD |

## Итог

1. **TIME_WAIT** существует для надежности сети и защиты от коллизий пакетов. Его длительность (`2 * MSL`) фиксирована ядром.
2. **Port Exhaustion** возникает при массовом создании короткоживущих соединений. Решается расширением диапазона эфемерных портов, `tcp_tw_reuse` и пулингом.
3. **Go-специфика:** `net/http.Transport` автоматически пулит соединения, но `IdleConnTimeout` должен превышать 60 секунд на Linux. `SO_REUSEPORT` снимает contention на `accept` и улучшает утилизацию CPU.
4. **Механика:** `TIME_WAIT` не грузит CPU, но забивает `conntrack` и память ядра. При переполнении начинаются silent drops, что маскируется под проблемы приложения.

Мы разобрали механику `TIME_WAIT`, причины port exhaustion и способы борьбы с ними на уровне ядра и Go-рантайма. Понимание этих ограничений критично для проектирования отказоустойчивых сервисов.
