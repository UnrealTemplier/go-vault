## VLAN, VXLAN и сегментация сети

Для Go-разработчика, работающего с микросервисами, контейнерами и облачной инфраструктурой, сеть перестает быть «абстракцией». Она становится ограничивающим фактором для задержек, пропускной способности и отказоустойчивости. Понимание того, как данные физически инкапсулируются, маршрутизируются и изолируются, напрямую влияет на проектирование `net.Dialer`, настройку connection pooling и диагностику `TIME_WAIT` или `MTU`-проблем.

## VLAN (802.1Q): Аппаратная сегментация на L2

**VLAN (Virtual Local Area Network)** стандарта IEEE 802.1Q позволяет логически разделить один физический коммутатор на несколько изолированных широковещательных доменов. Для Go-бэкенда это означает, что один и тот же сервер может обслуживать изолированные окружения (prod/staging/dev) без физического разделения портов.

### Устройство тега
При прохождении через tagged-порт коммутатор вставляет между полем `EtherType` и полезной нагрузкой Ethernet-кадра 4-байтный тег:
- **TPID (2 байта)**: `0x8100` указывает на наличие тега 802.1Q.
- **TCI (2 байта)**: 
  - Priority (3 бита) — QoS.
  - CFI (1 бит) — Canonical Format Indicator.
  - **VID (12 бит)** — VLAN ID. Значения `0` и `4095` зарезервированы. Доступно `4094` сети.

> [!info] Под капотом
> Коммутаторы инициализируют поиск VID в **TCAM** (Ternary Content-Addressable Memory). Это SRAM, позволяющая искать по маске за 1 такт. Однако TCAM очень дорогая и горячая. При большом количестве VLAN таблицы TCAM переполняются, и переключение контекста при маршрутизации между VLAN (L3 switch) требует вызова `ip route` и увеличения задержки на несколько микросекунд.

### Влияние на Go-приложения
1. **MTU Reduction**: Тег VLAN отнимает 4 байта. Стандартный MTU `1500` превращается в `1496`. Если Go-сервис отправляет пакет `1500` байт через tagged-порт, ядро Linux фрагментирует пакет или отбрасит его (PMTU Discovery). В Go это проявляется как `message too long` при `Write` на `net.UDPConn` или обрывы `net/http` при больших `Content-Length`.
2. **Broadcast Domains**: VLAN изолирует широковещательный трафик. Это критично для ARP, mDNS и Kubelet health-checks. В монолитах (PHP/Java) это часто игнорируется, но в Go-микросервисах с динамическим `net.Resolver` может привести к таймаутам DNS, если резолвер попадает в другой broadcast-домен.

## VXLAN: Программная сегментация поверх L3

В облаках и Kubernetes физическая изоляция через VLAN невозможна (лимит 4094, сложность миграции VM/контейнеров). Поэтому используется **VXLAN (RFC 7348)** — инкапсуляция L2-кадра поверх UDP/IP.

### Устройство VXLAN-пакета
Оригинальный Ethernet-кадр (включая VLAN-тег, если есть) оборачивается:
1. **VXLAN Header (8 байт)**: Flags + VNI (24 бита) + Reserved.
2. **Outer UDP Header (8 байт)**: Порт назначения `4785` (IANA).
3. **Outer IP Header (20-40 байт)**: Source/Dest IP инкапсулирующих узлов (VTEP).
4. **Outer Ethernet Header (14 байт)**: MAC-адрес шлюза/Next-Hop.

Общий оверхед: `8 + 8 + 20 + 14 = 50 байт`.

> [!warning] Ловушка / Gotcha
> **MTU Crisis**: Если базовый MTU `1500`, то доступное пространство для оригинального кадра составляет `1450`. При использовании VLAN (еще `-4`) и IPv6 (`-20`) доступный размер полезной нагрузки для TCP/Go-приложения падает до `1426`. Это вызывает фрагментацию на уровне UDP или принудительное уменьшение MSS. Решение: настройка **Jumbo Frames (MTU 9000)** на всех узлах VTEP.

### Почему UDP, а не TCP или ESP?
- **NAT Traversal**: UDP легко проходит через NAT-устройства, в отличие от ESP (IPsec) или протоколов с фиксированными номерами портов.
- **Middlebox Compatibility**: Маршрутизаторы и фаерволы привыкли маршрутизировать UDP. Инкапсуляция позволяет использовать стандартные L3-алгоритмы (ECMP, BGP) для балансировки нагрузки между VTEP.
- **Offloading**: Современные NIC поддерживают UDP checksum offload и VXLAN hardware offload, что снижает CPU overhead до `~0.5%` на поток.

```go
// Пример диагностики MTU и интерфейсов в Go
package main

import (
	"context"
	"fmt"
	"log"
	"net"
)

func checkInterfaceMTU(ctx context.Context) error {
	// net.Interfaces() обращается к netlink через syscall
	ifaces, err := net.Interfaces()
	if err != nil {
		return fmt.Errorf("net.Interfaces: %w", err)
	}

	for _, iface := range interfaces {
		// Ищем интерфейсы с MTU < 1500 (часто VXLAN или Docker bridge)
		if iface.MTU < 1500 {
			log.Printf("Interface %s: MTU=%d, Flags=%v", iface.Name, iface.MTU, iface.Flags)
		}
	}
	return nil
}
```

## Сегментация в экосистеме Go и контейнеризации

В современных кластерах (см. [[41. Kubernetes Networking. CNI, Pod Network, Service, Ingress.md]]) сегментация реализуется через **CNI (Container Network Interface)**. Плагин (Calico, Cilium, Flannel) создает виртуальные сетевые интерфейсы `veth`, назначает им IP из подсети кластера и настраивает маршрутизацию на хосте.

### Как Go видит эту сеть
Пакет `net` в Go работает поверх системного стека ядра Linux. При вызове `net.Dial("tcp", "service:8080")`:
1. `net.Resolver` выполняет DNS (обычно через CoreDNS в кластере).
2. `net.Dialer` вызывает `socket(AF_INET, SOCK_STREAM, IPPROTO_TCP)` и `connect()`.
3. Ядро смотрит в таблицу маршрутизации (`ip route`) и определяет исходящий интерфейс (например, `eth0` или `flannel.1` для VXLAN).
4. Если сегментация требует туннелирования, ядро инкапсулирует пакет в VXLAN. Это происходит в **Softnet** или через **eBPF** (в Cilium), что добавляет контекстные переключения.

> [!tip] Собеседование
> **Вопрос:** Почему VXLAN использует UDP-4784/4785, а не IP протокол 47 (GRE/IP-in-IP)?
> **Ответ:** GRE не проходит через NAT без дополнительных настроек. UDP позволяет использовать стандартный NAT-traversal и позволяет middlebox (фаерволам, балансировщикам) видеть и фильтровать трафик на основе UDP-портов. Кроме того, UDP позволяет избежать deadlock'а при маршрутизации туннеля, который сам зависит от маршрутизации.
> 
> **Вопрос:** Как Go-сервису правильно обрабатывать ошибки при недоступности сегмента?
> **Ответ:** Использовать `context.WithTimeout` для `net.Dial` и `net/http.Client`. В Kubernetes это часто `networkpolicy` или `service-mesh` (см. [[42. Service Mesh. Istio, Linkerd, sidecar proxy.md]]), которые возвращают `ECONNREFUSED` или `503` на L4/L7. Go должен корректно парсить `net.OpError` и отличать `timeout` от `network unreachable`.

## Итог
1. **VLAN** — L2-сегментация через 802.1Q тег (4 байта, 4094 сети). Влияет на MTU и broadcast-домены.
2. **VXLAN** — L2 поверх L3 через UDP-4785. VNI (24 бита, ~16 млн сетей). Оверхед +50 байт. Требует Jumbo Frames для производительности.
3. **Go-реальность**: Пакет `net` абстрагирует сеть, но физические ограничения сегментации (MTU, маршрутизация, DNS) определяют поведение connection pooling и задержек.
4. **Инфраструктура**: В контейнеризированных средах сегментация реализуется через CNI и eBPF, что требует понимания `ip route` и `netlink` для диагностики.
