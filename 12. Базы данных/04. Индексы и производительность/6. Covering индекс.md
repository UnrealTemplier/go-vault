
Covering индекс (покрывающий индекс) — это индекс, который содержит **все столбцы**, необходимые конкретному запросу, прямо в своих листовых страницах. Когда такой индекс используется, СУБД может полностью удовлетворить запрос, не обращаясь к таблице (heap). Операция превращается в **Index Only Scan**: чтение только структуры индекса, без случайных переходов к страницам данных.

Это мощнейшая оптимизация: исчезает самый дорогой элемент выполнения — случайный доступ к таблице для получения недостающих столбцов. Если B-Tree индекс заменяет полный перебор таблицы на логарифмический поиск, то covering индекс доводит идею до абсолюта, вообще убирая таблицу из уравнения.

### Как это работает: данные в листьях

Обычный B-Tree индекс ([[2. B Tree индекс под капотом]]) хранит в листьях только индексированные столбцы и физический указатель на строку (ctid в PostgreSQL, первичный ключ в InnoDB). Когда запросу нужны столбцы, отсутствующие в индексе, база данных вынуждена для каждой подходящей строки выполнить дополнительное чтение таблицы — **heap fetch**. Это превращает последовательное чтение листьев индекса в хаотичный набор случайных доступов к разным страницам таблицы.

Covering индекс решает эту проблему, включая дополнительные столбцы либо как часть ключа, либо как неключевые (included) столбцы. Последний вариант предпочтительнее: столбцы включаются только в листья, не увеличивая размер ключа во внутренних узлах и не влияя на сортировку. Это сохраняет высокую степень ветвления (fanout) и минимизирует размер индекса.

В PostgreSQL с версии 11 можно создать индекс с неключевыми столбцами:

```sql
CREATE INDEX idx_users_covering ON users (email) INCLUDE (name, created_at);
```

Индекс будет отсортирован по `email`. В листьях, помимо ключа и ctid, будут храниться значения `name` и `created_at`. Запрос:

```sql
SELECT name, created_at FROM users WHERE email = 'alice@example.com';
```

Может быть выполнен как Index Only Scan: база пройдёт по B-Tree до листа, найдёт запись и вернёт `name` и `created_at` непосредственно из индекса, никогда не прикасаясь к таблице.

### Index Only Scan и visibility map

В PostgreSQL недостаточно просто наличия столбцов в индексе. Из-за MVCC ([[7. MVCC. Multi Version Concurrency Control]]) каждая строка таблицы имеет несколько версий; только одна из них видима текущей транзакции. Индекс хранит только физические ссылки на строки и не содержит информации о видимости. Поэтому при чтении индекса необходимо проверить, видима ли соответствующая версия строки.

Для этого PostgreSQL использует **карту видимости** (visibility map) — битовую карту, хранящуюся для каждой страницы таблицы. Бит в ней означает, что все строки на данной странице видимы всем текущим и будущим транзакциям (то есть страница не содержит устаревших версий, помеченных `VACUUM`). Если бит установлен, PostgreSQL может извлечь столбцы из индекса и вернуть результат, не посещая страницу таблицы. Если бит сброшен, необходимо выполнить heap fetch — прочитать страницу и проверить видимость строки по заголовкам версий.

> [!warning] Ловушка / Gotcha
> Index Only Scan может оказаться иллюзией, если таблица активно изменяется и visibility map не поддерживается в актуальном состоянии. В плане запроса будет указано `Heap Fetches: N` — это количество обращений к таблице, которые потребовались для проверки видимости. Если `Heap Fetches` близко к количеству возвращённых строк, выгода от покрывающего индекса сводится на нет. Регулярный `VACUUM` (для PostgreSQL) обязателен.

### InnoDB и покрывающие индексы

В MySQL/InnoDB архитектура иная. Кластерный индекс (первичный ключ) хранит **всю строку** в листьях B+Tree — таблица физически является индексом, и поиск по первичному ключу всегда покрывает все столбцы. Дополнительного covering индекса не нужно.

Вторичные индексы в InnoDB хранят в листьях ключ индекса и значение первичного ключа. Если все столбцы, нужные запросу, содержатся во вторичном индексе (включая столбцы, добавленные в индекс), выполняется **Using index** (покрывающий запрос). Если же нужны столбцы, которых в индексе нет, происходит **bookmark lookup**: сначала ищется вторичный индекс, извлекается первичный ключ, затем по нему ищется кластерный индекс для чтения полной строки. Это удваивает число B-Tree обходов.

```sql
-- В MySQL можно создать вторичный покрывающий индекс:
CREATE INDEX idx_email_name ON users (email, name);
-- При запросе SELECT name FROM users WHERE email = ? индекс покрывает оба столбца.
```

### Mechanical Sympathy: где выигрыш

С точки зрения железа, покрывающий индекс превращает Random I/O в Sequential I/O или вообще в чистую работу с памятью. Разберём на примере.

Без covering индекса запрос с фильтром по ключу выполняет:
- Спуск по B-Tree (2–3 случайных чтения страниц индекса).
- Чтение листовой страницы индекса (1 случайное чтение).
- Для каждой подходящей строки — чтение случайной страницы таблицы. Если строк много и они раскиданы по разным страницам, это множество случайных доступов с латенцией диска.

С covering индексом:
- Спуск по B-Tree (2–3 чтения).
- Чтение листовой страницы индекса и извлечение всех нужных данных. Никаких обращений к таблице.

Особенно ярко эффект проявляется при сканировании диапазонов. В покрывающем индексе листья обычно связаны в список, и база данных просто идёт по ним, последовательно считывая страницы. Последовательное чтение на SSD может давать пропускную способность в сотни мегабайт в секунду, в то время как случайное чтение — пару сотен мегабайт в лучшем случае и массу IOPS.

Кроме того, все страницы индекса, участвующие в запросах, активнее удерживаются в буферном кэше (shared_buffers в PostgreSQL, buffer pool в InnoDB), потому что они меньше по размеру и более востребованы. Это снижает количество физических чтений.

### Практический пример с Go

Представьте веб-сервис, где необходимо отображать список пользователей с email и именем, отсортированных по дате регистрации:

```go
// Без covering индекса:
// SELECT name, email FROM users WHERE active = true ORDER BY created_at DESC LIMIT 50;

// Создадим покрывающий индекс:
// CREATE INDEX idx_users_active_covering ON users (active, created_at) INCLUDE (name, email);
```

В Go коде запрос выполняется как обычно, но планировщик выберет Index Only Scan.

Проверить это можно через EXPLAIN:

```go
func debugQueryPlan(ctx context.Context, db *sql.DB, query string, args ...any) {
    rows, err := db.QueryContext(ctx, "EXPLAIN (ANALYZE, BUFFERS) "+query, args...)
    if err != nil {
        log.Printf("explain error: %v", err)
        return
    }
    defer rows.Close()
    for rows.Next() {
        var line string
        rows.Scan(&line)
        log.Printf("PLAN: %s", line)
    }
}
```

Вывод может содержать:
```
Index Only Scan using idx_users_active_covering on users (cost=0.42..8.50 rows=50 width=48)
  Index Cond: (active = true)
  Heap Fetches: 0
```

Строка `Heap Fetches: 0` подтверждает, что ни одного обращения к таблице не было. Идеально.

### Компромиссы и ловушки

> [!info] Под капотом
> Добавление столбцов через `INCLUDE` увеличивает размер листовых страниц индекса. Если добавляется столбец с данными переменной длины (text, varchar, jsonb), размер индекса может вырасти существенно. Это снижает количество записей в листе, и для того же объёма данных потребуется больше страниц, что может повысить высоту B-Tree (хотя included-столбцы не влияют на внутренние узлы) и увеличить потребление дискового пространства.

> [!warning] Ловушка / Gotcha
> - **Обновление included-столбцов:** любое изменение значения в таком столбце вызывает обновление индекса, даже если ключ не меняется. Это увеличивает накладные расходы на запись и генерирует дополнительную работу в WAL ([[8. WAL. Write Ahead Log]]). Если столбец часто обновляется, стоит взвесить, перевешивает ли выгода от Index Only Scan затраты на модификацию индекса.
> - **Устаревшая visibility map:** как упоминалось, без правильной настройки `autovacuum` и регулярного `VACUUM` покрывающий индекс может превратиться в обычный Index Scan с кучей heap fetches. Следите за `pg_stat_user_tables.n_dead_tup` и `last_vacuum`.
> - **Ошибочное ожидание покрытия:** индекс не покрывает столбцы, не включённые явно. Если вы меняете запрос, добавляете новый столбец в `SELECT`, покрывающий индекс может перестать работать, и запрос неожиданно замедлится.
> - **Размер индекса:** покрывающий индекс с десятком included-столбцов может занимать больше места, чем сама таблица. Следует мониторить размеры и сравнивать общее потребление памяти буферного кэша.

### Сравнение с альтернативами

Можно было бы создать составной индекс, где все столбцы входят в ключ: `CREATE INDEX ON users (email, name, created_at)`. Однако тогда `name` и `created_at` станут частью ключа: они будут влиять на сортировку и занимать место во всех узлах дерева, а не только в листьях. Степень ветвления упадёт, индекс станет шире и медленнее для поиска. `INCLUDE` решает эту проблему, сохраняя ключ компактным.

### Когда covering индекс необходим

- Частые запросы с узкой выборкой столбцов, где каждый лишний heap fetch критичен для задержки.
- Аналитические запросы, сканирующие большие диапазоны (например, отчёты за период), где sequential read индекса многократно быстрее случайных скачков по таблице.
- Таблицы, где обновления редки, а чтения доминируют.

### Заключение

Covering индекс — инженерный компромисс, смещающий нагрузку с дорогого случайного доступа к таблице на более дешёвое последовательное чтение индекса. Для высоконагруженных систем на Go, взаимодействующих с PostgreSQL, это один из самых эффективных способов уменьшить задержки и увеличить пропускную способность. Однако он требует внимания к обслуживанию (VACUUM) и к тому, какие именно столбцы включать.

В следующей статье мы рассмотрим ещё более избирательный подход — [[7. Partial индекс]], который индексирует не все строки таблицы, а только их часть по заданному условию, экономя место и ускоряя определённые запросы.
