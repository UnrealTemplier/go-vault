
Hash индекс — это структура, реализующая ассоциативный массив на диске. В отличие от B-Tree, отсортированного по ключам, хеш-индекс использует **хеш-функцию**, которая преобразует значение ключа в номер корзины (bucket), где хранятся указатели на строки. Такой подход обеспечивает теоретически мгновенный O(1) доступ для запросов точного равенства (`WHERE key = val`), но лишает индекс каких-либо способностей к упорядочению, диапазонному поиску и префиксному сравнению.

Если B-Tree похож на упорядоченную книгу с предметным указателем, то hash-индекс — это ячейка камеры хранения: вы знаете номер ячейки по значению ключа и сразу идёте к ней, но не можете получить «все ключи между A и B» или отсортированный список без полного обхода всех ячеек.

### Хеш-функция и коллизии

Хеш-функция отображает ключ произвольной длины в целое число фиксированной длины (обычно 32 или 64 бита). Идеальная хеш-функция распределяет ключи равномерно по всем корзинам. На практике возникают **коллизии** — два разных ключа дают одинаковый хеш. Способы разрешения коллизий:

- **Цепочки (chaining):** Каждая корзина содержит список или массив записей с одинаковым хешем. Либо используется отдельная страница (overflow page).
- **Открытая адресация (open addressing):** При коллизии ищется следующая свободная корзина в массиве по определённому закону (линейное, квадратичное пробирование).

Реализации в реляционных СУБД обычно используют расширяемое хеширование (extendible hashing) или линейное хеширование (linear hashing), позволяющие динамически менять количество корзин по мере роста данных, избегая массовых перестроек.

### Устройство Hash индекса в PostgreSQL

PostgreSQL поддерживает тип индекса `hash` с самой первой версии, но до PostgreSQL 10 он не был WAL-логируемым, что делало его опасным при сбоях. С версии 10 hash-индексы получили полноценную поддержку WAL ([[8. WAL. Write Ahead Log]]), и теперь они безопасны для репликации и восстановления, однако остаются менее функциональными, чем B-Tree.

Внутренне PostgreSQL hash индекс использует **расширяемое хеширование**. Ключевые структуры:

- **Мета-страница (meta page):** хранит общие параметры индекса, количество корзин, максимальное количество корзин и контрольные данные.
- **Страницы корзин (bucket pages):** каждая корзина — это страница, содержащая массив записей «хеш-код + ctid (физический идентификатор строки)». Страница может вмещать десятки записей.
- **Страницы переполнения (overflow pages):** если корзина переполняется, к ней цепляется дополнительная страница, формируя цепочку. Это аналог метода цепочек.

Хеш-функция (по умолчанию `hashint4`, `hashtext` и т.д.) вычисляет 32-битный хеш. Номер корзины получается взятием остатка от деления хеша на количество корзин (маскируется по битовым операциям, так как количество корзин — степень двойки). При расширении количество корзин удваивается, и данные перераспределяются постепенно (одна корзина расщепляется за раз, что амортизирует затраты).

> [!info] Под капотом
> В исходном коде PostgreSQL (файлы `src/backend/access/hash/`) мета-страница описывается структурой `HashMetaPageData`. Количество корзин хранится в поле `hashm_maxbucket`. Функция `_hash_hashkey2bucket` определяет номер корзины по хеш-коду, используя битовую маску. Страница корзины содержит массив элементов типа `HashItem`, каждый из которых хранит `hashkey` и `ItemPointerData` — ссылку на строку в таблице (`ctid`).

Процесс поиска по hash-индексу:

1. Вычисляется хеш ключа.
2. По хеш-коду определяется номер корзины (через битовую маску или деление по модулю).
3. Читается соответствующая страница корзины (возможно, с переполнениями).
4. Внутри корзины выполняется последовательный или бинарный поиск по хеш-кодам (так как все записи в корзине имеют одинаковый хеш, поиск по точному ключу требует ещё и сравнения самих ключей для устранения коллизий).
5. По `ctid` читается строка таблицы (или возвращается при index-only scan, если все данные есть в индексе — но hash индексы обычно не хранят полные ключи явным образом, поэтому Index Only Scan для hash почти невозможен; требуется проверка видимости в таблице).

### Adaptive Hash Index в MySQL/InnoDB

InnoDB не позволяет пользователю создавать hash-индексы как самостоятельные объекты (команда `CREATE INDEX ... USING HASH` на самом деле создаёт B-Tree). Однако внутри InnoDB автоматически строит **Adaptive Hash Index (AHI)** — интеллектуальный хеш поверх часто используемых B-Tree индексов.

Когда InnoDB замечает, что определённые страницы B-Tree часто запрашиваются по точному ключу (например, повторяющиеся `SELECT ... WHERE pk = ?`), он строит в памяти хеш-таблицу, отображающую ключ напрямую в leaf page и запись, минуя спуск по B-Tree. Это радикально ускоряет частые точечные запросы. AHI полностью прозрачен для пользователя и сбрасывается при перезапуске или при активном изменении данных. Настройки: `innodb_adaptive_hash_index`.

### Mechanical Sympathy: Hash индекс и память

Хеш-индекс предельно прост с точки зрения CPU: вычисление хеша — это несколько арифметических операций (например, `hashlittle()` или murmurhash). Затем битовая операция нахождения номера корзины и чтение одной-двух страниц. Поскольку количество корзин обычно велико, вероятность попасть в переполненную корзину мала, и средняя длина цепочек близка к 1. Это означает 1 случайное чтение для поиска (и, возможно, ещё одно для таблицы).

Сравним с B-Tree из статьи [[2. B Tree индекс под капотом]]: там нужно 3–4 случайных чтения (корень, внутренние узлы, лист). Таким образом, hash-индекс даёт меньшее число дисковых операций для точного равенства. Но для сканирования диапазона hash-индекс потребовал бы полного обхода всех корзин, что многократно дороже B-Tree.

С точки зрения кэша CPU, в отличие от B-Tree, где ключи в странице отсортированы и бинарный поиск "прыгает" по массиву (вызывая промахи кэша), hash-индекс внутри корзины может хранить хеш-коды и ctid без сортировки, требуя линейного сканирования всей страницы. Но так как страница обычно 8 КБ и целиком загружается в кэш L1/L2, линейный обход может быть эффективнее бинарного поиска по случайным адресам (благодаря предсказанию переходов и конвейеризации). Однако при длинных цепочках переполнений случайный доступ становится преобладающим.

### Ограничения и ловушки

> [!warning] Ловушка / Gotcha
> - **Нет порядка.** Hash индекс не может ускорить `ORDER BY`, `MIN`, `MAX`, `BETWEEN`. Планировщик проигнорирует такой индекс для этих запросов.
> - **Префиксный поиск невозможен.** Запрос `WHERE col LIKE 'abc%'` или `WHERE col >= 'abc'` никогда не будет использовать hash-индекс.
> - **Размер индекса может расти неконтролируемо.** Хеш-индексы обычно занимают больше места, чем B-Tree, из-за резервирования пространства в корзинах под будущие вставки и страниц переполнения. Fillfactor у hash индекса может быть ниже (скажем, 75%).
> - **Не подходит для высоко-конкурентных вставок с уникальными ключами.** В отличие от B-Tree, где вставки в основном попадают в разные листья при случайных ключах, hash-индекс при росте активно расщепляет корзины, вызывая длительные блокировки при перестроении.
> - **Редко используется на практике.** В большинстве случаев B-Tree даёт почти такую же скорость для точечных запросов (благодаря кэшированию и малой высоте), плюс поддерживает диапазоны. Поэтому hash-индексы обычно выбираются только тогда, когда таблица гигантская, запросы строго на равенство и с частотой, где экономия одного случайного чтения критична.

> [!tip] Собеседование
> **Вопрос:** Почему PostgreSQL hash индекс не может участвовать в Index Only Scan?
> **Ответ:** Потому что hash индекс не хранит сами значения ключа — только их хеш. Даже если бы ключ хранился, из-за возможных коллизий и отсутствия возможности проверить видимость по MVCC ([[7. MVCC. Multi Version Concurrency Control]]) без обращения к таблице, index-only scan не применим. Всегда требуется заход в таблицу для подтверждения.

### Hash индекс в Go-разработке

В ORM или библиотеках для работы с PostgreSQL обычно не предусмотрено специального API для создания hash-индексов; вы пишете сырой SQL: `CREATE INDEX idx_whatever ON table USING hash (column)`. Затем при выполнении запросов с точным равенством PostgreSQL автоматически рассмотрит hash-индекс в плане выполнения. При анализе производительности через `EXPLAIN` вы увидите `Index Scan using idx_whatever` без указания направления сканирования (т.к. нет направления).

В Go коде вы бы могли использовать hash-таблицу в памяти (встроенную `map`) для локального кэша, схожего по семантике. Но нужно помнить, что `map` в Go не сохраняет порядок вставки и не поддерживает упорядоченный обход — ровно те же ограничения. Эффективная работа с hash-индексом в БД подразумевает понимание его внутреннего устройства и умение выбрать правильный тип индекса под нагрузку.

### Когда же применять hash индекс

- Таблица огромна, и даже 3 случайных чтения B-Tree вызывают проблемы с latency.
- Запросы исключительно по точному равенству (например, поиск по UUID или токену).
- Нет нужды в диапазонных запросах или сортировке.
- Данные относительно статичны (мало вставок/обновлений), чтобы избежать частых расщеплений.

В реальных системах hash-индексы встречаются редко, чаще используют B-Tree или более специализированные структуры. Но понимание их принципов необходимо для общей картины.

### Альтернативы в других СУБД

- **MySQL/MariaDB:** единственный способ получить hash-индекс — использовать таблицы с движком MEMORY (HEAP), где можно указать `USING HASH`. InnoDB даёт только Adaptive Hash Index без контроля пользователя.
- **SQLite:** не поддерживает hash индекс.
- **Oracle:** поддерживает hash clusters — группы данных, физически размещённые по хешу, но это другой механизм.
