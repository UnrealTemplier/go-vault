
В предыдущей статье [[11. Cost based optimizer]] мы увидели, что оптимизатор выбирает план на основе стоимости, а стоимость напрямую зависит от оценок количества строк — **cardinality**. Если оптимизатор ошибается в cardinality на порядок, он может выбрать план, который будет выполняться в сотни раз дольше, чем альтернативный.

Cardinality вычисляется по статистике, собираемой с таблиц и индексов. Понимание того, как эта статистика устроена, как собирается и как используется, — обязательная компетенция инженера, проектирующего высоконагруженные системы. В этой статье мы разберём статистику PostgreSQL изнутри, её влияние на производительность и способы поддержания её актуальности.

### Что такое cardinality

**Cardinality** (кардинальность) — оценка числа строк, которое вернёт конкретная операция (сканирование таблицы, фильтр, соединение). В плане запроса она отображается как `rows` (ожидаемое) и `actual rows` (фактическое). Именно её точность определяет, выберет ли оптимизатор Index Scan вместо Seq Scan, Nested Loop вместо Hash Join, и какой порядок соединений будет оптимальным.

Если реальная cardinality равна 10, а оптимизатор думает, что 1 000 000, он может выбрать Hash Join и занять гигабайт памяти для хеш-таблицы, хотя Nested Loop отработал бы мгновенно. И наоборот, недооценка cardinality ведёт к выбору Nested Loop с миллионами итераций там, где Hash Join был бы эффективен.

### Как PostgreSQL собирает статистику

Статистику собирает команда `ANALYZE`. Она запускается:

- **Вручную:** `ANALYZE table_name;`
- **Автоматически:** процессом `autovacuum`, который следит за долей изменённых строк (`autovacuum_analyze_scale_factor`, по умолчанию 10%) и минимальным порогом (`autovacuum_analyze_threshold`, по умолчанию 50).

При выполнении `ANALYZE` PostgreSQL не читает всю таблицу — это было бы слишком дорого для гигантских объёмов. Вместо этого берётся **случайная выборка** строк. Размер выборки определяется параметром `default_statistics_target` (по умолчанию 100), умноженным на коэффициент 300. То есть по умолчанию считывается до 30 000 случайных строк таблицы. Для столбцов можно установить индивидуальный target:

```sql
ALTER TABLE events ALTER COLUMN payload SET STATISTICS 1000;
```

Это увеличит размер выборки для `payload` до 300 000 строк, что даст более детальные MCV и гистограммы.

> [!info] Под капотом
> В коде PostgreSQL за сбор статистики отвечает модуль `src/backend/commands/analyze.c`. Функция `do_analyze_rel()` вызывает `acquire_sample_rows()`, которая выбирает случайные страницы и строки из таблицы. Для индексов статистика не собирается заново — они используют статистику таблицы и собственную `reltuples`/`relpages`, обновляемую при `ANALYZE` и операциях.

### Хранение статистики: pg_statistic и pg_stats

Статистика хранится в системной таблице `pg_statistic`, но напрямую читать её неудобно, потому что столбцы содержат массивы `anyarray`. Вместо этого используют представление `pg_stats`, которое приводит данные к понятному виду.

Ключевые столбцы `pg_stats` для каждого столбца таблицы:

- `null_frac` — доля строк с NULL.
- `avg_width` — средняя ширина значения в байтах.
- `n_distinct` — количество уникальных значений. Может быть положительным целым, или отрицательным (тогда абсолютное значение — доля уникальных, например -0.1 означает, что уникальны 10% строк).
- `most_common_vals` — массив наиболее частых значений (MCV).
- `most_common_freqs` — массив частот этих значений.
- `histogram_bounds` — массив границ гистограммы, покрывающей значения, не входящие в MCV.
- `correlation` — коэффициент корреляции между физическим порядком строк в таблице и значениями столбца (от -1 до 1). Используется для оценки стоимости Index Scan.

> [!tip] Собеседование
> **Вопрос:** Что означает `n_distinct = -1` и когда такое бывает?
> **Ответ:** `n_distinct = -1` означает, что все значения в столбце уникальны (число уникальных равно числу строк). Это эквивалентно `n_distinct = reltuples`. Используется, когда столбец в основном уникален, и оценщик кардинальности строит на этом предположение.

### Основные статистические показатели

#### Доля NULL
Используется для оценки селективности `IS NULL` и `IS NOT NULL`. Например, если `null_frac = 0.2`, то ожидаемое число строк с `IS NULL` = `reltuples × 0.2`.

#### Средняя ширина
Влияет на оценки использования памяти для сортировок и хеш-таблиц, а также на `width` в плане запроса.

#### Количество уникальных значений (n_distinct)
Напрямую определяет селективность равенства: `1 / n_distinct`, если значение не находится в MCV. Для `WHERE status = 'active'` при `n_distinct = 5` селективность равна 0.2.

#### Наиболее частые значения (MCV)
Массивы `most_common_vals` и `most_common_freqs` перечисляют самые популярные значения и их частоты. Если фильтр по значению, найденному в MCV, селективность равна частоте из `most_common_freqs`. Например, `status = 'active'` с частотой 0.75 даёт селективность 0.75. Остальные значения, не вошедшие в MCV, обрабатываются по `n_distinct` и гистограммам.

Размер MCV ограничен `statistics_target` (по умолчанию 100 элементов для `default_statistics_target = 100`). Если уникальных значений больше, остаток моделируется гистограммой.

#### Гистограммы
Массив `histogram_bounds` содержит `statistics_target + 1` границу, разбивающую значения столбца, не попавшие в MCV, на равные по числу элементов корзины. Это позволяет оценивать селективность диапазонов (`>`, `<`, `BETWEEN`). Например, для `WHERE created_at > '2025-01-01'` оптимизатор находит, в какую корзину попадает граница, и линейно интерполирует долю.

### Вычисление селективности

Для простого условия селективность (`selectivity`) — это доля строк, удовлетворяющих предикату. Оптимизатор вычисляет её так:

1. **Равенство (`=`, `!=`):** Если значение в MCV — частота из `most_common_freqs`. Иначе `(1 - sum(most_common_freqs)) / (n_distinct - len(MCV))`. Для `!=` — дополнение до 1.
2. **Диапазон (`>`, `<`, `BETWEEN`):** Используется гистограмма. Например, для `> const` определяется, сколько корзин полностью лежат правее, и линейно интерполируется неполная корзина.
3. **`LIKE 'prefix%'`:** Оценивается как диапазон по гистограмме с учётом префикса.
4. **`IN` и `OR`:** Комбинируются селективности с предположением независимости (что часто ошибочно).

> [!warning] Ловушка / Gotcha
> При комбинации нескольких условий (`WHERE a = 1 AND b = 2`) селективности перемножаются, если нет extended statistics. При сильной корреляции это даёт серьёзную недооценку или переоценку cardinality. Например, если `country` и `city` жёстко связаны, произведение селективности `country='RU'` и `city='Moscow'` может оказаться намного ниже реальной доли.

### Проблемы и искажения

**Устаревшая статистика.** После массовых вставок, обновлений или удалений `reltuples`, `relpages` и `pg_statistic` не обновляются до следующего `ANALYZE`. Это может привести к выбору Seq Scan на миллиардной таблице, которая, по мнению оптимизатора, всё ещё содержит 1000 строк. Решение — настроить агрессивный `autovacuum` или запускать `ANALYZE` принудительно после ET L-загрузок.

**Неравномерное распределение.** Если данные сильно неравномерны (например, несколько ключей занимают 99% таблицы), MCV и гистограммы должны это отразить. Но если `statistics_target` мал, частые значения могут не попасть в MCV, искажая оценки.

**Коррелированные столбцы.** Как упоминалось, предположение независимости — главный источник ошибок. Extended Statistics (PostgreSQL 10+) решает эту проблему: можно создать объект статистики, который собирает зависимости между столбцами.

```sql
CREATE STATISTICS s_city_country (dependencies) ON country, city FROM users;
ANALYZE users;
```

Теперь оптимизатор будет знать, что знание `country` существенно меняет распределение `city`, и не станет перемножать селективности наивно.

**Extended Statistics** также поддерживают:
- `ndistinct` — многоколоночное число уникальных значений (для группировок).
- `mcv` — многоколоночные MCV (собирает частоты комбинаций значений).

> [!info] Под капотом
> Extended Statistics хранятся в `pg_statistic_ext` и `pg_statistic_ext_data`. При `ANALYZE` собирается дополнительная выборка, вычисляются зависимости или частоты комбинаций. Это увеличивает время сбора статистики, особенно при большом числе колонок.

### Mechanical Sympathy: цена сбора статистики

`ANALYZE` читает выборку строк, что требует случайных или последовательных чтений страниц. Для таблиц в десятки гигабайт и `statistics_target = 1000` выборка может составить несколько сотен тысяч страниц, что вызывает значительный ввод-вывод и может повлиять на работающее приложение. Процесс `autovacuum` старается быть throttled (ограниченным), но при `ANALYZE` без ограничений (например, ручной) может занять заметное время и CPU на сортировку выборки.

Также само хранение расширенной статистики занимает память в `pg_statistic_ext_data`, и при большом количестве таких объектов может вырасти потребление разделяемой памяти (shared memory) планировщиком.

С точки зрения Go-разработчика: не вызывайте `ANALYZE` на каждый запрос; анализируйте периодически, ориентируясь на темп изменений данных. Используйте `ANALYZE` точечно после крупных миграций или загрузок.

### Практические рекомендации для Go-разработчика

**Мониторинг статистики.** Можно периодически запрашивать `pg_stats` и сравнивать с реальностью. Например, запрос:

```sql
SELECT relname, n_tup_ins, n_live_tup, last_analyze
FROM pg_stat_user_tables
WHERE last_analyze < now() - interval '1 day';
```

Покажет таблицы, давно не анализировавшиеся.

**Принудительный ANALYZE.** В Go-приложении, управляющем процессом ETL, можно вызвать `ANALYZE` после окончания загрузки:

```go
import "github.com/jackc/pgx/v5"

func analyzeTable(ctx context.Context, conn *pgx.Conn, tableName string) error {
    _, err := conn.Exec(ctx, fmt.Sprintf("ANALYZE %s", tableName))
    return err
}
```

**Проверка расхождений.** Сравнивайте ожидаемые и фактические строки в планах медленных запросов. Это можно автоматизировать, парся вывод `EXPLAIN (ANALYZE, FORMAT JSON)`:

```go
type PlanRow struct {
    Plan struct {
        RowsEstimated float64 `json:"Plan Rows"`
        ActualRows    float64 `json:"Actual Rows"`
    } `json:"Plan"`
}
// Вычисляем коэффициент расхождения, логируем при > 10x.
```

**Extended Statistics.** Если в приложении часты запросы с коррелированными условиями, создайте объекты extended statistics. Это можно сделать через миграции:

```go
// Пример миграции на Go-миграторе
migrator.Migrate(`CREATE STATISTICS s_users_role_tenant ON role, tenant_id FROM users`)
```

### Заключение

Кардинальность — это основа, на которой оптимизатор строит планы. Статистика, собираемая `ANALYZE`, переводит реальные данные в числовые оценки, позволяя Cost-Based Optimizer принимать верные решения. Но статистика требует заботы: настройки частоты сбора, размера выборки, учета корреляций. Для инженера на Go важно не только полагаться на автоматический `autovacuum`, но и активно мониторить здоровье статистики, реагировать на расхождения и понимать, как манипуляции со статистикой влияют на поведение продакшен-системы.
