
В предыдущих статьях мы разобрали, как InnoDB хранит страницы в памяти ([[2. InnoDB storage engine]]) и как ускоряет поиск с помощью B+ деревьев ([[3. Индексы в MySQL]]). Но базы данных редко работают в однопоточном режиме. В реальном мире ваш Go-бэкенд открывает сотни соединений к СУБД, и тысячи горутин пытаются одновременно читать и изменять одни и те же строки.

Чтобы этот хаос не привел к разрушению данных, существуют **Транзакции**. Транзакция переводит базу из одного консистентного состояния в другое. Если что-то идет не так — всё откатывается. 

Сегодня мы разберем, как транзакции работают в MySQL (InnoDB), почему дефолтное поведение MySQL отличается от стандартов и почему слепая вера в уровни изоляции может положить ваш production.

## ACID: Фундамент транзакций

Любая надежная СУБД обещает выполнение четырех свойств ACID:

1. **Atomicity (Атомарность):** «Всё или ничего». Если вы списываете деньги с одного счета и зачисляете на другой, обе операции либо выполнятся, либо откатятся. В InnoDB за это отвечает **Undo Log** (лог отката), хранящий старые версии строк.
2. **Consistency (Консистентность):** Транзакция не может нарушить правила БД (внешние ключи, уникальные индексы). Если уникальный индекс `idx_email` нарушен, вся транзакция отклоняется.
3. **Isolation (Изоляция):** Параллельные транзакции не должны влиять друг на друга. За это отвечают механизмы **Блокировок (Locking)** и **MVCC (Multi-Version Concurrency Control)**. Это самая сложная и настраиваемая часть.
4. **Durability (Долговечность):** Если клиент получил `COMMIT`, данные не пропадут даже при отключении электричества. За это отвечает **Redo Log (WAL)** и системные вызовы `fsync` к ОС.

## Уровни изоляции: Баланс между производительностью и аномалиями

Стандарт SQL-92 определяет четыре уровня изоляции. В MySQL (InnoDB) они реализованы со своими нюансами.

### 1. Read Uncommitted (RU)
Транзакция видит не закоммиченные изменения других транзакций. 
* **Аномалия:** *Dirty Read (Грязное чтение)*. Вы прочитали баланс клиента (1000), который другая транзакция только что обновила, но еще не закоммитила. Затем та транзакция упала и откатилась (Rollback). Вы работаете с фантомными деньгами, которых никогда не существовало.
* **Применение:** Практически никогда не используется на практике.

### 2. Read Committed (RC)
Транзакция видит только закоммиченные данные. Грязного чтения нет.
* **Как работает в InnoDB:** При каждом отдельном `SELECT`-запросе внутри транзакции InnoDB создает новый **Read View** (снимок видимости). Он смотрит в Undo Log и отфильтровывает строки, `DB_TRX_ID` которых принадлежит еще не завершенным транзакциям.
* **Аномалия:** *Non-repeatable Read (Неповторяющееся чтение)*. Вы делаете `SELECT` баланса (100). Затем другая транзакция делает `UPDATE` баланса на 50 и коммитит. Вы делаете тот же `SELECT` в своей транзакции и получаете 50. Данные изменились прямо в процессе вашей работы.
* **Интересный факт:** Это уровень изоляции по умолчанию в PostgreSQL и Oracle.

### 3. Repeatable Read (RR) — По умолчанию в MySQL
Транзакция видит мир таким, каким он был на момент её старта (или на момент первого `SELECT`).
* **Как работает в InnoDB:** **Read View** создается *один раз* при первом чтении и сохраняется до конца транзакции. Все последующие `SELECT` используют этот же снимок, читая старые версии строк из Undo Log. Вы всегда будете получать один и тот же результат.

> [!tip] Собеседование: Фантомное чтение и магия InnoDB
> **Вопрос:** Стандарт SQL-92 гласит, что на уровне Repeatable Read возможно *Phantom Read (Фантомное чтение)* — когда вы делаете `SELECT ... WHERE age > 18`, получаете 10 строк, затем кто-то вставляет 11-ю строку, и повторный `SELECT` вернет 11 строк. Происходит ли это в MySQL?
> **Ответ:** **НЕТ.** InnoDB защищает от фантомных чтений на уровне Repeatable Read. Для обычных `SELECT` это делается через MVCC (новая строка игнорируется, так как ее ID транзакции больше, чем в нашем Read View). Для блокирующих чтений (`SELECT ... FOR UPDATE`) это делается с помощью **Next-Key Locks**.

### 4. Serializable
Самый строгий уровень. Транзакции выполняются так, как будто они идут строго последовательно, одна за другой. 
В InnoDB это достигается тем, что СУБД неявно превращает *все* обычные `SELECT` в `SELECT ... FOR SHARE`, вешая разделяемые блокировки (Shared Locks) на все прочитанные строки и диапазоны.
* **Применение:** Крайне редко. Убивает конкурентность и превращает базу в узкое горлышко.

## Под капотом: Блокировки и Next-Key Locks

Чтобы понять, почему MySQL в режиме Repeatable Read не допускает фантомов, нужно заглянуть на уровень структуры B+ дерева.

Когда вы делаете `SELECT * FROM users WHERE id BETWEEN 10 AND 20 FOR UPDATE`, InnoDB блокирует не только существующие строки (Record Locks). Она также блокирует **пустоты (Gaps)** между индексами, чтобы никто не смог вставить туда новые данные.
Комбинация блокировки самой строки и промежутка перед ней называется **Next-Key Lock**.

```mermaid

    Node10["Index: 10"]
    Node15["Index: 15"]
    Node22["Index: 22"]

    Node7 -.->|"Gap Lock<br>Вставка 8,9 запрещена"| Node10
    Node10 -.->|"Gap Lock<br>Вставка 11-14 запрещена"| Node15
    Node15 -.->|"Gap Lock<br>Вставка 16-21 запрещена"| Node22
    
    style Node10 fill:#f96,stroke:#333,stroke-width:2px
    style Node15 fill:#f96,stroke:#333,stroke-width:2px
```

> [!warning] Ловушка / Gotcha: Deadlocks из-за Gap Locks
> Это архитектурная боль MySQL по умолчанию. Gap блокировки в Repeatable Read — главная причина взаимных блокировок (Deadlocks) в высоконагруженных системах. 
> Если две транзакции одновременно попытаются заблокировать пересекающиеся «пустоты» перед вставкой, они могут заблокировать друг друга.
> **Решение Senior-уровня:** Многие крупные компании (включая GitHub, Shopify и Booking) намеренно переводят MySQL на уровень **Read Committed** (`SET GLOBAL tx_isolation = 'READ-COMMITTED'`). На этом уровне InnoDB **отключает Gap Locks** для большинства операций, оставляя только блокировки конкретных строк (Row Locks). Это радикально снижает вероятность дедлоков за счет того, что мы миримся с возможными фантомами (которые в реальной бизнес-логике редко являются проблемой, если мы используем оптимистичные блокировки).

## Идиоматичный Go: Управление транзакциями

В Go пакет `database/sql` предоставляет отличные абстракции для работы с транзакциями, но требует дисциплины. Пул соединений в Go устроен так, что транзакция жестко привязывается к одному конкретному соединению (TCP-сокету).

Если вы забудете сделать `Commit` или `Rollback`, это соединение останется «висеть» в пуле в зависшем состоянии транзакции. Следующая горутина, которая возьмет этот коннект, получит непредсказуемое поведение или ошибку таймаута на локах БД.

```go
package repository

import (
	"context"
	"database/sql"
	"fmt"
)

func UpdateOrderStatus(ctx context.Context, db *sql.DB, orderID int, newStatus string) error {
	// Явно указываем уровень изоляции, если он отличается от глобального в MySQL
	opts := &sql.TxOptions{
		Isolation: sql.LevelReadCommitted, // Спасаемся от Gap Locks
		ReadOnly:  false,
	}

	tx, err := db.BeginTx(ctx, opts)
	if err != nil {
		return fmt.Errorf("begin tx: %w", err)
	}

	// КРИТИЧЕСКОЕ ПРАВИЛО: Всегда делаем defer tx.Rollback()
	// Если транзакция будет успешно закоммичена ниже, tx.Rollback()
	// безопасно вернет ошибку sql.ErrTxDone, которую defer проигнорирует.
	// Это гарантирует освобождение соединения при паниках или ранних return.
	defer tx.Rollback()

	// Используем ExecContext с переданным ctx. 
	// Если клиент отвалится (timeout/cancel), Go пошлет сигнал отмены в БД
	res, err := tx.ExecContext(ctx, "UPDATE orders SET status = ? WHERE id = ?", newStatus, orderID)
	if err != nil {
		return fmt.Errorf("update status: %w", err)
	}

	affected, _ := res.RowsAffected()
	if affected == 0 {
		return fmt.Errorf("order %d not found", orderID)
	}

	// Если мы дошли сюда, фиксируем изменения.
	// Это вызовет fsync(Redo Log) на стороне MySQL.
	if err := tx.Commit(); err != nil {
		return fmt.Errorf("commit tx: %w", err)
	}

	return nil
}
```

> [!info] Под капотом: Что происходит при `tx.Begin()`?
> Драйвер `go-sql-driver/mysql` берет свободное соединение из пула. Если `TxOptions` требует специфичного уровня изоляции, драйвер отправляет команду `SET TRANSACTION ISOLATION LEVEL ...`. Затем он отправляет команду `START TRANSACTION`. С этого момента данный тред MySQL (вспоминаем Thread-per-Connection из [[1. Архитектура MySQL]]) привязан к вашей горутине, пока вы не вызовете `Commit` или `Rollback`.

## Итог

1. Транзакции обеспечивают **ACID**, где InnoDB блистает своими механизмами Redo Log (Durability) и Undo Log (Atomicity).
2. MySQL по умолчанию использует уровень **Repeatable Read**, который, в отличие от стандарта, защищает еще и от фантомных чтений.
3. Защита от фантомов достигается дорогими **Next-Key Locks (Gap + Row lock)**, что часто приводит к дедлокам при высокой конкурентности. Снижение изоляции до **Read Committed** — классический паттерн оптимизации бэкенда.
4. В Go всегда используйте `defer tx.Rollback()` для защиты пула соединений от "осиротевших" транзакций.

Когда данные консистентны в рамках одного сервера, возникает новая проблема. Сервер может сгореть, или на него может обрушиться трафик, с которым один инстанс не справится. Нам нужно масштабироваться. В следующей статье мы разберем, как MySQL передает свои закоммиченные транзакции на другие узлы: [[5. Репликация в MySQL]].
