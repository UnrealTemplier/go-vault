
Если в вашем проекте появилась задача "Сделать поиск по статьям/товарам/пользователям", первая (и самая наивная) мысль бэкендера — использовать оператор `LIKE '%query%'`. 

Но `LIKE` обладает двумя фатальными недостатками:
1. **Производительность $O(N)$:** `LIKE '%...'` не может использовать классический B-Tree индекс, так как паттерн начинается с wildcard. Базе придется делать Sequential Scan, вычитывая каждую страницу с диска.
2. **Отсутствие лингвистики:** `LIKE` ищет точное совпадение подстроки. Запрос `LIKE '%бег%'` не найдет слово "бежал", а запрос `LIKE '%apple%'` не найдет "apples".

Вторая мысль — "Нужно поднимать ElasticSearch!". Это классический overengineering. ElasticSearch — мощный инструмент, но он приносит в систему проблему **распределенной консистентности** (данные в БД и в Elastic могут рассинхронизироваться), требует отдельного кластера, мониторинга и JVM-тюнинга.

В 95% случаев для полнотекстового поиска (Full Text Search, FTS) более чем достаточно встроенных механизмов PostgreSQL. И работают они невероятно быстро, если понимать их подкапотную механику.

## Типы данных: tsvector и tsquery

В основе FTS лежат два специализированных типа данных, которые преобразуют текст в математические множества для поиска.

### 1. `tsvector` (Документ)
База не ищет по сырому тексту. Текст предварительно нормализуется и превращается в `tsvector` — отсортированный список уникальных **лексем** (lexemes) с указанием их позиций в исходном тексте.

```sql
SELECT to_tsvector('russian', 'Мальчик быстро бежал по улице, пока другие мальчики играли');
-- Результат: 'бежа':3 'быстр':2 'друг':7 'игра':9 'мальчик':1,8 'пок':6 'улиц':5
```

> [!info] Под капотом: Пайплайн обработки текста
> Когда вы вызываете `to_tsvector`, текст проходит через жесткий конвейер в C-рантайме PostgreSQL:
> 1. **Парсер (Parser)**: Разбивает текст на токены (слова, числа, email, URL). Пунктуация выбрасывается.
> 2. **Словари (Dictionaries)**: Токены прогоняются через словари. 
>    - Сначала применяются словари **Stop-words** (стоп-слова). Предлоги, союзы ("по", "пока") просто удаляются, так как не несут смысловой нагрузки для поиска.
>    - Затем применяется словарь **Snowball (Стеммер)**. Он отбрасывает окончания и суффиксы, оставляя неизменяемый корень слова (лексему). Именно поэтому "мальчик" и "мальчики" превратились в одну лексему `'мальчик'`.

```mermaid
flowchart LR
    Text["Сырой текст<br>VARCHAR / TEXT"] -->|1 - Парсер| Tokens["Токены<br>Разбивка по пробелам"]
    Tokens -->|2 - Stop words| Filtered["Отфильтрованные<br>Без предлогов"]
    Filtered -->|3 - Stemming| Lexemes["Лексемы<br>Корни слов"]
    Lexemes --> TSV["tsvector<br>Бинарное множество"]
```

### 2. `tsquery` (Запрос)
Точно так же, как текст превращается в вектор, поисковый запрос пользователя должен быть превращен в `tsquery`. Это не просто строка, это дерево булевой логики.

В `tsquery` вы можете использовать операторы `&` (AND), `|` (OR), `!` (NOT) и `<->` (поиск фраз с учетом расстояния).

```sql
SELECT to_tsquery('russian', 'мальчик & (бег | игра)');
-- Результат: 'мальчик' & ( 'бег' | 'игр' )
```

Поиск осуществляется с помощью оператора совпадения `@@`:
```sql
SELECT to_tsvector('russian', 'Мальчик играл') @@ to_tsquery('russian', 'мальчик & игра');
-- Результат: true
```

## Индексация: GIN под капотом

Вычислять `to_tsvector` "на лету" при каждом `SELECT` — это убийство CPU. Текст нужно проиндексировать. 

Здесь на сцену выходит наш старый знакомый из статьи [[5. JSONB и работа с JSON]] — **GIN (Generalized Inverted Index)**.
Если B-Tree строится для целого значения, то GIN "выворачивает" документ наизнанку. Он берет каждую уникальную лексему из `tsvector` и создает список `CTID` (ссылок на физические строки в Куче), где эта лексема встречается.

> [!warning] Ловушка / Gotcha: Генерация tsvector
> Часто разработчики создают индекс прямо по функции:
> `CREATE INDEX idx_fts ON articles USING GIN (to_tsvector('russian', content));`
> Это рабочий, но хрупкий паттерн. Если локаль базы изменится, индекс может сломаться. 
> 
> **Production-ready подход** — использовать Generated Columns (появились в PG 12). Это отдельная колонка, которая автоматически пересчитывается при вставке/обновлении строки и хранится физически.

```sql
ALTER TABLE articles 
ADD COLUMN search_vector tsvector 
GENERATED ALWAYS AS (to_tsvector('russian', title || ' ' || content)) STORED;

CREATE INDEX idx_articles_search ON articles USING GIN (search_vector);
```
Теперь, при запросе `WHERE search_vector @@ to_tsquery('...')`, Postgres мгновенно найдет нужные `CTID` в GIN-индексе, не касаясь диска для чтения таблиц.

## Ранжирование (Ranking): Кто сверху?

Найти документы — полдела. Нужно отсортировать их по релевантности (как в Google). Для этого в Postgres есть функция `ts_rank` (и её вариация `ts_rank_cd`, учитывающая расстояние между словами).

Функция принимает документ (`tsvector`) и запрос (`tsquery`), и возвращает `float4` (чем больше, тем релевантнее).

```sql
SELECT id, title, ts_rank(search_vector, query) as rank
FROM articles, to_tsquery('russian', 'горутины & каналы') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 10;
```

### Mechanical Sympathy: Почему Ranking тормозит?

> [!tip] Собеседование
> **Вопрос:** Вы сделали полнотекстовый поиск с GIN-индексом. Сам поиск (`WHERE`) работает за 2 мс. Но как только вы добавляете `ORDER BY ts_rank(...)`, запрос начинает выполняться 500 мс. Почему?
> **Ответ:** Это фундаментальная архитектурная особенность GIN. Индекс GIN знает *в каких строках* есть слова. Но он **не хранит информацию о частоте слов и их позициях** (это заняло бы слишком много места в RAM).
> Чтобы вычислить `ts_rank`, база должна пойти в Кучу (Heap), прочитать 8-килобайтную страницу для *каждой* найденной строки (даже если их 100 000), извлечь оттуда сам `tsvector` и выполнить математику. Это порождает чудовищный Random I/O.
> **Решение:** Ограничивать выборку до ранжирования. Сначала найти топ-1000 строк (например, отфильтровав по дате или категории с помощью комбинированных индексов), а уже их ранжировать через подзапрос (CTE).

## Интеграция с Go

При написании бэкенда на Go возникает частая проблема: пользователь вводит в строку поиска "iphone 15 pro". Если вы передадите это напрямую в `to_tsquery('iphone 15 pro')`, Postgres выбросит ошибку синтаксиса, так как он ждет булевы операторы (ему нужно `iphone & 15 & pro`).

Никогда не склеивайте строку запроса через `strings.ReplaceAll(q, " ", " & ")` в Go! Это приведет к ошибкам при вводе спецсимволов. Используйте встроенную функцию **`websearch_to_tsquery`** (появилась в PG 11). Она понимает синтаксис Google (кавычки для фраз, минус для исключений) и никогда не падает с синтаксической ошибкой.

Пример идиоматичного кода с использованием `database/sql`:

```go
package repository

import (
	"context"
	"database/sql"
)

type Article struct {
	ID    int
	Title string
	Rank  float32
}

// SearchArticles выполняет полнотекстовый поиск
func SearchArticles(ctx context.Context, db *sql.DB, userInput string) ([]Article, error) {
	// Используем websearch_to_tsquery для безопасной обработки ввода пользователя.
	// CTE используется для ограничения выборки перед ранжированием.
	query := `
		WITH matched AS (
			SELECT id, title, search_vector
			FROM articles
			WHERE search_vector @@ websearch_to_tsquery('russian', $1)
			-- Фильтруем старые статьи, чтобы уменьшить Heap Hits при ранжировании
			ORDER BY created_at DESC 
			LIMIT 500
		)
		SELECT 
			id, 
			title, 
			ts_rank(search_vector, websearch_to_tsquery('russian', $1)) as rank
		FROM matched
		ORDER BY rank DESC
		LIMIT 10;
	`

	rows, err := db.QueryContext(ctx, query, userInput)
	if err != nil {
		return nil, err
	}
	defer rows.Close()

	var result []Article
	for rows.Next() {
		var a Article
		if err := rows.Scan(&a.ID, &a.Title, &a.Rank); err != nil {
			return nil, err
		}
		result = append(result, a)
	}

	if err = rows.Err(); err != nil {
		return nil, err
	}

	return result, nil
}
```

## Итог

1. PostgreSQL предоставляет FTS из коробки, который покрывает потребности большинства проектов без оверхеда на поддержку ElasticSearch.
2. В основе лежат типы `tsvector` (нормализованный документ) и `tsquery` (поисковый запрос).
3. Производительность достигается за счет **GIN-индекса**.
4. Всегда используйте `websearch_to_tsquery` для обработки сырого пользовательского ввода.
5. Помните про стоимость функции `ts_rank` — она требует физического чтения страниц из Кучи. Минимизируйте количество ранжируемых строк.

PostgreSQL невероятно богат на встроенные типы (от JSONB до геоданных и полнотекстового поиска). Но если даже этого не хватает, его архитектура позволяет загружать C-библиотеки прямо в рантайм БД, меняя её поведение до неузнаваемости. Именно эту мощную концепцию мы разберем в следующей статье: [[7. Расширения PostgreSQL]].
