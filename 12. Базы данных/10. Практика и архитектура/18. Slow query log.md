## 🐢 Slow Query Log: выявление и анализ медленных запросов в БД

**Slow Query Log (лог медленных запросов)** — это механизм регистрации запросов, выполнение которых заняло больше заданного порогового времени. Это один из основных инструментов диагностики производительности баз данных, позволяющий выявлять проблемные SQL-операторы, требующие оптимизации.

---

### 📌 Зачем нужен Slow Query Log?

1. **Выявление узких мест** — находит запросы, которые потребляют больше всего ресурсов и времени.
2. **Оптимизация производительности** — помогает определить, какие запросы нужно оптимизировать (добавить индексы, переписать SQL).
3. **Мониторинг и анализ** — позволяет отслеживать изменения производительности со временем.
4. **Планирование масштабирования** — показывает нагрузку на БД, помогает принимать решения о масштабировании.
5. **Соответствие SLA** — помогает гарантировать, что время отклика системы соответствует требованиям бизнеса.

---

### ⚙️ Как работает Slow Query Log

1. **Пороговое время (threshold)** — запросы, выполняющиеся дольше этого времени, записываются в лог.
2. **Запись в лог** — информация о запросе: текст SQL, время выполнения, пользователь, база данных, время начала, количество возвращенных строк и т.д.
3. **Хранение логов** — в файле, таблице (в MySQL) или системном каталоге (в PostgreSQL).

---

### 🛠️ Настройка и использование в разных СУБД

#### **1. MySQL / MariaDB**

**Ключевые параметры:**
- `slow_query_log` — включить/выключить логирование (`ON`/`OFF`).
- `long_query_time` — пороговое время в секундах (может быть с дробной частью, например, 0.1).
- `log_queries_not_using_indexes` — записывать запросы, не использующие индексы, даже если они быстрее порога.
- `slow_query_log_file` — путь к файлу лога.
- `log_output` — куда писать: `FILE`, `TABLE` (в таблицу `mysql.slow_log`) или `NONE`.

**Пример настройки:**
```sql
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 2;
SET GLOBAL log_queries_not_using_indexes = 'ON';
```

**Инструменты анализа:**
- `mysqldumpslow` — утилита для агрегации и суммирования медленных запросов (показывает похожие запросы, их количество, общее время).
- `pt-query-digest` (Percona Toolkit) — мощный инструмент для детального анализа логов, построения отчетов, выявления трендов.

**Пример использования mysqldumpslow:**
```bash
mysqldumpslow -s t /var/log/mysql/mysql-slow.log
```
(Показывает топ запросов по времени выполнения.)

**Рекомендации:**
- Включать на продакшене, но с аккуратным порогом (например, 1-2 секунды).
- Анализировать регулярно, особенно после изменений в данных или схеме.
- Для высоконагруженных систем: можно использовать семплирование (логировать только часть запросов).

---

#### **2. PostgreSQL**

**Ключевые параметры:**
- `log_min_duration_statement` — минимальное время выполнения в миллисекундах, при котором запрос попадает в лог.
- `log_line_prefix` — формат строки лога (можно добавить информацию о запросе).
- `log_statement` — можно логировать все запросы (`all`), но это создает большую нагрузку.

**Пример настройки:**
```sql
ALTER SYSTEM SET log_min_duration_statement = '1000';  -- 1 секунда
SELECT pg_reload_conf();
```

**Расширения:**
- `pg_stat_statements` — собирает статистику по всем выполненным запросам: количество вызовов, общее время, строки, планирование и выполнение. Это более современный и эффективный способ анализа, чем логи.
- `auto_explain` — автоматически записывает планы выполнения запросов, занимающих больше заданного времени.

**Пример использования pg_stat_statements:**
```sql
SELECT query, calls, total_exec_time, mean_exec_time, rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
```
(Показывает топ-10 запросов по общему времени выполнения.)

**Рекомендации:**
- Использовать `pg_stat_statements` для постоянного мониторинга.
- Настройка `log_min_duration_statement` полезна для разового анализа, но создает нагрузку на диск.

#### **3. Microsoft SQL Server**

**Инструменты:**
- **Extended Events (XEvents)** — современный механизм для отслеживания событий, включая длительные запросы. Можно настроить сессию для перехвата запросов с длительностью больше порога.
- **Query Store** — встроенный функционал для сбора статистики по запросам, планам выполнения и ресурсам. Позволяет видеть историю запросов и их производительности.
- **SQL Trace / SQL Server Profiler** — устаревшие, но все еще используемые инструменты для трассировки запросов.

**Пример настройки Extended Events:**
```sql
CREATE EVENT SESSION [LongRunningQueries] ON SERVER
ADD EVENT sqlserver.sql_statement_completed(
    ACTION(sqlserver.sql_text, sqlserver.database_name)
    WHERE ([duration] > 1000000))  -- 1 секунда в микросекундах
ADD TARGET package0.event_file(SET filename = N'C:\temp\LongRunningQueries.xel');
GO
ALTER EVENT SESSION [LongRunningQueries] ON SERVER STATE = START;
```

**Рекомендации:**
- Использовать Query Store для постоянного мониторинга.
- Extended Events — для детального анализа и отладки.

---

#### **4. Oracle Database**

**Инструменты:**
- **AWR (Automatic Workload Repository)** — собирает статистику по работе БД, включая SQL-запросы.
- **ASH (Active Session History)** — история активных сессий, позволяет видеть, какие запросы выполнялись в конкретные моменты времени.
- **SQL Trace / TKPROF** — трассировка SQL-запросов с последующим форматированием отчета.

**Пример включения трассировки для сессии:**
```sql
ALTER SESSION SET SQL_TRACE = TRUE;
-- или
EXEC DBMS_SESSION.SESSION_TRACE_ENABLE(waits => TRUE, binds => TRUE);
```
Затем отчет можно сгенерировать утилитой `tkprof`.

**Рекомендации:**
- Использовать AWR и ASH для регулярного мониторинга.
- SQL Trace — для детального анализа конкретных сессий.

### 📊 Анализ логов и инструменты

1. **Сбор и агрегация:**
   - Утилиты вроде `mysqldumpslow`, `pt-query-digest`.
   - Системы мониторинга: Prometheus + Grafana, Zabbix, Nagios (с плагинами).
   - Облачные решения: AWS RDS Performance Insights, Google Cloud SQL Insights.

2. **Выявление проблем:**
   - Запросы с высоким временем выполнения.
   - Запросы с высоким количеством сканируемых строк (full table scan).
   - Запросы с неэффективными планами выполнения (отсутствие индексов, неоптимальные join).

3. **Анализ планов выполнения:**
   - Использовать `EXPLAIN` (MySQL, PostgreSQL) или `EXPLAIN ANALYZE` для просмотра плана.
   - Искать: Full Table Scan, Nested Loop с большим количеством строк, неэффективные фильтры.

4. **Действия после анализа:**
   - Добавление/изменение индексов.
   - Переписывание SQL (избегать подзапросов, использовать правильные join).
   - Обновление статистики таблиц (ANALYZE, UPDATE STATISTICS).
   - Изменение конфигурации БД (например, увеличение буферов).

### 📈 Лучшие практики работы с Slow Query Log

1. **Регулярный анализ** — настроить еженедельный или ежедневный просмотр логов.
2. **Автоматизация** — использовать скрипты для агрегации отчетов и отправки по email/в мессенджеры.
3. **Тестирование изменений** — после оптимизации запроса проверять его производительность на тестовых данных.
4. **Мониторинг в реальном времени** — интеграция с системами мониторинга для алертинга при появлении медленных запросов.
5. **Документирование** — вести журнал оптимизаций и их результатов.
6. **Баланс между детализацией и нагрузкой** — не включать логирование всех запросов на высоконагруженных системах, использовать семплирование.

---

### 🔗 Связь с другими темами архитектуры БД

- **Индексы** — многие медленные запросы решаются добавлением правильных индексов.
- **Репликация и шардирование** — если запросы медленные из-за объема данных, возможно, пора подумать о масштабировании.
- **Кэширование** — некоторые запросы можно кэшировать на уровне приложения или БД.
- **Транзакции и блокировки** — медленные запросы могут быть вызваны блокировками, что требует анализа конкурентного доступа.

### 💡 Пример быстрой настройки в MySQL

1. Включить лог:
   ```sql
   SET GLOBAL slow_query_log = 'ON';
   SET GLOBAL long_query_time = 1;
   SET GLOBAL log_queries_not_using_indexes = 'ON';
   ```
2. Выполнить типичные запросы на продакшене (или нагрузочное тестирование).
3. Проанализировать лог:
   ```bash
   mysqldumpslow -s t /var/log/mysql/mysql-slow.log | head -20
   ```
4. Для топ-10 запросов выполнить `EXPLAIN` и найти узкие места.
5. Оптимизировать: добавить индексы, переписать запросы, обновить статистику.

---

Slow Query Log — это первый и важнейший шаг в оптимизации производительности базы данных. Его правильная настройка и регулярный анализ позволяют значительно улучшить отклик системы и предотвратить проблемы до того, как они станут критическими.
