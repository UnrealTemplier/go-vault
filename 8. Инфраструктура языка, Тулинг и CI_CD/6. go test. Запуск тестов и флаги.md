
В большинстве языков тестирование — это дело внешних библиотек: JUnit в Java, PyTest в Python, PHPUnit в PHP. Вы должны выбрать библиотеку, добавить её в зависимости, настроить раннер и конфигурацию.

В Go тестирование — это **часть языка**. Пакет `testing` встроен в стандартную библиотеку, а команда `go test` — в тулчейн. Это задает единый стандарт написания тестов для всей индустрии: любой Go-инженер, зайдя в любой проект, сразу понимает, как запустить тесты.

## Как работает `go test` (Under the hood)

Когда вы запускаете `go test`, происходит следующее:

1.  **Поиск**: Тулчейн находит файлы с суффиксом `_test.go` в указанном пакете.
2.  **Генерация main**: `go test` создает временный пакет `main`, который содержит скомпилированный код вашего пакета, тестовые функции и сгенерированный "харнесс" (тестовый раннер).
3.  **Компиляция и запуск**: Этот временный бинарник компилируется и запускается. Он последовательно выполняет функции, начинающиеся с `Test`.
4.  **Отчет**: Бинарник выводит результаты в stdout и возвращает код возврата 0 (успех) или 1 (провал).

> [!info] Под капотом
> Вы можете увидеть, какой именно код генерирует `go test`, используя флаг `-n` (dry run) или скомпилировав тест-бинарник вручную:
> ```bash
> go test -c -o my.test
> ```
> Это создаст исполняемый файл `my.test`, который можно запустить на другой машине, даже если там нет исходного кода или Go тулчейна. Это используется в CI для разделения этапов сборки и запуска тестов.

## Паттерн Table-Driven Tests

Go не предоставляет DSL для ассертов (вроде `Assert.Equal(expected, actual)`). В стандартном подходе используется простой `if` и `t.Error`. Чтобы избежать дублирования кода при тестировании разных сценариев, в Go принят идиоматичный паттерн **Table-Driven Tests**.

```go
func TestAdd(t *testing.T) {
    tests := []struct {
        name string
        a    int
        b    int
        want int
    }{
        {name: "simple add", a: 2, b: 2, want: 4},
        {name: "negative", a: -1, b: -1, want: -2},
        {name: "zero", a: 0, b: 0, want: 0},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := Add(tt.a, tt.b)
            if got != tt.want {
                t.Errorf("Add(%d, %d) = %d, want %d", tt.a, tt.b, got, tt.want)
            }
        })
    }
}
```

`t.Run` создает под-тесты, которые можно запускать параллельно или фильтровать независимо.

## Ключевые флаги `go test`

`go test` обладает мощным набором флагов, которые часто игнорируются новичками.

### 1. `-v` (Verbose)
Выводит詳細ную информацию о том, какие тесты запускаются. В режиме по умолчанию `go test` молчит, если все тесты прошли успешно. С `-v` вы увидите имена passed-тестов, что полезно для понимания, что процесс не завис.

### 2. `-run` (Filter)
Позволяет фильтровать тесты по регулярному выражению. Крайне полезно во время разработки, когда вы хотите запустить только один сломанный тест.

```bash
# Запустит только TestAdd и его подтесты
go test -run TestAdd

# Запустит только подтест с именем "negative" внутри TestAdd
go test -run TestAdd/negative
```

### 3. `-race` (Race Detector)
Один из главных козырей Go. Включает детектор гонок данных. Программа будет работать медленнее и потреблять больше памяти, но при обнаружении одновременного доступа к памяти без синхронизации, `go test` упадет с отчетом о Race Condition.

```bash
go test -race
```

> [!warning] Ловушка / Gotcha
> Детектор гонок не гарантирует нахождение *всех* гонок. Он детектирует только те, которые произошли во время конкретного запуска. Если ваш код зависит от порядка выполнения горутин (а он недетерминирован), гонка может не проявиться. Поэтому гонки лучше ловить на этапе нагрузочного тестирования или при запуске тестов с `-count=N`, где N > 1.

### 4. `-cover` (Coverage)
Выводит процент покрытия кода тестами. Для более детального отчета можно сгенерировать HTML-отчет.

```bash
go test -coverprofile=coverage.out
go tool cover -html=coverage.out
```

### 5. `-count` и Кэширование
Go кэширует результаты тестов! Если вы запустите `go test` дважды, а код не менялся, второй запуск будет мгновенным: `ok github.com/user/project 0.001s (cached)`.

Это отлично для CI, но может мешать при дебаггинге флапающих тестов (tests that fail intermittently). Флаг `-count=1` принудительно инвалидирует кэш и заставляет тесты отработать заново.

```bash
# Принудительный перезапуск тестов без кэша
go test -count=1
```

## Бенчмарки и Фаззинг

`go test` умеет не только проверять корректность, но и производительность.

**Бенчмарки**:
Функции, начинающиеся с `Benchmark`, запускаются только при указании флага `-bench`.
```go
func BenchmarkAdd(b *testing.B) {
    for i := 0; i < b.N; i++ {
        Add(1, 2)
    }
}
```
```bash
go test -bench=. -benchmem
```
Флаг `-benchmem` показывает количество аллокаций на одну операцию — критически важная метрика для Go.

**Фаззинг (Fuzzing)** (доступно с Go 1.18):
Автоматическая генерация случайных входных данных для поиска паник и крашей.

```go
func FuzzAdd(f *testing.F) {
    f.Add(1, 2) // Seed corpus
    f.Fuzz(func(t *testing.T, a, b int) {
        Add(a, b)
    })
}
```
Запуск: `go test -fuzz=FuzzAdd`.

## Параллелизм тестов

По умолчанию тесты в одном пакете запускаются последовательно. Если у вас есть медленные тесты (например, работа с БД или сетью), используйте `t.Parallel()`.

```go
func TestSlowDB(t *testing.T) {
    t.Parallel() // Этот тест будет запущен параллельно с другими Parallel-тестами
    // ... логика
}
```

Параллелизм ограничен флагом `-p` (по умолчанию равно `GOMAXPROCS`, обычно числу ядер CPU).

## Итог

1.  **Встроенность**: `go test` — мощнейший инструмент, не требующий внешних зависимостей.
2.  **Table-Driven Tests**: Единый идиоматичный паттерн для написания тестов.
3.  **Race Detector**: Всегда используйте `-race` при локальной разработке и в CI.
4.  **Кэширование**: Помните про кэш тестов и используйте `-count=1`, если нужно перезапустить.
5.  **Инструментарий**: `-cover`, `-bench` и `-fuzz` делают `go test` универсальным инструментом QA.

Мы проверили работоспособность кода. Однако тесты не спасают от всех проблем. Следующим уровнем защиты является статический анализ кода. В следующей статье мы разберем утилиту, которая ловит ошибки еще до запуска тестов: [[7. go vet. Статический анализ]].
