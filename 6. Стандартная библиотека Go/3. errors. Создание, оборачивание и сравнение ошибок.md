## Философия обработки ошибок в Go

В отличие от языков с исключениями (Java, C#, PHP), где контроль потока выполнения может быть разорван в любой точке стека вызовов, Go использует явную, предсказуемую модель: **ошибки — это значения**. Они возвращаются функциями, передаются как аргументы и проверяются в `if err != nil`.

Это не просто синтаксическое предпочтение. Это архитектурный выбор, который:
* Гарантирует, что каждый потенциальный сбой обрабатывается на уровне вызывающего кода.
* Позволяет компилятору лучше оптимизировать код (отсутствие скрытых блоков `catch` и раскрутки стека).
* Дает полный контроль над контекстом ошибки и стратегией восстановления.

Однако написание идиоматичного кода с ошибками требует понимания внутренней механики интерфейсов, аллокаций и правил сравнения.

> [!info] Под капотом
> Интерфейс `error` в Go — это всего одна строчка в спецификации:
> ```go
> type error interface {
>     Error() string
> }
> ```
> Несмотря на минимализм, передача любого значения через этот интерфейс запускает в рантайме механизм приведения типов (`runtime.convT2E`), который упаковывает данные и таблицу методов в структуру `runtime.iface`.

## 1. Интерфейс error и его устройство в памяти

В памяти интерфейс `error` представлен структурой `iface`:
```go
type iface struct {
    tab  *itab      // Указатель на таблицу методов и тип
    data unsafe.Pointer // Указатель на конкретные данные
}
```
Когда вы возвращаете ошибку, рантайм выделяет память под данные (если их еще нет в куче) и создает `iface`. Если функция возвращает `nil`, это буквально нулевой указатель. Но здесь кроется самая известная ловушка Go.

### Проблема: nil != nil
```go
func doWork() error {
    var p *MyError = nil
    return p // Вернет интерфейс, где data = nil, но tab != nil
}

func main() {
    err := doWork()
    if err != nil {
        fmt.Println("Ошибка есть!", err) // Сработает!
    }
}
```
Интерфейс равен `nil` **только когда и `tab`, и `data` равны `nil`**. Возврат указателя на пользовательский тип ошибки, который сам по себе `nil`, создает непустой интерфейс. Это частый источник багов в production-системах.

> [!warning] Ловушка / Gotcha
> **Никогда не возвращайте конкретный тип ошибки как `error`, если значение может быть `nil`.** Всегда возвращайте `error` напрямую или конвертируйте `nil` перед возвратом. Линтер `staticcheck` (правило `ST1005`) автоматически отлавливает это.

## 2. Три стратегии работы с ошибками

Архитектор должен выбирать стратегию в зависимости от того, как вызывающий код будет реагировать на сбой.

### Стратегия 1. Sentinel Errors (Маркерные ошибки)
Глобальные переменные, представляющие конкретное ожидаемое состояние.
```go
var ErrNotFound = errors.New("пользователь не найден")

// Использование
if errors.Is(err, ErrNotFound) {
    return http.StatusNotFound, nil
}
```
*Плюсы:* Простота, быстрое сравнение.
*Минусы:* Создают зависимость от пакета, сложно передавать контекст (например, `ID` сущности).

### Стратегия 2. Custom Error Types (Пользовательские типы)
Структуры, реализующие `error` и дополнительные методы.
```go
type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation failed: %s - %s", e.Field, e.Message)
}

// Использование
if e := (&ValidationError{}); errors.As(err, &e) {
    log.Printf("Invalid field: %s", e.Field)
}
```
*Плюсы:* Передача контекста, типобезопасность через `errors.As`.
*Минусы:* Больше кода, утечка внутренней структуры пакета наружу.

### Стратегия 3. Opaque Errors (Непрозрачные ошибки)
Функция возвращает просто `error`. Вызывающий код проверяет только `err != nil` и реагирует одинаково (логирует, возвращает 500).
*Плюсы:* Инкапсуляция, минимальная связность.
*Минусы:* Невозможно дифференцировать ответы на уровне API.

> [!tip] Собеседование
> **Вопрос:** Когда выбирать Sentinel, а когда Custom Type?
> **Ответ:** Sentinel — когда состояние однозначно, не требует дополнительных данных и должно сравниваться по значению (например, `io.EOF`, `sql.ErrNoRows`). Custom Type — когда нужна передача контекста (метаданные, HTTP-статусы, валидационные детали) или когда ошибка может иметь несколько вариантов с разными полями.

## 3. Цепочки ошибок и механизмы Go 1.13+

До Go 1.13 оборачивание ошибок требовало стороннего `pkg/errors`. Сейчас `fmt.Errorf` поддерживает `%w`, который создает цепочку, позволяя распаковывать её стандартными средствами.

### Механика Unwrap
Функция, создающая обертку, должна реализовать метод `Unwrap() error`. `errors.Is` и `errors.As` рекурсивно вызывают этот метод, поднимаясь по стеку вложенности.

```mermaid
flowchart TD
    A["Вызывающий код<br>errors.As err -> *DBError"] -->|"рекурсия Unwrap| B["Пакет репозитория<br>fmt.Errorf DB query failed %w"]
    B -->|"рекурсия Unwrap| C["Драйвер базы<br>lib/pq error"]
    
    D["runtime"] -->|"проверка интерфейса| E["Сравнение типов или значений"]
    B -.-> D
    C -.-> D
```

### Пример идиоматичного оборачивания
```go
func GetUser(ctx context.Context, id int64) (*User, error) {
    row := db.QueryRowContext(ctx, "SELECT name, email FROM users WHERE id = ?", id)
    var u User
    if err := row.Scan(&u.Name, &u.Email); err != nil {
        // Оборачиваем, добавляя контекст, но сохраняя оригинал
        return nil, fmt.Errorf("repository.GetUser id=%d: %w", id, err)
    }
    return &u, nil
}
```

### Go 1.20: errors.Join и множественная распаковка
`errors.Join` позволяет агрегировать несколько ошибок в одну (актуально для batch-операций или `defer` с несколькими `Close()`). `errors.Is` проверяет каждое вхождение, `errors.As` находит первое совпадение.

```go
var errs []error
// ... заполнение errs в цикле ...
if len(errs) > 0 {
    return errors.Join(errs...)
}
```

## 4. Mechanical Sympathy и цена аллокаций

Каждая созданная ошибка — это нагрузка на GC. В высоконагруженных сервисах (10k+ RPS) создание ошибок в горячих путях может вызывать микростоппы и рост `GC Assist Time`.

### Внутренние аллокации
1. `errors.New("текст")`: Возвращает указатель на структуру, содержащую строку. Строковый литерал лежит в read-only памяти бинарника, аллоцируется только оболочка `*errors.errorString` (16-24 байта в куче).
2. `fmt.Errorf("текст %w", err)`: Парсит строку формата, выделяет память под результат, создает новую структуру-обертку. **Значительно дороже**, чем `errors.New`.

### Оптимизация для Hot Paths
Если ошибка ожидается часто (например, `io.EOF` при чтении чанков), избегайте `fmt.Errorf` в цикле.
```go
// ❌ Плохо: создает новую аллокацию на каждой итерации
for {
    if err := read(); err != nil {
        return fmt.Errorf("read chunk %d: %w", i, err)
    }
}

// ✅ Хорошо: используем готовый sentinel или кэшируем ошибку
if errors.Is(err, io.EOF) {
    break // io.EOF — синглетон, сравнение мгновенное
}
```

> [!info] Под капотом
> Сравнение с исключениями в C++/Java: выброс исключения (`throw`) требует раскрутки стека (stack unwinding), поиска обработчиков `catch` и часто генерации таблицы исключений компилятором. Это может занимать сотни наносекунд и нарушать предсказание ветвлений CPU. В Go проверка `if err != nil` — это просто сравнение двух указателей с регистре CPU. Ветвление предсказывается идеально, если ошибки редки.

## 5. Ловушки и хардкорные вопросы с собеседований

| Ловушка | Описание | Решение |
|---------|----------|---------|
| `errors.As` без указателя | `errors.As(err, e)` скомпилируется, но паникует в рантайме. | Второй аргумент **всегда** должен быть указателем на интерфейс или тип ошибки: `&target`. |
| Сравнение через `==` | Работает только для sentinel-ошибок, созданных `errors.New`. Для оберток возвращает `false`. | Всегда используйте `errors.Is` для сравнения значений. |
| Потеря контекста | Возврат `err.Error()` вместо `err` в слое выше. | Передавайте `error` как есть. Преобразование в строку убивает возможность `Is`/`As`. |
| Дублирование сообщений | `fmt.Errorf("failed: %w", err)` + логирование того же текста в двух местах. | Логируйте на границе обработки. Возвращайте ошибки чистыми. |

> [!tip] Собеседование
> **Вопрос:** Почему `errors.As` требует указатель, а `errors.Is` нет?
> **Ответ:** `errors.Is` просто сравнивает значения/указатели или вызывает `Is(error) bool`, если он реализован. Ему не нужно менять переданное значение. `errors.As` должен **записать** найденную ошибку в переменную, чтобы вызывающий код мог работать с её конкретным типом. Без указателя изменение значения внутри функции не отразится на caller-е.
>
> **Вопрос:** Как сравнить две ошибки, если их типы приватны и нет методов `Is`?
> **Ответ:** Никак стандартными средствами. Это архитектурный запах. Если вы не можете сравнить ошибки через `Is`/`As`, используйте Opaque-стратегию или добавьте публичный `Is` метод в приватный тип, реализующий логику сравнения.

## 6. Сравнение с экосистемами других языков

| Аспект | Java / C# / PHP | Go |
|--------|-----------------|----|
| **Механизм** | `try-catch-finally`, исключения как объекты классов | Явные возвраты, `error` как интерфейс |
| **Стоимость** | High: раскрутка стека, аллокация объекта-исключения, нарушение branch prediction | Low: проверка nil-указателя, предсказуемый поток |
| **Контекст** | Stack trace генерируется автоматически при `throw` | Требуется явное оборачивание (`%w`) или `runtime.Callers` |
| **Обработка** | Можно проигнорировать (checked exceptions заставляют ловить, но `catch (Exception e) {}` убивает смысл) | Компилятор требует обработки, `err` нельзя проигнорировать без `_` (что линтуется) |
| **Stack Trace** | Всегда доступен | Отсутствует в stdlib. Для продакшена используют `runtime/debug.Stack()` или внешние либы, но это дорого. |

## Итог

1.  **Ошибки в Go — это данные.** Они передаются явно, не прерывают выполнение и требуют дисциплины.
2.  **Помните про `iface != nil`.** Возвращайте `nil`, а не `&MyError{}` со значением `nil`.
3.  **Используйте цепочки правильно.** `%w` для оборачивания, `errors.Is` для сравнения значений, `errors.As` для доступа к полям.
4.  **Оптимизируйте создание.** В горячих путях избегайте `fmt.Errorf`, кэшируйте sentinel-ошибки, используйте `errors.Join` для батчей.
5.  **Не изобретайте велосипеды со стек-трейсами.** В продакшене stack trace через `errors` пакет слишком дорог. Логируйте контекст (request ID, userID) в момент возникновения, а не ловите его на уровне API.

Понимание работы с ошибками закладывает фундамент для работы с потоками данных. В следующей статье мы перейдем к сердцу ввода-вывода в Go и разберем абстракции, на которых строится вся стандартная библиотека: [[4. io. Reader, Writer и философия потоков в Go.md]].
