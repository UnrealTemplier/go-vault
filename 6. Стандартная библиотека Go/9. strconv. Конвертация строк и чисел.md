## `strconv`: Мост между бинарным миром и текстом

Пакет `strconv` (string conversion) — это стандартный инструмент для преобразования примитивных типов (`int`, `float`, `bool`) в строки и обратно. В отличие от `fmt.Sprintf`, который универсален, но медленен из-за парсинга форматной строки и рефлексии, `strconv` предоставляет специализированные функции, которые компилируются в высокооптимизированный машинный код.

Для бэкенд-разработчика, работающего с HTTP (где всё — строки) и базами данных (где числа часто хранятся эффективно), умение быстро конвертировать данные без лишних аллокаций — ключевой навык оптимизации.

> [!info] Под капотом
> Функции `strconv` не используют регулярные выражения или сложные парсеры. Они работают побайтово, используя арифметику над символами ASCII. Например, чтобы превратить символ `'5'` в число `5`, используется простая операция: `byte('5') - '0'`. Это делает их экстремально быстрыми и предсказуемыми по времени выполнения.

## 1. Конвертация целых чисел: `Atoi`, `Itoa` и `ParseInt`

### `strconv.Atoi` vs `strconv.ParseInt`
Функция `Atoi` (ASCII to integer) — это просто удобная обертка над `ParseInt(s, 10, 0)`. Она возвращает `int` (размер которого зависит от архитектуры: 32 или 64 бита).

```go
// Atoi — быстрый путь для типичного int
n, err := strconv.Atoi("12345")

// ParseInt — когда нужна точность типа или другая система счисления
// Возвращает int64. Последний аргумент (0) означает "использовать размер int"
n64, err := strconv.ParseInt("12345", 10, 64) 
```

**Когда использовать что:**
*   **`Atoi`**: Когда вы работаете с обычными `int` (ID сущностей, счетчики) и вам важна краткость кода.
*   **`ParseInt`**: Когда нужно строго контролировать размерность (например, парсинг `int8` или `int64` для протоколов) или основание системы счисления (например, парсинг HEX-цветов `base=16`).

### `strconv.Itoa` и `FormatInt`
Обратная операция. `Itoa` — обертка над `FormatInt(int64(i), 10)`.

> [!warning] Ловушка / Gotcha
> **Переполнение при парсинге.**
> Если строка содержит число, превышающее максимальное значение типа (`math.MaxInt64`), `ParseInt` вернет ошибку `strconv.ParseInt: parsing "9223372036854775808": value out of range`.
> Однако, если вы используете `Atoi` на 32-битной системе для большого числа, поведение будет аналогичным. Всегда проверяйте `err != nil`. Игнорирование ошибки при парсинге пользовательского ввода — частая уязвимость, приводящая к непредсказуемому поведению (возврату `0`).

## 2. Работа с плавающей точкой: `ParseFloat` и `FormatFloat`

Конвертация `float64` сложнее из-за стандарта IEEE 754.

```go
f, err := strconv.ParseFloat("3.14159", 64)
s := strconv.FormatFloat(3.14159, 'f', 2, 64) // "3.14"
```

### Форматы вывода (`fmt` byte)
*   `'f'`: Десятичная запись (-ddd.dddd). Стандарт для валют и простых чисел.
*   `'e'`/`'E'`: Научная нотация (-d.dddde±dd).
*   `'g'`/`'G'**: Автоматический выбор: использует `'e'` если экспонента меньше -4 или больше/равна точности, иначе `'f'`. Убирает лишние нули. Идеально для общего назначения.
*   `'b'`: Двоичная экспонента (для отладки точности).

> [!info] Под капотом
> `FormatFloat` использует алгоритм Драгона4 (или его вариации) для гарантированно корректного округления. Это сложная математическая операция. Если вам нужно просто отбросить знаки после запятой для отображения, иногда быстрее умножить, привести к `int` и разделить, чем использовать полноценный форматтер, хотя `strconv` достаточно оптимизирован для большинства задач.

## 3. Mechanical Sympathy: Избегаем аллокаций с `Append`

Самая мощная, но часто игнорируемая часть пакета `strconv` — функции семейства `Append`.

Стандартные функции `Itoa` и `FormatFloat` возвращают **новую строку**, что требует аллокации памяти в куче. Функции `Append*` принимают существующий срез `[]byte` и дописывают результат в него.

```go
// ❌ Плохо: Аллокация новой строки
idStr := strconv.Itoa(userID)
key := "user:" + idStr 

// ✅ Хорошо: Нулевая аллокация (если буфер достаточной емкости)
var buf []byte
buf = append(buf, "user:"...)
buf = strconv.AppendInt(buf, int64(userID), 10)
// buf теперь содержит "user:123"
key := string(buf) // Единственная аллокация в конце, если нужно именно string
```

Это критически важно в горячих путях, таких как:
1.  Генерация ключей для Redis/Memcached.
2.  Формирование логов в формате JSON вручную.
3.  Сборка SQL-запросов (хотя лучше использовать параметризованные запросы).

Использование `Append` позволяет переиспользовать один и тот же буфер (например, из `sync.Pool` или `bytes.Buffer`), сводя давление на GC к минимуму.

## 4. Булевы значения и кватернионы

`strconv.ParseBool` принимает не только `"true"`/`"false"`, но и `"1"`/`"0"`, `"t"`/`"f"`, `"T"`/`"F"`, `"yes"`/`"no"` и т.д. Это удобно при парсинге переменных окружения или параметров запроса, где пользователи могут писать что угодно.

```go
// Принимает: 1, t, T, TRUE, true, True, 0, f, F, FALSE, false, False
val, err := strconv.ParseBool(os.Getenv("DEBUG"))
```

## 5. Сравнение подходов: `strconv` vs `fmt` vs `cast`

| Метод | Скорость | Аллокации | Гибкость | Использование |
|-------|----------|-----------|----------|---------------|
| `strconv.Itoa/ParseInt` | 🚀 Очень быстро | 1 (результат) | Низкая (только base 10) | Основная работа с числами |
| `strconv.AppendInt` | 🚀🚀 Максимально быстро | 0 (в существующий буфер) | Низкая | Высоконагруженные системы, генерация ключей |
| `fmt.Sprintf("%d")` | 🐢 Медленно | 2+ (интерфейсы + строка) | Высокая (форматирование) | Логи, человеко-читаемый вывод |
| `github.com/spf13/cast` | 🐇 Средне | Зависит от контекста | Очень высокая (any -> type) | Конфигурации, маппинг структур |

> [!tip] Собеседование
> **Вопрос:** Почему `strconv.Atoi` быстрее, чем `fmt.Sscanf("%d", &n)`?
> **Ответ:** `fmt.Sscanf` должен распарсить форматную строку `%d`, определить тип аргумента через рефлексию, обработать возможные пробелы и флаги формата. `strconv.Atoi` проходит по байтам строки один раз, выполняя простую арифметику `n = n * 10 + int(c - '0')`. Отсутствие рефлексии и парсинга шаблона дает выигрыш в порядок величины.
>
> **Вопрос:** Как безопасно конвертировать `string` в `int64` без аллокаций, если строка получена из `http.Request.URL.Query()`?
> **Ответ:** `URL.Query()` возвращает `map[string][]string`. Значения там уже строки. Используем `strconv.ParseInt(vals.Get("id"), 10, 64)`. Аллокаций не будет, так как мы не создаем новых строк, а парсим существующую. Если бы мы использовали `fmt`, была бы аллокация интерфейсов.

## 6. Ловушки и нюансы

### Локаль и разделители дробной части
`strconv` **всегда** использует точку `.` как разделитель дробной части, независимо от локали системы. Это соответствует стандарту RFC 4627 (JSON) и большинству сетевых протоколов. Если вы парсите ввод от пользователя, который может ввести запятую (например, `3,14` в русской локали), `ParseFloat` вернет ошибку. Вам придется предварительно заменять запятые на точки через `strings.Replace`.

### Точность `float32` vs `float64`
Go по умолчанию использует `float64`. Функции `ParseFloat` всегда возвращают `float64`. Если вам нужен `float32`, вы должны явно привести тип: `float32(f)`. При этом возможна потеря точности. `FormatFloat` позволяет указать битность (`32` или `64`), чтобы округление соответствовало типу.

```go
f64, _ := strconv.ParseFloat("0.1", 64)
f32 := float32(f64)
// f32 может не точно представлять 0.1 из-за ограничений IEEE 754 для 32 бит
```

## Итог

1.  **Используйте `strconv` вместо `fmt`** для конвертации чисел в продакшен-коде. Это быстрее и создает меньше мусора.
2.  **`Append` функции — ваши лучшие друзья** для высоконагруженных приложений. Они позволяют избежать аллокаций при сборке сложных строк из чисел.
3.  **Всегда проверяйте ошибки.** Парсинг внешних данных (HTTP, БД, конфиги) никогда не бывает гарантированно успешным.
4.  **Помните про локаль.** `strconv` работает только с точкой как разделителем десятичных дробей.
5.  **`Atoi` — это синтаксический сахар** для `ParseInt(s, 10, 0)`. Используйте его для удобства, но понимайте, что он возвращает платформо-зависимый `int`.
