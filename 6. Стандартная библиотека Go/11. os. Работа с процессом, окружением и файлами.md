## `os`: Абстракция над операционной системой

Пакет `os` предоставляет платформенно-независимый интерфейс к функционалу операционной системы. Он скрывает различия между Linux, Windows и macOS, предоставляя единый API для работы с файловой системой, переменными окружения, сигналами и процессами.

Однако «платформенно-независимый» не означает «абстрактный от железа». Напротив, функции пакета `os` являются прямыми обертками над системными вызовами (syscalls). Понимание того, какой syscall стоит за той или иной функцией, критически важно для оценки производительности и корректности работы приложения в production.

> [!info] Под капотом
> В основе большинства операций `os` лежит структура `*os.File`. Даже если вы работаете с сетевым сокетом, stdin/stdout или пайпом, внутри это файловый дескриптор.
> ```go
> type File struct {
>     *file // platform-specific implementation
> }
> 
> // Внутри file (на Linux):
> type file struct {
>     pfd         poll.FD // Интеграция с netpoller (epoll/kqueue)
>     name        string
>     dirinfo     *dirInfo // Кэш для readdir
>     nonblock    bool     // Флаг неблокирующего режима
>     stdoutOrErr bool     // Оптимизация для stdout/stderr
> }
> ```
> Поле `pfd` (Pollable File Descriptor) — это ключевой элемент, позволяющий Go реализовывать асинхронный I/O без блокировки OS-тредов, даже при работе с обычными файлами (хотя для дисковых файлов асинхронность ограничена возможностями ОС).

## 1. Работа с файловой системой: Syscalls и права доступа

### Открытие и создание файлов
Функции `os.Open`, `os.Create`, `os.OpenFile` транслируются в syscall `openat` (Linux) или `CreateFile` (Windows).

```go
// os.Open - только чтение
f, err := os.Open("config.yaml")

// os.Create - чтение/запись, создание, усечение (truncate). 
// Важно: права доступа 0666 модифицируются umask процесса!
f, err := os.Create("output.log") 

// os.OpenFile - полный контроль
// Флаги: O_RDWR | O_CREATE | O_APPEND
// Perm: 0640 (владелец: rw, группа: r, остальные: нет)
f, err := os.OpenFile("app.log", os.O_RDWR|os.O_CREATE|os.O_APPEND, 0640)
```

> [!warning] Ловушка / Gotcha
> **Umask и права доступа.**
> Аргумент `perm` в `os.Create` и `os.OpenFile` — это *максимально возможные* права. Реальные права вычисляются как `perm &^ umask`.
> Если `umask` системы равен `0022` (стандарт для многих серверов), то создание файла с `0666` даст реальные права `0644` (`rw-r--r--`).
> Если вам критичны строгие права (например, `0600` для приватных ключей), устанавливайте `umask` процесса в `0` при старте или явно вызывайте `f.Chmod(0600)` сразу после создания, но до записи данных, чтобы избежать окна уязвимости.

### Чтение и запись
Методы `f.Read()`, `f.Write()`, `f.Seek()` вызывают соответствующие syscalls `read`, `write`, `lseek`.
*   `Read` может вернуть `n < len(buf)` и `err == nil`. Это нормально. Вы должны вызывать `Read` в цикле, пока не получите `io.EOF`.
*   `Write` также может записать меньше байт, чем запрошено (особенно для pipe/socket), но для обычных файлов на локальной ФС это редкость.

### Метаданные и Stat
`os.Stat(path)` вызывает syscall `stat` (или `fstatat`). Он возвращает `os.FileInfo`, содержащий размер, время модификации, режим доступа.
**Важно:** `Stat` следует по символьным ссылкам (symlinks). Если нужно получить информацию о самом линке, используйте `os.Lstat`.

## 2. Переменные окружения: Глобальное состояние и гонки

Функции `os.Getenv`, `os.Setenv`, `os.Environ` работают с глобальной таблицей переменных окружения процесса.

> [!warning] Ловушка / Gotcha
> **Потокобезопасность `os.Setenv`.**
> Хотя чтение (`Getenv`) в Go относительно безопасно, запись (`Setenv`, `Unsetenv`) **не является потокобезопасной** на уровне всего процесса. Если две горутины одновременно вызовут `os.Setenv`, результат непредсказуем, а в некоторых ОС это может привести к состоянию гонки (data race) внутри libc.
> **Правило:** Устанавливайте переменные окружения только на этапе инициализации приложения (в `main` или `init`), до запуска горутин. В runtime используйте передачу конфигурации через структуры или `context`.

### Загрузка `.env` файлов
Стандартная библиотека **не умеет** парсить `.env` файлы. Это осознанное решение. Для продакшена рекомендуется передавать конфиги через ENV vars, установленные оркестратором (Kubernetes ConfigMaps/Secrets, Docker env), а не читать файлы `.env` из кода. Если всё же нужно, используйте сторонние библиотеки (например, `godotenv`), но делайте это один раз при старте.

## 3. Управление процессом и сигналами

### Graceful Shutdown
Один из самых важных паттернов в бэкенде на Go — корректное завершение работы при получении сигнала от ОС (SIGTERM, SIGINT).

```go
func main() {
    srv := &http.Server{Addr: ":8080"}
    
    // Канал для получения сигналов
    quit := make(chan os.Signal, 1)
    // Notify перенаправляет сигналы в канал
    signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
    
    go func() {
        <-quit
        log.Println("Shutting down server...")
        
        // Создаем контекст с таймаутом для graceful shutdown
        ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
        defer cancel()
        
        if err := srv.Shutdown(ctx); err != nil {
            log.Fatal("Server forced to shutdown:", err)
        }
    }()
    
    log.Println("Server starting...")
    if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        log.Fatal("ListenAndServe error:", err)
    }
    log.Println("Server exited properly")
}
```

> [!info] Под капотом
> `signal.Notify` регистрирует обработчик сигналов на уровне рантайма Go. Рантайм использует механизм `sigaction` (Linux) для перехвата сигналов. Вместо немедленного завершения процесса (как делает стандартный обработчик C), Go перенаправляет сигнал в ваш канал, позволяя выполнить логику очистки.

### Запуск внешних процессов (`os/exec`)
Хотя `os/exec` — это отдельный пакет, он тесно связан с `os`. Он позволяет запускать дочерние процессы.
**Важно:** Избегайте запуска шелл-команд (`sh -c "cmd"`) из-за риска инъекций. Используйте прямые вызовы:
```go
// ❌ Опасно и неэффективно
exec.Command("sh", "-c", "grep "+userInput+" /var/log/syslog")

// ✅ Безопасно
exec.Command("grep", userInput, "/var/log/syslog")
```

## 4. Mechanical Sympathy: Файловые дескрипторы и лимиты

Каждый открытый файл, сокет или пайп потребляет **файловый дескриптор (FD)**. Это ограниченный ресурс ядра ОС.
*   В Linux лимит на процесс задается через `ulimit -n` (часто 1024 по умолчанию, что катастрофически мало для бэкенда).
*   Утечка FD (забытый `f.Close()`) приведет к ошибке `too many open files` и краху сервиса.

**Правило:** Всегда используйте `defer f.Close()` сразу после проверки ошибки открытия.

```go
f, err := os.Open("data.txt")
if err != nil {
    return err
}
defer f.Close() // Гарантированное освобождение FD при выходе из функции
```

> [!tip] Собеседование
> **Вопрос:** Что произойдет, если вызвать `f.Close()` дважды?
> **Ответ:** Второй вызов вернет ошибку `os.ErrClosed` (или аналогичную, зависящую от ОС). Это безопасно, но указывает на логическую ошибку в коде. В некоторых случаях (если между закрытиями FD был переиспользован ОС для другого файла) это может привести к закрытию чужого ресурса, но в Go рантайм обычно отслеживает состояние `*os.File`, предотвращая такие катастрофы на уровне абстракции.
>
> **Вопрос:** Почему `os.Remove` может не освободить место на диске сразу?
> **Ответ:** В Unix-системах файл удаляется из директории (unlink), но данные на диске остаются, пока есть открытые файловые дескрипторы, указывающие на этот inode. Если ваша программа держит файл открытым после `os.Remove`, место не освободится до `f.Close()`.

## 5. Сравнение с другими языками

| Аспект | Java (NIO.2) | Python (`os`, `pathlib`) | Go (`os`) |
|--------|--------------|--------------------------|-----------|
| **Модель I/O** | Асинхронная (CompletableFuture), сложная | Синхронная (GIL блокирует), есть `asyncio` | Синхронная API, асинхронная реализация через G-M-P |
| **Обработка ошибок** | Исключения (`IOException`) | Исключения (`OSError`) | Явные возвращаемые значения `error` |
| **Управление ресурсами** | `try-with-resources` | `with open(...)` | `defer f.Close()` |
| **Производительность** | Высокая, но высокий overhead JVM | Низкая из-за GIL и интерпретатора | Высокая, близкая к C, минимальный runtime overhead |

## Итог

1.  **`os.File` — это абстракция над FD.** Помните про лимиты `ulimit` и всегда закрывайте файлы.
2.  **Права доступа зависят от `umask`.** Не полагайтесь слепо на аргумент `perm`.
3.  **ENV vars — глобальное состояние.** Не меняйте их в runtime конкурентно.
4.  **Graceful Shutdown обязателен.** Используйте `signal.Notify` и `context.WithTimeout` для корректного завершения HTTP-серверов и других долгоживущих процессов.
5.  **Syscalls дороги.** Группируйте операции с файлами, используйте буферизацию (`bufio`) поверх `os.File` для мелких чтений/записей.

Разобравшись с базовой работой с файлами и путями, мы часто сталкиваемся с необходимостью манипулировать этими путями: соединять части, извлекать расширения, нормализовать разделители. В следующей статье мы изучим правильные инструменты для этого: [[12. path, path_filepath и работа с путями]].
