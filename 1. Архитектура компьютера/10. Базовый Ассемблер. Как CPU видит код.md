## Читая мысли процессора

Мы знаем, что компилятор Go переводит наш код в машинные инструкции, которые понимает ISA (Архитектура набора команд). Но машинный код — это просто бесконечный поток байтов (`0x48 0x89 0x5C...`), который человеку читать невозможно.

Для того чтобы мы могли понимать, что делает процессор, был создан **Ассемблер** (Assembly). Это человекочитаемая мнемоника машинного кода. Одна строка ассемблера соответствует ровно одной машинной инструкции.

Нам не нужно уметь писать на ассемблере, но Senior-разработчик должен уметь его **читать**. Это позволяет понять, почему одна функция работает медленнее другой, как компилятор оптимизирует ваш код и что на самом деле происходит при передаче данных по указателю.

## Рабочее пространство: Регистры

Прежде чем смотреть на команды, нужно понять, где они работают. Представьте, что оперативная память (RAM) — это огромный склад в другом конце города, а регистры — это маленькая верстак прямо перед вами. Вы не можете складывать числа прямо на складе, вы сначала приносите их на верстак (в регистры), производите операцию и уносите результат обратно.

В архитектуре x86-64 (самой частой для бэкенда) есть основные регистры общего назначения:

*   **RAX** (Accumulator): Главный регистр. В него почти всегда записывается результат функции (возвращаемое значение).
*   **RBX, RCX, RDX**: Дополнительные регистры для временных вычислений.
*   **RSP (Stack Pointer)**: Указатель на вершину стека. Самый важный регистр для управления памятью и вызовов функций.
*   **RIP (Instruction Pointer)**: Тот самый `PC` (Счетчик команд) из статьи [[6. Цикл исполнения инструкции. Instruction Cycle]]. Он указывает на следующую команду.

## Словарь базовых команд

Почти любой код на ассемблере состоит из нескольких повторяющихся паттернов.

### 1. Пересылка данных: `MOV`
Команда `MOV` (от move) на самом деле не перемещает, а **копирует** данные.
`MOV RAX, RBX` — скопировать значение из RBX в RAX.
`MOV RAX, [0x1234]` — скопировать значение из памяти по адресу `0x1234` в регистр RAX.

### 2. Арифметика: `ADD`, `SUB`, `IMUL`, `IDIV`
Здесь всё просто:
`ADD RAX, 5` — прибавить 5 к значению в RAX.
`SUB RBX, RCX` — вычесть из RBX значение RCX.

### 3. Сравнение и прыжки: `CMP` и `Jxx`
Помните про [[8. Предсказание ветвлений. Branch Prediction и Спекулятивное исполнение]]? Вот так они выглядят в коде:
`CMP RAX, 10` — сравнить значение в RAX с числом 10 (на самом деле АЛУ вычитает их и ставит флаги).
`JE label` — **Jump if Equal**. Если результат сравнения был "равно", прыгнуть по адресу `label`.

### 4. Функции: `CALL` и `RET`
`CALL function_name` — сохранить текущий `RIP` в стек и прыгнуть по адресу функции.
`RET` — взять адрес из стека и вернуть `RIP` обратно, чтобы продолжить выполнение после вызова.

## Практика: Смотрим в "кишки" Go-функции

Давайте возьмем простейшую функцию на Go и посмотрим, во что она превращается.

```go
package main

func Add(a, b int) int {
	return a + b
}

func main() {
	Add(10, 20)
}
```

Чтобы увидеть ассемблерный выхлоп, в Go есть встроенный инструмент. Выполните в терминале:
`go tool compile -S main.go`

Вы увидите примерно такой вывод (упрощенно для x86-64):

```asm
TEXT main.Add(SB), NOSPLIT, $0-24
    MOVQ SP+8(FP), AX    ; Загрузить аргумент 'a' из стека в RAX
    MOVQ SP+16(FP), CX   ; Загрузить аргумент 'b' из стека в RCX
    ADDQ CX, AX          ; Сложить AX и CX, результат в AX
    RET                  ; Вернуть AX (как результат функции)
```

**Разбор по шагам:**
1. `MOVQ SP+8(FP), AX`: Процессор лезет в стек (память), находит там первое число (`a`) и копирует его на "верстак" в регистр `AX`.
2. `MOVQ SP+16(FP), CX`: То же самое для второго числа (`b`) в регистр `CX`.
3. `ADDQ CX, AX`: АЛУ складывает их. Результат теперь лежит в `AX`.
4. `RET`: Функция завершается. Поскольку результат по контракту ISA лежит в `RAX`, вызывающая сторона просто берет значение из этого регистра.

> [!info] Под капотом: ABI и Регистры
> Раньше Go передавал все аргументы функций через стек (как в примере выше), что было медленно. Но начиная с версии 1.17, в Go ввели **Register-based ABI**. Теперь аргументы передаются напрямую через регистры (RAX, RBX и т.д.), минуя медленную память. Это дало прирост производительности всей экосистемы Go на 5-10% без изменения кода!

## Mechanical Sympathy: Почему это важно для бэкендера?

Знание ассемблера позволяет вам видеть вещи, которые скрыты за синтаксисом Go:

1. **Инлайнинг (Inlining)**: Если вы увидите в выводе `go tool compile -S`, что вызова `CALL` для маленькой функции нет, а её код вставлен прямо в `main` — значит, компилятор сделал инлайнинг. Это убирает накладные расходы на сохранение контекста и прыжки по `RIP`.
2. **Стоимость интерфейсов**: Когда вы вызываете метод через интерфейс, в ассемблере вы увидите не простой `CALL`, а сложную операцию поиска адреса функции в `itab` (интерфейсном столе). Это объясняет, почему вызовы через интерфейсы чуть медленнее прямых вызовов.
3. **Промахи по кэшу**: Если в ассемблере вы видите много инструкций `MOV` из памяти с большими смещениями, это сигнал, что ваши данные разбросаны по куче, и вы, скорее всего, бьете по [[12. Кэши CPU (L1, L2, L3) и Кэш-линии|кэш-линиям]], вызывая тормоза.

> [!tip] Собеседование
> **Вопрос:** Что такое "подогрев" (warming up) кода или JIT-компиляция, и есть ли это в Go?
> **Ответ:** В языках с JIT (Java, C#, JavaScript) код сначала выполняется медленно (интерпретатор), а затем "горячие" участки перекомпилируются в оптимизированный машинный код на лету. В Go **нет JIT**. Код компилируется в машинные инструкции один раз (AOT — Ahead-of-Time). Это значит, что Go-приложение работает с максимальной скоростью с первой же миллисекунды после запуска, без фазы "прогрева".

## Итог

1. **Ассемблер** — это текстовое представление машинного кода.
2. **Регистры** — сверхбыстрая память внутри CPU, где происходят все реальные вычисления.
3. **Потоки данных**: Память (RAM) $\rightarrow$ Регистры $\rightarrow$ АЛУ $\rightarrow$ Регистры $\rightarrow$ Память.
4. **Инструментарий**: `go tool compile -S` — ваш главный микроскоп для изучения работы программы.
