## Конфликт ядер и иллюзия единства

В статье [[12. Кэши CPU (L1, L2, L3) и Кэш-линии]] мы рассматривали работу кэша в рамках одного ядра. Но современные процессоры — многоядерные. У каждого ядра есть свои собственные «быстрые» кэши L1 и L2, но общая оперативная память (RAM) и общий кэш L3.

Здесь возникает фундаментальная проблема: **Когерентность кэшей (Cache Coherence)**.

Представьте ситуацию: 
1. Ядро №1 читает переменную `count` из RAM в свой L1 кэш. Значение = `10`.
2. Ядро №2 читает ту же переменную `count` в свой L1 кэш. Значение = `10`.
3. Ядро №1 увеличивает `count` до `11` в своем L1.
4. Ядро №2 читает `count` из своего L1. Он всё ещё видит `10`!

Если бы всё оставалось так, многопоточное программирование было бы невозможно, так как каждое ядро видело бы свою «версию реальности». Чтобы этого не произошло, процессоры используют сложные протоколы синхронизации.

## Протокол MESI: Как ядра договариваются

Самый распространенный способ поддерживать когерентность — это протокол **MESI** (назван по первым буквам четырех состояний кэш-линии). Каждый байт в вашем L1 кэше помечен одним из этих состояний:

*   **M (Modified)**: Эта линия изменена мной, и она единственная актуальная версия в системе. В RAM сейчас лежат устаревшие данные. Если другому ядру понадобятся эти данные, я обязан их предоставить.
*   **E (Exclusive)**: Эта линия есть только у меня, и она идентична данным в RAM. Я могу изменить её в любой момент, переведя в состояние `M`, не уведомляя других.
*   **S (Shared)**: Эта линия есть и у меня, и у других ядер. Мы все просто читаем её.
*   **I (Invalid)**: Эта линия устарела (мусор). Её нельзя использовать, нужно заново запрашивать данные из L3 или RAM.

### Как это работает на практике (Bus Snooping)

Ядра постоянно «подслушивают» (Snooping) общую шину данных. 

Когда Ядро №1 хочет изменить переменную в линии, которая находится в состоянии **S (Shared)**, оно посылает по шине сигнал: *«Я собираюсь изменить эту линию, всем остальным — пометить её как I (Invalid)!»*. 

Ядро №2 слышит этот сигнал, видит, что у него в L1 лежит та же самая линия, и мгновенно помечает её как **I (Invalid)**. Теперь, когда Ядро №2 снова попытается прочитать `count`, оно увидит состояние `I`, поймает `Cache Miss` и будет вынуждено забрать актуальное значение `11` из кэша Ядра №1 или из L3.

> [!info] Под капотом
> Этот процесс называется **Cache Coherence Traffic**. Чем больше ядер пытаются одновременно изменить одну и ту же переменную, тем больше «шума» создается на шине. Процессор тратит больше времени на рассылку сигналов «инвалидации», чем на реальные вычисления. Это одна из главных причин, почему чрезмерная синхронизация (много мьютексов или атомиков на одну переменную) убивает производительность.

## False Sharing: Невидимый убийца производительности

Теперь мы подходим к самому коварному эффекту в многопоточном программировании — **False Sharing (Ложное разделение)**.

Помните из статьи [[12. Кэши CPU (L1, L2, L3) и Кэш-линии]], что процессор читает данные **линиями по 64 байта**? 

**Проблема возникает, когда две разные переменные, которые используются разными ядрами, случайно оказываются в одной кэш-линии.**

Представьте структуру:
```go
type Stats struct {
    Core1Count uint64 // 8 байт
    Core2Count uint64 // 8 байт
}
```
Эти две переменные лежат в памяти вплотную. Вместе они занимают 16 байт, а значит — **гарантированно попадают в одну кэш-линию (64 байта)**.

1. Ядро №1 постоянно обновляет `Core1Count`.
2. Ядро №2 постоянно обновляет `Core2Count`.

С точки зрения вашего кода на Go — это разные переменные, они не конфликтуют. Но с точки зрения железа — они находятся в **одной линии**. 
Когда Ядро №1 меняет `Core1Count`, оно помечает всю линию в кэше Ядра №2 как **Invalid (I)**. Ядро №2, чтобы обновить свою переменную `Core2Count`, вынуждено снова запрашивать всю линию из памяти. И так по кругу.

Ядра начинают «пинать» эту кэш-линию друг другу туда-сюда, создавая огромный трафик на шине. Это называется **Ping-pong effect**. Производительность падает в десятки раз, хотя вы не используете ни одного мьютекса!

## Mechanical Sympathy: Как лечить False Sharing в Go

Чтобы избавиться от False Sharing, нужно разнести конфликтующие переменные по разным кэш-линиям. Это делается с помощью **Padding (отступов)**.

Мы должны добавить в структуру «пустышки», чтобы переменные были разнесены минимум на 64 байта.

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

// Плохая структура: вызывает False Sharing
type BadStats struct {
	Core1Count uint64
	Core2Count uint64
}

// Хорошая структура: переменные разнесены по разным кэш-линиям
type GoodStats struct {
	Core1Count uint64
	_          [56]byte // Padding: заполняем остаток кэш-линии (64 - 8 = 56)
	Core2Count uint64
}

func main() {
	// Тест BadStats
	bad := &BadStats{}
	start := time.Now()
	var wg sync.WaitGroup
	wg.Add(2)
	go func() {
		for i := 0; i < 100_000_000; i++ { bad.Core1Count++ }
		wg.Done()
	}()
	go func() {
		for i := 0; i < 100_000_000; i++ { bad.Core2Count++ }
		wg.Done()
	}()
	wg.Wait()
	fmt.Printf("BadStats time: %v\n", time.Since(start))

	// Тест GoodStats
	good := &GoodStats{}
	start = time.Now()
	wg.Add(2)
	go func() {
		for i := 0; i < 100_000_000; i++ { good.Core1Count++ }
		wg.Done()
	}()
	go func() {
		for i := 0; i < 100_000_000; i++ { good.Core2Count++ }
		wg.Done()
	}()
	wg.Wait()
	fmt.Printf("GoodStats time: %v\n", time.Since(start))
}
```

> [!tip] Собеседование
> **Вопрос:** Вы заметили, что приложение с высокой степенью параллелизма работает медленнее при добавлении новых ядер CPU, хотя мьютексов в коде почти нет. В чем может быть причина?
> **Ответ:** Вероятно, имеет место **False Sharing**. Несколько часто изменяемых переменных попали в одну кэш-линию (64 байта), что вызывает постоянную инвалидацию кэшей между ядрами и перегрузку системной шины (MESI-трафик). Лечится добавлением паддинга (отступов) в структуры, чтобы разнести переменные по разным линиям.

## Итог

1. **Когерентность кэшей** — это механизм, который гарантирует, что все ядра видят актуальные данные.
2. **Протокол MESI** управляет состояниями линий (Modified, Exclusive, Shared, Invalid) и инвалидирует устаревшие данные в других ядрах.
3. **False Sharing** — это когда разные переменные в одной кэш-линии заставляют ядра бесконечно синхронизировать эту линию, что катастрофически замедляет код.
4. **Решение** — выравнивание и паддинг данных так, чтобы горячие переменные жили в разных кэш-линиях.

Мы разобрались с тем, как ядра CPU общаются между собой.
