## Проблема простоя железа

В статье [[6. Цикл исполнения инструкции. Instruction Cycle]] мы рассматривали классический цикл работы процессора: **Fetch $\rightarrow$ Decode $\rightarrow$ Execute $\rightarrow$ Write-back**. 

В старых процессорах (до 1990-х годов) этот цикл выполнялся строго последовательно. Устройство управления (CU) читало инструкцию из памяти, АЛУ её вычисляло, результат записывался в регистр. 
В чём проблема такого подхода? **Чудовищная неэффективность.**
Пока АЛУ занимается математикой (Execute), подсистемы выборки из памяти (Fetch) и декодирования (Decode) просто простаивают, потребляя электричество. 

Инженеры поняли: чтобы ускорить процессор, не обязательно бесконечно повышать тактовую частоту генератора (GHz). Нужно заставить блоки процессора работать параллельно.

## Конвейеризация (Pipelining)

Решение было подсмотрено у Генри Форда и конвейерных лент. Если разбить выполнение инструкции на независимые стадии, мы сможем запускать новую инструкцию на первой стадии еще до того, как предыдущая инструкция дойдет до последней стадии.

Классический пример — стирка одежды. Вы не ждете, пока одежда постирается, высохнет и будет поглажена, чтобы закинуть в стиральную машину следующую партию. Как только стиралка освободилась — вы загружаете новую партию, пока первая сушится.

В процессоре это выглядит так:

| Такт (Clock) | 1 | 2 | 3 | 4 | 5 | 6 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: |
| **Инструкция 1** | Fetch | Decode | Execute | Write | | |
| **Инструкция 2** | | Fetch | Decode | Execute | Write | |
| **Инструкция 3** | | | Fetch | Decode | Execute | Write |
| **Инструкция 4** | | | | Fetch | Decode | Execute |

При последовательном выполнении 4 инструкции заняли бы **16 тактов** (4х4).
Благодаря конвейеру, на 4-м такте процессор обрабатывает сразу четыре инструкции одновременно! В идеальных условиях, начиная с 4-го такта, процессор завершает по одной инструкции за **каждый такт**. 

>[!info] Под капотом
> В современных процессорах (например, архитектурах Intel Core или AMD Zen) конвейер состоит не из 4, а из **14-19 стадий**. Это называется *глубоким конвейером* (Superpipelining). Чем больше стадий, тем меньший объем работы делает транзистор на каждой стадии, а значит, можно сильнее повысить тактовую частоту процессора (до 4-5 ГГц).

## Проблемы конвейера (Hazards)

Конвейер звучит идеально, но в реальности он постоянно натыкается на препятствия, заставляющие его останавливаться. Эти остановки называются **Pipeline Stalls** (или Bubbles — пузыри).

### 1. Зависимость по данным (Data Hazards)
Посмотрите на этот Go-код:
```go
a := 10
b := a + 5
c := b * 2
```
Процессор не может вычислить `c` на 3-м такте конвейера, потому что для этого ему нужно значение `b`. А значение `b` еще не было записано в регистр на стадии *Write-back* предыдущей инструкции! Конвейер вынужден замереть и ждать, пропуская такты (вставляя пустые операции — `NOP`).

### 2. Конфликт управления (Control Hazards)
Что происходит с конвейером, если в коде есть `if`?
Устройство управления забирает инструкции из памяти (Fetch) заранее. Но если инструкция 1 — это условный переход (`JMP`), процессор не знает, перепрыгнем мы в блок `if` или нет, пока не дойдет до стадии *Execute*. А в этот момент в конвейер (на стадии Fetch и Decode) уже загружены следующие по порядку инструкции! Если условие срабатывает, процессор вынужден **сбросить весь конвейер** (Pipeline Flush) и начать читать инструкции по новому адресу. Это колоссальная потеря производительности.

## Суперскалярность и ILP

Конвейер позволяет выполнять 1 инструкцию за 1 такт (в идеале). Но как достичь показателя в 2, 3 или 4 инструкции за такт?
Эта метрика называется **IPC (Instructions Per Cycle)**.

> [!tip] Собеседование
> **Вопрос:** Почему современный процессор на 3 ГГц в несколько раз быстрее старого Pentium 4, который работал на частоте почти 4 ГГц?
> **Ответ:** Разница в IPC. Производительность CPU = Частота (Clock Rate) × IPC. Старые процессоры выполняли ~1 инструкцию за такт. Современные **суперскалярные** процессоры могут выполнять по 4-6 инструкций за один такт.

Чтобы стать **суперскалярным**, процессору добавили мускулов. Вместо одного АЛУ в него встроили несколько вычислительных блоков:
* 4 целочисленных АЛУ (ALU)
* 2 блока работы с плавающей запятой (FPU)
* 2 блока работы с памятью (Load/Store Units)

Теперь процессор может загрузить в конвейер сразу 4 инструкции `ADD`, если они независимы друг от друга. Это называется **ILP (Instruction-Level Parallelism** — параллелизм на уровне инструкций). Не путайте это с горутинами — ILP происходит аппаратно внутри одного логического ядра (треда).

## Исполнение вне порядка (Out-of-Order Execution)

Но как загрузить все эти многочисленные АЛУ, если в коде постоянно встречаются зависимости по данным? 
Инженеры придумали черную магию аппаратного уровня — **Out-of-Order Execution (OoOE)**.

Устройство управления в современных CPU не выполняет машинный код в том порядке, в котором его сгенерировал компилятор Go. Процессор имеет специальный буфер (Reorder Buffer), куда заглядывает на десятки инструкций вперед. 
Он динамически анализирует зависимости: "Ага, инструкции 1 и 2 зависят друг от друга, я их отложу. А вот инструкции 3, 4 и 5 независимы, я выполню их прямо сейчас на свободных АЛУ!". 
А после выполнения процессор "склеивает" результаты обратно в правильном порядке, так что программа (и программист) даже не подозревает о том, что инструкции выполнялись вразнобой.

## Mechanical Sympathy: Зависимости и Атомики в Go

Как эти знания помогают писать быстрый бэкенд на Go? 
Главное правило для достижения максимального ILP (суперскалярности): **минимизируйте длинные цепочки зависимостей данных в горячих циклах.**

Сравним два подхода к вычислениям в плотном цикле:

```go
package main

// Подход 1: Строгая зависимость по данным (Pipeline Stalls)
// Каждая итерация ждет результата предыдущей. IPC падает.
func SumArrayBad(arr[]int) int {
	sum := 0
	for i := 0; i < len(arr); i++ {
		sum += arr[i] // Зависимость от предыдущего значения sum
	}
	return sum
}

// Подход 2: Разворачивание цикла (Loop Unrolling) для ILP
// Независимые сумматоры. Процессор выполнит sum1 += и sum2 += 
// одновременно на разных АЛУ за 1 такт!
func SumArrayGood(arr[]int) int {
	sum1, sum2 := 0, 0
	// Шагаем по 2 элемента. 
	for i := 0; i < len(arr)-1; i += 2 {
		sum1 += arr[i]   
		sum2 += arr[i+1] 
	}
	return sum1 + sum2
}
```

>[!info] Под капотом
> К счастью, компилятор Go использует форму SSA (Static Single Assignment) при генерации машинного кода. Для простых циклов компилятор Go делает разворачивание (loop unrolling) автоматически. Но для сложных алгоритмов понимание ILP критически важно для ручной оптимизации.

Еще один важнейший аспект — **пакет `sync/atomic` и мьютексы (`sync.Mutex`)**.
Почему атомарные операции работают медленнее обычного `counter++`? 

```go
import "sync/atomic"

var counter int64

// Выполняется очень быстро (суперскалярно, в кэше L1)
func unsafeInc() {
	counter++ 
}

// Медленнее на порядки (на уровне железа)
func safeInc() {
	atomic.AddInt64(&counter, 1)
}
```
Атомарная операция (на архитектуре x86-64 это префикс `LOCK` перед инструкцией `ADD`) подает сигнал процессору: **"Останови Out-of-Order Execution, сбрось конвейер и заблокируй кэш-линию L1"**. 
Атомарные операции буквально отключают всю аппаратную магию ускорения, заставляя суперскалярный процессор работать как простейший последовательный калькулятор из 1980-х. Именно поэтому в высоконагруженных системах избегают глобальных счетчиков (используя шардирование счетчиков по горутинам).

## Итог

1. **Конвейеризация** позволяет накладывать стадии выполнения разных инструкций друг на друга.
2. **Суперскалярность** позволяет выполнять несколько инструкций за один такт (IPC > 1).
3. **OoOE** переставляет инструкции местами "на лету", чтобы избежать простоя АЛУ из-за зависимостей по данным.

Мы решили проблему зависимостей данных. Но у нас осталась главная угроза конвейеру — операторы `if` и ветвления (Control Hazards), которые заставляют сбрасывать конвейер. О том, как процессоры научились "заглядывать в будущее", чтобы решить эту проблему, читайте в статье: [[8. Предсказание ветвлений. Branch Prediction и Спекулятивное исполнение]].
