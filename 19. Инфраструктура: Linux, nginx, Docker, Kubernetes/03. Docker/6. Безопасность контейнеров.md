
Миф о том, что контейнеры обеспечивают полную изоляцию, подобно виртуальным машинам, — одна из самых опасных иллюзий в инфраструктуре. Контейнеры — это просто процессы с ограниченной видимостью (Namespaces) и потреблением (Cgroups), но они **разделяют ядро хоста**.

Если злоумышленник найдет уязвимость нулевого дня в ядре Linux или вы неправильно сконфигурируете контейнер, процесс вырвется из своей песочницы (Container Escape) и получит контроль над хост-системой и всеми соседними контейнерами. Безопасность контейнеров — это не бинарная величина, а глубоко эшелонированная оборона (Defense in Depth).

## Грех первый: Запуск от Root

Как мы обсуждали в статье [[4. Права доступа и безопасность]], UID 0 — это бог. По умолчанию Docker запускает контейнеры от root. Даже если внутри контейнера процесс изолирован в своем PID Namespace, он всё равно имеет UID 0 на хосте.

Если злоумышленник через RCE (Remote Code Execution) в вашем Go-коде получит доступ к shell внутри контейнера, он сразу же станет root. А если на хосте примонтирована директория с правами записи, злоумышленник сможет изменить файл `/etc/crontab` на хосте или подменить бинарники других сервисов.

> [!warning] Ловушка / Gotcha
> Многие разработчики думают: "Я использую User Namespace (параметр `--userns-remap` демона Docker), поэтому мой root внутри контейнера — это обычный пользователь на хосте, всё безопасно". 
> Проблема в том, что по умолчанию Docker **не включает** User Namespaces. И даже если вы включите их, это сломает многие функции (например, привязку к портам < 1024 без `CAP_NET_BIND_SERVICE` и монтирование NFS). Единственный надежный путь — **всегда использовать директиву `USER` в Dockerfile**.

## Linux Capabilities: Разделение полномочийRoot

Даже если вы запускаете контейнер от непривилегированного пользователя, ему могут быть выданы опасные привилегии. По умолчанию Docker выдает контейнерам набор из ~14 Linux Capabilities (например, `CAP_NET_RAW`, `CAP_CHOWN`).

Для типичного Go-бэкенда, который слушает HTTP на порту > 1024 и ходит в БД, подавляющее большинство capabilities не нужны.

> [!info] Под капотом
> Когда процесс делает системный вызов, требующий привилегий (например, `chown` или создание raw-сокета), ядро проверяет наличие соответствующего флага в структуре `cred` процесса, а не просто проверяет `uid == 0`. Удалив `CAP_NET_RAW`, вы лишаете злоумышленника возможности делать ping-флуд или IP-спуфинг даже если он захватит ваш контейнер.

**Безопасный запуск (Principle of Least Privilege):**
```bash
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE my-go-app
```
Сначала мы сбрасываем *все* capabilities, а затем добавляем только те, которые реально нужны (например, возможность слушать 80 порт).

## Seccomp: Фильтрация системных вызовов

**Seccomp (Secure Computing Mode)** — это механизм ядра Linux, позволяющий ограничить системные вызовы, которые может делать процесс. 

Go-рантайм использует очень ограниченный набор системных вызовов (эпохи epoll, чтение/запись, mmap и т.д.). Ему абсолютно не нужны `ptrace` (отладка), `mount` (монтирование ФС) или `reboot`.

Docker использует профиль Seccomp по умолчанию, который блокирует около 44 опасных системных вызовов (из сотен доступных в Linux). Но для продакшена лучше писать кастомные профили.

> [!tip] Собеседование
> **Вопрос:** Как злоумышленник может использовать системный вызов `ptrace` внутри контейнера, даже если он не root?
> **Ответ:** Если в контейнере не отключена capability `CAP_SYS_PTRACE` и не заблокирован syscall `ptrace` через Seccomp, злоумышленник, получивший доступ к контейнеру, может "прицепиться" (attach) к другим процессам внутри контейнера и читать их память. Если другой процесс хранит секреты (токены, пароли БД) в памяти, злоумышленник их украдет. Go-рантайму `ptrace` не нужен, его необходимо блокировать.

## Специфика Go: Статика против CGO

С точки зрения безопасности, у Go есть огромные преимущества перед C++, Python или Java:

1. **Отсутствие `LD_PRELOAD` атак**: Поскольку Go-бинарник при `CGO_ENABLED=0` линкуется статически и использует собственный загрузчик (в обход `glibc`), классическая атака через подмену динамических библиотек (например, запись зловредного `libc.so.6` в директорию) не сработает. Go просто не будет ее читать.
2. **Меньше Syscalls**: Статический Go-бинарник делает предсказуемый набор системных вызовов. Написать строгий Seccomp-профиль для Go-приложения намного проще, чем для Python, который динамически загружает модули и вызывает непредсказуемые syscalls.
3. **Distroless / Scratch**: Как обсуждалось в [[3. Multi stage build]], использование пустых образов означает, что в контейнере нет оболочки (`sh`, `bash`), нет `curl`, `wget` и пакетного менеджера. Если хакер попадет в контейнер через уязвимость в вашем коде (например, небезопасный `os.Exec`), ему будет нечем воспользоваться для горизонтального перемещения (Lateral Movement).

> [!warning] Ловушка / Gotcha
> Как только вы включаете CGO (`CGO_ENABLED=1`), вы теряете эти преимущества. Ваш бинарник начинает использовать системный `glibc`, открывая вектор для атак через уязвимости самой `glibc` (как это было с печально известным GHOST или buffer overflows в `dlopen`). Если CGO неизбежен (например, драйвер SQLite), обновляйте базовые образы (Alpine/Debian) еще чаще.

## Supply Chain Security: Уязвимости зависимостей

Ваш Go-код может быть идеален, но если в базовом образе Alpine или в сторонней библиотеке (например, `github.com/gorilla/websocket`) есть CVE (уязвимость), ваш контейнер скомпрометирован.

1. **Сканирование образов**: Интегрируйте в CI/CD пайплайн сканеры уязвимостей, такие как **Trivy** или **Snyk**. Они сверяют пакеты в образе с базами данных CVE.
   ```bash
   trivy image my-registry/my-go-app:latest
   ```
2. **Фиксация версий базового образа**: Никогда не используйте `FROM golang:latest`. Используйте точные дайджесты (SHA256) или минорные версии (например, `FROM golang:1.22.3-alpine3.19`), чтобы избежать непредсказуемых изменений.

## Защита секретов (Secrets)

Никогда не передавайте пароли, API-ключи или TLS-сертификаты напрямую в коде или через переменные окружения в Dockerfile (они остаются в истории слоев образа). 

В Docker есть механизм **Docker Secrets** (в Swarm) или просто монтирование секретов из внешних хранилищ (HashiCorp Vault, AWS Secrets Manager). В Kubernetes для этого существуют [[3. ConfigMap и Secret]].

Если вам нужно пробросить секрет в Go-контейнер для локальной разработки, используйте флаг `--secret` или монтируйте файл как Read-Only Volume:
```bash
docker run -v $(pwd)/db_password.txt:/run/secrets/db_password:ro my-go-app
```

## Итог

1. **Не запускайте от Root**: `USER appuser` — обязательное правило. Контейнер с root-процессом — это мина замедленного действия.
2. **Cap-drop ALL**: Удаляйте все Linux Capabilities и добавляйте только необходимые.
3. **Seccomp**: Ограничивайте системные вызовы. Go идеально подходит для жестких Seccomp-профилей.
4. **Scratch / Distroless**: Минимизация инструментов внутри контейнера усложняет жизнь хакеру при захвате.
5. **CGO — это риск**: Избегайте CGO, чтобы закрыть векторы атак на динамический линковщик и использовать статическую безопасность Go.

Изоляция на уровне Docker — это отличный фундамент, но управление тысячами контейнеров, их сетями, секретами и масштабированием требует оркестратора. Мы переходим к вершине инфраструктурной эволюции — Kubernetes. В следующей статье мы разберем его архитектуру: [[1. Kubernetes. Архитектура]].
