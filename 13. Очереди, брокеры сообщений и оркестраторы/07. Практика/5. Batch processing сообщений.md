## Закон амортизации в распределенных системах

В статьях про [[3. Параллелизм обработки сообщений]] мы ускорили обработку, распараллелив вычисления. Но параллелизм не спасет, если каждая итерация цикла делает сетевой запрос в базу данных. 

Если вы вставляете 1000 строк в PostgreSQL по одной, вы совершаете 1000 сетевых Round-Trip Time (RTT) и 1000 транзакционных коммитов (fsync на диске WAL-лога). В высоконагруженных системах узким местом является не CPU, а **I/O и сеть**. 

Решение — **Batch Processing (пакетная обработка)**. Вместо `INSERT INTO tbl VALUES ($1)` мы делаем `INSERT INTO tbl VALUES ($1), ($2), ... ($1000)`. Это амортизирует стоимость одного сетевого прыжка и одного дискового I/O на сотни записей.

## Под капотом: Слайсы, Escape Analysis и GC

Для Go-разработчика пакетная обработка означает накопление сообщений в слайсе. И вот тут кроется главная ловушка управления памятью.

Когда вы делаете `append(msg, batch)`, рантайм Go проверяет, хватает ли емкости (cap). Если нет, Go аллоцирует новый массив в куче (Heap) большего размера (обычно в 2 раза больше старого), копирует туда данные старого слайса и добавляет новый элемент. 

> [!info] Под капотом
> При пакетной обработке в цикле постоянные `append` с превыщениемcapacity вызовут множественные аллокации в куче. Сборщик мусора (GC) будет вынужден постоянно сканировать и освобождать старые массивы слайсов.
> 
> **Правило Mechanical Sympathy:** Если вы знаете размер батча (например, 1000), всегда выделяйте слайс с запасом емкости на старте: `batch := make([]Message, 0, 1000)`. Это заставит рантайм выделить один непрерывный кусок памяти. При заполнении батча и его очистке через `batch = batch[:0]`, мы **не освобождаем память**, а просто сбрасываем длину (len). Бэкапирующий массив (backing array) остается в куче и переиспользуется. Это снижает нагрузку на GC до нуля.

## Идиоматичный Batch Collector на Go

Существует два триггера для отправки батча:
1. **Размер (Size):** Накопили N сообщений — отправили.
2. **Время (Time):** Сообщений мало, но прошло T секунд. Если ждать дальше, нарушится SLA обработки (увеличится Lag). Нужно отправлять то, что есть.

В Go паттерн "Size + Time" элегантно реализуется через `select` с каналом и `time.Ticker`.

```go
type BatchProcessor struct {
	batchSize    int
	flushInterval time.Duration
	process      func(ctx context.Context, batch []Message) error
}

func (p *BatchProcessor) Run(ctx context.Context, ch <-chan Message) error {
	// Предвыделяем слайс с нужной емкостью!
	batch := make([]Message, 0, p.batchSize)
	
	// ВАЖНО: Используем time.NewTicker, а не time.After в select!
	// time.After создает новый канал (и таймер) на каждую итерацию цикла,
	// который не очистится GC, пока не истечет его время.
	ticker := time.NewTicker(p.flushInterval)
	defer ticker.Stop()

	for {
		select {
		case <-ctx.Done():
			// При получении SIGTERM мы обязаны сбросить остатки батча
			if len(batch) > 0 {
				if err := p.process(ctx, batch); err != nil {
					return fmt.Errorf("flush on shutdown: %w", err)
				}
			}
			return nil

		case msg, ok := <-ch:
			if !ok {
				return nil // Канал закрыт
			}
			batch = append(batch, msg)
			// Триггер по размеру
			if len(batch) >= p.batchSize {
				if err := p.process(ctx, batch); err != nil {
					return fmt.Errorf("batch process: %w", err)
				}
				// Очищаем батч, сохраняя емкость (cap)
				batch = batch[:0] 
				// Сбрасываем тикер, чтобы не отправлять пустой батч сразу после заполнения по размеру
				ticker.Reset(p.flushInterval) 
			}

		case <-ticker.C:
			// Триггер по времени
			if len(batch) > 0 {
				if err := p.process(ctx, batch); err != nil {
					return fmt.Errorf("ticker batch process: %w", err)
				}
				batch = batch[:0]
			}
		}
	}
}
```

## Брокеры и нативный Batching

Важно понимать, как ваш брокер работает "под капотом". В статье [[5. Prefetch и QoS]] мы обсуждали RabbitMQ, а в [[3. Producer и consumer]] — Kafka. У них кардинально разный подход к батчам.

### Kafka: Батч — это базовая единица
Kafka физически хранит данные сегментами, а продюсер отправляет их пачками (`linger.ms` и `batch.size`). Консьюмер Kafka тоже вычитывает данные батчами (Fetch Request). Когда вы итерируетесь по `*kafka.ConsumerMessage`, вы уже читаете данные из сетевого буфера, который пришел одним куском. 
Поэтому в Kafka ваш "БД-батч" можно собирать прямо в цикле `for`, читая сообщения из канала (как в коде выше), и делать `CommitOffsets` после успешного `INSERT` в БД.

### RabbitMQ: Батчинга нет, есть Prefetch
RabbitMQ отправляет сообщения по одному, как только они появляются в очереди. Если вы используете `qos.PrefetchCount = 1000`, брокер просто зафloodит ваш консьюмер тысячей сообщений по TCP, создавая огромную нагрузку на память. В RabbitMQ собирать батчи сложнее: вы должны сами контролировать, как быстро вы Ack'аете сообщения, чтобы не словить OOM.

## Ловушки батчинг-обработки

### 1. Проблема Poison Message в батче
Это самый частый вопрос на хардовых собеседованиях. Что, если вы собрали батч из 500 сообщений, сделали `INSERT ... VALUES (...), ...`, и база данных вернула ошибку `unique_violation` или `foreign_key_violation`? Весь батч откатился. 

Если вы просто заNACKаете весь батч в брокер, при следующей попытке 499 валидных сообщений будут пытаться обработаться снова вместе с 1 битым. И так до бесконечности (Poison Message остановит обработку всей партиции/очереди).

> [!tip] Собеседование
> **Вопрос:** Как обработать ошибку БД при вставке батча, если одно из сообщений невалидно (Poison Message)?
> **Ответ:** Паттерн **Batch Fallback to Single**. Если батч-запрос упал, мы не ретраим весь батч. Мы разбиваем его и обрабатываем каждое сообщение по одному в отдельной транзакции (или с `INSERT ON CONFLICT IGNORE`). Валидные пройдут, а битое сообщение изолируется, логируется и отправляется в [[8. Dead Letter Queue]] или [[6. Handling poison messages|обрабатывается отдельным механизмом]].

Пример реализации Batch Fallback:

```go
func (p *BatchProcessor) processWithFallback(ctx context.Context, batch []Message) error {
	// Пытаемся вставить батч
	err := p.db.BatchInsert(ctx, batch)
	if err == nil {
		// Успех! Ack'аем все сообщения
		p.ackAll(batch)
		return nil
	}

	// Батч упал. Разбиваем на единичные обработки.
	p.logger.Warn("Batch insert failed, falling back to single processing", "error", err)
	for _, msg := range batch {
		if singleErr := p.db.SingleInsert(ctx, msg); singleErr != nil {
			// Это и есть Poison Message
			p.logger.Error("Poison message detected", "msg_id", msg.ID, "error", singleErr)
			msg.Nack(false) // Отправляем в DLQ без requeue
		} else {
			msg.Ack()
		}
	}
	return nil // Ошибки на уровне батча нет, мы её обработали
}
```

### 2. Идемпотентность и дубликаты в батчах
При использовании [[4. Idempotent handlers]] с `UNIQUE CONSTRAINT` в PostgreSQL, вставка батча может упасть, если *хотя бы одно* сообщение является дубликатом из прошлой неудачной попытки. 
В PostgreSQL есть элегантное решение для батчей — `ON CONFLICT DO NOTHING` или `ON CONFLICT DO UPDATE`. Это позволяет базе данных игнорировать дубликаты прямо внутри батча, не сваливая всю транзакцию.

### 3. Проблема Backpressure
Если ваша БД медленная, а брокер отдает сообщения быстро, `append(batch, msg)` будет бесконечно расти, превышая `cap`, аллоцируя гигабайты в куче и вызывая OOM. 
Размер `ch <-chan Message` и параметр Prefetch в брокере — ваши главные ограничители. Если канал полон, консьюмер заблокируется, и сеть между брокером и приложением будет простаивать, ожидая, пока БД переварит батч.

## Итог

1. Пакетная обработка (Batching) — критически важный инструмент для уменьшения сетевых RTT и дисковых I/O.
2. В Go используйте предвыделение слайсов `make([]T, 0, cap)` и очистку `batch[:0]`, чтобы минимизировать аллокации и помочь GC.
3. Всегда используйте `time.Ticker` в связке с триггером по размеру, чтобы сообщения не застревали в буфере в периоды низкой нагрузки.
4. Помните про ловушку Poison Message: упавший батч нужно ретраить по одной записи (Batch Fallback), иначе одно битое сообщение остановит весь консьюмер.
5. При graceful shutdown критически важно сбросить (flush) остатки частично заполненного батча.

Батчинг значительно ускоряет систему, но что делать с теми сообщениями, которые ломают логику даже при одиночной обработке? В следующей статье мы детально разберем [[6. Handling poison messages]].
