## Иллюзия Exactly-Once и неизбежность дубликатов

В статье [[4. Модели доставки. At most once, at least once, exactly once]] мы выяснили суровую правду распределенных систем: доставка `Exactly-Once` в чистом виде невозможна. Большинство брокеров (Kafka, RabbitMQ) гарантируют `At-Least-Once`. Это значит, что дубликаты сообщений — это не баг, а нормальная часть жизни системы. 

Если консьюмер упал после обработки сообщения, но перед отправкой ACK (как мы видели в [[2. Консьюмеры и graceful shutdown]]), или если сеть моргнула, а таймаут репликации в Kafka истек, брокер отправит сообщение повторно. 

Если ваш обработчик не является **идемпотентным**, один и тот же платеж спишется дважды, один и тот же email улетит клиенту два раза, а счетчик лайков увеличится на 2 вместо 1.

**Идемпотентность** — свойство операции, при котором многократное ее выполнение приводит к тому же результату, что и однократное. В_terms_ обработки сообщений это означает: *если я получу сообщение с `message_id=123` десять раз, состояние системы изменится ровно так же, как если бы я получил его один раз*.

## Откуда берется ключ: Producer vs Broker

Чтобы отбросить дубликат, нам нужно его опознать. Откуда взять уникальный идентификатор?

1. **Брокер (Kafka, RabbitMQ):** Брокер присваивает сообщениям внутренние ID (например, `DeliveryTag` в RabbitMQ или связка `Topic-Partition-Offset` в Kafka). Но использовать их для идемпотентности **нельзя**. При репликации или пересылке сообщения в Dead Letter Queue внутренний ID меняется.
2. **Producer (Правильный подход):** Продюсер генерирует `Idempotency Key` (часто UUIDv7) на своей стороне и кладет его в заголовок сообщения (Headers) или в тело. Этот ключ должен быть глобально уникальным и неизменным на протяжении всего жизненного цикла сообщения.

> [!info] Под капотом
> Почему UUIDv7, а не UUIDv4? В высоконагруженных БД (PostgreSQL) первичные ключи используются для кластерных индексов (B-Tree). UUIDv4 абсолютно случаен, что приводит к случайным вставкам в разные страницы индекса (Page Splits). Это вызывает трэшинг кэша CPU (Cache Misses на уровне L1/L2) и деградацию IOPS диска.
> UUIDv7 содержит префикс временной метки (millisecond unix timestamp). Такие ключи монотонно возрастают, поэтому вставки идут в "горячий" правый край B-Tree. Это обеспечивает локальность данных (Data Locality) и потрясающую производительность на запись.

## Паттерны идемпотентности в Go

Существует три фундаментальных подхода к обеспечению идемпотентности, от простых к сложным.

### 1. Optimistic Locking через БД (Естественная идемпотентность)

Самый надежный способ — положиться на ограничения целостности данных в вашей БД (PostgreSQL). Если бизнес-логика позволяет, мы используем `UNIQUE CONSTRAINT` на идемпотентный ключ или бизнес-поле.

**Пример:** Обработка транзакции пополнения баланса.

```sql
-- Схема БД
CREATE TABLE transactions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    idempotency_key VARCHAR(255) UNIQUE NOT NULL, -- Ключ от продюсера
    user_id BIGINT NOT NULL,
    amount DECIMAL(19,2) NOT NULL
);
```

В Go мы делаем `INSERT`. Если мы получим дубликат, БД выбросит ошибку нарушения уникального ограничения (в `pgx` это `pgerrcode.UniqueViolation`).

```go
func (h *Handler) HandleTopUp(ctx context.Context, msg Message) error {
	var event TopUpEvent
	if err := json.Unmarshal(msg.Body, &event); err != nil {
		return fmt.Errorf("unmarshal: %w", err)
	}

	_, err := h.db.Exec(ctx, `
		INSERT INTO transactions (idempotency_key, user_id, amount) 
		VALUES ($1, $2, $3)`, 
		event.IdempotencyKey, event.UserID, event.Amount,
	)

	if err != nil {
		var pgErr *pgconn.PgError
		// Проверяем, что ошибка - это нарушение UNIQUE CONSTRAINT (код 23505)
		if errors.As(err, &pgErr) && pgErr.Code == pgerrcode.UniqueViolation {
			// Дубликат! Операция уже была выполнена ранее.
			// Мы НЕ возвращаем ошибку, чтобы консьюмер мог сделать ACK.
			h.logger.Warn("Дубликат транзакции, пропускаем", "key", event.IdempotencyKey)
			return nil 
		}
		// Другая ошибка БД (нет соединения, диск отвалился) - возвращаем ошибку
		return fmt.Errorf("db exec: %w", err)
	}

	// Успешная вставка (первая обработка)
	return nil
}
```

> [!warning] Ловушка / Gotcha
> Важнейший момент в коде выше: при обнаружении дубликата мы возвращаем `nil`! Если вы вернете ошибку, консьюмер посчитает, что сообщение не обработано, и попытается обработать его снова (NACK + Retry). Это создаст бесконечный цикл (Poison Message) из-за того, что БД всегда будет отказывать в INSERT, а консьюмер всегда будет ретраить.

### 2. Отдельное хранилище идемпотентности (Redis / Memcached)

Подход выше идеален, если сама суть операции — это запись в БД. Но что, если обработка сообщения — это вызов стороннего API (например, отправка SMS через Twilio)? Мы не можем сделать `UNIQUE CONSTRAINT` в API провайдера. Нам нужно состояние *до* вызова API.

Паттерн: сохраняем ключ в распределенном кэше (Redis) с TTL.
1. Пытаемся записать ключ в Redis через `SET key value NX EX 3600`.
2. Если `NX` вернул `OK` — ключа не было, выполняем работу.
3. Если `NX` вернул `nil` — дубликат, пропускаем.

```go
func (h *Handler) HandleSendSMS(ctx context.Context, msg Message) error {
	var event SMSEvent
	// ...
	
	// Пытаемся захватить блокировку по ключу идемпотентности
	set, err := h.redis.SetNX(ctx, "idemp:"+event.IdempotencyKey, 1, 1*time.Hour).Result()
	if err != nil {
		return fmt.Errorf("redis setnx: %w", err) // Ошибка связи с Redis -> NACK
	}
	if !set {
		// Ключ уже существует -> Дубликат
		return nil 
	}

	// Вызов стороннего API
	_, err = h.twilioClient.SendMessage(ctx, event.Phone, event.Text)
	if err != nil {
		// ВНИМАНИЕ: Если API упало, мы должны УДАЛИТЬ ключ из Redis!
		// Иначе при следующем ретрае мы сочтем сообщение дубликатом и SMS никогда не отправится.
		h.redis.Del(ctx, "idemp:"+event.IdempotencyKey)
		return fmt.Errorf("twilio send: %w", err)
	}

	return nil
}
```

> [!warning] Ловушка / Gotcha
> Подход с Redis имеет уязвимое окно (Race Condition): что, если наше приложение успешно записало ключ в Redis, вызвало API, но упало *до* отправки ACK в брокер? Брокер пришлет сообщение снова. Redis скажет "Ключ занят", и мы проигнорируем сообщение. По сути, мы свели гарантию доставки к `At-Most-Once`. 
> Если вам нужна строгая идемпотентность с внешними сайд-эффектами, Redis не подходит. Нужен паттерн [[6. Outbox pattern]] или [[7. Inbox pattern]].

### 3. Конечные автоматы (State Machines)

Этот паттерн применяется, когда сущность уже существует в БД и переходит из состояния в состояние (например, заказ: `Created` -> `Paid` -> `Shipped`).

Вместо отдельной таблицы для ключей, мы используем статус самой сущности. Запрос в БД выглядит так:

```sql
UPDATE orders 
SET status = 'PAID', paid_at = NOW() 
WHERE id = $1 AND status = 'CREATED';
```

Если консьюмер получит сообщение об оплате дважды, первый `UPDATE` изменит статус на `PAID` и вернет 1 затронутую строку. Второй `UPDATE` не найдет заказ со статусом `CREATED` и вернет 0 строк. В Go мы анализируем `RowsAffected()`:

```go
	tag, err := h.db.Exec(ctx, sqlUpdateOrder, orderID)
	if err != nil {
		return err
	}
	if tag.RowsAffected() == 0 {
		// Либо заказ не существует, либо он уже в статусе PAID (дубликат)
		return nil 
	}
```

## Mechanical Sympathy: Цена идемпотентности

Идемпотентность не бесплатна. Каждый паттерн требует дополнительного I/O:
1. **БД CONSTRAINT:** Требует поиска по индексу на каждой вставке. Используйте UUIDv7 или ULID, чтобы B-Tree индекс не фрагментировался и поиск происходил в горячем кэше.
2. **Redis/Memcached:** Добавляет сетевой хоп (round-trip) на каждую операцию. Если вы обрабатываете 50 000 msg/s, это 50 000 запросов в Redis в секунду. Это может стать узким местом сети.
3. **Кросс-системная консистентность:** Если вы пишете данные в PostgreSQL, а ключи идемпотентности проверяете в Redis, у вас нет транзакционной связи. Ошибка между коммитом в PG и удалением ключа в Redis приведет к неконсистентности.

Единственный способ обеспечить строгую идемпотентность без потери скорости на сетевых хопах в сторонние системы — это **Inbox Pattern**, где мы записываем сообщение в локальную БД приложения в рамках той же транзакции, что и бизнес-данные, элиминируя сетевые прыжки.

## Итог

1. Дубликаты в асинхронных системах — это норма. Обработчики **обязаны** быть идемпотентными.
2. Генерируйте `Idempotency Key` на стороне Продюсера. Встраивайте в ключ монотонно возрастающие компоненты (UUIDv7) для блага B-Tree индексов БД.
3. Используйте `UNIQUE CONSTRAINT` в БД как самый надежный и транзакционно-целостный метод.
4. При использовании Redis для идемпотентности не забывайте удалять ключ в случае падения сайд-эффекта, иначе вы потеряете сообщения.
5. Если бизнес-логика позволяет, используйте конечные автоматы (State Machines) через `UPDATE ... WHERE status = ...`.

Идемпотентные обработчики дают нам уверенность в том, что ретраи (о которых мы говорили в [[9. Retry стратегии и exponential backoff]]) не сломают бизнес-логику. Но что если мы хотим обрабатывать сообщения не по одному, а пачками, чтобы сэкономить на сетевых round-trips до БД? В следующей статье мы разберем: [[5. Batch processing сообщений]].
