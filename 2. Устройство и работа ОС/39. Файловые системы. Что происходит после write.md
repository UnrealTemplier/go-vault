## Прерывая иллюзию мгновенной записи

Когда метод `os.WriteFile()` или `bufio.Writer.Flush()` возвращает `nil`, подавляющее большинство разработчиков подсознательно считает данные «сохранёнными». Это опасная иллюзия. В современных ОС успешный возврат из `write()` означает лишь одно: ядро приняло данные и поместило их в оперативную память. Физическая запись на носитель происходит асинхронно, с переупорядочиванием, буферизацией и потенциальным откатом при сбое питания.

Для Senior/Lead инженера понимание этого пути критично: оно определяет выбор между консистентностью, производительностью и отказоустойчивостью. Разберём полный жизненный цикл данных от системного вызова до флеш-ячеек.

## Виртуальная файловая система и Page Cache

При вызове `write()` трафик проходит через **VFS (Virtual File System)** — абстрактный слой, скрывающий различия между ext4, xfs, btrfs, tmpfs и другими ФС. VFS переводит вызов в блочно-ориентированную операцию и отдаёт данные в **Page Cache**.

Page Cache — это ядро механизма буферизации ввода-вывода. Вместо того чтобы каждый раз обращаться к диску, ядро использует незанятую оперативную память для кэширования блоков дисковых данных. Это работает как read-ahead (предварительная загрузка) и write-back (отложенная запись).

> [!info] Под капотом
> В ядре Linux данные хранятся в структуре `struct page`. Каждая страница (обычно 4 КБ) содержит:
> - `mapping` — указатель на `address_space` (представляет файл в кэше).
> - `private` — указатель на `buffer_head` (связь с конкретным блоком диска).
> - `flags` — биты состояния: `PG_dirty`, `PG_uptodate`, `PG_locked`.
> Когда вы вызываете `write()`, ядро находит или выделяет страницу, копирует данные через `copy_from_user()`, помечает её как `PG_dirty` и увеличивает счётчик `mapping->dirty_pages`.

## Асинхронный writeback и планировщик

Данные не улетают на диск сразу. Они лежат в Page Cache, пока не сработает один из триггеров:
1. **Переполнение кэша**: количество `dirty` страниц превышает `vm.dirty_ratio` (по умолчанию ~5-10% от RAM).
2. **Таймер**: процесс `kworker` (ранее `pdflush`) запускается каждые 5 секунд (параметр `vm.dirty_writeback_centisecs`).
3. **Приложение требует синхронизации**: вызов `fsync()`, `fdatasync()` или `file.Sync()` в Go.

Планировщик IO (`blk-mq`) собирает запросы, сортирует их по физическому адресу диска и отправляет контроллеру. Это позволяет объединять случайные записи в последовательные, что критично для HDD и оптимизирует wear-leveling для SSD.

## Журналирование: цена консистентности

Если система упадёт, пока данные лежат в `PG_dirty`, мы получим повреждённый файл. Для защиты от этого файловые системы используют **журналирование (Journaling)**.

Журнал — это выделенная область диска, куда перед записью основных данных записываются транзакции изменений метаданных (и опционально данных). При восстановлении после краша журнал проигрывается, гарантируя согласованность метаданных.

> [!warning] Ловушка / Gotcha
> Журналирование защищает от **несогласованности**, но не от **потери данных**. Если вы записали 100 МБ в кэш, а `write()` вернул успех, и система теряет питание — данные уйдут в небытие. Журнал не содержит ваших пользовательских данных, если они ещё не были committed in the journal. Это фундаментальное различие между *durability* (устойчивость) и *consistency* (консистентность).

## Принудительная синхронизация и обход кэшей

Чтобы гарантировать запись на физический носитель, используются механизмы принудительной синхронизации:

| Механизм | Поведение |
|----------|-----------|
| `fsync(fd)` | Выталкивает данные и метаданные файла из кэша на диск. Блокирующий вызов. |
| `fdatasync(fd)` | Выталкивает только данные. Метаданные (права, время) могут остаться в кэше. Быстрее. |
| `O_SYNC` / `O_DSYNC` | Флаги открытия файла. Заставляют каждый `write()` синхронизироваться. Убивает производительность. |
| `O_DIRECT` | Полностью обходит Page Cache. Данные идут напрямую из user-space буфера на диск. Требует выравнивания по 512/4096 байт. |

```go
package main

import (
	"os"
	"log"
)

func main() {
	// Идиоматичный подход: буферизация + явная синхронизация
	file, err := os.OpenFile("data.bin", os.O_CREATE|os.O_WRONLY|os.O_APPEND, 0644)
	if err != nil {
		log.Fatalf("open: %v", err)
	}
	defer file.Close()

	// bufio.Writer работает в user-space, не трогая Page Cache
	// Пока не вызовется Flush(), syscall write() не сработает
	bw := bufio.NewWriterSize(file, 64*1024)
	
	// ... запись данных ...
	
	// Принудительно выталкиваем буфер в ядро
	if err := bw.Flush(); err != nil {
		log.Fatalf("flush: %v", err)
	}

	// Гарантируем попадание данных на физический носитель
	// Аналог sync_file_range(fd, 0, 0, SYNC_FILE_RANGE_WAIT_BEFORE|SYNC_FILE_RANGE_WRITE)
	if err := file.Sync(); err != nil {
		log.Fatalf("sync: %v", err)
	}
}
```

## Mechanical Sympathy: Путь байта до NAND-ячеек

Для написания высоконагруженных систем важно видеть полную картину:

1. **CPU Cache Coherency**: Данные из register CPU попадают в L1/L2/L3 кэш. При `write()` ядро копирует их в RAM. Контроллер памяти может отложить запись в DRAM для оптимизации bandwidth.
2. **DMA Engine**: Диск не читает данные через CPU. DMA-контроллер напрямую переносит данные из RAM в буфер диска. Это освобождает CPU, но требует согласования кэшей (`wbinvd` или `clflush`).
3. **Disk Controller Cache**: У современных SSD/HDD есть собственная RAM (write-back cache). Контроллер обещает OS: "Я получил данные, можно возвращать". На самом деле данные могут ещё не записаться на NAND/HDD. При отключении питания они исчезнут.
4. **NAND Flash Wear Leveling**: SSD не пишет по фиксированным адресам. Контроллер перемещает данные, чтобы равномерно изнашивать ячейки. Это добавляет задержку (latency) и требует фоновых операций Garbage Collection.

> [!tip] Собеседование
> **Вопрос:** Почему `fsync()` может быть в 100-1000 раз медленнее `write()`?
> **Ответ:** `write()` — это копирование в RAM (байт/нс). `fsync()` требует: 1) Выталкивания dirty pages из Page Cache (может быть сотни мегабайт). 2) Ожидания завершения DMA-запросов на физическом устройстве. 3) Обновления метаданных ФС (журнал, inode). 4) Возможного ожидания ответа от контроллера диска (особенно на HDD или медленных VPS). На SSD задержка определяется не поиском по пластинам, а временем программирования NAND-ячеек и фоновым GC.

## Go-идиомы и производительность

В Go взаимодействие с этой цепочкой происходит на нескольких уровнях:

- `os.File` делает прямые системные вызовы (`syscall.Write`).
- `bufio` откладывает syscall до заполнения буфера в user-space. Это уменьшает количество переключений контекста и syscall-ов.
- `file.Sync()` вызывает `fsync` или `fdatasync` в зависимости от ОС. На Linux это `sync_file_range`.
- Для hot-pathей (например, базы данных или логгеры) используют `O_DIRECT`, чтобы исключить двойное копирование (user -> page cache -> disk controller) и снять нагрузку с Page Cache.

> [!warning] Ловушка / Gotcha
> Использование `O_DIRECT` требует строгого выравнивания буферов и смещений на размер блока ФС (обычно 4 КБ). Нарушение выравнивания приведёт к `EINVAL` или скрытому копированию в буфер ядра, что убьёт преимущество обхода кэша. Всегда проверяйте `os.File` на поддержку `O_DIRECT` через `file.Readdir(0)` или тестовый `syscall`.

## Итог и переход

Путь данных после `write()` — это не прямой провод, а сложная система буферизации, переупорядочивания и асинхронного выталкивания. Понимание этой цепи позволяет:
- Выбирать между `fsync` и `fdatasync` в зависимости от требований к консистентности.
- Правильно настраивать `dirty_ratio` и `vm.dirty_background_ratio` для балансировки нагрузки на IO.
- Избегать пиков latency в микросервисах, вызванных фоновым writeback.
