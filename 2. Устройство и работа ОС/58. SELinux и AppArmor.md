
## Mandatory Access Control: Почему `chmod` больше не достаточно

Традиционные Unix-права доступа ([[56. Безопасность ОС. Права доступа, users, groups.md]]) работают на уровне **DAC (Discretionary Access Control)**. Владелец файла или группа решает, кто и как может его читать или исполнять. Это удобно, но уязвимо: если процесс `nginx` или ваш Go-демон скомпрометируют, атакующий получит все права текущего пользователя.

**MAC (Mandatory Access Control)** ломает эту парадигму. Даже если процесс запущен от `root`, MAC-система может запретить ему доступ к файлу, сетевому порту или системному вызову, если политика явно не разрешает это. В Linux двумя главными реализациями MAC являются `[[58. SELinux и AppArmor.md]]` и их подсистема **LSM (Linux Security Modules)**.

Для бэкенд-разработчика это не просто "фича безопасности". Это критический слой, который определяет, сможет ли ваше приложение запуститься в продакшене, как будут работать контейнеры и какие `capabilities` вам потребуются для обхода ограничений.

## LSM: Архитектура безопасности ядра

До появления LSM безопасность была "зашита" в конкретные подсистемы ядра. LSM вынесла эту логику в единый каркас, позволяя динамически подключать разные MAC-движки без перекомпиляции ядра.

> [!info] Под капотом
> LSM работает через **хуки (hooks)**, встроенные в критические пути ядра: VFS (файлы), сетевой стек, `ptrace`, `sched`, `ipc`. Когда программа делает системный вызов, ядро не просто проверяет DAC-биты. Оно проходит по цепочке LSM-хуков. Если хотя бы один модуль возвращает ошибку, операция блокируется.
> 
> В исходниках ядра это выглядит как массив структур `security_hook_heads`. При инициализации ядра каждый загруженный модуль (SELinux, AppArmor, YAMA и др.) регистрирует свои функции в этих хуках. Проверка происходит на уровне C-указателей, что дает минимальные накладные расходы (~5-10 ns на хук при пропуске).

## SELinux: Контексты, xattrs и политика

SELinux (Security-Enhanced Linux) от NSA использует контекстно-ориентированную модель. Каждому процессу и каждому файлу присваивается **контекст** в формате:
`user:role:type:level`

Политика определяет правила типа `allow httpd_t var_log_t :file { write };`. Процесс с типом `httpd_t` может писать в файлы с типом `var_log_t`, но не в `/etc/shadow`, даже если запущен от `root`.

**Как это работает на уровне FS:**
SELinux хранит контексты в **расширенных атрибутах (xattrs)** инодов. При каждом доступе к файлу ядро делает `getxattr()`, извлекает контекст и сверяет с политикой. Это добавляет накладные расходы на каждый syscall доступа к файлу, поэтому политики тщательно оптимизируются.

Режимы работы:
- `Enforcing` — жесткое блокирование нарушений.
- `Permissive` — логирование нарушений без блокировки (идеально для отладки политик).
- `Disabled` — полная отключение LSM-проверок SELinux.

## AppArmor: Профильно-ориентированный подход

AppArmor (Application Armor) от Canonical использует более простой, путь-ориентированный подход. Вместо контекстов инодов он привязывает профиль к **путьу исполняемого файла** или `[[53. Контейнеры под капотом. namespaces и cgroups.md]]`.

Пример профиля:
```text
/usr/bin/myapp {
  #include <abstractions/base>
  /etc/app/config r,
  /var/log/myapp w,
  network inet tcp,
  deny /etc/shadow r,
}
```

AppArmor хранит профили в `/etc/apparmor.d/`, компилирует их в бинарный формат ядра и загружает через `apparmor_parser`. Проверки происходят по имени исполняемого файла и его `[[53. Контейнеры под капотом. namespaces и cgroups.md]]`, что делает AppArmor идеальным для контейнеризованных сред и Debian/Ubuntu-экосистемы.

> [!tip] Собеседование
> **Вопрос:** SELinux vs AppArmor: что выбрать и почему?
> **Ответ:** SELinux дает гранулярный контроль на уровне контекстов (тип файла, роль, уровень), что критично для высоконагруженных кластеров с сотнями сервисов. AppArmor проще в настройке, привязан к путям и бинарникам, быстрее отлаживается и является стандартом в Ubuntu/Kubernetes. Для Go-микрофреймворков и контейнеров AppArmor часто предпочтительнее из-за предсказуемости и отсутствия необходимости переписывать политику при смене пути бинарника.

## Влияние на Go-разработку и контейнеризацию

Go по умолчанию собирает **статически линкованные бинарники**. Это ломает путь-ориентированную модель AppArmor, если вы не знаете точный путь запуска бинарника. Решения:
1. Использовать `sha256` хеш бинарника в профиле AppArmor (`profile /usr/bin/myapp { ... }` или `profile /usr/bin/myapp sha256:abc123 { ... }`).
2. Настроить SELinux с `semanage fcontext` и `restorecon` для динамического назначения контекстов.
3. Использовать `CAPABILITIES` вместо запуска от root, чтобы минимизировать поверхность атаки.

В Go работа с capabilities осуществляется через пакет `golang.org/x/sys/unix`:

```go
package main

import (
	"fmt"
	"log"
	"os"

	"golang.org/x/sys/unix"
)

func dropPrivileges() error {
	// 1. Убираем все capabilities
	header := unix.CapHeader{Version: unix.LINUX_CAPABILITY_VERSION_3}
	data := unix.CapData{Permitted: 0, Effective: 0, Inheritable: 0}
	if err := unix.Capset(&header, &data); err != nil {
		return fmt.Errorf("capset drop: %w", err)
	}

	// 2. Оставляем только CAP_NET_BIND_SERVICE для прослушивания <1024 портов
	dataPermitted := unix.CapData{
		Permitted:   1 << unix.CAP_NET_BIND_SERVICE,
		Effective:   1 << unix.CAP_NET_BIND_SERVICE,
		Inheritable: 0,
	}
	if err := unix.Capset(&header, &dataPermitted); err != nil {
		return fmt.Errorf("capset set: %w", err)
	}

	// 3. Переключаемся на non-root пользователя
	return os.Chown(".", 1000, 1000) // Пример смены владельца
}

func main() {
	if err := dropPrivileges(); err != nil {
		log.Fatal(err)
	}
	// Инициализация HTTP-сервера и т.д.
}
```

> [!warning] Ловушка / Gotcha
> В контейнерах Docker/K8s по умолчанию применяется базовый профиль AppArmor `docker-default`. Он разрешает множество операций, но блокирует `ptrace`, `mount` и доступ к `/proc/sys`. Если ваш Go-приложение делает `os.Open("/proc/self/status")` или использует `debug/pprof`, вы получите `EACCES` в Enforcing-режиме.
> **Решение:** Либо отключите AppArmor для теста (`docker run --security-opt apparmor=unconfined`), либо напишите кастомный профиль, либо используйте `--cap-add=SYS_PTRACE` и настройте политики доступа к `/proc`.

## Отладка и мониторинг MAC

Когда политика блокирует действие, ядро логирует событие в `dmesg` или `auditd`.
- **SELinux:** `ausearch -m avc -ts recent`, `sealert -a /var/log/audit/audit.log`, `audit2allow` для генерации правил.
- **AppArmor:** `dmesg | grep audit`, `aa-status`, `aa-logprof` для автоматического создания профилей на основе логов.

В Go приложениях стоит настроить graceful shutdown с обработкой `SIGTERM` и `SIGKILL`, так как MAC-блокировки не влияют на сигналы ядра. Однако, если процесс пытается сохранить состояние в файл, который заблокирован политикой, `os.WriteFile` вернет `EACCES`, и приложение упадет без graceful-обработки. Всегда оборачивайте критичные I/O в `if err != nil { log and handle }`.

> [!tip] Собеседование
> **Вопрос:** Как MAC влияет на производительность Go-сервера?
> **Ответ:** В `Permissive` режиме накладные расходы минимальны (только логирование). В `Enforcing` каждая проверка добавляет ~10-50 ns на syscall из-за вызовов LSM-хуков и чтения xattrs. Для высоконагруженных I/O-линий это может складываться в миллисекунды на миллион запросов. Оптимизация политик (минимизация `deny` правил, использование `include` в AppArmor) и отказ от избыточных проверок на каждый файл критичны для latency-чувствительных сервисов.

## Итог

1. **MAC (SELinux/AppArmor)** работает поверх DAC, блокируя доступ на уровне ядра через LSM-хуки, даже для `root`.
2. **SELinux** использует контексты и xattrs, давая гранулярный контроль. **AppArmor** привязан к путям/бинарникам, проще в поддержке.
3. **Go-статические бинарники** требуют особого подхода к политикам (хеш-валидация или строгие пути).
4. **Контейнеры** по умолчанию накладывают свои MAC-ограничения. Их нужно явно настраивать под задачи приложения.
5. **Отладка** ведется через `auditd`/`dmesg`, а не через стандартные логи приложения.
