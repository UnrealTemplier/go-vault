## Что происходит, когда CPU «забывает» задачу

Когда вы запускаете `go run main.go`, вы ожидаете линейного выполнения кода. Но в многозадачной ОС ваш процесс делит процессорное время с тысячами других. Когда таймер прерывает выполнение вашей программы, ядро ОС решает, что пришло время переключиться на другой процесс. Это событие называется **Context Switch (Переключение контекста)**.

Для разработчика это выглядит как «магия», но для процессора это тяжелая инженерная операция. В контексте Go понимание стоимости контек-switch критически важно, потому что бэкенд, созданный без учета этой цены, упирается в потолок производительности уже на 100+ RPS.

## Стоимость переключения: от регистров до кэш-линий

Переключение контекста — это не просто прыжок на другой адрес. Это полный сброс состояния вычислительной системы. Основные компоненты стоимости:

1. **Состояние процессора (Registers):** Нужно сохранить ~60-100 регистров x86-64 (RAX, RBX, RSP, RIP, RFLAGS и др.) в структуру ядра. Восстановление требует атомарных записей.
2. **Виртуальная память (Page Tables):** Каждый процесс имеет свое пространство адресов. Переключение требует обновления указателя базового каталога (CR3 в x86-64). Это меняет отображение виртуальных адресов на физические.
3. **TLB (Translation Lookaside Buffer):** Кэш трансляций адресов привязан к конкретному процессу. При смене CR3 TLB становится невалидным. Требуется `TLB Flush` (очистка кэша), что стоит сотни тактов CPU.
4. **CPU Кэши (L1/L2/L3):** Кэш содержит данные предыдущего процесса. При переключении он становится «грязным» и бесполезным для нового. Новый процесс страдает от `cache misses`, пока кэш не заполнится его данными.
5. **Конвейер процессора (Pipeline):** Branch prediction сбрасывается. Процессор теряет все предвычисленные инструкции и начинает заново декодировать код новой задачи.

> [!info] Под капотом
> В современных ОС (Linux, Windows, macOS) используется **ASID (Address Space ID)** или **PID для TLB**. Это позволяет не чистить весь TLB полностью, а помечать записи как устаревшие. Однако обновление кэш-линий процессора и переключение `mm_struct` все равно остаются дорогими.
> Средняя стоимость контек-switch в Linux: **10,000 – 50,000 тактов CPU** (~3-15 микросекунд на современных частотах). Это вечность для бэкенда.

## Под капотом Linux: `switch_to` и `pt_regs`

В ядре Linux переключение контекста реализовано через макрос `switch_to` (архитектура x86_64). Он не просто сохраняет данные, он выполняет строгую последовательность:

```c
// Упрощенная логика arch/x86/include/asm/switch_to.h
#define switch_to(prev, next, last) \
do { \
    __switch_to(prev, next); \
    /* Сохранение RIP/RSP флагов в pt_regs */ \
    save_prev_state(prev); \
    /* Переключение mm_struct и page tables */ \
    switch_mm(prev->mm, next->mm, next); \
    /* Обновление кэша и TLB */ \
    flush_tlb_mm(next->mm); \
    /* Восстановление состояния */ \
    restore_next_state(next); \
} while (0)
```

Ключевая структура ядра, хранящая состояние задачи: `task_struct` (включает `thread_info` и `pt_regs`). Именно туда ядро «сваливает» контекст, когда решает, что горутине/потоку пора отдохнуть.

## Go Runtime: Green Threads вместо OS Threads

Здесь Go совершает революцию. В Go **не используется** один 1:1 маппинг горутин на потоки ОС. Вместо этого применяется **M:N планировщик**:
- `M` (Machine) — поток ОС (sys thread).
- `G` (Goroutine) — легковесная задача.
- `P` (Processor) — логический процессор, привязанный к `M`.

Когда горутина блокируется на I/O или ожидании канала, Go runtime **не делает системный вызов** для переключения контекста. Вместо этого:
1. Текущий `G` сохраняется в стеке текущего `M`.
2. `P` ищет новую runnable `G` в своей локальной очереди.
3. Если очередь пуста, `P` выполняет `sysmon` (монитор системных вызовов) или уходит в спящий режим, освобождая `M` для других потоков.

Переключение между `G` в Go — это **Green Thread Context Switch**. Оно происходит полностью в User Space:
- Сохраняются только указатели стека, `PC` (program counter) и регистры.
- Не затрагиваются page tables и TLB.
- Не происходит переключения привилегий (Ring 3 -> Ring 0).

Стоимость Go context-switch: **~1,000 – 2,000 тактов CPU**. В 10-20 раз быстрее системного переключения.

> [!warning] Ловушка / Gotcha
> **Миф:** «Больше горутин = больше контек-свитчей = тормоза».
> **Реальность:** В Go контек-switch между горутинами на одном `P` стоит копейки. Проблема возникает, когда `P` не может найти runnable `G` и вынужден делать syscall `sched_yield` или `pthread_yield`, либо когда `P` блокируется на сетевом I/O. Именно для этого нужен `netpoller` (см. [[38. Как netpoller Go использует epoll и kqueue]]).

## Mechanical Sympathy: Кэш-линии, TLB и конвейер процессора

Для Senior/Lead инженера важно понимать, как контек-switch бьет по производительности на уровне железа:

### 1. Cache Thrashing (Кэш-разрушение)
Если горутина `A` активно работает с `map[string]int`, она загружает кэш-линии L1 в процессор. При переключении на `Goroutine B` (или другой процесс) эти линии вытесняются. Когда `A` возвращается, она получает `L1 miss`. Каждый `L1 miss` стоит ~4 такта, `L2 miss` ~12 тактов, `L3 miss` ~40+ тактов. В высоконагруженном сервисе это превращается в миллисекунды задержки.

### 2. NUMA и локальность памяти
На серверах с NUMA (Non-Uniform Memory Access) память физически привязана к конкретному CPU сокету. Если `M` перепрыгивает между сокетами из-за контек-switch, доступ к памяти становится асинхронным и медленным. Go runtime пытается удерживать `P` на одном CPU через `sched_gomaxprocs` и `runtime.lockOSThread`.

### 3. Branch Prediction & Pipeline Flush
При каждом контек-switch предсказатель переходов сбрасывается. Процессор теряет все предвыполненные инструкции. Если вы запускаете тысячи легких горутин, которые постоянно блокируются и разблокируются, вы получаете постоянные pipeline stalls, которые не видны в `top` (CPU load = 0%), но убивают latency.

```bash
# Пример: как увидеть влияние на CPU
go tool pprof -http=:8080 http://localhost:6060/debug/pprof/profile?seconds=30
# Ищите "runtime.schedule" и "runtime.gopark" в flat profile.
# Высокий % в gopark часто означает избыточное ожидание каналов/мап.
```

## Оптимизация: Как Go минимизирует OS Context Switch

1. **`GOMAXPROCS`**: По умолчанию равен количеству физических ядер. Установка > ядер не ускоряет CPU-bound задачи, а только увеличивает вероятность OS context-switch и cache misses.
2. **`netpoller`**: Использует `epoll` (Linux) / `kqueue` (BSD/macOS) / `IOCP` (Windows) для асинхронного I/O. Горутина, ожидающая сеть, **не блокирует поток ОС**. Она просто снимается с `P` и ставится в очередь `netpoll`. Когда сокет готов, `sysmon` возвращает `G` на `P`. OS context-switch не происходит.
3. **Work Stealing**: Если `P` голодает, он «крадет» задачи у других `P`. Это делается через атомарные операции в User Space, без вызова ядра.
4. **`runtime.LockOSThread()`**: Используется в редких случаях (например, для `cgo` или специфичного I/O), чтобы запретить планировщику Go перемещать поток. Это *увеличивает* риск OS context-switch, поэтому применяется с осторожностью.

> [!tip] Собеседование
> **Вопрос:** «Почему в Go говорят, что горутины дешевые, но при 100k горутин сервис начинает тормозить?»
> **Ответ:** «Сами горутины и их переключение в User Space действительно дешевые. Тормоза возникают из-за: 1) Конкуренции за `P` (scheduler lock contention), 2) Вытеснения кэша процессора (cache thrashing) при частом переключении на одной логической машине, 3) Блокировок `sync.Mutex`/`sync.RWMutex`, которые заставляют горутины уходить в `gopark` и требуют syscall `futex` только при длительном ожидании. Оптимизация лежит в области уменьшения критических секций и использования `netpoller` для I/O-bound задач».

## Итог

1. **Context Switch** — это тяжелая операция ядра: сохранение регистров, сброс TLB, инвалидация кэшей, сброс конвейера.
2. **Стоимость**: ~10k-50k тактов CPU. В многопоточных системах это становится узким местом.
3. **Go Runtime** избегает OS context-switch через M:N планировщик и Green Threads. Переключение между `G` стоит ~1k-2k тактов.
4. **Реальная угроза производительности** — не количество горутин, а **конкуренция за `P`**, **cache misses** и **неэффективное использование I/O** (когда горутины блокируют потоки ОС).
5. Для бэкенда правило простое: **1 CPU = 1 `P`**. Не заставляйте планировщик Go и ядро ОС бороться за кэш-линии.
