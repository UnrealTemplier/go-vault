## Процесс и Поток: Границы и Контекст

В многозадачных операционных системах процессы и потоки (threads) являются фундаментальными единицами исполнения. Понимание разницы между ними критично для проектирования высоконагруженных систем, так как напрямую влияет на потребление памяти, latency при запуске новых задач и стоимость переключения контекста.

**Процесс** — это изолированный контейнер ресурсов ОС. Каждый процесс обладает собственным виртуальным адресным пространством, дескрипторами файлов, обработчиками сигналов, идентификаторами пользователя/группы и переменными окружения. Изоляция гарантирует, что краш или утечка памяти в одном процессе не затронет другие, но делает коммуникацию между ними дорогой (требует IPC или сетевого стека).

**Поток выполнения** — это легковесная единица исполнения внутри процесса. Потоки разделяют адресное пространство и ресурсы родительского процесса, но обладают собственным контекстом исполнения: стеком, набором регистров CPU и программным счетчиком. Именно разделение ресурсов и определяет кардинальную разницу в стоимости их создания и переключения.

> [!info] Под капотом
> В современных ядрах Linux (начиная с 2.6) **всё является потоком**. И процессы, и потоки представлены одной внутренней структурой `task_struct`. Разница лишь в флагах и указателях:
> *   Если несколько `task_struct` ссылаются на один и тот же `mm_struct` (структура управления виртуальной памятью), они считаются потоками одного процесса.
> *   Если `mm_struct` уникален — это отдельный процесс.
> Это архитектурное решение упростило реализацию планировщика, но не изменило семантику для разработчика.

## Под капотом: Как ОС представляет процессы и потоки

Когда вы запускаете приложение, ядро выделяет ему адресное пространство. В Linux оно управляется через **VMA (Virtual Memory Areas)** и **Page Tables**. 

Процесс содержит:
*   **Text segment**: исполняемый код (read-only).
*   **Data segment**: глобальные переменные.
*   **Heap**: динамически выделяемая память (через `brk` или `mmap`).
*   **Stack**: стеки всех потоков процесса.
*   **File descriptors**: таблица открытых файлов/сокетов.

Поток же — это просто набор регистров и стек внутри этого адресного пространства. При создании потока ядро:
1. Выделяет новый `task_struct`.
2. Резервирует память для стека потока (обычно 8 МБ в Linux по умолчанию).
3. Добавляет новый стек в VMA родительского процесса.
4. Делит время CPU с другими потоками/процессами через планировщик.

Поскольку потоки делят память, они могут обмениваться данными через глобальные переменные или кучу. Это удобно, но требует явной синхронизации.

## Стоимость создания и переключения

Здесь кроется главная инженерная дилемма. Чем "легче" сущность, тем дешевле её создание, но тем чаще возникает необходимость в синхронизации.

### Создание (Fork vs pthread_create)
Запуск нового процесса через `fork()` требует дублирования или копирования структуры адресного пространства. Исторически это делало `fork()` очень дорогой операцией. Современные ОС используют **Copy On Write (CoW)**: страницы памяти не копируются физически, а помечаются как "только для чтения". При попытке записи ядро выделяет новую страницу и копирует данные. Это ускоряет `fork()` в разы, но всё равно требует выделения PTE (Page Table Entry) и синхронизации TLB.

Создание потока (`pthread_create` / `CreateThread`) обходится дешевле. Ядру не нужно настраивать новое адресное пространство. Достаточно выделить стек, инициализировать `task_struct` и добавить его в runqueue.

### Context Switch
Переключение между потоками одного процесса быстрее, чем между процессами. Почему?
1. **Сохранение состояния**: Регистры, PC, флаги сохраняются в `task_struct`.
2. **Смена стека**: Указатель `RSP` меняется на стек нового потока.
3. **TLB Flush (Критично!)**: При переключении между **процессами** ядро должно очистить кэш трансляции адресов (TLB), иначе CPU продолжит использовать виртуальные адреса старого процесса, что приведет к `Segmentation Fault` или утечке данных. Очистка TLB (`CR3` register write на x86) стоит сотни тактов CPU.
4. **При переключении потоков одного процесса** виртуальное адресное пространство не меняется. MMU (Memory Management Unit) остается в том же контексте. TLB не нужно сбрасывать. Это экономит 50-200 тактов на переключение.

> [!warning] Ловушка / Gotcha
> "Легковесность" потоков иллюзорна. Создание 100 000 OS-потоков "завесит" процесс из-за:
> 1. Расхода 8 МБ стека на поток (~800 ГБ памяти).
> 2. Постоянных context switch между ними, что вытеснит полезную нагрузку.
> 3. Конкуренции за L1/L2 кэш CPU. Кэш выстраивается по линиям (cache lines). Частое переключение потоков, работающих с одними и теми же данными, вызывает `cache thrashing` — данные вытесняются из кэша до того, как будут использованы.

## Go-подход: Горутины против OS-потоков

Go радикально меняет парадигму. Вместо 1:1 маппинга (как в Java, C# или Python), Go использует **M:N scheduler** (M горутин на N потоков ОС).

```go
package main

import (
	"fmt"
	"runtime"
	"time"
)

func worker(id int) {
	// Горутина исполняется легковесно
	fmt.Printf("Горутина %d: работаю\n", id)
}

func main() {
	// Ограничиваем количество физических потоков ОС
	runtime.GOMAXPROCS(4)
	
	// Создаем 1000 горутин
	for i := 0; i < 1000; i++ {
		go worker(i)
	}
	
	// Даем время на выполнение
	time.Sleep(100 * time.Millisecond)
	
	fmt.Printf("Физических потоков (GOMAXPROCS): %d\n", runtime.GOMAXPROCS(0))
	fmt.Printf("Активных горутин: %d\n", runtime.NumGoroutine())
}
```

### Ключевые отличия Go от классических языков:
| Характеристика | Java / C# / Python Threads | Go Goroutines |
|----------------|----------------------------|---------------|
| **Маппинг** | 1:1 (Один = Один OS-thread) | M:N (Многие к Многим) |
| **Стартовый стек** | Фиксированный (~1-2 МБ) | Динамический (~2 КБ, растет до 1 ГБ) |
| **Создание** | Дорого (syscall, стек аллокация) | Дешево (спейс в куче + структура `g`) |
| **Планирование** | ОС (Kernel scheduler) | User-space (Go runtime scheduler) |
| **Блокировка** | Блокирует OS-thread | Блокирует только горутину, тред (M) освобождается |

Go-рантайм (`runtime.scheduler`) хранит горутины в структурах `g`. При вызове `go func()` выделяется `g` в куче/стеке, инициализируется стек и добавляется в runqueue планировщика. Если все 4 потока (`p` в архитектуре Go) заняты, горутина ждет в локальной или глобальной очереди. При блокирующем syscall или channel operation горутина "усыпается" (`gopark`), а её тред (`m`) переназначается на другую горутину. Это называется **work stealing** и гарантирует высокий утилизация CPU без оверхеда контекстных переключений ядра.

> [!tip] Собеседование
> **Вопрос:** Почему `GOMAXPROCS` по умолчанию равен количеству логических ядер? Что будет, если установить его в 1?
> **Ответ:** Go использует многопоточность для выполнения CPU-bound задач. Если `GOMAXPROCS=1`, все горутины будут исполняться последовательно на одном ядре. Это уберет оверхед от переключения контекста между тредми ОС, но убьет параллелизм. При блокировке одной горутины (например, на сетевом IO через `netpoller`) остальные горутины будут простаивать, так как планировщик не сможет перекинуть их на другой физический тред.
> **Вопрос:** Как Go справляется с race condition, если горутины делят память?
> **Ответ:** Go не налагает ограничений на доступ к памяти. Память действительно общая. Ответственность за синхронизацию лежит на разработчике. Go предоставляет `sync.Mutex`, `sync.RWMutex`, `sync.WaitGroup`, `channel` (как IPC-механизм) и `atomic` пакет. Отсутствие implicit locking (как в Java `synchronized` или C# `lock`) — сознательный выбор в пользу производительности и предсказуемости.

## Сравнение с C++/C#/PHP

*   **PHP**: Фундаментально не имеет потоков. Задачи исполняются последовательно в одном треде процесса (или через `fork` на каждый запрос в PHP-FPM). Масштабирование идет горизонтально (Nginx + множество процессов PHP). Go позволяет обрабатывать тысячи соединений в одном процессе без `fork`.
*   **C# / Java**: Используют `ThreadPool`. Потоки тяжелые, создание нового потока требует syscall. Для IO-bound задач используют `async/await`, который по сути является state-machine, но под капотом часто возвращает тред в пул. Go решает IO-bound задачи нативно через `netpoller` (эпиолинг), не возвращая треды в пул.
*   **C++**: Позволяет создавать `std::thread`, но требует ручного управления жизненным циклом, синхронизацией и памятью. Высокий риск data race и undefined behavior. Go предоставляет safe concurrency primitives out-of-the-box.

## Итог

1. **Процесс** — граница изоляции ресурсов. **Поток** — единица исполнения внутри процесса.
2. Создание процесса дорого из-за работы с Page Tables и CoW. Создание потока дешевле, но требует выделения стека.
3. Context switch между процессами требует сброса TLB, что на 2-3 порядка дороже переключения потоков.
4. Go заменяет тяжелые OS-потоки на легковесные горутины (`g`), управляемые user-space планировщиком. Это дает экономии памяти (2 КБ vs 8 МБ) и позволяет обрабатывать миллионы соединений.
5. Общая память горутин требует явной синхронизации. Go не спасает от race condition, но предоставляет инструменты (`sync`, `channel`, `atomic`) для безопасной работы.
