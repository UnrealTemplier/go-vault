## Почему диск — узкое горлышко современной системы

Для бэкенд-разработчика, привыкшего измерять производительность в микросекундах, главное правило остаётся неизменным: **операции с диском на 3-4 порядка медленнее, чем с оперативной памятью**. Даже NVMe SSD не спасает от физических ограничений носителя. Именно поэтому современная ОС не читает и не пишет данные напрямую на диск при каждом запросе приложения. Она использует **Page Cache** (кэш страниц) — механизм, который превращает часть RAM в быстрый буфер для дисковых операций.

Понимание того, как работает буферизация на уровне ОС, критически важно для Go-разработчика. Это определяет выбор между `bufio` и прямым `syscall`, влияние `fsync` на latency, поведение сервиса под нагрузкой и причины внезапных скачков CPU usage.

## Page Cache: Первый рубеж защиты от медленного железа

Когда вы вызываете `os.ReadFile()` или `file.Write()`, данные не попадают на диск мгновенно. Они проходят через несколько слоёв копирования и кэширования.

> [!info] Под капотом
> В ядре Linux кэш страниц реализован через массив структур `struct page`. Каждая страница (стандартный размер 4 КБ) индексируется в иерархической структуре `xarray` (ранее `radix tree`) по ключу: `устройство (major:minor) + блок + смещение`. 
> Когда процесс вызывает `read()`, ядро сначала делает быстрый хеш-поиск в `xarray`. Если страница найдена — **Page Hit**. Если нет — **Page Miss**, запускается подсистема ввода-вывода, ядро блокирует горутину (или тред ОС) и ждёт завершения I/O.

### Механизмы чтения и записи: Hit, Miss и Dirty Pages

**Чтение (Read)**
ОС использует стратегию **Read-Ahead**. Если вы читаете последовательные блоки, планировщик I/O предугадывает потребность и загружает следующие страницы в кэш до того, как приложение их запросит. Это превращает случайный диск в последовательный по скорости.

**Запись (Write)**
Записи обрабатываются по принципу **Write-Back**. Когда вы вызываете `write()`, данные копируются из буфера пользователя в Page Cache и помечаются как `dirty` (грязные). Физическая запись на диск откладывается. Этим занимаются фоновые потоки ядра (`flush` / `kworker`), которые периодически синхронизируют грязные страницы с диском.

> [!warning] Ловушка / Gotcha
> У ядра есть лимиты на количество грязных страниц: `vm.dirty_background_ratio` и `vm.dirty_ratio`. Если приложение пишет данные быстрее, чем фоновые потоки успевают сбрасывать их на диск, и заполняется `dirty_ratio`, процесс записи **блокируется** до освобождения места в кэше. Это может вызвать внезапные паузы (STW-подобное поведение) в вашем сервисе, даже если CPU свободен.

> [!tip] Собеседование
> **Вопрос:** В чём разница между `write()` и `fsync()`?
> **Ответ:** `write()` копирует данные в Page Cache и возвращает управление немедленно. Данные могут ещё не существовать на диске. `fsync()` (или `Sync()` в Go) принудительно сбрасывает все грязные страницы, относящиеся к файлу, на физическое устройство и гарантирует durability. Это дорогая операция, так как она прерывает write-back поток и вызывает реальное I/O.

## Два уровня буферизации: Прикладной vs Системный

В Go буферизация работает на двух независимых уровнях. Понимание их взаимодействия — ключ к оптимизации.

| Уровень | Инструмент | Что делает | Влияние на производительность |
|:---|:---|:---|:---|
| **Прикладной** | `bufio`, `io.Writer` | Собирает байты в буфер в User Space | Уменьшает количество системных вызовов `write()` и переходов контекста |
| **Системный** | Page Cache (OS) | Кэширует данные в RAM после `write()` | Уменьшает обращения к диску, скрывает latency I/O |

Если вы используете `file.Write()` с маленькими чанками без `bufio`, вы платите за:
1. Переключение контекста User/Kernel Space.
2. Копирование данных из буфера Go в буфер ядра (`copy_to_user`).
3. Аллокацию структур ядра для каждого вызова.

Page Cache не спасает от этого. Он сработает только *после* того, как данные попадут в ядро.

## Управление кэшированием в Go: Идиомы и инструменты

### 1. `bufio` — стандартный подход
```go
f, err := os.OpenFile("log.txt", os.O_APPEND|os.O_CREATE|os.O_WRONLY, 0644)
if err != nil {
    return err
}
defer f.Close()

// bufio.Buffer выделяется в куче, если размер > 32 КБ
w := bufio.NewWriterSize(f, 1<<16) 
defer w.Flush() // Принудительный сброс буфера в syscall write()

_, err = w.Write([]byte("data"))
if err != nil {
    return err
}
```
> [!tip] Собеседование
> **Вопрос:** Когда `bufio.NewWriter` начинает аллоцировать память в куче?
> **Ответ:** В стандартной библиотеке Go буфер размером до 4 КБ (или до размера, указанного в `NewWriterSize`, если он мал) может выделяться в стеке вызывающей функции, если компилятор (escaper) определяет, что указатель не выходит за пределы области видимости. При больших буферах или возврате Writer из функции всегда происходит Escape Analysis и аллокация в куче.

### 2. `os.O_DIRECT` — обход Page Cache
Иногда кэш ОС мешает производительности (например, при реализации собственной СУБД или при работе с большими бинарными файлами). `O_DIRECT` заставляет ядро копировать данные напрямую из буфера пользователя в устройство, минуя Page Cache.

```go
f, err := os.OpenFile("data.bin", os.O_RDWR|os.O_CREATE|os.O_DIRECT, 0644)
if err != nil {
    return err
}
defer f.Close()

// ОЧЕНЬ ВАЖНО: буфер для O_DIRECT должен быть выровнен по размеру страницы (обычно 4096)
// и выделен с помощью sync.Pool или posix_memalign
buf := make([]byte, 4096) 
_, err = f.Write(buf)
if err != nil {
    return err
}
```

> [!warning] Ловушка / Gotcha
> `O_DIRECT` требует строгого выравнивания адресов и размеров буферов по границе страницы (обычно 4 КБ). Невыполнение условия вернёт `EINVAL`. Кроме того, данные, записанные через `O_DIRECT`, **не кэшируются** автоматически. Последующее чтение вернёт Page Miss. Используйте его только для предсказуемой, высоконагруженной последовательной I/O нагрузки.

### 3. `mmap` — маппинг файла в память
Позволяет работать с файлом как с большим массивом в памяти. Ядро само управляет загрузкой страниц в Page Cache при обращении к ним (Page Fault).

```go
import (
    "os"
    "golang.org/x/sys/unix"
)

f, err := os.Open("large.bin")
if err != nil {
    return err
}
defer f.Close()

stat, err := f.Stat()
if err != nil {
    return err
}

// Маппинг только для чтения (PROT_READ)
data, err := unix.Mmap(int(f.Fd()), 0, int(stat.Size()), unix.PROT_READ, unix.MAP_SHARED)
if err != nil {
    return err
}
defer unix.Munmap(data)

// Работа с данными как с slice: data[0], data[1000]...
// Ядро подгрузит страницы в Page Cache лениво при доступе.
```

## Mechanical Sympathy: Влияние на CPU, GC и память

1. **Копирование памяти:** При стандартном `write()` данные проходят путь: `User Buffer -> Kernel Buffer -> Page Cache -> Disk`. Это минимум два копирования (`copy_to_user` / `copy_from_user`), каждое из которых затрагивет CPU cache и шину памяти. `O_DIRECT` убирает копирование в ядро, но требует ручного выравнивания.
2. **Влияние на GC:** Большие буферы `bufio` увеличивают давление на Garbage Collector. Если вы пишете высоконагруженный шлюз, создавайте Writer'ы в `sync.Pool`, чтобы избежать постоянных аллокаций и последующих циклов сбора мусора.
3. **Память под давлением (Memory Pressure):** Если система испытывает нехватку RAM, ядро начнёт агрессивно выгружать Page Cache (clean pages) или принудительно сбрасывать dirty pages. Это может вызвать внезапный рост latency дисковых операций. В Go это проявляется как скачки latency в метриках, даже если приложение не изменило логику.
4. **Сравнение с PHP/Java:** В PHP `file_put_contents()` часто работает с маленькими чанками, полагаясь на кэш ОС. В Java `FileOutputStream` также требует явной буферизации. Go даёт разработчику полный контроль: вы можете выбрать между удобством `bufio`, производительностью `O_DIRECT` или ленивым кэшированием `mmap`, понимая цену каждого выбора.

## Итоги

1. **Page Cache** — фундаментальный механизм ОС, превращающий RAM в быстрый буфер для диска. Работает на уровне страниц (4 КБ) и индексируется через `xarray`.
2. **Dirty Pages и Write-Back** — записи откладываются для пакетной обработки. Блокировка при превышении лимитов `dirty_ratio` — частая причина неочевидных пауз.
3. **`bufio` не дублирует Page Cache** — он уменьшает syscall overhead. Используйте его всегда при последовательной записи.
4. **`fsync()` стоит дорого** — принудительный сброс грязных страниц прерывает write-back цикл и увеличивает latency. Используйте только при необходимости гарантии durability.
5. **`O_DIRECT` и `mmap`** — инструменты для обхода стандартного цикла копирования. Требуют понимания выравнивания памяти и особенностей планировщика I/O.
