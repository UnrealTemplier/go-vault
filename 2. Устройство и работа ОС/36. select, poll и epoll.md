## Введение: Проблема масштабирования блокирующих I/O

Когда бэкенд-разработчик пишет первое приложение, он использует блокирующие вызовы: `read()`, `write()`, `accept()`. Пока соединение ждет данных, поток спит. Это работает для десятков клиентов, но рушится при переходах к тысячам или миллионам одновременных подключений. Запускать отдельный тред ОС или процесс на каждое соединение — неоправданная трата памяти и контекстных переключений.

Решение — **I/O мультиплексирование**. Один поток (или горутина, обернутая в тред ОС) регистрирует множество файловых дескрипторов и спрашивает у ядра: «Какие из них готовы к чтению или записи прямо сейчас?». Ядро возвращает только готовые, а остальные дескрипторы игнорируются до следующего опроса.

В Linux эволюция этих механизмов шла от простых битовых масок до сложных событийно-ориентированных структур. Понимание этой эволюции критично, потому что именно `epoll` лежит в основе производительности Go-серверов.

## Эволюция решений: от select к epoll

Исторически появилось три основных API для I/O мультиплексирования:
1. `select` (BSD, 1983) — битовые маски, жесткий лимит дескрипторов.
2. `poll` (System V, 1980-е) — массив структур, снятие лимита, но сохранение O(n).
3. `epoll` (Linux 2.6+, 2002) — событийно-ориентированный интерфейс, O(1) в худшем случае, прямая интеграция с драйверами устройств.

> [!tip] Собеседование
> **Вопрос:** В чем фундаментальная разница между `select/poll` и `epoll` с точки зрения сложности алгоритма?
> **Ответ:** `select` и `poll` имеют сложность O(n) при каждом вызове, потому что ядро должно пройти по *всем* зарегистрированным дескрипторам, чтобы проверить их статус. `epoll` имеет сложность O(1) для ожидания, так как использует механизм обратных вызовов в ядре: готовый дескриптор сразу попадает в список, и ядро возвращает его без сканирования.

## select: Жесткие ограничения и O(n)

```c
#include <sys/select.h>
int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set *exceptfds, struct timeval *timeout);
```

`select` оперирует битовыми масками (`fd_set`). Размер маски фиксирован на этапе компиляции (`FD_SETSIZE`, обычно 1024). Это означает, что вы не можете мониторить больше 1024 дескрипторов одновременно.

**Под капотом:**
1. Пользовательское приложение заполняет структуры `fd_set` в User Space.
2. При вызове `select` эти структуры **полностью копируются** в Kernel Space.
3. Ядро перебирает все дескрипторы в переданном диапазоне (O(n)), проверяя флаги готовности.
4. Ядро копирует готовую маску обратно в User Space.
5. Приложение сканирует маску, чтобы найти нужные дескрипторы.

> [!warning] Ловушка / Gotcha
> `fd_set` — это **маска**, а не список. Если вы зарегистрируете дескрипторы 5, 500 и 900, `select` все равно просканирует все 1024 бита. При 100k активных соединений это гарантированно убьет производительность. Кроме того, `select` модифицирует переданные структуры `fd_set` на месте, поэтому перед каждым вызовом их нужно пересоздавать через `FD_ZERO` и `FD_SET`.

## poll: Устранение лимита, но сохранение проблемы

```c
#include <poll.h>
int poll(struct pollfd *fds, nfds_t nfds, int timeout);
```

`poll` заменил битовые маски на массив структур `struct pollfd`. Лимит `FD_SETSIZE` исчез (ограничением становится только доступная RAM). Однако архитектурная проблема осталась:
- Массив `pollfd` все равно копируется в ядро при каждом вызове.
- Ядро все равно перебирает весь массив (O(n)).
- Нет разделения между «интересующими дескрипторами» и «готовыми дескрипторами».

> [!info] Под капотом
> `poll` по сути является оберткой над `select` внутри ядра Linux. В исходниках Linux (`fs/select.c`) можно увидеть, что `poll` и `select` используют одни и те же внутренние функции (`core_sys_select`, `do_select`). Разница лишь в том, как пользовательское пространство передает данные.

## epoll: Архитектура и работа "под капотом"

`epoll` полностью переосмыслил подход. Он разделяет дескрипторы на два независимых набора:
1. **Интерес (Interest list)** — дескрипторы, которые мы хотим мониторить.
2. **Готовность (Ready list)** — дескрипторы, которые сейчас активны.

```c
#include <sys/epoll.h>
int epoll_create(int size);
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
```

**Ключевые структуры в ядре:**
- `struct eventpoll` — главный объект, создается при `epoll_create`. Содержит красное-черное дерево (`rb_root`) для интересующих дескрипторов и список готовых (`rdllist`).
- `struct epitem` — узел дерева. Связывает файловый дескриптор с обратным вызовом (`ep_poll_callback`).
- `struct epoll_filefd` — обертка для валидации типа дескриптора (только sockets, pipes, eventfd, etc.).

**Как это работает:**
1. `epoll_ctl(EPOLL_CTL_ADD)` вставляет дескриптор в красное-черное дерево ядра.
2. Ядро регистрирует обратный вызов (`ep_poll_callback`) в драйвере устройства или сетевом стеке.
3. Когда сетевая карта получает пакет или сокет готов к чтению, драйвер вызывает обратный вызов.
4. Обратный вызов помещает дескриптор в `rdllist` (список готовности) в ядре.
5. `epoll_wait` просто копирует содержимое `rdllist` в User Space. Если список пуст — горутина/поток засыпает в ядре без сканирования.

> [!tip] Собеседование
> **Вопрос:** Почему `epoll` быстрее `select` при малом количестве активных соединений?
> **Ответ:** При малой активности `epoll` может быть даже медленнее из-за накладных расходов на работу с красным-черным деревом и аллокации `epitem`. `epoll` раскрывает свой потенциал при высокой нагрузке (10k+ соединений), где O(1) ожидания и отсутствие копирования всей маски дают экспоненциальный выигрыш.

## Triggered modes: Level vs Edge

`epoll` поддерживает два режима уведомления, что критично для проектирования сетевых демонов:

| Режим | Поведение | Аналог в Go |
|---|---|---|
| **Level-Triggered (LT)** | `epoll_wait` возвращает дескриптор, пока он остается готовым. Даже если вы прочитали 1 КБ из 1 МБ, следующий `epoll_wait` снова вернет этот дескриптор. | Стандартный режим Go `netpoller`. Безопасен, прост в реализации, но может вызывать "thundering herd" при высокой нагрузке. |
| **Edge-Triggered (ET)** | Возвращает дескриптор **только при изменении состояния** (появились новые данные). Требуется чтение до `EAGAIN/EWOULDBLOCK`. Требует неблокирующих сокетов (`O_NONBLOCK`). | Используется в высоконагруженных прокси (NGINX, HAProxy). Снижает количество пробегов, но усложняет обработку ошибок. |

> [!warning] Ловушка / Gotcha
> При использовании `EPOLLET` обязательно нужно читать данные до тех пор, пока `recv` не вернет `-1` с кодом `EAGAIN`. Если вы остановитесь раньше, `epoll` больше не вернет этот дескриптор, пока не придет новый пакет. Это частая причина "зависания" соединений в production.

## Механическая симпатия: Почему это важно для Go

Go runtime не создает отдельный тред ОС на каждую горутину. Он использует **M:N модель планирования**. Когда горутина выполняет блокирующий сетевой вызов (`net.Read`), Go runtime:
1. Отвязывает горутину от текущего системного треда (M).
2. Регистрирует сокет в `epoll` через `netpoll`.
3. Переводит тред в `syscall` или спит.
4. Когда `epoll_wait` возвращает готовый дескриптор, runtime будит соответствующую горутину и планирует её на доступный тред.

Это означает, что `epoll` — это не просто "удобный API", а фундамент, на котором держится асинхронность Go. Без эффективного I/O мультиплексирования Go-сервер не смог бы обрабатывать сотни тысяч соединений на одном физическом CPU.

> [!info] Под капотом
> В исходниках Go (`src/runtime/netpoll_epoll.go`) можно увидеть прямую обертку над `epoll_create1`, `epoll_ctl` и `epoll_wait`. Go использует `EPOLLONESHOT` для дескрипторов, чтобы избежать гонок при пробуждении горутин, и самостоятельно управляет повторной регистрацией дескрипторов после каждого чтения/записи.

## Go runtime и netpoller: Ключевые детали

Go's `netpoller` — это не просто обертка над `epoll`. Это сложный механизм, который:
- Кэширует буферы (`sync.Pool`) для сетевых пакетов, чтобы избежать аллокаций в куче.
- Использует `epoll` с флагом `EPOLLRDHUP` для быстрого обнаружения разрывов соединений.
- Интегрируется с планировщиком через `runtime.gopark` и `runtime.wakeG`, чтобы минимизировать контекстные переключения.

При высокой нагрузке `netpoller` может стать узким местом, если не соблюдать идиомы:
- Избегайте долгих блокирующих операций внутри `net.Conn`.
- Используйте `bufio` для буферизации, чтобы сократить количество вызовов `epoll_ctl` и `epoll_wait`.
- В Go 1.20+ добавлена поддержка `epoll` с `EPOLLEXCLUSIVE`, что снижает "thundering herd" при пробуждении множества горутин.

## Итог

1. `select` и `poll` ушли в историю из-за O(n) сложности и копирования данных между User/Kernel Space.
2. `epoll` использует событийно-ориентированную архитектуру с обратными вызовами в ядре, обеспечивая O(1) ожидание.
3. `epoll` разделяет дескрипторы на интерес и готовность, что критично для масштабируемости.
4. Режимы LT и ET имеют разные trade-offs: LT безопасен, ET требует неблокирующих сокетов и полного чтения.
5. Go runtime напрямую зависит от `epoll` через `netpoller`. Понимание его работы — обязательный шаг к оптимизации высоконагруженных Go-сервисов.
