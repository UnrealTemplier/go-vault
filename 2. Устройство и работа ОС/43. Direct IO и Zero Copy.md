## Классический путь IO и его фундаментальная проблема

Прежде чем разбирать оптимизации, нужно чётко понимать, что мы оптимизируем. В стандартном POSIX-пути записи файла данные проходят сложный маршрут:

1. **User Space**: Приложение вызывает `write()` и копирует данные из своего буфера в ядро.
2. **Page Cache**: Ядро копирует данные в `struct page` внутри `[[42. Буферизация IO и Page Cache]]`.
3. **Disk Scheduler**: Данные лежат в кэше. В момент, когда кэш переполняется или срабатывает таймер, ядро передаёт дескрипторы страниц в `[[45. Scheduler диска и очередь запросов]]`.
4. **DMA Engine**: Контроллер диска забирает данные из RAM напрямую на физический носитель, минуя CPU.

**Где теряется производительность?**
- **Двойное копирование**: Данные копируются из User Buffer → Page Cache (CPU), а затем Page Cache → Disk (DMA).
- **Загрязнение CPU Cache**: Копирование через `memcpy` заполняет L1/L2/L3 кэш процессора служебными данными, вытесняя кэш приложения.
- **Контекстные переключения**: Каждая граница User/Kernel требует сохранения регистров, очистки TLB и переключения стека.

Для высоконагруженного бэкенда (базы данных, стриминг, прокси) эти накладные расходы становятся узким местом. Здесь вступают `[[Direct IO и Zero Copy]]`, которые меняют физику передачи данных.

## Direct IO (O_DIRECT): Обход Page Cache

**Direct IO** — режим работы с файлами, при котором данные передаются напрямую между буфером пользовательского процесса и блочным устройством, полностью минуя `Page Cache`.

### Как это работает под капотом
Когда вы открываете файл с флагом `O_DIRECT`, ядро Linux меняет путь в структуре `struct file_operations`:
- Обычный путь: `read_iter` → `filemap_read` → `page_cache_read`
- Direct IO путь: `read_iter` → `direct_IO` → `blk_rq_map_user` → DMA

DMA-контроллер получает физический адрес вашего буфера в RAM и пишет/читает данные напрямую. CPU вообще не участвует в копировании payload. Это радикально снижает CPU utilization при работе с большими объёмами данных.

> [!warning] Ловушка / Gotcha
> **Требование выравнивания (Alignment)**
> Direct IO **требует** выравнивания как адреса буфера, так и смещения в файле. Обычно это 512 байт или 4 КБ (размер сектора/страницы). Если передать невыровненный буфер, ядро вернёт `EINVAL` и операция завершится ошибкой. Это жёсткое аппаратное ограничение DMA-контроллеров.

### Когда стоит использовать Direct IO?
- **Базы данных и движки хранения**: PostgreSQL (`fsync`, `direct_io`), Redis, ClickHouse. Они управляют кэшированием самостоятельно, чтобы избежать дублирования данных в Page Cache и `[[42. Буферизация IO и Page Cache]]`.
- **Высокопропускной стриминг**: Логгеры, агрегаторы метрик, где важны стабильные задержки, а не кэширование.
- **Избегание cache thrashing**: Если вы работаете с файлами, которые не помещаются в RAM, стандартный кэш будет постоянно вытеснять другие полезные данные (горячие страницы). Direct IO устраняет этот шум.

### Когда Direct IO вреден?
- **Случайный доступ к мелким файлам**: Без кэширования каждое чтение идёт на диск. Latency вырастет в 100-1000 раз.
- **Частые маленькие записи**: Overhead на выравнивание и отсутствие буферизации убьёт throughput.
- **Не все ФС поддерживают его стабильно**: XFS, ext4, btrfs работают хорошо. NFS, FUSE, tmpfs могут игнорировать флаг или работать медленно.

## Zero Copy: Устранение копирования на уровне ядра

**Zero Copy** — это не отдельный syscall, а архитектурный паттерн, при котором данные передаются между двумя kernel-space буферами (например, файл → сокет) без участия User Space.

### Механизм работы
Классический пример: отправка статического файла по сети.
1. `sendfile()` syscall вызывает ядро.
2. Ядро читает файл в `Page Cache` (как обычно).
3. Вместо копирования в user buffer, ядро **привязывает дескрипторы страниц** (`struct page`) к буферу сокета.
4. Сетевой стек (TCP/IP) формирует дескрипторы пакетов, ссылаясь на те же `struct page`.
5. DMA-контроллер NIC (сетевой карты) забирает данные из `Page Cache` прямо в сетевой буфер.

**CPU видит только один контекстный переход** (на вызов `sendfile`). Данные никогда не покидают ядро. В современных ядрах (Linux 2.6.4+) используется `splice()` и `copy_file_range()`, которые делают то же самое для любых файловых дескрипторов.

> [!info] Под капотом
> В ядре Linux данные передаются через **Page Descriptors**. Когда вы вызываете `mmap()`, вы не копируете данные, вы просто создаёте отображение адресного пространства процесса на существующие `struct page`. При первом обращении возникает `Page Fault`, ядро мапит страницу, и дальше CPU работает с ней как с обычной памятью. Это основа `mmap`-оптимизации.

### Go-реализация Zero Copy
В Go вам не нужно писать обёртки над `sendfile`. Стандартная библиотека делает это автоматически:
```go
// net/http автоматически использует sendfile на Linux для static files
http.ServeFile(w, r, filepath)

// io.Copy оптимизирована: если src - *os.File, а dst - net.Conn,
// Go 1.16+ автоматически вызоде syscall sendfile
io.Copy(dst, src)
```
Для работы с огромными файлами (>2GB) используется `mmap`:
```go
import "golang.org/x/exp/mmap" // или syscall.Mmap

f, _ := os.Open("huge.bin")
data, _ := mmap.Map(f, syscall.PROT_READ, syscall.MAP_PRIVATE)
defer mmap.Unmap(data)
// data[] работает как обычный slice, но данные не загружаются в RAM полностью.
// Загрузка происходит постранично через Page Fault.
```

## Сравнение стратегий и Mechanical Sympathy

| Характеристика | Standard IO | Direct IO (`O_DIRECT`) | Zero Copy (`sendfile`/`mmap`) |
|:---|:---|:---|:---|
| **Путь данных** | User ↔ Kernel ↔ Disk | User ↔ Disk | Kernel ↔ Kernel |
| **Копирования CPU** | 2+ (memcpy) | 0 (только syscall setup) | 0 (дескрипторы страниц) |
| **Кэш CPU (L1/L2/L3)** | Загрязняется payload-ом | Чистый (нет копирования) | Чистый |
| **Кэш Page Cache** | Используется | Обходится | Используется (для read) |
| **Задержка (Latency)** | Высокая (контексты) | Низкая (для больших блоков) | Минимальная |
| **Throughput** | Ограничен CPU | Ограничен диском/DMA | Ограничен шиной/сетью |
| **Сложность** | Низкая | Высокая (выравнивание) | Средняя (зависит от ФС) |

> [!tip] Собеседование
> **Вопрос:** В чём разница между `O_DIRECT` и Zero Copy?
> **Ответ:** `O_DIRECT` оптимизирует путь **User ↔ Disk**, устраняя Page Cache. Он нужен, когда приложение само хочет управлять кэшем или избегать загрязнения RAM. Zero Copy оптимизирует путь **Kernel ↔ Kernel** (например, File → Socket), устраняя переход User Space. Данные могут лежать в Page Cache, но не копируются в user buffer.
>
> **Вопрос:** Почему `O_DIRECT` требует выравнивания?
> **Ответ:** DMA-контроллеры работают с физическими страницами памяти. Невыровненные адреса могут привести к тому, что данные разобьются на две физические страницы, что потребует двух DMA-транзакций или промежуточного буфера в ядре, что ломает концепцию bypass. Жёсткое требование — гарантия атомарности DMA-операции.

## Go-специфика и практическая реализация

При переходе с PHP/Java/C# на Go важно понять: **Go не скрывает IO-реализм**. Если вы пишете высоконагруженный сервис, вы должны контролировать путь данных.

### 1. Работа с `O_DIRECT` в Go
```go
package main

import (
    "os"
    "syscall"
    "unsafe"
)

func writeDirectIO(path string, data []byte) error {
    // 1. Выравнивание буфера (4KB)
    aligned := make([]byte, len(data)+4096, len(data)+4096+4096)
    aligned = aligned[:len(data)]
    copy(aligned, data)
    
    // 2. Открытие с флагом
    f, err := os.OpenFile(path, os.O_WRONLY|os.O_CREATE|os.O_TRUNC|syscall.O_DIRECT, 0644)
    if err != nil {
        return err
    }
    defer f.Close()

    // 3. Запись. Offset тоже должен быть выровнен!
    _, err = f.WriteAt(aligned, 0)
    return err
}
```
> [!warning] Ловушка / Gotcha
> В Go 1.16+ флаг `syscall.O_DIRECT` доступен через `os.O_DIRECT`. Не используйте устаревший `syscall.O_DIRECT` напрямую без проверки совместимости на разных ОС (macOS/Windows не поддерживают O_DIRECT, код упадёт или проигнорирует флаг).

### 2. Управление GC давлением
Direct IO и Zero Copy критически важны для контроля GC. Каждая аллокация буфера в User Space создаёт объект в куче. При обработке 10GB логов в память вы получите:
- 10GB аллокаций
- Частые STW паузы (Stop-The-World) для Scavenge/Mark
- Конкуренция с другими горутинами

Используя `mmap` или `O_DIRECT` с пулом выровненных буферов, вы выносите работу за пределы кучи Go. GC видит только дескрипторы файлов, а не payload.

## Ловушки и вопросы с собеседований

### Corner Cases
1. **`O_DIRECT` на SSD vs HDD**: На SSD разница менее очевидна, так как нет механического seek time. Но на HDD Direct IO даёт огромный выигрыш за счёт последовательного чтения блоков.
2. **Сетевые сокеты и `sendfile`**: `sendfile` не работает с некоторыми типами сокетов (например, `AF_UNIX` в старых ядрах, или зашифрованные TLS-сокеты, где нужен user-space парсинг).
3. **`fsync()` и `O_DIRECT`**: Даже с Direct IO данные не гарантированно записаны на диск после `write()`. Нужно вызывать `f.Sync()` или `fsync()`. DMA асинхронна.

### Типичные вопросы на Middle+/Senior
- **Как `net/http` сервит статические файлы в Go?** Автоматически использует `sendfile` (Zero Copy) на Linux, `WriteFile` на других ОС. Это можно увидеть в `src/net/http/fs.go`.
- **Почему `io.Copy` иногда быстрее, чем `bufio.NewReader` + `bufio.Writer`?** `io.Copy` на Linux вызывает `sendfile`. `bufio` добавляет лишнее копирование в промежуточный буфер и контекстные переходы.
- **Как отладить падение с `EINVAL` при `O_DIRECT`?** Проверьте выравнивание буфера (`uintptr(unsafe.Pointer(&data[0])) % 4096 == 0`) и смещения файла. Используйте `strace` для отслеживания syscalls.

## Итог

1. **Standard IO** удобен, но платит за это двойным копированием и загрязнением CPU Cache.
2. **Direct IO (`O_DIRECT`)** убирает Page Cache, связывая User Buffer напрямую с диском через DMA. Требует жёсткого выравнивания, идеален для БД и больших блобов.
3. **Zero Copy (`sendfile`/`mmap`)** устраняет переход User ↔ Kernel для передачи данных. Данные остаются в Page Cache, но не копируются в user space. Максимальный throughput для сетевых сервисов.
4. В Go эти примитивы доступны через `os.O_DIRECT`, `syscall.Mmap` и автоматически в `net/http`/`io`. Контроль пути данных — ключ к предсказуемой latency и контролю GC.

Мы разобрали, как данные перемещаются между памятью и диском, минимизируя CPU overhead. Следующим логичным шагом будет понимание физического уровня: как устроены сами диски, SSD и контроллеры, которые исполняют эти DMA-запросы.
