## systemd: Инициатор, менеджер и надзиратель

В современных дистрибутивах Linux (RHEL, Ubuntu, Debian, Alpine-based images) **systemd** занимает PID 1. Это не просто «менеджер процессов», а полнофункциональная инициализационная система, управляющая жизненным циклом демонов, изоляцией ресурсов, журналированием и активацией сокетов. Для Go-разработчика, целящегося на Senior/Lead, понимание systemd критично: именно он определяет, как ваше приложение стартует, получает ресурсы, реагирует на падения и интегрируется в инфраструктуру.

В отличие от классических SysVinit-скриптов, systemd работает **параллельно**, использует **cgroups** для ограничения ресурсов и **namespaces** для изоляции. Это напрямую влияет на производительность, безопасность и наблюдаемость Go-приложения.

## Unit-файл: Анатомия конфигурации Go-приложения

Конфигурация сервиса в systemd хранится в `.service` файлах. Для production-ready Go-приложения минимально достаточный unit выглядит так:

```go
// go.mod
require github.com/coreos/go-systemd/v22 v22.5.0
```

```go
// main.go
package main

import (
	"fmt"
	"log"
	"net"
	"os"
	"strconv"

	"github.com/coreos/go-systemd/v22/daemon"
)

func main() {
	// 1. Поддержка systemd socket activation
	if listenFDs, ok := os.LookupEnv("LISTEN_FDS"); ok {
		fd, err := strconv.Atoi(listenFDs)
		if err != nil || fd < 1 {
			log.Fatalf("invalid LISTEN_FDS: %v", listenFDs)
		}
		// systemd передает FD начиная с 3. Берем первый.
		conn, err := net.FileConn(os.NewFile(3, ""))
		if err != nil {
			log.Fatalf("failed to get connection from FD 3: %v", err)
		}
		defer conn.Close()
		// ... обработка conn ...
	} else {
		// Fallback для локальной разработки
		conn, err := net.Listen("tcp", ":8080")
		if err != nil {
			log.Fatal(err)
		}
		defer conn.Close()
	}

	// 2. Отправка READY сигнала в systemd
	_, err := daemon.SdNotify(false, daemon.SdNotifyReady)
	if err != nil {
		log.Printf("failed to notify systemd: %v", err)
	}
	defer daemon.SdNotify(false, daemon.SdNotifyStopping)

	// Запуск сервера
	log.Println("server started")
}
```

```ini
# /etc/systemd/system/myapp.service
[Unit]
Description=Go Backend Application
After=network.target postgresql.service
Requires=postgresql.service

[Service]
Type=notify
User=appuser
Group=appgroup
WorkingDirectory=/opt/myapp
ExecStart=/opt/myapp/bin/myapp
ExecReload=/bin/kill -SIGUSR1 $MAINPID
Restart=on-failure
RestartSec=5s
WatchdogSec=30s
LimitNOFILE=65536
MemoryMax=2G
CPUQuota=80%
EnvironmentFile=/etc/myapp/env

[Install]
WantedBy=multi-user.target
```

> [!info] Под капотом
> `Type=notify` означает, что systemd **не считает сервис запущенным**, пока не получит сигнал через Unix Domain Socket `/run/systemd/notify`. Это решает проблему race condition в `Type=simple`, когда `ExecStart` возвращает код 0, но инициализация внутри процесса еще не завершилась (например, подключение к БД не установлено).

## Socket Activation: Устранение TCP-накладных расходов

systemd умеет открывать TCP/UDP/Unix-сокеты **до** запуска процесса. При первом входящем соединении он `fork()` и `exec()` процесс, передавая готовый FD через переменную окружения `LISTEN_FDS`.

**Зачем это Go-разработчику:**
1. **Zero-downtime деплой:** systemd держит сокет открытым. Новый процесс получает его, старый завершается после `SIGUSR1` и обработки активных соединений.
2. **Устранение холодного старта:** TCP-рукопожатие (SYN/SYN-ACK/ACK) и инициализация TLS занимают 1-3 RTT. С активацией сокета приложение стартует мгновенно при первом запросе, а сокет уже в состоянии `LISTEN`.
3. **Балансировка нагрузки:** systemd может передавать сокет сразу нескольким инстансам сервиса (socket-based activation).

```go
// Пример получения FD от systemd
func getListenFD() (net.Listener, error) {
	if fdStr, ok := os.LookupEnv("LISTEN_FDS"); ok {
		fd, err := strconv.Atoi(fdStr)
		if err != nil {
			return nil, fmt.Errorf("invalid LISTEN_FDS: %w", err)
		}
		f := os.NewFile(uintptr(fd), "")
		conn, err := net.FileConn(f)
		if err != nil {
			return nil, fmt.Errorf("net.FileConn failed: %w", err)
		}
		return conn, nil
	}
	return net.Listen("tcp", ":8080")
}
```

> [!warning] Ловушка / Gotcha
> Не используйте `os.NewFile(uintptr(fd), "").Close()` после получения `net.FileConn`. `net.FileConn` создает обертку, которая **не закроет** базовый FD при `Close()`. Это намеренно: systemd должен контролировать жизненный цикл сокета. Закрывать сокет должен только systemd или явный `systemctl stop`.

## Под капотом: cgroups, namespaces и запуск процесса

Когда вы запускаете `systemctl start myapp`, происходит следующий цепочный процесс:

1. **Парсинг unit-файла:** systemd читает директивы, создает **cgroup v2** группу `system.slice/myapp.service`.
2. **Изоляция ресурсов:** `MemoryMax`, `CPUQuota`, `LimitNOFILE` применяются к cgroup через `/sys/fs/cgroup/`. Это **жесткие** ограничения ядра. Если Go-приложение превысит `MemoryMax`, ядро убьет процесс (OOMKilled), даже если GC работает корректно.
3. **Формирование процесса:** 
   - `fork()` -> `clone()` с флагами `CLONE_NEWPID`, `CLONE_NEWNS`, `CLONE_NEWNET` (если настроено в `ProtectSystem`, `ProtectHome`, `PrivateTmp`).
   - `setuid`/`setgid` для `User`/`Group`.
   - `chdir` в `WorkingDirectory`.
   - Установка `EnvironmentFile`.
4. **Exec:** `execve()` бинарника. Передача `LISTEN_FDS` через env.
5. **Наблюдение:** systemd отслеживает PID главного процесса (`$MAINPID`). Если он падает, systemd проверяет `Restart=` и `RestartSec=`.

**Сравнение с другими подходами:**
- **PHP-FPM:** Тоже поддерживает socket activation, но управляет только воркерами PHP. systemd универсален и работает с любым бинарником.
- **Docker/K8s:** В контейнеризованном мире systemd часто заменяется на `tini` или `dumb-init` (PID 1 в контейнере). Однако на bare-metal или в гибридных кластерах systemd остается стандартом.
- **C++/C#:** В этих экосистемах часто используют `systemd` для управления демоном, но Go-приложения выигрывают от `Type=notify` благодаря отсутствию тяжелых фреймворков.

> [!tip] Собеседование
> **Вопрос:** В чем разница между `LimitNOFILE=65536` в systemd и `ulimit -n` в bash?
> **Ответ:** `ulimit` — это ограничение процесса в текущей сессии shell. `LimitNOFILE` в systemd применяется к процессу **до** его запуска, на уровне ядра и cgroup. Даже если пользователь выполнит `ulimit -n 1000`, systemd переопределит это значение на указанное в unit-файле. Для Go-бэкенда это критично: `netpoller`, goroutines, открытые соединения к БД и кэшам быстро исчерпают лимит по умолчанию (1024).

## Watchdog, Health Checks и graceful shutdown

В production Go-приложения должны сообщать о своем здоровье. systemd поддерживает **Watchdog** (наблюдатель), который убьет процесс, если тот не отчитается в течение `WatchdogSec`.

```go
// Пинг каждые 15 секунд (WatchdogSec=30s)
ticker := time.NewTicker(15 * time.Second)
defer ticker.Stop()

go func() {
	for {
		select {
		case <-ticker.C:
			_, err := daemon.SdNotify(false, daemon.SdNotifyWatchdog)
			if err != nil {
				log.Printf("watchdog notify failed: %v", err)
			}
		}
	}
}()
```

**Graceful Shutdown в связке с systemd:**
systemd посылает `SIGTERM` при `stop` или `reload`. Задержка определяется `TimeoutStopSec` (по умолчанию 90s). Внутри этого окна приложение должно:
1. Остановить прием новых соединений (`listener.Close()`).
2. Завершить обработку активных запросов.
3. Отключиться от БД/брокеров.
4. Отправить `SdNotifyStopping` (опционально, для ускорения таймаута).
5. Завершиться с кодом 0.

Если приложение не завершится за `TimeoutStopSec`, systemd пришлет `SIGKILL`. Это вызовет `core dump`, который можно исследовать через `coredumpctl`.

> [!warning] Ловушка / Gotcha
> Не используйте `SIGKILL` (9) в `ExecReload=`. Он не дает приложению возможности освободить ресурсы. Используйте `SIGUSR1` или `SIGHUP`, которые вы перехватываете в Go через `os/signal` и запускаете graceful shutdown.

## Ловушки и вопросы на собеседовании

1. **`Type=simple` vs `Type=notify`**
   - `simple`: systemd считает процесс запущенным сразу после `exec`. Если приложение падает в инициализации, systemd не узнает об этом, пока не упадет PID.
   - `notify`: ждет сигнала `READY=1`. Надежнее для production.
   
2. **`KillMode`**
   - `control-group` (default): убивает все процессы в cgroup.
   - `process`: убивает только `$MAINPID`. Оставшиеся процессы (например, воркеры) могут создать zombies или повиснуть в памяти.
   - `mixed`: `SIGTERM` для `$MAINPID`, `SIGKILL` для группы. Часто используется для Go-приложений с отдельными воркерами.

3. **`Restart=always` vs `Restart=on-failure`**
   - `always` перезапускает сервис даже при успешном завершении (`exit 0`). Для демонов это нормально, но для скриптов приведет к бесконечному циклу.
   - `on-failure` перезапускает только при ненулевом коде или сигнале.

4. **Вопрос на интервью:** *Как systemd управляет памятью Go-приложения? Что произойдет, если приложение превысит `MemoryMax`?*
   **Ответ:** systemd применяет `MemoryMax` через cgroup v2. Ядро Linux отслеживает RSS и cache процесса. При превышении лимита ядро инициирует OOM. Если процесс не может освободить память, он получает `SIGKILL`. Для Go это особенно опасно из-за `madvise(MADV_DONTNEED)` и работы GC: память может быть возвращена ядру не мгновенно, что вызовет ложные срабатывания или OOMKilled. Рекомендуется использовать `MemoryHigh` (soft limit) вместо жесткого `MemoryMax` для Go-приложений.

## Итог

systemd — это не просто «менеджер сервисов», а фундамент операционной системы для управления процессами в production. Понимание его работы позволяет:
1. Корректно настраивать жизненный цикл Go-приложения (`Type=notify`, `Watchdog`, `Restart`).
2. Использовать `LISTEN_FDS` для ускорения холодного старта и zero-downtime деплоя.
3. Правильно ограничивать ресурсы через cgroups, не ломая работу GC и netpoller.
4. Отлаживать падения через `journalctl` и `coredumpctl`.
