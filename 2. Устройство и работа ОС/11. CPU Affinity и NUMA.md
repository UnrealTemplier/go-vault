## Управление процессами и памятью на уровне ядра

В предыдущей статье [[17. NUMA (Non-Uniform Memory Access) и Hyper-Threading]] мы разобрали физическую топологию сервера. Однако современная ОС не просто констатирует факт существования NUMA-узлов, она активно управляет тем, **где** будет исполняться код и **откуда** будет браться память. 

Для Go-разработчика переход от понимания архитектуры к управлению ресурсами на уровне ядра — это граница между написанием кода, который «просто работает», и системами, которые стабильно держат SLA под нагрузкой. В Linux управление этими ресурсами осуществляется через системные вызовы планировщика, политики памяти и иерархию контрольных групп.

## CPU Affinity: Жесткая привязка к ядрам

**CPU Affinity (Связь процессора)** — это механизм ядра Linux, который ограничивает набор логических процессоров, на которых процесс или поток может исполняться. Это не рекомендация, а жесткое ограничение. Если процесс имеет affinity к ядрам `[0, 1, 2]`, планировщик **никогда** не перенесет его на ядро `3`, даже если все первые три ядра заняты (в режиме `SCHED_OTHER`), а ядро `3` простаивает.

### Как это работает под капотом
В ядре Linux affinity хранится в структуре задачи `task_struct` в поле `cpus_mask`. При вызове системного API `sched_setaffinity(pid, mask, sizeof(mask))` ядро:
1. Валидирует маску (битовую карту) доступных CPU.
2. Устанавливает флаг `SCHED_AFFINITY` для задачи.
3. Отключает задачу от глобальной очереди балансировки нагрузки (`load_balance`).

> [!info] Под капотом
> В пространстве пользователя API обертывается библиотекой pthread: `pthread_setaffinity_np(pthread_t thread, size_t cpusetsize, const cpu_set_t *cpuset)`. Internally это вызывает `sched_setaffinity` для текущего потока. Для Go-программ это означает, что привязка применяется к конкретному OS thread (`M` в терминологии рантайма), а не ко всей горутинам автоматически.

### Утилиты и управление
* `taskset -c 0-3 ./myapp` — установка affinity для всего процесса до старта.
* `ps -T -p <PID>` — проверка текущей маски.
* `cgroups v2 cpuset` — современный способ изоляции (см. [[52. Ограничения ОС. ulimit, cgroups, quotas]]).

> [!warning] Ловушка / Gotcha
> Жесткая привязка отменяет автоматическую балансировку нагрузки планировщика. Если вы привяжете Go-сервис к ядрам 0-3, а нагрузка вырастет, эти ядра будут загружены на 100%, пока остальные простаивают. Affinity оправдан только когда вы **знаете**, что локальность данных кэша (L1/L2/L3) критичнее баланса нагрузки.

## Как планировщик ОС работает с Affinity

Планировщик CFS (Completely Fair Scheduler) в Linux стремится к идеальной справедливости, распределяя CPU time пропорционально приоритету (`nice`). Когда вы применяете affinity, вы вводите задачу в режим **Sticky Task**. 

Планировщик:
1. Проверяет `cpus_mask` при каждом `schedule()`.
2. Если текущее ядро вышло из маски, он ищет первое доступное ядро в маске.
3. Если ни одно ядро в маске не свободно, задача переходит в состояние `TASK_UNINTERRUPTIBLE` (ожидание) или получает минимальный квант на доступном ядре, создавая «бутылочное горлышко» на физическом уровне.

Для высоконагруженных Go-сервисов это означает, что `GOMAXPROCS` должен строго соответствовать количеству ядер в маске affinity. Иначе горутины будут биться за доступ к ограниченному пулу `P` (logical processors), что приведет к спинам в планировщике Go и росту latency.

## NUMA-ориентированное управление памятью в Linux

ОС не просто видит NUMA-топологию, она активно управляет распределением памяти, чтобы минимизировать `Remote Access`. 

### Системные вызовы управления памятью
1. `mbind(addr, len, mode, nodes, maxnode, flags)` — привязывает диапазон виртуальной памяти к конкретным NUMA-узлам.
2. `set_mempolicy(mode, nodemask, maxnode)` — устанавливает политику для будущих аллокаций текущего потока.

В Linux по умолчанию действует политика `MPOL_PREFERRED`, где память аллоцируется в узле, к которому принадлежит вызвавший поток. Если локальная память переполнена, ОС падает в `MPOL_INTERLEAVE` (чередование по страницам) или `MPOL_BIND` (строго заданные узлы).

> [!info] Под капотом
> При вызове `malloc` (или `mmap` от [[15. malloc под капотом. Откуда ОС берет память]]) Go рантайм вызывает `sys_mmap`. Ядро Linux проверяет текущую `mempolicy` потока. Если политика `MPOL_BIND` задает узел `0`, а в нем нет свободных свободных страниц, ядро вернет `ENOMEM` без попыток аллокации на удаленных узлах. Это критично для систем с ограниченным RAM.

### Автоматическое NUMA-балансирование
Начиная с ядра 2.6.38+, Linux включает `kernel.numa_balancing`. Если оно активно:
1. Ядро отслеживает `Page Fault` и `TLB Miss` для процессов.
2. При обнаружении высокой доли удаленных обращений (Remote Page Faults > Local), оно планирует миграцию страниц через `ksm` и `migratepages`.
3. **Проблема:** Эта миграция происходит в фоне, блокирует страницы и создает всплески CPU. При ручной привязке (Affinity) автоматическая миграция часто мешает, поэтому в production её отключают:
   ```bash
   echo 0 > /proc/sys/kernel/numa_balancing
   ```

## Интеграция с cgroups и оркестраторами

В контейнеризованных средах ручное управление через `taskset` или `numactl` невозможно. Управление ресурсами делегируется **cgroups v2**.

### cgroups v2: cpuset и memory
Контрольная группа `cpuset` в cgroups v2 позволяет:
* Задать `cpuset.cpus` и `cpuset.mems` (узлы памяти).
* Включить `cpuset.sched_load_balance` (влияет на планировщик).
* Включить `cpuset.mems_hardwall` (запрещает аллокацию памяти вне заданных узлов).

```bash
# Пример изоляции сервиса в узле 0
mkdir /sys/fs/cgroup/io_service
echo 0 > /sys/fs/cgroup/io_service/cpuset.cpus
echo 0 > /sys/fs/cgroup/io_service/cpuset.mems
echo $$ > /sys/fs/cgroup/io_service/cgroup.procs
```

### Kubernetes CPU Manager и Topology Manager
K8s абстрагирует NUMA через два компонента:
1. **CPU Manager** (`cpuManagerPolicy: static`): Гарантирует, что под получит exclusive доступ к физическим ядрам. Включается для stateful-сервисов (базы данных, высоконагруженные API).
2. **Topology Manager** (`topologyManagerPolicy: restricted`): Отклоняет под, если их запросы не укладываются в один NUMA-узел. Это предотвращает случайное «размазывание» процесса по разным узлам памяти.

> [!tip] Собеседование
> **Вопрос:** Как вы будете деплоить высоконагруженный Go-сервис на NUMA-сервере в Kubernetes, чтобы избежать remote memory access?
> **Ответ:** 
> 1. Включить `cpuManagerPolicy: static` и `topologyManagerPolicy: restricted` в kubelet.
> 2. Запросить ресурсы в `resources.requests` точно равные количеству ядер одного NUMA-узла (например, `cpu: "8"`).
> 3. Установить `GOMAXPROCS=8` внутри контейнера.
> 4. Отключить автоматическое NUMA-балансирование на ноде (`sysctl -w kernel.numa_balancing=0`).
> 5. Использовать `nodeSelector` или `affinity` для размещения подов на нодах с идентичной топологией.

## Mechanical Sympathy: Влияние на Go-рантайм

Go runtime не является полностью aware к NUMA-политикам ОС из коробки. Он полагается на глобальный пул памяти и динамическую балансировку `P` (logical processors). 

### Ключевые точки взаимодействия:
1. **Старт рантайма:** При инициализации Go вызывает `sched_getaffinity` для определения доступных ядер. Если вы привязали процесс через `numactl` или cgroups, рантайм автоматически увидит только доступные ядра.
2. **Аллокация кучи:** Go выделяет память через `mmap` большими chunk-ами (arenas). Если процесс запущен на узле `0`, куча будет в основном в локальной RAM. Но при росте `GOMAXPROCS` и переходе горутин на другие узлы (если affinity не настроен), горутины начнут работать с «чужой» памятью, вызывая `Remote TLB Miss`.
3. **GC Residency:** В Go 1.21+ появился флаг `GODEBUG=gcresidency=1`. Он выводит в лог, сколько времени GC работает на каждом логическом процессоре. Если вы видите дисбаланс (например, 90% GC на ядрах 0-3, а 10% на 4-7), это прямой признак того, что память распределена неравномерно относительно физического расположения `P`.

> [!warning] Ловушка / Gotcha
> Установка `GOMAXPROCS` больше, чем количество физических ядер в одном NUMA-узле, на сервере с NUMA-топологией почти гарантированно приведет к падению производительности из-за:
> * Переполнения кэша L3 при переключении между узлами.
> * Конкуренции за межпроцессорную шину UPI/Infinity Fabric.
> * Неэффективной работы GC, который сканирует кучу, физически размазанную по разным сокетам.

## Итог

1. **CPU Affinity** — это инструмент жесткого контроля планировщика через `sched_setaffinity`. Он отключает глобальный баланс ради локальности кэша и предсказуемости latency.
2. **NUMA-управление в Linux** строится на `mbind`/`set_mempolicy` и автоматически балансирует страницы через `ksm`. В production автоматическую миграцию страниц обычно отключают.
3. **В контейнеризации** ответственность за изоляцию переходит на cgroups v2 (`cpuset`) и K8s CPU/Topology Manager. Ручная настройка `taskset` в Kubernetes не работает.
4. **Для Go-инженера:** `GOMAXPROCS` должен отражать количество ядер в **одном** NUMA-узле. Используйте `GODEBUG=gcresidency=1` для валидации распределения памяти. Понимание этого механизма отличает Senior-архитектора, который просто «поднимает сервис», от Lead-инженера, который проектирует систему под железо.
