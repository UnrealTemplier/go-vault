## Общая память: Самый быстрый канал связи между процессами

В иерархии **[[31. Межпроцессное взаимодействие. IPC]]** общая память (Shared Memory) стоит на вершине по производительности. В отличие от `pipe`, `socket` или `message queue`, она исключает копирование данных между User Space и Kernel Space. Два или более процессов получают доступ к одним и тем же физическим страницам RAM через свои виртуальные адресные пространства.

Это фундамент для СУБД (InnoDB buffer pool, Redis), кэширующих систем, высокочастотных трейдингов и shared-object библиотек. Но за скорость приходится платить сложностью синхронизации и риском `Data Race`.

> [!info] Под капотом
> Общая память реализуется на уровне виртуальной памяти. Когда процесс вызывает `mmap()`, ядро выделяет виртуальные страницы, но **Page Table Entry (PTE)** указывает на один и тот же **Page Frame Number (PFN)** в физической RAM. Никакого копирования данных не происходит — только дублирование записей в таблицах страниц.

## Как ОС реализует общую память

В Linux существует два исторических API:
1. **System V IPC** (`shmget`, `shmat`, `shmdt`) — устаревший, но до сих пор поддерживаемый. Использует уникальный ключ (`key_t`) для поиска сегмента.
2. **POSIX Shared Memory** (`shm_open`, `mmap`, `munmap`) — современный стандарт. Работает через файловую систему `/dev/shm` (tmpfs в RAM).

Под капотом POSIX использует `shm_open()` для создания файла в `/dev/shm`, а затем `mmap()` для отображения этого файла в память процесса. Поскольку `shm_open` возвращает файловый дескриптор, можно применять `ftruncate()`, чтобы задать размер сегмента.

```go
package main

import (
	"fmt"
	"os"
	"syscall"
	"unsafe"
)

func main() {
	// 1. Создаем или открываем сегмент в /dev/shm
	fd, err := syscall.ShmOpen("/my_shared_seg", syscall.O_RDWR|syscall.O_CREAT, 0644)
	if err != nil {
		// production-ready: всегда обрабатываем ошибки
		panic(fmt.Sprintf("shm_open failed: %v", err))
	}
	defer syscall.ShmUnlink("/my_shared_seg")

	// 2. Устанавливаем размер (например, 4096 байт)
	if err := syscall.ShmSetSize(fd, 4096); err != nil {
		panic(fmt.Sprintf("shm_set_size failed: %v", err))
	}

	// 3. Отображаем в память процесса
	ptr, err := syscall.Mmap(fd, 0, 4096, syscall.PROT_READ|syscall.PROT_WRITE, syscall.MAP_SHARED)
	if err != nil {
		panic(fmt.Sprintf("mmap failed: %v", err))
	}
	defer syscall.Munmap(ptr)

	// ptr теперь указывает на область, видимую другими процессами
	// Работа с ptr требует синхронизации!
	_ = ptr
}
```

> [!warning] Ловушка / Gotcha
> `MAP_SHARED` vs `MAP_PRIVATE`. Если использовать `MAP_PRIVATE`, изменения не будут видны другим процессам и нарушат правило Copy-On-Write. Всегда используйте `MAP_SHARED` для межпроцессной общей памяти.

## Корень зла: Race Conditions и Data Races

Когда несколько процессов пишут в одну область без синхронизации, возникает **Race Condition**. Если это приводит к неопределенному поведению из-за чтения/записи без атомарности — это **Data Race**.

Пример классической гонки:
```c
// Процесс А и Процесс Б одновременно делают:
counter = counter + 1;
```

На уровне ассемблера это разваливается в три инструкции:
1. `LOAD R1, [addr]` (прочитать из памяти в регистр)
2. `ADD R1, R1, 1` (инкрементировать)
3. `STORE [addr], R1` (записать обратно)

Если Процесс А прерывается между шагами 1 и 3, а Процесс Б успевает выполнить весь цикл, результат будет `counter + 1` вместо `counter + 2`. Потерянное обновление (Lost Update).

## Синхронизация на уровне ОС

Чтобы использовать общую память, нужно наложить примитивы синхронизации. В Go стандартный `sync.Mutex` **не работает** между процессами, так как он блокируется только в рамках адресного пространства текущего процесса.

Для cross-process синхронизации используются:
- **POSIX Semaphores** (`sem_open`, `sem_wait`, `sem_post`) — работают с общей памятью или как именованные объекты.
- **Futexes** — низкоуровневый примитив Linux, сочетающий блокировку в User Space с системным вызовом только при конфликте.
- **Atomic Operations** — инструкции процессора (`LOCK XADD`, `CMPXCHG`), гарантирующие атомарность на уровне шины/кэша.

> [!tip] Собеседование
> **Вопрос:** Почему нельзя использовать `sync.Mutex` из Go для синхронизации общей памяти между двумя разными Go-процессами?
> **Ответ:** `sync.Mutex` использует атомарные операции над памятью внутри текущего адресного пространства. Он не создает именованных объектов в ядре. Второй процесс просто не увидит состояние мьютекса, так как его виртуальная память выделена отдельно. Для cross-process нужны именованные семафоры (`sem_open`) или futex-подобные механизмы, либо `futex` через `cgo`/`syscall`.

## Mechanical Sympathy: Кэш, MESI и False Sharing

Даже если вы используете атомарные инструкции, производительность общей памяти может быть убита архитектурой CPU.

### 1. Cache Coherence (MESI Protocol)
Современные CPU имеют локальные кэши L1/L2. Когда два ядра пишут в один кэш-лайн (64 байта), включается протокол **MESI** (Modified, Exclusive, Shared, Invalid).
- Ядро 1 пишет в байт 0 кэш-лина. Статус меняется на `Modified`.
- Ядро 2 пытается прочитать байт 4. Оно должно инвалидировать свою копию кэш-лина и забрать свежие данные у Ядра 1 через шину.
- Частые записи в один кэш-лайн вызывают **Cache Thrashing**.

### 2. False Sharing (Ложная совместная запись)
Если два процесса/потока пишут в разные переменные, но они попадают в один кэш-лайн (например, `struct { int a; int b; }`), процессор считает, что происходит конфликт. Он будет постоянно инвалидировать кэш-лайн, хотя данные не пересекаются.

**Решение:** Padding (выравнивание по границе кэш-лина).
```go
// Go-аналог padding для избежания false sharing
type SafeCounter struct {
    Value int64
    _     [56]byte // padding до 64 байт (размер кэш-лайна)
}
```

### 3. Memory Barriers (Fences)
Компилятор и CPU могут переставлять инструкции для оптимизации. `mfence` (x86) или `dmb` (ARM) запрещают это, гарантируя порядок операций для других ядер. Атомарные операции в Go (`atomic.AddInt64`) автоматически генерируют необходимые барьеры.

## Go и shared memory: Философия и реальность

Go имеет четкую философию:
> *"Don't communicate by sharing memory; share memory by communicating."* — Кен Томпсон / Роб Пайк

Это означает, что горутины должны обмениваться данными через каналы (`chan`), а не через общую память. Это устраняет `Data Race` на уровне языка и делает код безопасным.

**Но в реальности:**
1. **Горутины внутри одного процесса** действительно делят память. `sync.Mutex`, `atomic` пакет и `unsafe.Pointer` работают быстро, но требуют дисциплины.
2. **Cross-process shared memory** в Go поддерживается через `syscall.Mmap` или `golang.org/x/sys/unix`. Однако синхронизация требует внешних примитивов (`semaphore`, `futex`).
3. **Race Detector** (`go run -race`) внедряет Instrumentation на этапе компиляции и отслеживает все чтения/записи. Он работает только в рамках одного процесса. Для cross-process отладки нужны `valgrind` или `helgrind`.

> [!tip] Собеседование
> **Вопрос:** Когда в Go все-таки оправдано использовать общую память, а не каналы?
> **Ответ:** 
> 1. Высоконагруженные компоненты, где overhead каналов (`chan`) слишком велик (например, internal metrics collector, shared ring buffer).
> 2. Взаимодействие с C/C++ библиотеками через `cgo`, где данные уже лежат в общей памяти.
> 3. Cross-process кэширование, где производительность критичнее простоты кода (например, `memcached`-подобные структуры).
> В таких случаях используют `unsafe.Pointer`, выравнивание по кэш-лайну, атомарные операции и иногда `cgo` для POSIX семафоров.

## Ловушки и типичные вопросы на интервью

| Ловушка / Вопрос | Пояснение |
|------------------|-----------|
| **Data Race vs Deadlock** | Data Race — неопределенное поведение из-за асинхронных доступов к памяти. Deadlock — состояние, где процессы ждут друг друга бесконечно. Data Race часто приводит к невоспроизводимым крашам, Deadlock — к зависанию. |
| **Как обнаружить Race Condition?** | `go run -race` (компилятор + runtime). Для cross-process: `valgrind --tool=helgrind`, `clang -fsanitize=thread`. |
| **Почему `mmap` быстрее `read/write`?** | `read/write` вызывает копирование данных: User Buffer -> Kernel Buffer -> Socket/Disk -> Kernel Buffer -> User Buffer. `mmap` убирает копирование, оставляя только один переход в ядро для установления отображения. |
| **Copy-On-Write и общая память** | Если вы `mmap`-ите файл с `MAP_PRIVATE`, изменения не видны другим процессам и не сохраняются на диск. Для общей памяти всегда `MAP_SHARED`. CoW экономит память при `fork()`, но не при `mmap` для IPC. |
| **Что такое `futex`?** | Fast Userspace Mutex. Сначала проверяет переменную в User Space (без syscall). Если свободна — сразу захватывает. Если занята — делает `futex()` syscall, чтобы уснуть в ядре. Основа `sync.Mutex` в Go. |

## Итог

Общая память — это скальпель, а не молоток. Она дает максимальную пропускную способность и минимальную задержку, но перекладывает всю ответственность за консистентность на разработчика. Понимание того, как виртуальная память отображается на физическую RAM, как работает MESI и почему возникает False Sharing, отличает Senior-инженера от Middle.

В Go общая память используется точечно, в обход философии языка, когда производительность критична. В остальных случаях каналы и `context` остаются идиоматичным выбором.
