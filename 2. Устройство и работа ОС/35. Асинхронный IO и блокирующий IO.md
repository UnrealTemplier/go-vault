## Разрыв между скоростью CPU и скоростью I/O

Основная проблема бэкенд-разработки никогда не в вычислениях, а в ожидании внешних ресурсов. Процессор совершает миллиарды операций в секунду, а чтение с диска, сетевой пакет или ответ от базы данных занимают микросекунды, миллисекунды или даже секунды. Отношение скорости CPU к скорости I/O может достигать `10^6` и выше.

Архитектура обработки ввода-вывода определяет, как приложение управляет этим разрывом. В Go мы привыкли писать `conn.Read(buf)` и не задумываться, что происходит, но именно здесь решается вопрос горизонтальной масштабируемости: почему один процесс на Go держит `100k` соединений, а на PHP или Java тот же процесс упадет по `OOM` или упрется в лимит потоков ОС.

## Блокирующий I/O: Классический подход

В классической модели **Блокирующий I/O (Blocking I/O)** вызов функции ввода-вывода (например, `read`, `recv`, `os.ReadFile`) останавливает выполнение текущего потока до момента завершения операции.

```go
func processRequest(conn net.Conn) {
    buf := make([]byte, 4096)
    // Вызов блокирует системный тред (M) до прихода данных
    n, err := conn.Read(buf)
    if err != nil {
        // обработка ошибки
    }
    handle(buf[:n])
}
```

### Что происходит на уровне ОС?
1. Пользовательский код вызывает `recvfrom` (или аналогичный syscall).
2. Ядро проверяет состояние сокета. Если данных нет, ядро переводит процесс в состояние ожидания (`TASK_INTERRUPTIBLE` в Linux).
3. **Контекстный смен (Context Switch)**: планировщик ОС снимает текущий тред с CPU, сохраняет его регистры на стек ядра, переключается на другой тред.
4. Процессор простаивает или переключается на другую задачу. Кэш L1/L2 процесса становится невалидным (cache cold).
5. Сетевая карта получает пакет, генерирует прерывание, драйвер копирует данные в буфер ядра (`sk_buff`), будит тред.
6. Тред переходит в состояние `TASK_RUNNING`, syscall возвращает количество прочитанных байт.

> [!warning] Ловушка / Gotcha
> В классических языках (C, PHP, Java до версий 17) каждый блокирующий I/O требует отдельного системного треда ОС. Треды ОС «тяжелые»: `1-2 МБ` стека на тред, дорогие аллокации в ядре, частые контекстные смены. Это фундаментальный лимит масштабируемости.

## Неблокирующий I/O: Отказ от сна

**Неблокирующий I/O (Non-blocking I/O)** заставляет syscall возвращать управление немедленно, даже если операция не завершена. Вместо ожидания приложение получает код ошибки `EAGAIN` или `EWOULDBLOCK` и должно опрашивать ресурс повторно.

```go
import (
    "os"
    "syscall"
)

func setNonblocking(fd int) error {
    // Устанавливаем флаг O_NONBLOCK на файловый дескриптор
    flags, err := syscall.FcntlInt(uintptr(fd), syscall.F_GETFL, 0)
    if err != nil {
        return err
    }
    flags |= syscall.O_NONBLOCK
    return syscall.FcntlInt(uintptr(fd), syscall.F_SETFL, flags)
}
```

Прямое использование неблокирующих сокетов в лоб приводит к **busy-waiting** (неэффективному опросу), который убивает CPU. Поэтому неблокирующий режим всегда используется в паре с механизмом **I/O Multiplexing** (мультиплексирование), который позволяет одному потоку следить за тысячами дескрипторов.

## Синхронный vs Асинхронный: Терминологическая ловушка

На собеседованиях и в документации часто путают пары понятий. Важно разделять два независимых измерения:

1. **Блокирующий vs Неблокирующий** — как ведет себя вызов *в момент старта* (ждет завершения или возвращает сразу).
2. **Синхронный vs Асинхронный** — как приложение получает *уведомление о завершении* (ожидает в том же потоке или получает callback/event).

| Модель | Блокирующий вызов | Уведомление о завершении |
|:---|:---|:---|
| **Sync Blocking** (PHP, Java Thread-per-Request) | Ждет | В том же потоке |
| **Sync Non-blocking** (C `epoll` + `O_NONBLOCK`) | Возвращает `EAGAIN` | Опрос (`poll`/`select`) |
| **Async Blocking** (POSIX `aio_*`) | Ждет | Callback / сигнал |
| **Async Non-blocking** (Go `netpoll`, Node.js) | Возвращает управление | Event loop / Scheduler |

> [!tip] Собеседование
> **Вопрос:** «Является ли Go асинхронным языком?»
> **Ответ:** Технически нет. Go использует **синхронный неблокирующий** I/O. Вызов `conn.Read()` выглядит синхронным (возвращает данные или ошибку), но под капотом горутина «паркуется», а системный тред освобождается. Когда данные приходят, планировщик Go пробуждает горутину. Это не callback-стиль и не сигналы ядра, а кооперативное переключение в пространстве пользователя.

## Под капотом ОС: Состояния процесса и очереди ожидания

Когда тред ОС блокируется на I/O, он не просто «засыпает». Ядро помещает его в специальную очередь ожидания (`wait_queue`), привязанную к конкретному устройству или сокету.

1. **Аллокация структуры ожидания**: Ядро выделяет `struct wait_queue_entry`, привязывает её к сокету.
2. **Смена состояния**: `task->state = TASK_INTERRUPTIBLE`. Планировщик видит, что тред не может исполняться, и исключает его из runqueue.
3. **Пробуждение**: При поступлении данных драйвер вызывает `wake_up()` на очереди сокета. Ядро ищет треды в очереди, переводит их в `TASK_RUNNING` и ставит в очередь планировщика.

Это объясняет, почему `fork` + blocking I/O плохо масштабируется: каждый «спящий» тред занимает память и места в таблице процессов ядра (`task_struct`). При `100k` соединений вы получаете `100k` структур ядра, что приводит к фрагментации памяти и замедлению планировщика.

## Go Runtime: Как горутина «не блокирует» тред

Go решает проблему масштабируемости через **netpoller** и модель планировщика `M-P-G`.

Когда горутина вызывает блокирующий syscall (например, `read` на сокете):
1. **Hand-off**: Планировщик видит, что системный тред `M` ушел в syscall. Он не ждет его, а «паркует» (`M-park`), переводя в состояние ожидания новых горутин.
2. **Перенос в netpoll**: Горутина `G` не уничтожается. Она переносится в очередь ожидания `netpoll` и регистрируется в epoll/kqueue как событие готовности к чтению.
3. **Освобождение M**: Тред `M` освобождается и может обслуживать другие горутины из глобальной или локальной очереди.
4. **Событие**: Когда данные приходят, ядро будит netpoller. Netpoller собирает готовые горутины, переносит их обратно в планировщик Go и пробуждает их на доступных тредах.

```go
// Псевдокод логики netpoll (упрощенно)
func netpoll(block bool) []g {
    events := epoll.Wait(&ev, 64, timeout) // syscall epoll_wait
    var ready []g
    for _, ev := range events {
        if ev.Events&EPOLLIN != 0 {
            // Сокет готов к чтению
            g := pollCache.get(ev.Fd)
            ready = append(ready, g)
        }
    }
    return ready
}
```

> [!info] Под капотом
> В Linux Go использует `epoll` в режиме `EPOLLET` (Edge-Triggered). Это означает, что событие генерируется *один раз* при изменении состояния. Netpoller должен считать все доступные данные за один вызов `read`, иначе он не получит повторного уведомления. В Go это гарантируется внутренними буферами сокетов и логикой `netFD`.
> В BSD/macOS используется `kqueue`, в Windows — `IOCP` (Completion Ports). Все они реализуют одну концепцию: асинхронное уведомление о готовности дескрипторов без блокировки тредов ОС.

## Сравнение подходов: Go vs PHP / Java / Node.js

| Характеристика | PHP (FPM) | Java (Virtual Threads / NIO) | Node.js | Go |
|:---|:---|:---|:---|:---|
| **Базовая модель** | Process per request | Thread per request (или VThreads) | Single-threaded Event Loop | Multi-threaded Scheduler |
| **Блокировка I/O** | Блокирует процесс | Блокирует тред ОС | Не блокирует (async/await) | Блокирует горутину, не тред |
| **Масштабирование** | `ulimit` процессов | `OOM` / Context Switch overhead | Callback hell / Event loop contention | `100k+` goroutines на 1 CPU |
| **Стоимость контекста** | Высокая (fork/exec) | Высокая (kernel thread switch) | Низкая (JS stack) | Очень низкая (user-space switch) |

Go не изобретает велосипед, а комбинирует лучшие практики: неблокирующий I/O от BSD/Linux + кооперативное планирование от C# / Erlang. Это дает предсказуемую производительность без сложной асинхронной кодогенерации.

> [!warning] Ловушка / Gotcha
> **CPU Bound vs IO Bound:** Горутины экономят память и контекстные смены *только* при ожидании I/O. Если горутина делает тяжелные вычисления (шифрование, сортировка, парсинг), она блокирует системный тред `M` намеренно. В этом случае Go не переключает контекст, и вы теряете преимущество. Для CPU-bound задач используйте `runtime.LockOSThread()` или выносите в отдельные процессы.

## Итог

1. **Блокирующий I/O** переводит тред ОС в состояние ожидания, вызывая дорогой контекстный смен и холодный кэш.
2. **Неблокирующий I/O** возвращает управление сразу, но требует опроса (`poll`/`epoll`) для предотвращения busy-waiting.
3. **Синхронный vs Асинхронный** — это про уведомление о завершении, а не про блокировку вызова.
4. **Go** использует синхронный неблокирующий I/O. Горутина «спит» в netpoller, а системный тред освобождается для других задач.
5. Масштабируемость Go до `100k+` соединений обеспечивается отсутствием блокировки тредов ОС и дешевым переключением контекста в user-space.

Разобравшись с тем, как приложения *ожидают* данные, мы переходим к механизму, который делает это ожидание возможным без busy-waiting. В следующей статье мы разберем: `[[36. select, poll и epoll.md]]`, чтобы понять, как один тред эффективно следит за тысячами сокетов.
